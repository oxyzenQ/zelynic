## Explicit Pre-Real-Write Freeze Guarantees

The following guarantees are in effect and must remain in effect until a
dedicated implementation phase explicitly and deliberately removes them with
its own phase entry, review, commit, and validation:

1. **No live PID move**: The guarded real writer seam always returns blocked/
   not implemented. No code path in the current build performs or triggers a PID
   move. The seam is dead code at runtime.

2. **No real cgroup.procs write**: No code in the current build writes to any
   `cgroup.procs` file. The guarded real writer has no I/O capability. The
   mkdir executor creates directories only (no PID movement).

3. **No rollback write**: No code in the current build writes a PID back to an
   original cgroup for rollback purposes. The rollback path exists only as a
   model in move_transaction.rs (skeleton only, execution blocked).

4. **No cleanup mutation**: The only cleanup performed in the current build is
   the mkdir executor removing empty, operation-owned target cgroup directories.
   No PID-related cleanup is performed.

5. **No limiter attach**: No code in the current build attaches a bandwidth
   limiter to any process or cgroup. The attach-live path is hard-blocked and
   returns "not implemented."

6. **No nftables/tc/Zelynic state mutation**: No code in the current build
   modifies nftables rules, tc qdisc/class/filter configuration, or Zelynic
   persistent state files.

7. **No persistent state write**: No code in the guarded real writer or any
   related module (move_executor, move_transaction, failure_simulation,
   fake_writer) writes to `/run/zelynic/state.json` or any other persistent
   state file.

8. **No CLI path enabled**: The guarded real writer module is not wired to any
   CLI command. No CLI flag combination can reach it. The `--attach-live` flag
   enters the experimental attach gate, which evaluates only the move executor
   seam (phase 3c), not the guarded real writer.

9. **No /proc access**: The guarded real writer performs no /proc reads. Its
   input model is pure data; it does not inspect `/proc/<pid>/cgroup`,
   `/proc/<pid>/status`, or any other /proc entry.

10. **No /sys access**: The guarded real writer performs no /sys reads or
    writes. Its target validation is string-based (starts-with check); it does
    not inspect `/sys/fs/cgroup` on the filesystem.

11. **No filesystem mutation from guarded_real_writer**: The guarded real writer
    module performs zero filesystem operations. It builds and renders pure data
    structures in memory. No files are created, modified, or deleted.

## Blocked-Path Proof Summary

The following 9 blocked-path properties have been verified by code audit
(phase 5e) and integration audit (phase 5g):

1. **guarded_real_writer is internal-only**: Declared as `mod guarded_real_writer`
   in `src/systemd_wrapper/mod.rs` (private). API re-exports are `pub(crate)` only.
   No `pub` re-export exists. No external crate or CLI layer can access it.

2. **No CLI command calls it**: `grep` across `src/cli/` and `src/commands/`
   returns zero references to `guarded_real_writer`,
   `build_guarded_real_writer_plan`, or `render_guarded_real_writer_plan`.

3. **Attach-live remains hard-blocked**: The `--attach-live` flag enters the
   experimental attach gate, which evaluates the move executor seam (phase 3c).
   The seam always returns `status: "blocked"` and `execution: "not implemented"`.
   The guarded real writer is NOT wired into this path.

4. **Mkdir-live remains mkdir-only**: The `--mkdir-live` flag triggers the mkdir
   executor, which creates/removes directories only. It reports
   `cgroup_procs_writes: "disabled"` and never writes to `cgroup.procs`. The
   guarded real writer is NOT wired into this path.

5. **Move executor remains blocked**: `move_executor.rs` (phase 3c) always
   returns blocked/not implemented. Every mutating step produces a blocked
   result. The executor does not call or depend on the guarded real writer.

6. **Move transaction remains skeleton/model-only**: `move_transaction.rs`
   (phase 3b) produces `status: "skeleton only; not executed"` and
   `execution: "blocked"`. All 10 transaction steps remain planned/model-only.

7. **Failure simulation remains fake/model-only**: `failure_simulation/`
   (phases 4b–4d) provides pure models for 12 failure scenarios with canonical
   deny lines. All simulation is in-memory; no real system access.

8. **Fake writer remains fake/model-only**: `failure_simulation/fake_writer/`
   (phases 4c–4d) simulates cgroup.procs writes with injectable failure modes
   in memory. All output includes deny lines stating no real operations.

9. **No runtime path calls guarded_real_writer**: The functions
   `build_guarded_real_writer_plan` and `render_guarded_real_writer_plan`
   appear only within the `guarded_real_writer/` directory itself. No other
   source file in the crate imports or calls them.

## Next-Phase Entry Criteria Before Any Real cgroup.procs Write

Before any future phase can implement a real cgroup.procs write, ALL of the
following conditions must be satisfied. Missing any single condition blocks the
write unconditionally:

1. **Separate explicit implementation phase required**: A dedicated phase
   (beyond 5h) must be created, reviewed, and committed. The phase must have its
   own design document, test plan, and validation gates. The guarded real writer
   seam's hard-block must be explicitly and deliberately changed in that phase
   with full justification and review.

2. **Separate explicit operator confirmation required**: Before the first real
   smoke, the operator must explicitly confirm readiness after reviewing all
   abort conditions, manual recovery procedures, and expected output honesty.
   Confirmation must be recorded (e.g., in a smoke log or checklist).

3. **Root-only**: The caller must have euid == 0. This is enforced by gate 1 in
   the guarded real writer seam and must be enforced at runtime before any
   write. Non-root callers are blocked unconditionally.

4. **System-scope-only**: The target PID must be in system scope (not
   user.slice). This is enforced by gate 2 in the seam and must be enforced at
   runtime. User-scope PIDs are blocked unconditionally.

5. **Single disposable sleep PID only**: The target must be a single PID
   running a disposable process such as `sleep 3` or `sleep 10`. No browser,
   terminal, desktop, or long-running application processes are permitted. No
   multi-PID process trees are permitted. This constraint is enforced by gate 3
   (single non-zero PID) in the seam.

6. **Original cgroup captured and verified**: The original cgroup path of the
   target PID must be captured from `/proc/<pid>/cgroup` and verified to be a
   valid cgroup v2 path, not under `/zelynic/`, and non-empty. This is enforced
   by gates 4 and 5 in the seam.

7. **Target cgroup under /sys/fs/cgroup/zelynic only**: The target cgroup must
   be a path under the Zelynic cgroup namespace with no path traversal. This is
   enforced by gate 6 in the seam (`is_safe_writer_target`).

8. **Immediate rollback required**: After the move, the PID must be moved back
   to its original cgroup immediately. The `--rollback-required` flag must be
   present. If rollback fails, manual recovery is required and must be reported
   loudly.

9. **Rollback path reviewed**: The rollback sequence must be reviewed against the
   operator checklist from phase 5b. The operator must understand the exact
   commands for rollback write, rollback verification, and cleanup.

10. **Manual recovery reviewed**: The manual recovery procedures from phase 5b
    (11 steps) must be reviewed. The operator must understand how to inspect PID
    cgroup, verify the original cgroup exists and is writable, and move the PID
    back manually if automated rollback fails.

11. **Exact smoke commands reviewed**: The exact smoke commands from the phase 5a
    manual smoke plan (14 steps) and the phase 5b operator checklist must be
    reviewed step-by-step before execution.

12. **No limiter/nft/tc/state**: The real write must not attach a bandwidth
    limiter, must not modify nftables rules, must not modify tc qdisc/class/
    filter configuration, and must not mutate Zelynic persistent state.

13. **Abort on ambiguity**: If any preflight check, runtime observation, or
    post-move verification produces ambiguous output, the operation must abort
    immediately. Ambiguous results must never be interpreted optimistically.
