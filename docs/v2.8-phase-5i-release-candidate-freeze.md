## Purpose

Phase 5i produces the release-candidate freeze and full validation index for the
v2.8 Experimental PID Move Lab. This report covers the entire v2.8 timeline from
phase 1 (design only) through phase 5h (final pre-real-write validation), documents
the current validated state, declares final safety guarantees, and defines the
release-candidate decision: v2.8 can be frozen as a safety/research milestone
without any live PID movement having been implemented.

Phase 5i is **docs/report only**. No Rust code changes. No runtime behavior
changes. No live PID move. No cgroup.procs write.

## Final v2.8 RC Safety Guarantees

The following guarantees are the permanent safety record of the v2.8
Experimental PID Move Lab release candidate. These guarantees remain in effect
until explicitly and deliberately changed in a future post-RC phase with its own
entry criteria, review, commit, and validation:

1. **No live PID move implemented**: No code path in v2.8 performs or triggers a
   PID move. The guarded real writer seam always returns blocked/not implemented
   and is dead code at runtime.

2. **No real cgroup.procs write implemented**: No code in v2.8 writes to any
   `cgroup.procs` file. The guarded real writer has no I/O capability. The
   mkdir executor creates/removes directories only.

3. **No rollback write implemented**: No code in v2.8 writes a PID back to an
   original cgroup for rollback purposes. The rollback path exists only as a
   model in move_transaction.rs (skeleton only, execution blocked).

4. **No cleanup mutation from guarded_real_writer**: The guarded real writer
   performs zero filesystem operations. It builds and renders pure data
   structures in memory. No files are created, modified, or deleted.

5. **No limiter attach**: No code in v2.8 attaches a bandwidth limiter to any
   process or cgroup. The attach-live path is hard-blocked and returns "not
   implemented."

6. **No nftables/tc/Zelynic state mutation**: No code in v2.8 modifies
   nftables rules, tc qdisc/class/filter configuration, or Zelynic persistent
   state files.

7. **No persistent state write**: No code in the guarded real writer or any
   related module writes to `/run/zelynic/state.json` or any other persistent
   state file.

8. **No CLI path enabled for live PID move**: The guarded real writer module is
   not wired to any CLI command. No CLI flag combination can reach it. The
   `--attach-live` flag enters the experimental attach gate, which evaluates
   only the move executor seam (phase 3c), not the guarded real writer.

9. **Guarded_real_writer remains internal-only and hard-blocked**: Declared as
   a private module in `src/systemd_wrapper/mod.rs`. API re-exports are
   `pub(crate)` only. No external crate or CLI layer can access it. The seam
   always returns blocked/not implemented.

10. **Failure_simulation remains fake/model-only**: All 12 failure scenarios
    (F1-F12) are pure model code. No real system access. All output includes
    canonical deny lines.

11. **Fake_writer remains fake/model-only**: All cgroup.procs write simulation
    is in-memory with injectable failure modes. No real filesystem writes. All
    output includes canonical deny lines.

12. **Mkdir-live remains mkdir-only**: The `--mkdir-live` flag triggers the
    mkdir executor, which creates/removes directories only. It reports
    `cgroup_procs_writes: "disabled"` and never writes to `cgroup.procs`.
    No PID movement, no nft/tc/state writes.

## Release-Candidate Decision

### v2.8 is frozen as a safety/research milestone

v2.8 can be released as a safety/research milestone without any live PID
movement having been implemented. The entire v2.8 timeline (phases 1 through 5h)
produced:

- 1 master design document
- 3 implementation design documents (phases 3a, 4a, 5c)
- 1 operator checklist with exact command tables (phase 5b)
- 1 readiness document with smoke plan (phase 5a)
- 4 audit/freeze/validation reports (phases 2c, 3e, 4e, 5e)
- 1 integration audit / blocked-path proof (phase 5g)
- 1 final pre-real-write validation / release gate (phase 5h)
- 1 output audit document (phase 3d)
- 1 release-candidate freeze / full validation index (this report, phase 5i)

The codebase includes:
- 1 mkdir-only executor (the only real filesystem write in v2.8)
- 1 guarded real writer seam (hard-blocked, dead code at runtime)
- 1 move executor seam (hard-blocked, phase 3c)
- 1 move transaction skeleton (execution blocked, phase 3b)
- 1 failure simulation model (pure model/fake-only, phases 4b–4d)
- 1 fake writer injection harness (pure fake/in-memory, phases 4c–4d)
- 645 unit tests, all passing
- 84 source files, all under 1000 LOC

### Future real PID move must be a separate explicit post-RC phase

The first real PID move is NOT part of v2.8. It requires a separate, dedicated
post-RC implementation phase with its own:

- Design document
- Test plan and test implementation
- Code changes (removing the hard-block in the guarded real writer seam)
- CLI gate (new explicit flag or flag combination)
- Root smoke review and execution
- Operator confirmation

### Future real PID move requirements

The first real PID move, when eventually implemented in a future post-RC phase,
must satisfy ALL of the following:

1. **Root-only**: The caller must have euid == 0.
2. **System-scope-only**: The target PID must be in system scope (not user.slice).
3. **Single disposable sleep PID only**: The target must be a single PID running
   `sleep 3` or `sleep 10`. No browser, terminal, desktop, or long-running
   application processes. No multi-PID process trees.
4. **Immediate rollback required**: After the move, the PID must be moved back
   to its original cgroup immediately. The `--rollback-required` flag must be
   present.
5. **No limiter/nft/tc/state**: The real move must not attach a bandwidth
   limiter, must not modify nftables rules, must not modify tc configuration,
   and must not mutate Zelynic persistent state.
6. **Explicit operator confirmation**: Before execution, the operator must
   explicitly confirm readiness after reviewing all abort conditions, manual
   recovery procedures, and expected output honesty.
7. **Abort on ambiguity**: If any preflight check, runtime observation, or
   post-move verification produces ambiguous output, the operation must abort
   immediately. Ambiguous results must never be interpreted optimistically.
