## 1. Purpose

Phase 15 freezes the future contract for `zelynic ledger export --json`
before any implementation work begins. This phase chooses and documents
one future command shape, explicitly rejects alternative shapes, and
codifies the design rules that any future export implementation must
satisfy. No export functionality is implemented in this phase — the
command remains design-gated and rejected at dispatch.

The primary design question answered in this phase is: **how does a
user explicitly select a source ledger file for export, and where does
the exported JSON go?** The answer is frozen as contract for all future
phases.

## 2. Chosen Future Command Shape

```
zelynic ledger export --json --file <PATH>
```

This is the only accepted future shape for ledger export. The command
reads a ledger JSON file from the user-specified `<PATH>`, validates
it, and emits the exported ledger JSON to stdout. Key characteristics:

- `--json` is required (export always outputs JSON).
- `--file <PATH>` is required and specifies the source ledger file to
  read. There is no implicit default file path, no environment
  variable, and no auto-discovery.
- Output goes to stdout only. There is no `--output`, `--write`, or
  `--save` flag. If a user wants to save the output, they redirect
  stdout to a file using shell redirection (`> output.json`).

### 2.1 Design Rationale

The `--json --file <PATH>` shape was chosen for six reasons:

1. **Explicit source selection**: The user must type `--file` and a
   path. There is no ambiguity about which file is being read. This
   extends the Phase 11/12 principle of explicit user consent to the
   export path.

2. **No output file flag**: By emitting to stdout only, the command
   avoids all file-write safety concerns (overwrite, atomic write,
   backup, permissions, directory creation). The user controls output
   via shell redirection, which is their responsibility and their
   shell's responsibility. This means the export command itself never
   writes to the filesystem.

3. **Composable with shell pipelines**: Stdout output allows piping
   to `jq`, `grep`, `wc`, or other tools without temporary files.
   This is the Unix philosophy and aligns with how `usage --sample
   --json` already works.

4. **Consistent with inspect --file**: The `--file <PATH>` flag
   already exists on `ledger inspect` (Phase 12). Reusing the same
   flag name on `ledger export` provides a consistent CLI surface:
   `--file <PATH>` always means "read from this file" across both
   subcommands.

5. **No silent state**: Unlike a default-path export that might
   silently read and emit a file the user didn't specify, `--file
   <PATH>` makes the data source visible in the command invocation.
   This aligns with the Phase 8 principle of "no hidden state."

6. **Testable at the parser level**: Both `--file` presence and
   absence are easily tested with clap's `try_parse_from()` without
   any filesystem access.

## 3. Rejected Alternatives

### 3.1 Implicit Default Path Export (Rejected)

**Shape**: `zelynic ledger export --json` reads
`~/.local/share/zelynic/network-ledger-v1.json` automatically.

**Rejection reason**: Violates the Phase 8 principle of "no hidden
state" and "explicit user command only." A user running `ledger export
--json` today (after a future implementation phase) might not know
which file is being exported. Different machines would produce
different output depending on whether a ledger file exists. This
violates determinism. The user must explicitly specify the source file
via `--file`.

### 3.2 Export from Live State (Rejected)

**Shape**: `zelynic ledger export --json` reads live system state
(cgroup counters, nft counters, /proc/net/dev) and constructs a
ledger on the fly.

**Rejection reason**: `ledger export` is an export/reporting command
for persisted ledger data, not a live monitoring command. Reading live
state is the domain of `zelynic usage --sample`. Mixing live state into
ledger export would blur the boundary between observation (usage) and
reporting (ledger), making it unclear what data source the output
represents. Live state also introduces nondeterminism (different
results each time).

### 3.3 `--output <PATH>` for Output File (Rejected)

**Shape**: `zelynic ledger export --json --file <SRC> --output <DST>`
writes the export to a destination file.

**Rejection reason**: Adding `--output <PATH>` requires designing the
entire file-write safety model (atomic writes, overwrite protection,
backup strategy, directory creation, permissions) before the export
feature itself is useful. This creates a coupling between export
semantics and write safety that makes both harder to review and test.
By emitting to stdout only, export can be implemented and tested
independently of any future write functionality. If a write flag is
desired later, it can be added in a dedicated phase with its own
design freeze.

### 3.4 `--overwrite` Flag (Rejected)

**Shape**: `zelynic ledger export --json --file <SRC> --output <DST>
--overwrite` allows overwriting an existing file.

**Rejection reason**: This presupposes `--output <PATH>` which is
rejected above. Furthermore, `--overwrite` is a dangerous flag that
makes it easy to accidentally destroy data. The Phase 8 safety
principles explicitly prohibit silent overwrites. If file output is
ever added, it should require a separate, explicit opt-in phase with
its own design freeze.

### 3.5 Auto-Export from Persistence Without Explicit Request (Rejected)

**Shape**: `zelynic ledger export --json` automatically exports from
the persistence store if one exists.

**Rejection reason**: This combines the worst aspects of implicit
default path export with silent behavior. The output would change
depending on whether persistence has been enabled and a ledger file
exists on the system. Scripts would produce different results on
different machines. This violates determinism and the "explicit user
command only" principle.

### 3.6 Combined Inspect/Export/Save in One Command (Rejected)

**Shape**: `zelynic ledger --file <PATH> --save --export --json`
combines multiple operations.

**Rejection reason**: Single-responsibility principle. Each command
should do one thing. `inspect` reads and displays; `export` reads and
emits structured JSON; `save` writes to disk. Combining these in one
command creates ambiguity about side effects (does `--file` mean read
from or write to?), complicates error handling, and makes it harder to
reason about safety guarantees.

### 3.7 Export from Live /proc/sysfs Directly (Rejected)

**Shape**: `zelynic ledger export --json --live` reads
`/proc/net/dev` and sysfs to construct and export ledger data.

**Rejection reason**: This would make `ledger export` a live monitoring
command, which is the domain of `zelynic usage`. It would also
introduce nondeterministic output, OS-specific behavior, and potential
permission issues. The export command should work on a known, validated
ledger JSON file, not on live system state.

### 3.8 Export Enforcement/Limiter State as Ledger Data (Rejected)

**Shape**: `zelynic ledger export --json --include-enforcement`
includes nft counters, tc class state, or cgroup bandwidth as if they
were ledger entries.

**Rejection reason**: Enforcement state (nft counters, tc policer
counters, cgroup stats) is operational diagnostic data, not ledger
data. The ledger schema is a record of historical network accounting
observations, not a dump of kernel state. Mixing enforcement data into
the ledger JSON would corrupt the schema's semantic meaning and create
false impressions about what the data represents.

## 4. Design Rules

The following design rules are frozen as contract for all future export
implementation phases. Any implementation of `ledger export --json
--file <PATH>` must satisfy every rule before the export code path is
activated.

### Rule 1: Export Remains Gated in Phase 15

In Phase 15, `zelynic ledger export --json` remains design-gated at
dispatch. No export code path is activated. This rule is implemented
by the existing dispatch gate in `src/commands/mod.rs`.

### Rule 2: Explicit `--file <PATH>` Required for Source

Future export must require `--file <PATH>` to specify the source ledger
file. There is no short form, no positional argument, no implicit
default. The `PATH` argument must be a non-empty string provided by
the user on the command line.

### Rule 3: Export is Read-Only

Future export must be a read-only operation. It reads a file, validates
it, and emits JSON to stdout. No file is written, no directory is
created, no state is mutated. The command is pure: same input file
always produces the same stdout output.

### Rule 4: Path Validation Before Read

Future export must validate the path using the same conservative
validation as `ledger inspect --file` (Phase 12/13): no empty paths,
no parent traversal (`..`), no suspicious filename characters, no
symlinks, no directories, no nonexistent files. The existing
`validate_inspect_file_path()` function (or a shared equivalent) must
be used.

### Rule 5: Schema Validation Before JSON Output

Future export must parse and validate the ledger JSON before emitting
any output. The existing `deserialize_ledger_from_json()` pipeline
must be used, which checks: JSON syntax, schema_version, per-entry
`validate_safety()` (read_only, attribution_scope,
enforcement_status, combined_bytes consistency). Invalid files must
produce an honest error and no stdout output.

### Rule 6: JSON Emitted to Stdout Only

Future export must emit the ledger JSON to stdout. There is no
`--output <PATH>`, no `--write <PATH>`, no `--save <PATH>`. If a user
wants to save the output, they use shell redirection. The export
command itself never writes to the filesystem.

### Rule 7: No File Write

Future export must not write, backup, rename, delete, truncate, or
migrate any file. No temporary files are created. No lock files are
created. The operation is strictly read-only.

### Rule 8: No Directory Creation

Future export must not create any directories. If the user wants to
redirect output to a file in a non-existent directory, that is the
user's shell's responsibility.

### Rule 9: No Backup, Migration, Rename, Truncate, Delete, Overwrite

Future export must not perform any of these operations. The command
reads one file and writes nothing.

### Rule 10: No Silent Default Path Read

Future export must not read from any default path unless the user
explicitly passes that path via `--file`. No XDG auto-discovery, no
environment variable, no implicit fallback.

### Rule 11: No Live /proc/sysfs Read

Future export must not read from `/proc/net/dev`, `/sys/class/net/`,
or any other live system interface. The data source is exclusively the
user-specified JSON file.

### Rule 12: No nft/tc/cgroup/PID/Zelynic Limiter State Mutation

Future export must not modify nftables rules, tc classes/filters,
cgroup assignments, process state, or any Zelynic limiter state. No
limiter attach, no policy install, no cleanup.

### Rule 13: Honest Non-Mutating Errors

All errors during export (path rejection, file not found, parse error,
schema validation failure) must be reported as honest error messages.
No error path may silently mutate state, write files, or fall back to
alternative data sources. The error message prefix `ledger export:`
(or `ledger export --file:`) must be stable and identifiable.

### Rule 14: Export JSON Schema is Separate

The future export JSON schema is the ledger serialization schema
(`schema_version: 1`, with `entries` array, `host_id`, timestamps,
etc.). It is explicitly documented as separate from:

- The v3.0 usage JSON schema (produced by `usage --sample --delta
  --json`), which has `command`, `sample_mode`, `interfaces`,
  `totals`, etc.
- Any future inspect-specific JSON schema if one diverges.

The export JSON is produced by the existing
`serialize_ledger_to_json()` function, which is the same function used
by `ledger inspect --json`. This means export JSON and inspect JSON
have the same structure (both are ledger serialization), but they are
semantically different operations (export reads a user file; inspect
may use fixture data).

### Rule 15: v3.0 Usage JSON Schema Unchanged

The v3.0 usage JSON schema (schema_version 1) produced by `zelynic
usage --sample --delta --json` remains frozen. No field is added,
removed, or renamed. The schema is the same as in v3.0.1.

### Rule 16: Ledger Inspect Behavior Unchanged

All existing `ledger inspect` behavior remains unchanged:
- `ledger inspect` (fixture text)
- `ledger inspect --json` (fixture JSON)
- `ledger inspect --file <PATH>` (explicit file text)
- `ledger inspect --file <PATH> --json` (explicit file JSON)

## 6. JSON Schema Boundary

The codebase has three distinct JSON output contexts:

| Context | Command | Schema | Status |
|---------|---------|--------|--------|
| v3.0 usage | `usage --sample --delta --json` | Usage schema (schema_version 1, command, sample_mode, interfaces, totals, honesty, warnings) | Frozen |
| Ledger inspect | `ledger inspect --json` or `--file <PATH> --json` | Ledger schema (schema_version 1, host_id, created_at, entries[]) | Active |
| Future export | `ledger export --json --file <PATH>` | Ledger schema (same serialization as inspect) | Design only |

The future export will use the same `serialize_ledger_to_json()` as
inspect. This is explicitly documented: export JSON and inspect JSON
share the same serialization schema, but export is semantically a
different operation (reads user file, not fixture).

## 7. Future Activation Checklist

Before any future phase enables `ledger export --json --file <PATH>`,
the following must be completed:

1. Add `file: Option<String>` (hidden) to `LedgerCommands::Export` in
   `src/cli.rs`.
2. Add `--file` to the export clap definition with `hide = true`.
3. Implement a handler function `handle_ledger_export(json: bool,
   file: Option<&str>)` in `src/commands/ledger.rs` (or a new module).
4. The handler must reject `--json` without `--file` (export requires
   an explicit source file).
5. The handler must reuse `validate_inspect_file_path()` for path
   validation.
6. The handler must reuse `read_ledger_file()` for file reading and
   schema validation.
7. The handler must reuse `serialize_ledger_to_json()` for stdout
   output.
8. The handler must not write any file.
9. Add deterministic tests proving all 16 design rules above.
10. Add error matrix tests (analogous to Phase 13) proving stable
    error keywords for every error class.
11. Update all design docs with completion notes.
12. Run full validation suite (fmt, clippy, all test suites,
    build.sh check-all, release build).

new runtime behavior and adds the following invariants:

- Phase 15 design doc exists and states future shape `ledger export
  --json --file <PATH>`.
- Phase 15 design doc explicitly rejects implicit default path export.
- Phase 15 design doc explicitly rejects output file / overwrite.
- Phase 15 design doc says future export is read-only.
- Phase 15 design doc says future export validates path before read.
- Phase 15 design doc says future export validates schema before JSON
  output.
- Phase 15 design doc says future export does not read live
  /proc/sysfs.

  nft/tc/cgroup/PID state.
