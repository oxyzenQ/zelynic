# v3.0 Phase 14b: Usage Delta JSON Validation Freeze + Test Split

## Purpose

Phase 14b is a validation/documentation/split phase that freezes the pure delta JSON
model and serialization tests from phase 14 before wiring `--delta --json` to the CLI.
This phase splits the oversized test file to comply with the project's 1000 LOC policy,
freezing the test suite for maintainability before the next implementation phase.

This phase does NOT implement `--delta --json`, does NOT change CLI behavior, does NOT
change runtime code, does NOT change the JSON schema, does NOT add dependencies, and
does NOT add live reads beyond existing behavior.

## What Changed

Phase 14b is a refactor/split-only phase. The only changes are test file reorganization
and documentation updates. No runtime behavior was modified, no public API was changed,
no CLI output was changed, and no JSON schema was changed.

### Test File Split

`src/accounting/tests/usage_delta_json.rs` was 973 LOC with 66 tests, approaching the
1000 LOC policy limit. The file was split into a directory module with six focused files:

- `tests/usage_delta_json/mod.rs` (80 LOC): Shared imports, sample data constants
  (START_SAMPLE, END_SAMPLE_NORMAL, END_SAMPLE_RESET, END_SAMPLE_ADDED_INTERFACE,
  END_SAMPLE_REMOVED_INTERFACE, MULTI_IFACE_WITH_LO), parse helper function, and module
  declarations for sub-modules.
- `tests/usage_delta_json/success.rs` (489 LOC, 33 tests): Success build and
  validation tests — normal delta success, interface names, start/end counter fields,
  delta bytes/packets computation, totals correctness, loopback counts, sample summaries,
  top-level fields (schema_version, command, source_path, source_label, sample_mode,
  sample_count, delta_wait_ms, read_count), counter reset detection, interface
  added/removed detection, empty snapshots, u64::MAX counters, saturating totals,
  zero delta, loopback flag, multiple counter resets, serialization success, round-trip
  success/normal/counter-reset, determinism, sample summaries in JSON.
- `tests/usage_delta_json/errors.rs` (197 LOC, 17 tests): Error case tests — first
  read error (zero interfaces, honesty flags, zero totals), second read error (start
  sample present, honesty flags, summary correct), unsupported flag error, error output
  warnings, serialization error JSON, round-trip error JSON, deserialize rejection of
  malformed/missing fields, error type display, error JSON excludes samples.
- `tests/usage_delta_json/warnings.rs` (56 LOC, 6 tests): Warning-specific tests —
  default 13 warnings count and content, counter reset warnings appended on detection,
  delta incomplete warning, not-per-app warning, success output includes all defaults.
- `tests/usage_delta_json/honesty.rs` (156 LOC, 7 tests): Honesty flag tests — all 19
  flags present in success output, default honesty constants verification, first/second
  read error honesty, error preserves all 19 flags, serialized JSON includes all honesty
  flags (success and error).
- `tests/usage_delta_json/safety.rs` (32 LOC, 3 tests): Structural safety tests — no
  CLI flag added, tests do not read real /proc/net/dev, no filesystem write APIs used.

All 66 usage_delta_json tests are preserved with identical behavior. The test count
has not changed. No tests were added or removed.

## Current State

At phase 14b freeze:

| Command | Behavior |
|---------|----------|
| `zelynic usage --sample --delta` | Finite two-sample text output |
| `zelynic usage --sample --delta --json` | **Enabled** — produces valid delta JSON output |
| `zelynic usage --delta` | Rejected by clap (requires `--sample`) |

The delta JSON model exists and is fully tested, but is NOT wired to the CLI. The
`--delta --json` combination remains rejected before any live read. This is
intentional — CLI wiring is deferred to a future phase.
