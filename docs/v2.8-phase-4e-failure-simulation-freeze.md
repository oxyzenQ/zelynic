# Zelynic v2.8 Phase 4e — Failure Simulation Freeze / Validation Report

## Overview

This document freezes the failure simulation subsystem (phases 4a through 4d)
and records its validated state. Phase 4e is **report-only**: it produces
this freeze document and updates related documentation but introduces **no
runtime changes, no PID movement, no `cgroup.procs` writes, no limiter
attach, and no state mutation**.

The purpose of phase 4e is threefold: (1) summarize everything that was
designed, implemented, and tested across phases 4a–4d; (2) record the
current validation state so that future phases have a known-good baseline;
and (3) define the explicit entry criteria for phase 5 (the first real PID
move attempt, if and when it occurs).

## Phase 4e Boundary

### What Phase 4e Produces

- This freeze/validation report (`docs/v2.8-phase-4e-failure-simulation-freeze.md`).
- Updates to `docs/experimental-pid-move-lab.md` with phase 4d completed and
  phase 4e current/freeze status.
- Updates to `CHANGELOG.md` with a phase 4e entry.
- No Rust code changes. No runtime behavior changes. No new CLI flags.

### What Phase 4e Does NOT Do

- Does NOT implement live PID movement.
- Does NOT write `cgroup.procs`.
- Does NOT add runtime mutation of any kind.
- Does NOT add limiter attach, nftables/tc enforcement, or state writes.
- Does NOT change CLI behavior.
- Does NOT modify any Rust source file.
- Does NOT perform root smoke tests.
- Does NOT write test code (only a report is produced).

## Phase 4 Subsystem Summary

### Phase 4a: Failure Simulation Design (Completed)

**Commit**: `b5e4f0b`
**Deliverable**: `docs/v2.8-phase-4a-failure-simulation-design.md`

Phase 4a produced the foundational design document that defines every credible
failure mode in the 10-step PID move transaction. The design enumerates 12
failure scenarios (F1–F12), each with detailed specifications for rollback
behavior, cleanup behavior, output honesty requirements, target cgroup
leftover policy, manual recovery needs, and explicitly forbidden claims.

Key design artifacts from phase 4a:

- **12 failure scenarios** covering failure before target cgroup creation (F1),
  failure after target creation but before PID move (F2), failure after PID
  move at various transaction boundaries (F3–F6), failure during cleanup (F7),
  stale/dead PID at any point (F8), original cgroup disappearing (F9),
  target becoming non-empty (F10), permission denied (F11), and unexpected
  errnos (F12).
- **9 universal failure rules** that apply to all scenarios without
  exception: no false rollback claims, rollback attempted once if move may have
  occurred, rollback failure reported loudly, no limiter attach after partial
  move, no nft/tc/state write in failure path, no deletion of non-empty
  cgroup, no deletion outside `/sys/fs/cgroup/zelynic/`, no retry loops for
  PID movement, and PID location explicitly stated.
- **PID location label taxonomy**: five canonical labels — `not moved`,
  `verified in target`, `verified restored`, `rollback unverified`, `unknown`.
- **Test plan**: 21 fake filesystem unit tests (T-FS-01 through T-FS-21),
  13 fake writer injection tests (T-FW-01 through T-FW-13), 13 render tests
  (T-RN-01 through T-RN-13), and 7 canonical deny-line persistence tests
  (T-CD-01 through T-CD-07).

No code was written in phase 4a. The design document serves as the source of
truth for all subsequent phases.

### Phase 4b: Failure Simulation Model + 73 Fake Tests (Completed)

**Commit**: `a25b080`
**Deliverables**: `src/systemd_wrapper/failure_simulation/mod.rs` (724 LOC),
`src/systemd_wrapper/failure_simulation/tests/mod.rs` (933 LOC)

Phase 4b implemented the pure Rust model and 73 tests for all 12 failure
scenarios defined in phase 4a. The model is entirely pure-function-based with
no filesystem I/O, no real cgroup interaction, and no system mutation of any
kind.

Key implementation artifacts from phase 4b:

- **`FailureScenario` enum** (F1–F12) with `id()`, `name()`, and
  `failure_point()` methods.
- **`PidLocationStatus` enum** matching the five canonical labels from the
  phase 4a taxonomy.
- **`RollbackDecision` enum**: `NotNeeded`, `AttemptOnce`,
  `AttemptedAndSucceeded`, `AttemptedAndFailed`, `ManualRecoveryRequired`.
- **`CleanupDecision` enum**: `NotNeeded`, `RemoveEmptyOperationOwned`,
  `LeaveBecauseNonEmpty`, `LeaveBecauseUnsafe`, `ManualCleanupRequired`.
- **`SimulationResult` struct**: captures scenario ID, failure point, PID
  location, rollback/cleanup decisions, target_may_remain flag,
  manual_recovery_required flag, 7 canonical deny lines, honesty lines, and
  forbidden claims.
- **Pure functions**: `build_failure_simulation_matrix()` (builds all 12
  entries), `simulate_failure_scenario()` (pure simulation with match arms
  for each scenario), `render_failure_simulation_result()` (structured output
  rendering with deny lines and honesty markers).

The 73 tests cover: matrix completeness (12 scenarios, PID location labels,
rollback decisions, cleanup decisions), per-scenario correctness for all 12
failure scenarios, render output markers, canonical deny lines (7 per output),
no false rollback claims, no retry loops, FailureScenario enum helpers,
render structure verification, universal failure rules (all 9), determinism,
and simulation result field correctness.

**Fix commit**: `7839bb5` — wired the test submodule into the parent module
via `#[cfg(test)] mod tests;` which brought the test count from the module's
internal count to full crate visibility (453 → 526 tests at the time).

## Freeze Guarantees

The following guarantees are frozen as part of phase 4e. They apply to all
phase 4 subsystems (4a design, 4b model, 4c fake writer, 4d render matrix)
and remain in effect until explicitly revised in a future phase:

1. **No live PID move**: The failure simulation subsystem does not perform any
   live PID movement. All PID move operations are simulated in memory.

2. **No real cgroup.procs write**: No code in the failure simulation subsystem
   writes to `/sys/fs/cgroup` or any `cgroup.procs` file on the real
   filesystem. All write operations are modeled as pure function return values.

3. **No limiter attach**: The failure simulation subsystem does not attach
   any bandwidth limiter, BPF program, or tc class. Limiter attach is
   explicitly forbidden in all failure outputs via canonical deny lines.

4. **No nftables/tc/Zelynic state mutation**: The failure simulation subsystem
   does not modify nftables rulesets, tc qdisc/class configurations, or any
   Zelynic persistent state. All state changes are modeled as data structures
   in memory.

5. **No persistent state write**: The failure simulation subsystem does not
   write to any persistent storage. No files are created, modified, or deleted
   outside the Rust compilation target directory.

6. **No CLI enablement for live PID move**: The failure simulation subsystem
   has no CLI command, flag, or code path that would enable live PID movement.
   The module is crate-private (`pub(crate)`) and test-focused.

7. **All simulation is fake/model-only**: Every function in the failure
   simulation subsystem operates on in-memory data structures only. No real
   system calls, no real kernel interactions, no real cgroup hierarchy
   operations. The rendered output is explicitly labeled as "fake/model-only"
   in every entry.

## Phase 5 Entry Criteria

Phase 5 (the first real PID move attempt, if and when it occurs) must satisfy
all of the following criteria before any work begins:

1. **Phase 4 freeze report complete**: This document (phase 4e) must be
   committed, pushed, and CI-green.

2. **CI green**: All GitHub Actions CI checks must pass on the main branch.

3. **All failure simulation tests wired and passing**: The 151 failure
   simulation tests and 78 fake writer tests must all pass in CI and locally.

4. **Root smoke commands must be reviewed before execution**: Any root smoke
   command proposed for phase 5 must be reviewed line-by-line before execution.
   No root command runs without explicit prior approval.

5. **First real PID move remains blocked until phase 5 / next explicit plan**:
   The phase 4 subsystem explicitly blocks all real PID movement. Phase 5
   must have its own design document, safety review, and explicit plan before
   any real PID move code is written.

6. **Future real move constraints**: When phase 5 or a future phase does
   implement the first real PID move, it must adhere to the following hard
   constraints:
   - **Root-only**: The real PID move must require root (CAP_SYS_ADMIN) and
     must not be possible from unprivileged contexts.
   - **System-scope-only**: The real PID move must only operate within the
     systemd system scope (not user scope).
   - **Single disposable sleep PID only**: The target PID must be a freshly
     spawned `sleep` process with a short lifetime (e.g., `sleep 3`).
     No long-lived processes, no multi-PID scenarios, no production PIDs.
   - **Immediate rollback required**: After the move write, the PID must be
     written back to the original cgroup immediately. The rollback must be
     attempted even if the move verification failed.
   - **No limiter attach**: The first real PID move must not attach any
     bandwidth limiter, BPF program, or tc enforcement. The move is cgroup-only.
   - **No nft/tc/state mutation**: The first real PID move must not modify
     nftables rules, tc configuration, or Zelynic persistent state. The move is
     cgroup-only.
