## Artifact Naming Audit

The existing release infrastructure derives version dynamically from the git tag
or Cargo.toml. When `v3.0.0` is tagged and the release workflow triggers, it
will produce:

### GitHub Actions Release Workflow (`.github/workflows/release.yml`)

The workflow uses `${GITHUB_REF_NAME#v}` to extract the version from the pushed
tag. For a `v3.0.0` tag:

- **Archive**: `zelynic-v3.0.0-x86_64-linux.tar.gz`
- **Checksum**: `zelynic-v3.0.0-x86_64-linux.tar.gz.sha256`

Artifact naming is consistent and will use v3.0.0 automatically.

### Local Package Script (`package.sh`)

The script reads version from `Cargo.toml` via `grep '^version = ' Cargo.toml`.
For v3.0.0:

- **Archive (glibc)**: `dist/zelynic-v3.0.0-x86_64-linux.tar.gz`
- **Archive (musl)**: `dist/zelynic-v3.0.0-x86_64-linux-musl.tar.gz`
- **Archive (aarch64, if available)**: `dist/zelynic-v3.0.0-aarch64-linux.tar.gz`
- **Checksums**: `<archive>.sha256` alongside each archive

Artifact naming is consistent and will use v3.0.0 automatically.

No artifacts were generated, renamed, or uploaded in this phase.

## Checksum Readiness Audit

The existing release infrastructure generates SHA256 checksums automatically:

### GitHub Actions

The workflow runs `sha256sum "${PKG_NAME}.tar.gz" > "${PKG_NAME}.tar.gz.sha256"`
during artifact preparation. The checksum file will reference
`zelynic-v3.0.0-x86_64-linux.tar.gz` and contain the correct SHA256 hash.

### Local Package Script

The script runs `sha256sum "$archive_name" > "${archive_name}.sha256"` for each
built archive. The checksum files will reference v3.0.0 archives.

For manual verification during release, the exact command is:

```bash
sha256sum zelynic-v3.0.0-x86_64-linux.tar.gz
```

No checksum files were generated or modified in this phase. The checksum process
is documented and ready for use during release.
