# Zelynic v2.8 Phase 3a — Single PID Move + Immediate Rollback Design

## Overview

This document captures the design for the first actual PID move experiment in the
Zelynic v2.8 Experimental PID Move Lab. Phase 3a is **design-only**: it produces
this document and updates related documentation but introduces **no runtime
changes, no PID movement, no `cgroup.procs` writes, and no limiter attach**.
The existing move-only executor skeleton (`src/systemd_wrapper/move_transaction.rs`)
remains hard-blocked at `status: "skeleton only; not executed"` and `execution:
"blocked"`.

This design builds directly on the validated patterns from phase 2b (mkdir-only
executor, `src/systemd_wrapper/mkdir_executor.rs`) and the existing pure models
from v2.7 (gate checklist, target preflight, operation journal, original cgroup
capture, PID safety checks). The mkdir executor proved that Zelynic can safely
perform real cgroup filesystem writes with explicit verification, honest output, and
conservative cleanup. Phase 3a designs the next narrow write step: writing a
single PID into a `cgroup.procs` file and immediately rolling back.

## Phase 3a Boundary

### What Phase 3a Produces

- This design document (`docs/v2.8-phase-3a-single-pid-rollback-design.md`).
- Updates to `docs/experimental-pid-move-lab.md` with phase 3a status.
- Updates to `CHANGELOG.md` with a design-only entry.
- No Rust code changes. No runtime behavior changes. No new CLI flags.

### What Phase 3a Does NOT Do

- Does NOT implement live PID movement.
- Does NOT write `cgroup.procs`.
- Does NOT add runtime mutation of any kind.
- Does NOT add limiter attach, nftables/tc enforcement, or state writes.
- Does NOT change CLI behavior.
- Does NOT add new command-line flags.
- Does NOT modify any Rust source file.

The first actual PID move (when eventually implemented in a future phase, not
3a) will follow the design specified in this document.

## First Actual PID Move Experiment Specification

### Hard Constraints

The first PID move experiment must satisfy every one of the following
constraints. If any constraint cannot be satisfied, the move must not proceed:

| Constraint | Requirement |
|------------|-------------|
| **Root only** | euid == 0. Enforced by the probe gate at entry. |
| **System scope only** | `--scope-mode system`. User scope remains blocked. |
| **Single PID only** | Exactly one PID discovered in the scope's `cgroup.procs`. Multi-PID scopes must block. |
| **Disposable command only** | The child process must be a short-lived, stateless command. Recommended: `sleep 3` or `sleep 10`. |
| **Immediate rollback** | The PID must visit the target cgroup and return to its original cgroup within the same operation. No persistent placement. |
| **No limiter attach** | `zelynic strict` must NOT be called. No nftables, no tc, no bandwidth enforcement. |
| **No nft/tc/state mutation** | No nftables rules, no tc qdisc/class/filter changes, no `/run/zelynic/state.json` writes. |
| **No persistent Zelynic state write** | The operation journal preview is rendered in output but never persisted to disk. |
| **No multi-PID process trees** | Scopes that produce more than one PID in `cgroup.procs` must be blocked. |
| **No user scope** | `--scope-mode user` is blocked at the probe gate. No user-scope mutation. |
| **No long-running apps** | Commands like browsers, terminals, desktop environments, or daemons must NOT be used. Only `sleep` or `true`. |
| **No browser/terminal/desktop processes** | Explicitly forbidden as test subjects. Only disposable system utilities. |
| **No bandwidth limiting claim** | Output must state "Bandwidth limiting is not active from this command." |

### Why These Constraints Exist

The first PID move is the most dangerous write Zelynic will ever perform. If
something goes wrong during the move, a PID could be stranded in an empty
cgroup with no resource controller, no network access via cgroup-based routing, or
no way to be restored. The constraints minimize blast radius:

- **Single PID + disposable**: If the move fails and rollback fails, the worst
  outcome is a `sleep 3` process exiting on its own in an empty cgroup. No data
  loss, no service disruption, no user-visible breakage.
- **Root + system scope**: Ensures the operation has permission to write
  `cgroup.procs` and that the cgroup hierarchy is the unified system hierarchy.
- **Immediate rollback**: The PID visits the target and returns. There is no
  window where the PID is persistently in an unmanaged cgroup.
- **No limiter attach**: The purpose of this experiment is to validate the move
  and rollback mechanism, not to apply bandwidth limits. Bandwidth enforcement
  is a separate concern for a future milestone.
- **No nft/tc/state**: These subsystems introduce their own failure modes. Phase
  3 isolates the PID move from all other mutation types.

## Planned Transaction Model

The PID move experiment follows a deterministic, ordered sequence of steps.
Each step must complete successfully before the next step begins. On any
failure after the move step (step 8), rollback is attempted immediately.

### Step Sequence

```
Step 1: Gate Re-Evaluation
  Re-evaluate every safety gate from the experimental attach gate checklist.
  Gates must be checked at the moment of move, not cached from probe time.
  If any gate fails, abort without writing anything.

Step 2: PID Liveness Recheck
  Verify /proc/<pid> still exists and is a directory.
  The PID may have exited between discovery and this moment.
  If the PID is gone, abort and report honestly.

Step 3: Original Cgroup Validation
  Re-read /proc/<pid>/cgroup and verify the path matches the captured original.
  The original cgroup must:
    - Be a valid cgroup v2 path (not empty, not "0::").
    - Not be under /sys/fs/cgroup/zelynic/ (restoring into a Zelynic target
      would be unsafe — the PID would be in an operation-owned cgroup with
      no controller).
  If the original cgroup cannot be validated, abort without writing anything.

Step 4: Target Cgroup Preparation
  Create/prepare /sys/fs/cgroup/zelynic (parent namespace directory).
  Create/prepare /sys/fs/cgroup/zelynic/target_<name> (target cgroup).
  This reuses the mkdir-only pattern validated in phase 2b.
  If mkdir fails, abort without writing cgroup.procs.

Step 5: Move PID to Target
  Write the single PID string to /sys/fs/cgroup/zelynic/target_<name>/cgroup.procs.
  This is the first cgroup.procs write in Zelynic's history.
  The write must be synchronous (no background threads, no async).
  On write failure, skip to rollback (step 8) with a move failure report.

Step 6: Verify PID in Target
  Read /sys/fs/cgroup/zelynic/target_<name>/cgroup.procs.
  Parse the PIDs and verify the moved PID appears in the list.
  Kernel reaping of zombie PIDs may cause the PID to appear as Tgid; check both.
  If the PID is not found in the target, proceed to rollback with a verification
  failure report (do not assume the move failed — the PID may have moved but
  verification may have a race).

Step 7: Record Move Success
  Update the operation result model with:
    - move_status: "moved"
    - target_cgroup_path: the target path
    - moved_pid: the PID value
  This is an in-memory model update only, not a state file write.

Step 8: Rollback — Move PID Back
  Write the single PID string to the original cgroup's cgroup.procs.
  This restores the PID to its pre-move location.
  On rollback write failure, report loudly and leave diagnostics.
  Do NOT retry the rollback write. Do NOT attempt alternative restore targets.
  A single rollback attempt is the maximum; repeated writes risk further damage.

Step 9: Verify PID Restored
  Read the original cgroup's cgroup.procs.
  Verify the PID appears in the list.
  If verification fails, report the failure but do NOT attempt another rollback.
  The PID may already be restored but the read may have a race; do not assume
  the worst or the best.

Step 10: Cleanup Target Cgroup
  Remove /sys/fs/cgroup/zelynic/target_<name> only if:
    - The target cgroup.procs is empty or absent (no PIDs remain).
    - No child cgroups exist under the target.
    - The removal succeeds.
  If any condition fails, leave the target in place and report the leftover.
  Do NOT remove /sys/fs/cgroup/zelynic (parent namespace may remain).
```

### Step Dependency Graph

```
Step 1 (gate re-eval)
    |
Step 2 (PID liveness)
    |
Step 3 (original cgroup validation)
    |
Step 4 (target cgroup preparation)
    |
Step 5 (move PID to target)
    |
Step 6 (verify PID in target)
    |
Step 7 (record move success)
    |
Step 8 (rollback — move PID back)  <-- must always execute after step 5
    |
Step 9 (verify PID restored)
    |
Step 10 (cleanup target cgroup)
```

Steps 1-4 are pre-move gates. If any fails, the operation aborts without any
`cgroup.procs` write. Steps 5-7 are the move phase. Steps 8-10 are the rollback
phase and must always execute after step 5, even if step 6 or 7 fails.

### Relationship to Existing Skeleton

The existing `move_transaction.rs` skeleton already models this sequence:

```rust
operations: vec![
    op("1. verify gate checklist is still valid"),
    op("2. verify exactly one PID"),
    op("3. verify source/original cgroup path captured"),
    op("4. verify target cgroup path is under /sys/fs/cgroup/zelynic/"),
    op("5. future write: create/prepare target cgroup directory"),
    op("6. future write: write PID into target cgroup cgroup.procs"),
    op("7. future verify: PID appears in target cgroup.procs"),
    op("8. future rollback: write PID back into original cgroup cgroup.procs"),
    op("9. future verify: PID restored to original cgroup"),
    op("10. future cleanup: remove target cgroup only if safe/empty"),
],
rollback: vec![
    rollback("write PID back to original cgroup.procs"),
    rollback("verify PID restored to original cgroup"),
    rollback("remove target cgroup only if safe and empty"),
],
```

The phase 3a design expands this skeleton into a detailed specification with
failure policies, verification semantics, and honest output requirements. When a
future phase implements the actual write, it will follow this design and
update the skeleton from `"future write"` to the real implementation.

### Relationship to Mkdir Executor

The target cgroup preparation in step 4 reuses the exact pattern from
`mkdir_executor.rs`:

- `is_safe_mkdir_target(base_root, target)` for path validation.
- `fs::create_dir(base_root)` for parent namespace (ok if already exists).
- `fs::create_dir(target_cgroup_path)` for target cgroup.
- `Path::is_dir(target_cgroup_path)` for post-creation verification.
- `attempt_cleanup_target(target_cgroup_path)` for safe cleanup.

The mkdir executor validated this pattern with 25 unit tests. The PID move
executor will call the same preparation and cleanup functions, adding the
`cgroup.procs` write and rollback steps around them.

## Failure Policy

### Pre-Move Failures (Steps 1-4)

If any pre-move step fails, the operation aborts immediately. No `cgroup.procs`
write has occurred, so no rollback is needed. The failure must be reported
honestly in the output with a clear description of which step failed and why.

Specific pre-move failure modes and expected responses:

| Failure Mode | Step | Response |
|-------------|------|----------|
| Gate checklist fails (missing consent, root, etc.) | 1 | Abort. Print gate failure. No mkdir, no write. |
| PID exited between discovery and move | 2 | Abort. Print "PID <pid> is no longer alive." No mkdir, no write. |
| Original cgroup unreadable or changed | 3 | Abort. Print "Original cgroup <path> could not be validated." No mkdir, no write. |
| Original cgroup is under /zelynic/ | 3 | Abort. Print "Original cgroup is Zelynic-managed; refusing move." No mkdir, no write. |
| Target mkdir fails (permission, no space, etc.) | 4 | Abort. Print mkdir failure. No cgroup.procs write. Cleanup any partial directories if safe. |

### Move Failures (Steps 5-7)

If the move write (step 5) fails, rollback is attempted immediately. If the
move succeeds but verification (step 6) fails, rollback is still attempted — the
PID may be in the target even if verification had a race.

| Failure Mode | Step | Response |
|-------------|------|----------|
| cgroup.procs write to target fails | 5 | Attempt rollback (step 8). Report move failure loudly. |
| PID not found in target after write | 6 | Attempt rollback (step 8). Report verification failure. |
| PID in target but unexpected additional PIDs present | 6 | This should not happen (target was freshly created and empty). Attempt rollback. Report anomaly. |

### Rollback Failures (Steps 8-9)

Rollback failure is the most serious failure mode. If rollback cannot restore the
PID to its original cgroup, the PID may be stranded in the Zelynic target cgroup
until it exits on its own (for `sleep 3`, this is at most 3 seconds).

| Failure Mode | Step | Response |
|-------------|------|----------|
| cgroup.procs write to original cgroup fails | 8 | Report loudly: "ROLLBACK FAILURE: PID <pid> could not be restored to <original_cgroup>." Leave diagnostics. Do NOT retry. Do NOT attempt alternative targets. |
| PID not found in original after rollback write | 9 | Report: "Rollback verification could not confirm PID <pid> in <original_cgroup>." Do NOT retry. The PID may have already been restored (race) or may be stranded. |
| Original cgroup directory disappeared | 8 | Report loudly: "ROLLBACK FAILURE: Original cgroup <path> no longer exists." Do NOT create the original cgroup. Do NOT guess a restore target. |

### Cleanup Failures (Step 10)

| Failure Mode | Response |
|-------------|----------|
| Target cgroup.procs is non-empty | Leave target in place. Report "leftover target: <path>". Do NOT force-remove. |
| Target has child cgroups | Leave target in place. Report "leftover target: <path>". Do NOT recurse. |
| fs::remove_dir fails | Leave target in place. Report "leftover target: <path>: <reason>". |

### Universal Failure Rules

1. **Every failure after step 5 must attempt rollback.** There are no
   exceptions. Even if verification of the move (step 6) fails, rollback must
   be attempted because the PID may actually be in the target.
2. **No retry loops.** A single rollback attempt is the maximum. Repeated
   `cgroup.procs` writes risk further destabilizing the cgroup state.
3. **No limiter attach even if move succeeds.** The PID move experiment does
   not trigger `zelynic strict` or any bandwidth enforcement, regardless of the
   move outcome.
4. **No state persistence.** The operation journal preview is rendered in output
   but never written to `/run/zelynic/state.json` or any other file.
5. **No cleanup of non-empty cgroups.** If the target cgroup has PIDs or child
   cgroups, it is left in place. A leftover empty directory is safer than an
   incorrect forced removal.
6. **No cleanup outside `/sys/fs/cgroup/zelynic/`.** Only operation-owned
   cgroups under the Zelynic namespace may be touched during cleanup. External
   cgroups are never modified or removed.
7. **Report all failures honestly.** Every failure mode has a specific, human-
   readable error message. The output must never claim success when a failure
   occurred.

## Honest Output Requirements

When the PID move experiment is eventually implemented, its output must include
all of the following explicit statements:

| "PID <pid> move failed: <reason>" | After step 5 (failure) |
| "PID <pid> verified in target cgroup." | After step 6 (success) |
| "PID verification inconclusive: <reason>" | After step 6 (failure) |

### Rollback Section

| Statement | Condition |
|-----------|-----------|
| "PID <pid> restored to original cgroup <path>." | After step 8 (success) |
| "ROLLBACK FAILURE: PID <pid> could not be restored to <path>." | After step 8 (failure) |
| "PID <pid> verified in original cgroup." | After step 9 (success) |
| "Rollback verification inconclusive." | After step 9 (failure) |
| "Target cleanup: performed/skipped/failed." | After step 10 |

### Root Smoke Test (Future, Not Now)

The root smoke test will be performed when the actual PID move is implemented in
a future phase. Phase 3a does not perform this test. The planned smoke command:

```bash
sudo zelynic run --execute --scope-mode system --probe-live --attach-live \
  --experimental-single-pid-attach --i-understand-this-moves-pids \
  --rollback-required --mkdir-live -d 500kbit -u 500kbit -- sleep 3
```

Expected behavior when implemented:
1. Launch transient systemd scope with `sleep 3`.
2. Discover exactly one PID in scope `cgroup.procs`.
3. Capture original cgroup from `/proc/<pid>/cgroup`.
4. Create `/sys/fs/cgroup/zelynic/target_sleep`.
5. Write PID to target `cgroup.procs`.
6. Verify PID in target.
7. Write PID back to original `cgroup.procs`.
8. Verify PID restored.
9. Remove target cgroup.
10. Print honest move+rollback results.
11. Print safety footer: "No limiter attach was performed."
12. Scope transitions to inactive/dead after `sleep 3` exits.

Post-operation verification:
- `systemctl list-units --type=scope | grep zelynic` — no leftover scope.
- `zelynic status` — no active limits.
- `nft list table inet zelynic` — unchanged.
- `tc qdisc show` — unchanged.
- `/run/zelynic/` — unchanged.
