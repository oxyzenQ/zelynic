# Zelynic v3.0 Phase 4: `zelynic usage --sample` CLI Gate Design

This document captures the design for the v3.0 phase 4 `zelynic usage --sample`
CLI preview gate. Phase 4 is **design-only**: it produces this document and
updates related documentation but introduces no CLI command registration, no
live filesystem reads, and no runtime behavior changes.

> **Phase 5 note**: Phase 5 implements the single-shot read-only `zelynic usage
> --sample` CLI command as defined in this gate document. See the main design
> document (`docs/v3.0-live-read-only-usage-lab.md`) for phase 5 details.
>
> **Phase 5b note**: Phase 5b freezes the first CLI implementation before adding
> JSON, delta sampling, interface filtering, or persistence. See
> `docs/v3.0-phase-5b-usage-sample-validation-freeze.md` for the freeze report.

## Purpose

Phase 4 defines the activation gate, contract, and safety constraints for the
future `zelynic usage --sample` command. The command is not implemented in this
phase — only the design of the gate that must be satisfied before implementation
can proceed. This follows the established Zelynic pattern of progressive consent
gates (as used for `--attach-live`, `--experimental-single-pid-attach`, and
`--mkdir-live` in v2.7–v2.8).

## Why Phase 4 Is Design-Only (No Code Changes)

Phase 4 does not add a `Usage` variant to the `Commands` enum in `src/cli.rs` or
register any CLI path for `zelynic usage`. This is an intentional design
decision based on the following analysis:

1. **Clap derive architecture**: The `Commands` enum in `src/cli.rs` uses
   `#[derive(Subcommand)]`. Adding a `Usage` variant would register it as a
   live CLI subcommand even if the dispatch handler immediately prints
   "not implemented" and exits. Users would see `usage` in `zelynic --help`
   output, creating an expectation of functionality that does not exist.

2. **Existing progressive gate pattern**: Zelynic already has a well-established
   pattern for experimental features: flags are added to existing commands
   (e.g., `--attach-live` on `zelynic run`), not as new top-level commands
   that print "blocked." The `usage` command is conceptually different — it is
   a new top-level command, not an experimental flag on an existing command.

3. **Clear activation boundary**: The reader seam is fully implemented and tested
   (phases 2–3), but the output contract (`--json`, `--delta`, `--interface`),
   the CLI wiring, and the manual smoke review have not been validated yet.
   Adding the CLI variant before these are ready would create a premature
   surface that could be confusing or misleading.

4. **Precedent from phase 1**: Phase 1 was also design-only for the same
   reason — the command contract was defined first, implementation deferred.
   Phase 4 continues this pattern by defining the activation gate before the
   CLI registration.

## Future Command Contract

### `zelynic usage --sample`

Display a single live snapshot of network interface counters from
`/proc/net/dev`. This is the simplest and safest invocation: one read of the
kernel's network statistics file, one parse through the existing parser, one
render with honesty disclaimers.

```
$ zelynic usage --sample
zelynic usage --sample — live read-only interface counters
Source: live_proc_net_dev
Attribution: interface-level only (not per-app)
No quota enforcement active. No network blocking active.
No limiter attach performed. No nft/tc/Zelynic state mutation performed.
No ledger persistence performed. No eBPF used.
No cgroup mutation. No PID movement.
Counters may reset after reboot or interface reset.
Filesystem write not performed. State mutation not performed.

Interface     RX           TX           Since
wlan0         1.23 GiB     342 MiB      boot
eth0          0 bytes       0 bytes      boot
usb0          45.6 MiB     12.3 MiB     boot (tethering)

Total (excl. loopback):  RX 1.28 GiB  TX 354 MiB
```

### Command Properties

| Property | Value |
|----------|-------|
| Subcommand | `usage` (new top-level `Commands` variant) |
| Required flag | `--sample` (required in phase 4 implementation; bare `usage` deferred) |
| Data source | `/proc/net/dev` (hardcoded, no arbitrary path input) |
| Attribution | Interface-level only (not per-app) |
| Enforcement | None — display only, no blocking/throttling |
| Persistence | None — no ledger file read or write |
| Loop/watch | None — single sample only |
| JSON output | Not in phase 4 — deferred to phase 5 |
| Delta sampling | Not in phase 4 — deferred to phase 6 |

## Hard Safety Boundary

The `zelynic usage --sample` command must never:

| Prohibited Action | Reason |
|-------------------|--------|
| Read arbitrary file paths | Source is hardcoded to `/proc/net/dev` |
| Accept user-supplied file paths | Path is not a CLI parameter |
| Write any file | Read-only by design |
| Mutate system state | Observes only, never modifies |
| Block network traffic | Display only, no enforcement |
| Enforce quota limits | No budget tracking or throttling |
| Attach rate limiters | No limiter integration |
| Load/attach eBPF programs | No eBPF in v3.0 |
| Mutate nftables/tc rules | No rule changes |
| Move PIDs or write cgroup.procs | No cgroup interaction |
| Read sysfs | Only `/proc/net/dev` |
| Read or write ledger files | No persistence in v3.0 |
| Implement loop/watch behavior | Single sample only |
| Generate JSON output | Deferred to phase 5 |
| Compute delta between samples | Deferred to phase 6 |

## Required Output Honesty Lines

Every render of `zelynic usage --sample` output must include all 13 honesty
disclaimers that are already tested in the `live_proc_net_dev` render module:

1. "read-only /proc/net/dev seam" — identifies the data source and read boundary
2. "Attribution: interface-level only (not per-app)" — counters are aggregate
3. "No quota enforcement active" — display does not enforce budgets
4. "No network blocking active" — display does not drop or reject packets
5. "No limiter attach performed" — display does not configure limiters
6. "No nft/tc/Zelynic state mutation performed" — no rule or state changes
7. "No ledger persistence performed" — no ledger file I/O
8. "No eBPF used" — no eBPF programs loaded or attached
9. "No cgroup mutation" — no cgroup interaction
10. "No PID movement" — no PID manipulation
11. "Counters may reset after reboot or interface reset" — counter semantics
12. "Filesystem write not performed" — explicit write denial
13. "State mutation not performed" — explicit mutation denial

These disclaimers are already implemented in
`render_live_proc_net_dev_read_plan()` and tested by 38 render tests in
`src/accounting/tests/live_proc_net_dev/render.rs`. The CLI implementation
must reuse this existing render function — no new render logic.

## CLI Activation Criteria

The `zelynic usage` command must not be registered in the `Commands` enum until
all of the following criteria are satisfied:

### Gate 1: Phase Completion

- Phase 4 (this document) must be completed and committed.
- Phase 5 (JSON output contract) must be designed or implemented.
- A designated implementation phase must be explicitly approved.

### Gate 2: Reader Seam Integrity

- `read_live_proc_net_dev()` must remain the sole filesystem read function.
- The source path must remain hardcoded to `/proc/net/dev`.
- `ContentReader` trait must have no path parameter.
- Source-level boundary audit tests (FORBIDDEN_PATHS, FORBIDDEN_FS_WRITE_APIS)
  must pass.
- All 75 live_proc_net_dev tests must pass.

### Gate 3: No Arbitrary Path Input

- The CLI must not accept a `--source` or `--file` flag.
- The CLI must not accept an `--interface` flag that reads from sysfs.
- The only accepted path is the hardcoded `/proc/net/dev`.

### Gate 4: Output Honesty

- All 13 honesty disclaimers must be present in every render.
- Unit tests must prove each disclaimer is present (existing 38 render tests).
- No output may claim enforcement, blocking, persistence, or per-app attribution.

### Gate 5: No Enforcement/Blocking/Persistence Claims

- No test may assert that blocking, throttling, or enforcement is active.
- No test may assert that ledger files are read or written.
- No test may assert that PIDs are moved or cgroups are modified.
- The `filesystem_write_performed` flag must remain `false` for injected content.
- The `state_mutation_performed` flag must remain `false` always.

### Gate 6: No Background Loop

- The implementation must use a single-shot read → parse → render pattern.
- No `loop {}`, no polling interval, no `tokio::spawn`, no daemon mode.
- `--sample` is the only accepted invocation in the initial phase.

### Gate 7: No JSON Until Phase 5

- `--json` output is deferred to a separate JSON contract phase.
- Phase 4 implementation must not produce JSON output.
- The JSON schema (schema_version, interfaces, totals, honesty) is defined but
  not implemented.

### Gate 8: Manual Smoke Review

- A developer must run `zelynic usage --sample` on a real system.
- The output must be reviewed for correctness (interface names, byte values,
  human-readable formatting).
- The output must be reviewed for honesty (all 13 disclaimers present).
- The output must be reviewed for safety (no enforcement claims, no path
  exposure).
- The review must be documented in a phase completion note.

### Gate 9: Clap Integration Pattern

When the `Usage` variant is added to `Commands`, the following pattern must be
used to maintain the progressive gate approach:

```rust
// In Commands enum:
Usage(UsageArgs),

// UsageArgs with --sample as required flag:
#[derive(Args, Debug)]
pub struct UsageArgs {
    #[arg(long, required = true)]
    sample: bool,

    // Future flags (all optional, all hard-blocked initially):
    // --json: deferred to phase 5
    // --interface <NAME>: deferred
    // --delta --interval <DUR>: deferred to phase 6
    // --no-loopback: deferred
    // --all-interfaces: deferred (this is the default)
}
```

The dispatch handler in `commands/mod.rs` must:
1. Call `build_live_proc_net_dev_read_plan()` or the injected reader.
2. Call `render_live_proc_net_dev_read_plan()` to produce output.
3. Print the rendered output.
4. Exit without mutation, persistence, or enforcement.

## Implementation Plan (Future, Not Phase 4)

Phase 4 does not implement any code. The actual implementation will occur in a
designated future phase and must follow this sequence:

### Step 1: Add `Usage` variant to `Commands`

Add `Usage(UsageArgs)` to the `Commands` enum in `src/cli.rs` with `--sample`
as a required flag.

### Step 2: Add dispatch handler

Add a `handle_usage` function in `src/commands/` that:
- Calls `build_live_proc_net_dev_read_plan()` to create a plan.
- Uses `read_live_proc_net_dev()` to read `/proc/net/dev`.
- Calls `render_live_proc_net_dev_read_plan()` to render the output.
- Prints the rendered string to stdout.
- Returns `Ok(())` — no mutation, no persistence.

### Step 3: Wire into dispatch

Add `Commands::Usage(args) => handle_usage(args)` to the match in
`commands::dispatch()`.

### Step 4: Validate

- All existing tests must continue to pass.
- Manual smoke test on a real system.
- Output review for honesty and correctness.
- `./build.sh check-all` must pass.

## Relationship to Existing Architecture

### Reuse of Reader Seam

The `usage --sample` implementation must reuse the existing reader seam from
`src/accounting/live_proc_net_dev.rs`:

- `read_live_proc_net_dev()` for the actual filesystem read.
- `render_live_proc_net_dev_read_plan()` for the output rendering.
- The existing parser `parse_proc_net_dev()` for content parsing.
- The existing `InterfaceCounterSnapshot` and related types.

No new parser, no new render function, no new model types. Only CLI wiring.

### Reuse of Output Contract

The render output already includes:
- Read-only seam statement
- Interface-level attribution disclaimer
- All 13 honesty disclaimers
- Mutation flags (all false)
- Source path and label
- Snapshot summary with interface names and byte counts
- Total (excluding loopback) line
- Counter reset warning

This output format is already tested and validated. The CLI implementation must
use it as-is.

### Relationship to `zelynic list --live`

The existing `zelynic list --live` command provides real-time per-process
bandwidth monitoring via nftables counters. The `zelynic usage --sample` command
is distinct:

| Property | `list --live` | `usage --sample` |
|----------|--------------|-------------------|
| Data source | nftables counters | `/proc/net/dev` |
| Attribution | Per-process (PID-based) | Per-interface only |
| Refresh | Continuous (TUI) | Single sample |
| Purpose | Real-time monitoring | Snapshot usage report |
| Requires | Active strict rules | No limiter setup |

The two commands serve different purposes and use different data sources.
`usage --sample` does not depend on `zelynic strict` being active.
