# Zelynic v3.0 Phase 5b: `usage --sample` Live CLI Validation / Output Honesty Freeze

This document captures the validation freeze for the first live read-only CLI
command implemented in v3.0 phase 5. Phase 5b freezes the `zelynic usage --sample`
implementation before adding JSON output, delta sampling, interface filtering, or
persistence. No new features are added in this phase — only validation, audit,
and documentation.

## Purpose

Phase 5b is a **validation and freeze checkpoint** that locks the current
`zelynic usage --sample` implementation before any feature expansion. The freeze
serves three purposes: (1) audit the first live read-only CLI command for output
honesty and safety boundary compliance, (2) document the current validation
state and test coverage, and (3) establish explicit criteria that must be
satisfied before JSON, delta sampling, filtering, or persistence can be added.

This follows the Zelynic pattern of progressive freezes (as used in v2.9 phase
10 persistence seam freeze and v2.9 phase 11 RC freeze).

## Live CLI Behavior Proof

### `zelynic usage` without `--sample` does not silently read `/proc/net/dev`

Clap enforces `--sample` as a required flag on the `Usage` command variant. When
invoked without `--sample`, clap rejects the invocation with an error message.
The `handle_usage_no_sample()` function exists as a fallback but is not reached
in normal clap parsing — clap rejects before dispatch.

```
$ zelynic usage
error: the following required arguments were not provided:
  --sample

Usage: zelynic usage --sample
```

This guarantees that no live filesystem read is performed when the user omits
the `--sample` flag.

### `zelynic usage --sample` performs one read of `/proc/net/dev`

The handler calls `read_live_proc_net_dev()` exactly once, which performs a single
`std::fs::read_to_string("/proc/net/dev")`. The result is parsed once and rendered
once. There is no loop, no retry, no polling.

### Command exits after one sample

The `handle_usage_sample()` function reads once, renders once, prints once, and
returns `Ok(())`. The process exits normally after the single render. There is no
background thread, no daemon mode, no interval loop.

### No loop/watch mode

The implementation is single-shot. There is no `loop {}`, no `tokio::spawn`, no
polling interval, no `--watch` or `--interval` flag on the `Usage` command.

### No arbitrary path input

The `Usage` variant in `Commands` has only `sample: bool`. There is no `--source`,
`--file`, `--interface`, or any other path-accepting argument. The source path is
hardcoded to `/proc/net/dev` in `DEFAULT_LIVE_SOURCE_PATH`. The `ContentReader`
trait has no path parameter.

### No sysfs read

The reader reads only `/proc/net/dev`. Source-level audit constants
(`FORBIDDEN_PATHS`) verify the module source does not contain sysfs or cgroup
paths. No `/sys/class/net`, `/sys/fs/cgroup`, or other sysfs paths are accessed.

### No ledger persistence

The usage handler does not read or write ledger files. The
`filesystem_write_performed` flag is always `false`. No `LedgerPersistencePlan`
is created. The v2.9 hard-blocked persistence seam remains in place.

### No state mutation

The `state_mutation_performed` flag is always `false`. No nftables rules, tc
qdisc/filters/classes, or Zelynic runtime state is modified. No PIDs are moved.
No cgroups are created or modified.

### No enforcement/blocking

The usage command is display-only. It does not block, throttle, or enforce any
quota. It does not attach rate limiters. It does not drop or reject any packets.
`zelynic strict` remains the only validated active limiter path.
