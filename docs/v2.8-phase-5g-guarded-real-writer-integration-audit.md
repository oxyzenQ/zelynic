# v2.8 Phase 5g: Guarded Real Writer Integration Audit / Blocked-Path Proof

**Phase**: 5g
**Type**: Audit / blocked-path proof
**Status**: Complete

## Purpose

Phase 5g provides a formal integration audit proving that the guarded real writer
seam (introduced in phase 5d, frozen in phase 5e, refactored in phase 5f)
remains **internal, hard-blocked, non-mutating, and unreachable from any live
CLI or runtime path**. This audit covers the entire module graph, CLI surface,
and output safety properties.

## Phase History Summary

### Phase 5d: Guarded Real Writer Seam

The guarded real writer seam was introduced as a narrow code interface modeling
the future real writer boundary for the first live PID move. The seam provides
two pure functions (`build_guarded_real_writer_plan` and
`render_guarded_real_writer_plan`) that validate 7 gate conditions and always
return a hard-blocked result. No I/O, no filesystem access, no /proc access, no
/sys access. The seam was implemented as a single 945 LOC file
(`src/systemd_wrapper/guarded_real_writer.rs`) containing input/result models,
gate validation logic, rendering, and 45 tests.

### Phase 5e: Guarded Real Writer Seam Freeze / Non-Exposure Audit

Phase 5e produced a freeze/non-exposure audit report confirming that the seam
module was registered as internal only, that no CLI command referenced it, that
the attach-live path remained hard-blocked, and that all related modules
(failure_simulation, fake_writer, move_executor, move_transaction) remained
blocked, model-only, or skeleton-only. The audit documented 11 future phase
entry criteria and confirmed explicit seam freeze guarantees.

### Phase 5f: LOC Split / Maintainability Refactor

Phase 5f refactored the single 945 LOC file into a directory module structure to
reduce LOC pressure and improve maintainability. The split preserved exact
behavior, API compatibility, and all 45 tests with identical results.

## Blocked-Path Proof

### 1. Module is Internal Only

The guarded_real_writer module is declared in
`src/systemd_wrapper/mod.rs` as:

```rust
mod guarded_real_writer;
```

This is a private module declaration. It is not `pub`. The module's public API
is re-exported only as `pub(crate)`, making it visible within the crate but not
to external callers. No `pub` re-export exists.

Verified by `grep` search: `build_guarded_real_writer_plan` and
`render_guarded_real_writer_plan` appear only within the guarded_real_writer
directory itself (`mod.rs`, `render.rs`, `tests.rs`). No other source file in the
crate imports or calls these functions.

### 2. No CLI Command Calls It

The CLI layer (`src/cli.rs`, `src/cli/tests.rs`) contains zero references to
`guarded_real_writer`, `build_guarded_real_writer_plan`, or
`render_guarded_real_writer_plan`. The CLI commands (run, scope-runner,
probe, attach) do not have any code path that reaches the guarded real writer
module.

Verified by `grep` search across `src/cli/` and `src/commands/`: no matches.

### 3. No Runtime Path Calls It

The runtime execution path flows through:
- `src/commands/run.rs` → scope runner command dispatch
- `src/systemd_wrapper/scope_runner.rs` → scope lifecycle
- `src/systemd_wrapper/scope_probe.rs` → PID/cgroup discovery
- `src/systemd_wrapper/experimental_attach_gate.rs` → gate checklist

None of these files import or reference `guarded_real_writer`. The guarded
real writer module is dead code in the current build. It is compiled (with
`#![allow(dead_code, unused_imports)]`) but never invoked at runtime.

### 4. Attach-Live Path Remains Hard-Blocked

The `--attach-live` flag, when combined with all required consent flags, enters
the experimental attach gate checklist. The gate checklist evaluates the move
executor seam (phase 3c), which always returns `status: "blocked"` and
`execution: "not implemented"`. The gate output renders the seam section as
report-only text, then terminates with the existing hard-block message:
"Experimental PID move is not implemented yet."

The guarded real writer (phase 5d) is NOT wired into the experimental attach
gate. The attach gate uses only `move_executor` (phase 3c), which predates the
guarded real writer seam. The guarded real writer is a separate, independent
module not connected to any runtime path.

### 5. Mkdir-Live Path Remains Mkdir-Only

The `--mkdir-live` flag triggers the mkdir-only experiment in
`src/systemd_wrapper/mkdir_executor.rs`. This executor:
- Creates cgroup directories (mkdir only)
- Never writes to `cgroup.procs`
- Never moves PIDs
- Never touches nftables/tc/state
- Reports `cgroup_procs_writes: "disabled"` in its output

The mkdir executor has no reference to `guarded_real_writer`. The guarded real
writer seam does not participate in the mkdir-only path.

### 6. Move Executor Remains Blocked

`src/systemd_wrapper/move_executor.rs` (phase 3c) always returns
`status: "blocked"` and `execution: "not implemented"`. Every mutating step
produces a "blocked/not-implemented" result. The seam explicitly states:
"phase 3c is executor-seam only" and "live PID move is not implemented." The
move executor does not call or depend on the guarded real writer module.

### 7. Move Transaction Remains Skeleton/Model-Only

`src/systemd_wrapper/move_transaction.rs` (phase 3b) produces a skeleton with
`status: "skeleton only; not executed"` and `execution: "blocked"`. All 10
transaction steps remain "planned" / model-only / execution-blocked. The move
transaction has no reference to the guarded real writer module.

### 8. Failure Simulation Remains Fake/Model-Only

`src/systemd_wrapper/failure_simulation/` (phases 4b–4d) provides pure
models for failure scenarios. All 12 failure scenarios (F1–F12) produce
simulation results with canonical deny lines stating no real operations were
performed. The fake writer (`failure_simulation/fake_writer/`) simulates
cgroup.procs writes in-memory without touching the real filesystem. Neither
module references the guarded real writer.

### 9. Fake Writer Remains Fake/Model-Only

`src/systemd_wrapper/failure_simulation/fake_writer/` (phases 4c–4d) simulates
write operations with injectable failure modes. All output includes 7 canonical
deny lines: no real cgroup.procs write, no live PID move, no limiter attach, no
nft/tc/Zelynic state mutation, no persistent state write, no CLI path for
live PID move, fake/model-only. The fake writer has no reference to the guarded
real writer module.

## Output/Safety Proof

### Canonical Deny Lines

The guarded real writer seam output always includes exactly 7 canonical deny
lines. These are defined as `CANONICAL_DENY_LINES` in `model.rs`:

1. "No live PID move was performed."
2. "No real cgroup.procs write was performed."
3. "No limiter attach was performed."
4. "No nftables, tc, or Zelynic state changes were made."
5. "No persistent state write was performed."
6. "No CLI path for live PID move was enabled."
7. "Guarded real writer seam is hard-blocked."

### Forbidden Claims Absent

The rendered output never claims:
- PID was moved (verified by 7 tests across valid and negative paths)
- cgroup.procs was written (verified by test)
- rollback was performed (verified by test)
- cleanup mutation occurred (verified by test)
- limiter was attached (verified by test)
- bandwidth limiting is active (verified by test)
- nft/tc/state was mutated (verified by test)
- persistent state was written (verified by explicit deny line)

### Non-Mutating Result Fields

Every `GuardedRealWriterResult` has hardcoded non-mutating values:
- `pid_location`: always `GuardedPidLocation::NotMoved`
- `rollback_attempted`: always `false`
- `cleanup_attempted`: always `false`
- `cgroup_procs_writes_performed`: always `false`
- `limiter_attach_performed`: always `false`
- `nft_tc_state_mutation_performed`: always `false`
- `status`: always `"blocked"`

These fields are struct literals, not computed from any external state. No
branch in the code can produce a different value.

### Comprehensive Negative-Path Mutation Sweep

Test `negative_path_outputs_never_claim_mutation` sweeps 7 negative-path
scenarios (non-root, user scope, zero PID, missing original cgroup,
zelynic-managed original, invalid target, missing rollback consent) and asserts
that none of the rendered outputs claim: PID moved, cgroup.procs written,
limiter attached, bandwidth limiting active, rollback performed, or state
mutation. All 7 scenarios pass.

## Future Real Move Activation Criteria

The guarded real writer seam is the designated entry point for the future first
real PID move. Activating it requires ALL of the following:

1. **Separate explicit implementation phase**: A dedicated phase (beyond 5g)
   must implement the real writer. The seam currently returns blocked; changing
   this behavior requires its own phase, review, and commit.

2. **Separate explicit CLI gate**: A new CLI flag or flag combination must be
   introduced that is distinct from all existing flags. The current flag set
   (`--execute`, `--probe-live`, `--attach-live`,
   `--experimental-single-pid-attach`,
   `--i-understand-this-moves-pids`, `--rollback-required`) gates the
   mkdir-only experiment and the blocked attach path; the real move requires
   an additional explicit gate.

3. **Separate root smoke review**: Before any real move code executes, the
   exact root smoke commands must be reviewed by the operator. The manual smoke
   plan from phase 5a and the operator checklist from phase 5b define the
   required steps and abort conditions.

4. **Root-only**: The caller must have euid == 0. The seam's gate 1 enforces
   this at the model level; the real implementation must enforce it at runtime
   before any write.

5. **System-scope-only**: The target PID must be in system scope (not
   user.slice). The seam's gate 2 enforces this at the model level.

6. **Single disposable sleep PID only**: The target must be a single PID running
   a disposable process (e.g., `sleep 3` or `sleep 10`). No browser, terminal,
   desktop, or long-running application processes are permitted. Multi-PID
   process trees are forbidden.

7. **Immediate rollback required**: After the move, the PID must be moved back
   to its original cgroup immediately. The `--rollback-required` flag must be
   present. If rollback fails, manual recovery is required.

8. **No limiter/nft/tc/state**: The real move must not attach a bandwidth
   limiter, must not modify nftables rules, must not modify tc qdisc/class/filter,
   and must not mutate Zelynic persistent state.

9. **Operator must review exact commands before execution**: Before the first
   real move, the operator must have reviewed the exact commands from the phase
   5b operator checklist and confirmed all abort conditions are clear.
