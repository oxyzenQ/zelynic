# Zelynic v3.0 Phase 9: Delta Sampling Design for `usage --sample --delta`

This document designs the delta sampling behavior for a future `zelynic usage --sample
--delta` command. Phase 9 is **design-only**: it produces this document and updates
related documentation but does NOT implement the `--delta` CLI flag, does NOT add any
Rust code changes, does NOT add interval sampling, does NOT introduce loop/watch
mode, and does NOT introduce any runtime behavior changes. The design freezes the
contract that future implementation phases must follow.

## Purpose

Phase 9 designs how delta sampling will work once implemented in future phases
(10-14). Delta sampling extends the existing single-shot `usage --sample` command
by taking two `/proc/net/dev` snapshots separated by a wait duration, computing
the per-interface byte/packet differences using the existing `SessionDelta` model
from v2.9, and rendering the result as either human-readable text or structured
JSON. This phase defines the contract, sampling rules, output honesty
requirements, JSON delta contract direction, and the phase plan for implementation.
No implementation occurs in this phase.

The delta sampling design is firmly grounded in the existing v2.9 `session_delta`
module (`src/accounting/session_delta.rs`), which already provides a pure model
for computing interface-level usage differences between two
`InterfaceCounterSnapshot` values with explicit counter reset detection, overflow-safe
saturating totals, and comprehensive safety disclaimers. The implementation phases
will wire this model into the live reader seam.

## Existing Foundation: `session_delta` Module

The v2.9 `session_delta` module (`src/accounting/session_delta.rs`) provides the
pure model that delta sampling will build upon. Key components:

- `build_session_delta(start: &InterfaceCounterSnapshot, end: &InterfaceCounterSnapshot) -> SessionDelta`:
  Computes per-interface deltas for RX/TX bytes and packets with deterministic
  ordering, overflow-safe saturating totals, and explicit counter reset warnings.
- `SessionDelta`: Contains `rows` (per-interface delta rows), `warnings` (counter
  reset/decrease warnings), `total_rx_delta_bytes`, `total_tx_delta_bytes`,
  `total_combined_delta_bytes`, `interface_count`, `source_label`,
  `attribution_scope` ("interface-level only"), `enforcement_status`
  ("inactive/not implemented"), `read_only` ("model-only").
- `SessionDeltaRow`: Per-interface delta with `rx_delta_bytes`, `tx_delta_bytes`,
  `combined_delta_bytes`, `rx_delta_packets`, `tx_delta_packets`, reset flags
  (`rx_reset`, `tx_reset`, `rx_packets_reset`, `tx_packets_reset`),
  `present_in_start`, `present_in_end`.
- `CounterResetWarning`: Interface name, counter field, start/end values with
  Display formatting ("counter reset/decrease on 'iface' field 'rx_bytes': start=X end=Y
  (delta reported as 0 -- exact usage unknown)").
- `safe_delta(start: u64, end: u64) -> (u64, bool)`: Returns `(end - start, false)`
  if end >= start, or `(0, true)` if end < start (counter reset). Never produces
  negative values.
- `render_session_delta()`: Human-readable output with 7 safety disclaimers
  (read-only session delta model, interface-level only, not per-app attribution,
  no quota enforcement active, no network blocking active, no limiter attach performed,
  no nft/tc/Zelynic state mutation performed, no live /proc or sysfs read performed).
- 33 tests in `src/accounting/tests/session_delta.rs`: normal delta, multiple
  interfaces, totals, zero delta, start-only/end-only interfaces, counter reset
  detection for RX/TX bytes/packets, no silent underflow, deterministic ordering,
  overflow-safe saturating totals (u64::MAX), render honesty, render warnings.

This module is pure model — no live system reads, no filesystem access, no
enforcement. It operates on in-memory `InterfaceCounterSnapshot` values only.

## Future Delta Behavior Design

### Two-Sample Protocol

Delta sampling requires exactly two reads of `/proc/net/dev`:

1. **First sample (start snapshot)**: Read `/proc/net/dev` exactly once. Parse the
   content using the existing `parse_proc_net_dev()` parser. Produce an
   `InterfaceCounterSnapshot` with `SourceLabel::LiveProcNetDev`.

2. **Wait duration**: An explicit, user-specified wait duration between samples.
   The wait duration must be provided by the caller via a future `--interval`
   flag (e.g., `--interval 5s`). No implicit or default wait duration is used.
   The wait is a simple `std::thread::sleep` — no timer wheel, no async runtime,
   no daemon behavior.

3. **Second sample (end snapshot)**: Read `/proc/net/dev` exactly once again.
   Parse the content identically to the first sample. Produce a second
   `InterfaceCounterSnapshot` with `SourceLabel::LiveProcNetDev`.

4. **Delta computation**: Pass both snapshots to `build_session_delta(start, end)`.
   The function computes per-interface deltas with counter reset detection,
   overflow-safe saturating totals, and deterministic ordering.

### Sampling Rules

| Rule | Description |
|------|-------------|
| First sample reads `/proc/net/dev` exactly once | Single `std::fs::read_to_string`, single parse. No retries, no loop. |
| Wait duration must be explicit | Caller must provide `--interval <duration>`. No implicit default. No silent default of 1s or 0s. |
| Second sample reads `/proc/net/dev` exactly once | Single `std::fs::read_to_string`, single parse. No retries, no loop. |
| Compute deltas using existing `session_delta` model | `build_session_delta(start, end)`. No new delta computation logic. |
| Detect counter reset/decrease | If end counter < start counter for any field, delta = 0 with warning. No silent underflow. |
| Do not silently underflow | Negative deltas are impossible — `safe_delta()` returns `(0, true)` on reset. |
| No background loop | Delta is computed once after two samples. No repeated sampling. |
| No daemon behavior | Command exits after output. No long-running process. |
| No state persistence | Snapshots are held in memory only for the duration of the command. No disk writes. |
| Interface-level only | Deltas are per-interface aggregate counters. Not per-app, not per-PID, not per-cgroup. |
| Source path hardcoded to `/proc/net/dev` | No arbitrary path accepted. Same boundary as existing `usage --sample`. |

### Counter Reset Handling

The existing `session_delta` model handles counter resets as follows:

- If a counter value in the end snapshot is lower than in the start snapshot,
  this indicates a counter reset (e.g., interface down/up, driver reload, or
  reboot).
- The delta for that counter is reported as **0** with an explicit
  `CounterResetWarning`.
- No negative values are ever produced.
- The warning includes the interface name, counter field, start value, and end
  value.
- The warning message explicitly states "exact usage unknown" for that counter.

This behavior is already tested by the 33 session_delta tests and is reused
directly by the delta sampling implementation.

### Error Handling

Errors can occur at three points in the delta sampling flow:

1. **First sample read error**: `/proc/net/dev` read or parse failure on the first
   sample. The command must report the error and exit. No delta computation
   attempted.

2. **Wait interruption**: If the wait is interrupted (e.g., by signal), the
   command should exit gracefully. No partial delta output.

3. **Second sample read error**: `/proc/net/dev` read or parse failure on the
   second sample. The command must report the error and exit. No partial delta
   using only the first sample.

Error reporting follows the same contract as `usage --sample --json`: error JSON
with canonical error type strings (`read_error`, `parse_error`) and full honesty
flags preserved.

## Future Command Shape

### `zelynic usage --sample --delta`

The primary delta sampling command. Takes two samples separated by an explicit
interval, computes deltas, and displays the result.

```
$ zelynic usage --sample --delta --interval 5s
zelynic usage --sample --delta (5s interval): live read-only delta
Source: live_proc_net_dev (two samples: start + end)
Attribution: interface-level only (not per-app)
No quota enforcement active. No network blocking active.
No limiter attach performed. No nft/tc/Zelynic state mutation performed.
No ledger persistence performed. No eBPF used.
No cgroup mutation. No PID movement.
Counters may reset after reboot or interface reset.
Filesystem write not performed. State mutation not performed.

Interface     RX delta     TX delta     Since
wlan0         61.5 MiB    22.5 MiB    5s sample
eth0          0 bytes      0 bytes     5s sample

Delta totals:  RX 61.5 MiB  TX 22.5 MiB  Combined 84.0 MiB
Interfaces: 2
```

**Constraints**:
- `--delta` requires `--sample` (clap enforces).
- `--interval <duration>` is required when `--delta` is used (clap enforces).
- Single-shot: two reads, one delta computation, one output, exit.
- No loop, no watch, no repeated sampling.
- Read-only. No enforcement. No blocking.

### Error JSON for Delta

Errors during delta sampling use the same error contract as snapshot errors:

| Error Scenario | Error Type | Description |
|----------------|-------------|-------------|
| First sample read failure | `read_error` | Cannot read `/proc/net/dev` for the start snapshot. |
| First sample parse failure | `parse_error` | Cannot parse `/proc/net/dev` content for the start snapshot. |
| Second sample read failure | `read_error` | Cannot read `/proc/net/dev` for the end snapshot. |
| Second sample parse failure | `parse_error` | Cannot parse `/proc/net/dev` content for the end snapshot. |

Error JSON includes full honesty flags (all constant in v3.0). No delta-specific
error types are needed — the existing three error types (`read_error`,
`parse_error`, `unsupported_flag_error`) are sufficient.

### sampled_at Policy for Delta

The `sampled_at` policy for delta sampling follows the same strict
caller-provided-only policy as the snapshot contract:

- `start_sampled_at`: Omitted when not provided. Never silently generated.
- `end_sampled_at`: Omitted when not provided. Never silently generated.
- No `SystemTime::now()` or `Instant::now()` call in the production output path.
- If timestamps are added in a future phase, both start and end timestamps
  must be explicitly provided.
