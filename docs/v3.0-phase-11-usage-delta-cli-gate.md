# v3.0 Phase 11: `--delta` CLI Gate Design / Blocked Flag Contract

## Purpose

Phase 11 defines the CLI gate and activation rules for the future
`zelynic usage --sample --delta` command before any actual two-sample delta
behavior is implemented. This phase is **design-only** and **docs-only**: it
produces this gate document and updates related documentation but does not add
any Rust code, does not register any CLI flag, and does not perform any live
filesystem reads.

The gate document serves as a contract between the delta output model
(implemented in phase 10) and the future CLI wiring phase (phase 12). It
defines exactly what conditions must be satisfied before `--delta` can be
exposed as a clap flag on the `Usage` variant, what the initial blocked
behavior should look like, and what hard prohibitions remain in effect.

Phase 11 enforces the same hard safety boundary as all v3.0 phases. In

| Two-sample live `/proc/net/dev` read | Hard-blocked |
| Interval sampling | Hard-blocked |
| Loop/watch mode | Hard-blocked |
| Interface filtering | Hard-blocked |
| Persistence (ledger file read/write) | Hard-blocked |

| nftables/tc rule mutation | Hard-blocked |
| Zelynic runtime state mutation | Hard-blocked |

| Rust code changes (unless hard-blocked flag) | Hard-blocked (preferred docs-only) |

Only existing live filesystem read remains `/proc/net/dev` through the

## Future Command Shape

### Primary Command: `zelynic usage --sample --delta`

The future delta command takes exactly two live snapshots of `/proc/net/dev`,
separated by a bounded wait duration, computes the per-interface delta using
the existing `SessionDelta` model, and renders the result using the
`UsageDeltaOutput` model (implemented in phase 10).

```
$ zelynic usage --sample --delta
zelynic usage --sample --delta: live read-only delta (single-shot)
Source: live_proc_net_dev (two reads)
Attribution: interface-level only (not per-app)
No quota enforcement active. No network blocking active.
No limiter attach performed. No nft/tc/Zelynic state mutation performed.
No ledger persistence performed. No eBPF used.
Counters may reset after reboot or interface reset.
Delta computed from two /proc/net/dev reads.

Interface     RX Delta     TX Delta     Status
wlan0         45.6 MiB     12.3 MiB     ok
eth0          0 bytes       0 bytes      ok

Total delta:  RX 45.6 MiB  TX 12.3 MiB
Delta interval: <bounded wait>
```

**Status**: Not yet implemented. Requires phase 12 activation.
**Constraints**: Interface-level only. Read-only. No enforcement. No loop.

## Absolute Constraints (Inherited and Delta-Specific)

The following constraints apply to `--delta` in all phases including the
future implementation. These are non-negotiable:

### Inherited Constraints

| Constraint | Description |
|-----------|-------------|
| Interface-level only | Delta is computed per-interface, not per-app or per-process |
| No per-app attribution | Delta data does not identify which process sent/received bytes |
| No quota enforcement | Delta display does not enforce any bandwidth budget |
| No network blocking | Delta display does not block, drop, or reject any traffic |
| No persistence | Delta results are not written to any ledger file |
| No eBPF | Delta computation uses only `/proc/net/dev`, not eBPF |
| No nft/tc mutation | Delta display does not modify nftables or tc configuration |
| No cgroup/PID mutation | Delta display does not move PIDs or write cgroup.procs |
| No loop/watch | Delta is single-shot, not continuous monitoring |
| No arbitrary path | Source path is hardcoded to `/proc/net/dev` |
| No filesystem write | Delta computation does not write any file |
| Read-only | Delta is observed and reported, never used for enforcement |

### Delta-Specific Constraints

| Constraint | Description |
|-----------|-------------|
| Exactly two reads | Delta requires exactly two reads of `/proc/net/dev`, no more |
| Bounded wait | Wait between reads is explicit and bounded (no infinite sleep) |
| No background loop | After two reads and render, the command exits |
| No daemon behavior | Delta command does not run as a background process |
| Counter reset handling | Counter resets/decreases are detected and reported as warnings |
| No state mutation | Delta computation does not modify any runtime state |
| Single-shot only | One delta computation per invocation, no repeated sampling |

## Activation Criteria

The following criteria must all be satisfied before the `--delta` flag is
enabled on the `Usage` CLI variant. These are explicit gates that prevent
premature activation:

### Gate 1: Phase Completion

All prerequisite phases must be completed and committed:

- Phase 9: Delta sampling design document (completed)
- Phase 10: Pure delta output model + render tests (completed)
- Phase 11: CLI gate design / blocked flag contract (this phase, current)

### Gate 2: Two-Read Behavior Reviewed

The exact two-read behavior must be reviewed and documented:

- First read: exactly `/proc/net/dev` via existing `read_live_proc_net_dev()`
- Wait duration: explicit value, bounded, documented (e.g., 1s default)
- Second read: exactly `/proc/net/dev` via same reader function
- No additional reads between first and second
- No reads after second (render only)

### Gate 3: Reader Seam Integrity

The existing reader seam must remain intact:

- Source path hardcoded to `/proc/net/dev` (no arbitrary paths)
- ContentReader trait used for test injection
- Live read via `read_live_proc_net_dev()` only
- Source label honestly set to `live_proc_net_dev`
- Parse errors and read errors distinguished and handled

### Gate 4: No Arbitrary Path Input

The `--delta` flag must not introduce any path-related CLI argument:

- No `--source` or `--file` argument
- No `--path` argument
- Source is always `/proc/net/dev`

### Gate 5: Output Honesty Verified

All required honesty disclaimers must be present in the delta render output:

- Interface-level only (not per-app attribution)
- No quota enforcement active
- No network blocking active
- No limiter attach performed
- No nft/tc/Zelynic state mutation performed
- No ledger persistence performed
- No eBPF used
- No cgroup mutation performed
- No PID movement performed
- Filesystem write not performed
- State mutation not performed
- Counters may reset after reboot or interface reset
- Delta-specific: counter reset/decrease warnings included
- Delta-specific: delta may be incomplete warning when resets detected

### Gate 6: No Enforcement/Blocking/Persistence Claims

The delta output must never claim or imply:

- Per-app attribution capability
- Quota enforcement action
- Network blocking action
- Traffic shaping action
- Persistent data storage

### Gate 7: No Background Loop

The delta command must be single-shot:

- Two reads, one delta computation, one render, then exit
- No repeated sampling
- No sleep-after-render loop
- No daemon/watch mode

### Gate 8: Reset/Decrease Handling Reviewed

Counter reset and decrease handling must be reviewed:

- `safe_delta()` from `session_delta.rs` detects wraps (counter reset)
- Warnings are generated for interfaces where counters decreased
- Render output includes reset/decrease warnings
- Delta may be marked as incomplete when resets are detected
- No panic on counter arithmetic (saturating arithmetic used)

### Gate 9: Text Output Reviewed

The human-readable delta text output must be reviewed:

- Uses `render_usage_delta()` from phase 10
- All 14 safety disclaimers present
- Counter reset warnings displayed
- Human-readable byte formatting (IEC binary prefixes)
- Totals line present
- Interface-level delta rows present

### Gate 10: JSON Output Contract Reviewed

Before `--delta --json` is enabled, the JSON contract must be reviewed:

- Delta-specific fields defined (delta.interfaces, delta.totals, warnings)
- Honesty flags extended for delta context
- Schema version unchanged or incremented
- Round-trip serialization verified
- Error JSON contract for delta (read errors on first/second read)

### Gate 11: Tests Prove No Persistence/Enforcement Claims

Tests must prove the delta output does not claim persistence or enforcement:

- All honesty flags are constant `false` (except delta-model/CLI/live flags)
- No "quota" string in output
- No "blocked" string in output
- No "enforced" string in output
- No ledger file path in output
- No filesystem write in delta computation path

### Gate 12: Manual Smoke Output Reviewed

Manual smoke test of delta output must be reviewed by developer:

- `cargo run -- usage --sample --delta` produces expected text output
- All safety disclaimers visible
- No unexpected side effects
- Command exits after single delta computation
- Exit code is 0 on success

## Initial CLI Behavior (Pre-Activation)

Before all activation criteria are satisfied, the `--delta` flag follows
these rules:

### Option A: No Flag Registered (Preferred for Phase 11)

The preferred approach for phase 11 is **docs-only** with no `--delta` flag
registered in clap. This avoids exposing a misleading command that would be
immediately rejected. The existing `no_json_delta_interval_flags_on_usage_command`
test in `src/commands/usage.rs` already verifies that `--delta` is not a
recognized flag on the `Usage` variant.

### Option B: Hard-Blocked Flag (If Repo Pattern Favors)

If the repository pattern strongly favors adding the flag with a hard block
(referencing the phase 4 gate approach), the following behavior applies:

- `--delta` is added to `Usage` variant with `#[arg(long)]`
- `handle_usage_sample()` checks for `delta == true` before any I/O
- If `delta` is true, the handler prints a rejection message and exits:
  ```
  error: --delta is not yet implemented. Delta sampling requires a future
  implementation phase. Use `zelynic usage --sample` for single-shot
  interface counters.
  ```
- The handler does NOT read `/proc/net/dev` when `--delta` is true
- The handler does NOT perform any second read
- The handler does NOT enter any wait/sleep
- The handler does NOT compute any delta
- A test verifies the rejection message is emitted
- A test verifies no live read occurs when `--delta` is passed

**Decision for phase 11**: Option A (docs-only, no flag) is preferred.
The repo already has a structural test confirming `--delta` is not recognized.
Adding a blocked flag is deferred to phase 12 if needed.

### `usage --delta` Without `--sample`

If `--delta` were ever registered without `requires = "sample"`, it must
NOT read `/proc/net/dev` on its own. The `--sample` flag is the gateway
for any live filesystem read. Without `--sample`, `--delta` must be rejected
with a message like:

```
error: --delta requires --sample. No live read performed.
```

### `usage --sample --delta --json` Pre-Activation

The JSON delta output is deferred to phase 13. Before phase 13:

- `--delta --json` is not a valid combination
- If both flags are present, the handler must reject before any delta computation
- The rejection message must clearly state that delta JSON is not yet implemented

### `--interval` Availability

The `--interval` flag remains unavailable until a future explicit phase:

- `--interval` is not registered in clap
- The structural test `no_json_delta_interval_flags_on_usage_command` confirms
  this
- No interval parsing or duration handling exists
- The wait duration between two delta reads is an implementation detail of
  phase 12, not a user-configurable parameter at this stage

## Relationship to Existing Architecture

### Reader Seam Reuse

The delta implementation (phase 12) will reuse the existing reader seam:

- `read_live_proc_net_dev()` for the first live read
- `read_live_proc_net_dev()` for the second live read (same function, called twice)
- `ContentReader` trait for test injection of both reads
- `parse_proc_net_dev()` for parsing both reads
- `InterfaceCounterSnapshot` as the intermediate type for both reads

### SessionDelta Reuse

The delta computation (phase 12) will reuse the existing `SessionDelta` model:

- `build_session_delta()` computes per-interface deltas between two snapshots
- `safe_delta()` handles counter wraps and produces `CounterResetWarning`
- `SessionDelta` provides per-interface `SessionDeltaRow` with rx/tx/combined bytes

### UsageDeltaOutput Reuse

The delta rendering (phase 12) will reuse the phase 10 output model:

- `build_usage_delta_from_session_delta()` transforms `SessionDelta` to
  `UsageDeltaOutput`
- `render_usage_delta()` renders the human-readable output
- All 14 safety disclaimers are included automatically

### JSON Model Extension (Phase 13)

The delta JSON output will extend the existing JSON contract:

- Reuse `UsageJsonOutput` structure as a base
- Add delta-specific fields (interfaces with delta bytes, warnings, etc.)
- Reuse existing honesty flags and add delta-specific flags
- `serialize_usage_json()` / `deserialize_usage_json()` extended

## Prohibited Actions (Explicit Denial List)

The following actions are explicitly prohibited in all delta-related phases
(11 through 14):

1. **No delta CLI implementation** until all activation criteria are satisfied
2. **No two-sample live read** until phase 12 is activated
3. **No interval sampling** — the wait between reads is an implementation detail
4. **No loop/watch mode** — delta is single-shot only
5. **No interface filtering** — delta shows all interfaces (like `--sample`)
6. **No persistence** — delta results are not saved to any file
7. **No ledger file read/write** — the v2.9 hard-blocked persistence seam remains
8. **No eBPF** — delta uses only `/proc/net/dev` counters
9. **No quota enforcement** — delta is observation only, not enforcement
10. **No network blocking** — delta does not drop, reject, or shape traffic
11. **No limiter attach** — delta does not interact with `zelynic strict`
12. **No nft/tc mutation** — delta does not modify firewall or traffic control rules
13. **No state mutation** — delta computation does not modify Zelynic runtime state
14. **No PID move** — delta does not move processes between cgroups
15. **No cgroup.procs write** — delta does not write to any cgroup file
16. **No sysfs read** — delta does not read `/sys/fs/cgroup` or any sysfs path
17. **No filesystem write** — delta computation produces output to stdout only
18. **No arbitrary path read** — source is always `/proc/net/dev`
