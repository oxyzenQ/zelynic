# Zelynic v2.8 Phase 4a — Failure Simulation Design

## Overview

This document captures the design for failure simulations that will test the
robustness of the first real single-PID move experiment defined in phase 3a
(`docs/v2.8-phase-3a-single-pid-rollback-design.md`). Phase 4a is
**design-only**: it produces this document and updates related documentation
but introduces **no runtime changes, no PID movement, no `cgroup.procs`
writes, no limiter attach, and no state mutation**.

The purpose of phase 4a is to enumerate every credible failure mode in the
10-step transaction model, define the exact response for each, specify what
the output must say, and plan the test infrastructure needed to validate
these responses before any live PID move is attempted. When the failure
simulation tests are eventually implemented (in a future phase), they will
inject faults into a fake filesystem or fake writer abstraction and verify
that the rollback decision model, output rendering, and cleanup behavior
handle each failure correctly.

## Phase 4a Boundary

### What Phase 4a Produces

- This design document (`docs/v2.8-phase-4a-failure-simulation-design.md`).
- Updates to `docs/experimental-pid-move-lab.md` with phase 4a status.
- Updates to `CHANGELOG.md` with a design-only entry.
- No Rust code changes. No runtime behavior changes. No new CLI flags.

### What Phase 4a Does NOT Do

- Does NOT implement live PID movement.
- Does NOT write `cgroup.procs`.
- Does NOT add runtime mutation of any kind.
- Does NOT add limiter attach, nftables/tc enforcement, or state writes.
- Does NOT change CLI behavior.
- Does NOT add new command-line flags.
- Does NOT modify any Rust source file.
- Does NOT perform root smoke tests.
- Does NOT write test code (only a test plan is specified).

## Transaction Step Reference

The failure scenarios reference the 10-step transaction model from phase 3a:

| Step | Name | Type | Writes? |
|------|------|------|---------|
| 1 | Gate Re-Evaluation | Pre-move gate | No |
| 2 | PID Liveness Recheck | Pre-move gate | No |
| 3 | Original Cgroup Validation | Pre-move gate | No |
| 4 | Target Cgroup Preparation | Pre-move write | Yes (mkdir) |
| 5 | Move PID to Target | Move write | Yes (cgroup.procs) |
| 6 | Verify PID in Target | Move verify | No (read) |
| 7 | Record Move Success | In-memory update | No |
| 8 | Rollback PID to Original | Rollback write | Yes (cgroup.procs) |
| 9 | Verify PID Restored | Rollback verify | No (read) |
| 10 | Cleanup Target Cgroup | Cleanup write | Yes (rmdir) |

Steps 1-3 are pure gates. Step 4 is a filesystem mkdir. Steps 5 and 8 are
`cgroup.procs` writes. Steps 6 and 9 are reads. Step 10 is an rmdir.

## Failure Scenario Catalog

### Scenario F1: Failure Before Target Cgroup Creation (Step 4)

**Trigger**: Gate failure (step 1), PID dead (step 2), or original cgroup
invalid (step 3) prevents the operation from reaching step 4.

**Sub-scenarios**:
- **F1a**: One or more consent gates fail at re-evaluation (step 1).
- **F1b**: PID exited between probe discovery and the move moment (step 2).
- **F1c**: Original cgroup path changed or is unreadable (step 3).
- **F1d**: Original cgroup is under `/sys/fs/cgroup/zelynic/` (step 3).

**Expected rollback behavior**: No rollback needed. No `cgroup.procs` write
has occurred. No PID has moved. The operation aborts cleanly.

**Expected cleanup behavior**: No cleanup needed. No target cgroup was
created. No directories were created in this operation. The operation leaves
the system state exactly as it was before entry.

**Expected output honesty**:
- "Operation aborted at step <N>: <reason>."
- "No PID was moved."
- "No cgroup.procs write was performed."
- "No target cgroup was created."
- PID location: "not moved" (the PID was never moved).

**Target cgroup may remain**: No. No target was created.

**Manual recovery required**: No.

**What must never be claimed**: "Rollback was performed." "PID was moved."
"Target cgroup was created." "cgroup.procs was written." "Limiter was
attached."

---

### Scenario F2: Failure After Target Cgroup Creation but Before PID Move (Step 4→5 boundary)

**Trigger**: Target cgroup mkdir succeeds (step 4) but something prevents
the PID move write (step 5) from being attempted.

**Sub-scenarios**:
- **F2a**: Target cgroup created successfully, but PID exits between step 4
  and step 5.
- **F2b**: Target cgroup created successfully, but an unexpected error (e.g.,
  signal, panic) prevents step 5 from executing.

**Expected rollback behavior**: No PID move rollback is needed because no PID
was moved. The target cgroup was created and must be cleaned up.

**Expected cleanup behavior**: Attempt to remove the target cgroup directory
if it is empty and operation-owned. This reuses the `attempt_cleanup_target`
pattern from `mkdir_executor.rs`. If the target is non-empty (which should not
happen since no PID was moved), leave it in place and report.

**Expected output honesty**:
- "Target cgroup created: <path>."
- "PID move not attempted: <reason>."
- "No PID was moved."
- "No cgroup.procs write was performed."
- "Target cleanup: performed/skipped/leftover."
- PID location: "not moved" (the PID was never written to any cgroup.procs).

**Target cgroup may remain**: Yes, if cleanup fails or if the target is
unexpectedly non-empty. This is safe — the directory is empty or contains
only system-generated files, and it is under the Zelynic namespace.

**Manual recovery required**: No. The leftover target is an empty directory
under `/sys/fs/cgroup/zelynic/`. Manual removal with `rmdir` is safe if
desired.

**What must never be claimed**: "Rollback was performed." "PID was moved."
"cgroup.procs was written." "Limiter was attached." "PID was restored."

---

### Scenario F3: Failure After PID Move but Before Target Verification (Step 5→6 boundary)

**Trigger**: The PID write to target `cgroup.procs` succeeds (step 5), but
target verification cannot proceed.

**Sub-scenarios**:
- **F3a**: Move write succeeds, PID exits before verification read (step 6).
  The PID is already dead, so rollback to the original cgroup may succeed
  (kernel may accept the write for a zombie PID) or fail.
- **F3b**: Move write succeeds, but target `cgroup.procs` read fails with
  EACCES or ENOENT (step 6).

**Expected rollback behavior**: Rollback MUST be attempted (step 8), even
though verification was skipped. The PID may be in the target cgroup.
Write the PID string to the original cgroup's `cgroup.procs`. This is a
single attempt — no retry.

**Expected cleanup behavior**: After rollback (success or failure), attempt
to remove the target cgroup if empty and operation-owned. If the target is
non-empty (PID may still be there), leave it in place.

**Expected output honesty**:
- "PID <pid> written to target cgroup."
- "Target verification skipped: <reason>."
- "Rollback attempted: <success/failure>."
- If rollback succeeds: "PID <pid> restored to original cgroup <path>."
- If rollback fails: "ROLLBACK FAILURE: PID <pid> could not be restored to
  <path>."
- "Target cleanup: performed/skipped/leftover."
- PID location: "unknown" (move succeeded but verification was skipped; PID
  may or may not be in target).

**Target cgroup may remain**: Yes, if the PID is still in the target or if
cleanup fails.

**Manual recovery required**: Only if rollback failed and the PID is still
alive and stranded in the target cgroup. For a disposable `sleep 3`, the PID
will exit on its own within seconds. For the first experiment, this is
acceptable.

**What must never be claimed**: "PID was successfully restored." (if
verification was skipped). "Limiter was attached." "Bandwidth limiting is
active." "PID location verified."

---

### Scenario F4: Failure After Target Verification but Before Rollback (Step 6→8 boundary)

**Trigger**: Target verification succeeds (step 6), record succeeds (step 7),
but rollback cannot proceed.

**Sub-scenarios**:
- **F4a**: Verification succeeds, PID exits between verification and rollback.
  The rollback write may succeed for a zombie or may fail with ESRCH.
- **F4b**: Verification succeeds, but the original cgroup directory
  disappears between step 6 and step 8.

**Expected rollback behavior**: Rollback MUST be attempted (step 8). Write
the PID string to the original cgroup's `cgroup.procs`. If the original
cgroup no longer exists, report loudly and do NOT attempt to create it or
guess an alternative restore target.

**Expected cleanup behavior**: After rollback attempt, attempt to remove the
target cgroup if empty. If the target is non-empty (PID may still be there),
leave it in place.

**Expected output honesty**:
- "PID <pid> verified in target cgroup."
- "Rollback attempted: <success/failure>."
- If rollback write fails (original gone): "ROLLBACK FAILURE: Original cgroup
  <path> no longer exists. Do not attempt alternative restore."
- PID location: "rollback attempted — status: <restored/failed/unknown>."

**Target cgroup may remain**: Yes.

**Manual recovery required**: Only if rollback failed and PID is still alive.
For `sleep 3`, the PID exits on its own.

**What must never be claimed**: "PID was successfully restored." (if rollback
failed). "Alternative restore target used." "Original cgroup was created."

---

### Scenario F5: Failure During Rollback Write (Step 8)

**Trigger**: The rollback `cgroup.procs` write to the original cgroup fails.

**Sub-scenarios**:
- **F5a**: `cgroup.procs` write returns EACCES (permission denied on the
  original cgroup).
- **F5b**: `cgroup.procs` write returns ENOENT (original cgroup was removed
  between step 6 and step 8).
- **F5c**: `cgroup.procs` write returns EBUSY (kernel or cgroup subsystem
  locked).
- **F5d**: `cgroup.procs` write returns ESRCH (PID no longer exists — zombie
  already reaped).

**Expected rollback behavior**: The rollback write has already failed. No
further rollback attempts are made. The single attempt was the maximum.

**Expected cleanup behavior**: Attempt to remove the target cgroup if empty
and operation-owned. The PID may or may not be in the target at this point.
If the target is non-empty, leave it in place.

**Expected output honesty**:
- "ROLLBACK FAILURE: cgroup.procs write to <original_cgroup> failed:
  <errno>/<error message>."
- "PID <pid> location: unverified. May be in target cgroup
  <target_path>."
- "No further rollback attempts. Single attempt was the maximum."
- "Target cleanup: performed/skipped/leftover."
- PID location: "unknown" (rollback failed; PID may be in target or may have
  exited).

**Target cgroup may remain**: Yes, if non-empty or cleanup fails.

**Manual recovery required**: For `sleep 3`, no — the PID exits on its own.
If the PID were a longer-lived process (forbidden in the first experiment),
manual `echo <pid> > /proc/<pid>/cgroup_path/../original/cgroup.procs` might
be needed, but this is outside Zelynic's scope.

**What must never be claimed**: "PID was restored." "Rollback succeeded."
"Alternative restore target used." "Retry will be attempted."

---

### Scenario F6: Failure After Rollback Write but Before Rollback Verification (Step 8→9 boundary)

**Trigger**: Rollback write succeeds (step 8) but verification cannot proceed.

**Sub-scenarios**:
- **F6a**: Rollback write succeeds, PID exits before verification read.
- **F6b**: Rollback write succeeds, original cgroup `cgroup.procs` read fails
  with EACCES.

**Expected rollback behavior**: Rollback write already completed. No further
rollback action is needed. The PID was written to the original cgroup, but we
cannot confirm it arrived.

**Expected cleanup behavior**: Attempt to remove target cgroup if empty and
operation-owned.

**Expected output honesty**:
- "PID <pid> rollback write completed."
- "Rollback verification skipped: <reason>."
- "PID <pid> location: rollback write completed but not verified."
- PID location: "rollback unverified" (write succeeded but verification was
  skipped; PID is likely restored but not confirmed).

**Target cgroup may remain**: Yes, if cleanup fails.

**Manual recovery required**: No. The rollback write succeeded. Even if
verification was skipped, the PID was written back to the original cgroup.
The most likely reason for verification failure is that the PID exited, in
which case the question is moot.

**What must never be claimed**: "PID was successfully restored and verified."
"PID location verified." "Verification confirmed restoration."

---

### Scenario F7: Failure During Target Cleanup (Step 10)

**Trigger**: Target cgroup removal fails.

**Sub-scenarios**:
- **F7a**: `fs::remove_dir` returns EBUSY (cgroup subsystem has internal
  references).
- **F7b**: `fs::remove_dir` returns ENOTEMPTY (unexpected PID in target
  `cgroup.procs`).
- **F7c**: `fs::remove_dir` returns EACCES (permission issue on the target
  directory).

**Expected rollback behavior**: Rollback is already complete by this point.
The PID has been restored. Cleanup failure does not trigger any re-rollback.

**Expected cleanup behavior**: The cleanup attempt itself has failed. Leave
the target cgroup in place and report.

**Expected output honesty**:
- "Target cleanup failed: <reason>."
- "Leftover target cgroup: <path>."
- "PID <pid> restoration: already completed."
- PID location: "restored" (cleanup failure does not affect PID location).

**Target cgroup may remain**: Yes — this is the primary scenario where the
target remains. An empty directory under `/sys/fs/cgroup/zelynic/` is safe.

**Manual recovery required**: No. The leftover is an empty directory. Manual
`rmdir /sys/fs/cgroup/zelynic/target_<name>` is safe if desired. The parent
namespace directory `/sys/fs/cgroup/zelynic` is intentionally left in place
across all operations.

**What must never be claimed**: "Cleanup succeeded." "Target was removed."
"All operation-owned state was cleaned up."

---

### Scenario F8: Stale/Dead PID During Transaction

**Trigger**: The PID exits (or becomes a zombie) at any point during the
transaction.

**Sub-scenarios**:
- **F8a**: PID exits between gate re-evaluation (step 1) and PID liveness
  recheck (step 2). Caught by liveness recheck. Operation aborts at step 2.
- **F8b**: PID exits between liveness recheck (step 2) and target preparation
  (step 4). Target may be created but no PID write occurs. Abort before step 5.
- **F8c**: PID exits between target preparation (step 4) and move write
  (step 5). Target exists but no PID write occurs. Abort before step 5.
- **F8d**: PID exits between move write (step 5) and target verification
  (step 6). Move may have succeeded. Rollback attempt (step 8) may fail
  because the PID is already reaped. See scenario F3a.
- **F8e**: PID exits between verification (step 6) and rollback write
  (step 8). Rollback attempt may fail. See scenario F4a.
- **F8f**: PID exits between rollback write (step 8) and cleanup (step 10).
  Rollback already completed. Cleanup proceeds normally.

**Expected rollback behavior**: Depends on when the PID exits:
- Before step 5: No rollback needed. No PID was moved.
- After step 5: Rollback MUST be attempted. The write to `cgroup.procs` may
  succeed for a zombie PID (kernel behavior varies). If it fails with ESRCH,
  report the rollback failure.

**Expected cleanup behavior**: Standard cleanup — remove target if empty and
operation-owned.

**Expected output honesty**: For all sub-scenarios:
- "PID <pid> liveness: <alive at step N / dead at step N>."
- PID location: "not moved" (before step 5), "unknown" (after step 5 without
  verification), or "restored/unverified" (after rollback attempt).

**Target cgroup may remain**: Only if cleanup fails or target is non-empty.

**Manual recovery required**: No. For `sleep 3`, the PID exits on its own.

**What must never be claimed**: "PID was moved" (if it was dead before step 5).
"PID was restored" (if rollback write failed). "PID is alive" (if it is not).

---

### Scenario F9: Original Cgroup Disappears During Transaction

**Trigger**: The original cgroup directory is removed or becomes inaccessible
between capture and rollback.

**Sub-scenarios**:
- **F9a**: Original cgroup disappears between capture (during probe) and step 3
  (original cgroup validation). Caught by validation. Operation aborts at
  step 3.
- **F9b**: Original cgroup disappears between step 3 (validation) and step 8
  (rollback write). This is the dangerous case — the PID has been moved but
  the rollback target no longer exists.

**Expected rollback behavior**:
- F9a: No rollback needed. No PID was moved. Abort at step 3.
- F9b: Rollback attempt MUST still be made. The `cgroup.procs` write will
  fail with ENOENT. Report loudly: "ROLLBACK FAILURE: Original cgroup
  <path> no longer exists."

**Expected cleanup behavior**: Standard cleanup — remove target if empty and
operation-owned.

**Expected output honesty**:
- F9a: "Original cgroup <path> no longer valid or accessible."
- F9b: "ROLLBACK FAILURE: Original cgroup <path> no longer exists at
  rollback time. No alternative restore target will be used."
- PID location (F9b): "unknown" (PID was moved, rollback target gone).

**Target cgroup may remain**: Yes.

**Manual recovery required**: For F9b with a live PID, the PID is stranded in
the Zelynic target cgroup. For `sleep 3`, this resolves itself. For any
other process, manual intervention with `echo <pid> > /sys/fs/cgroup/.../
cgroup.procs` to any valid cgroup would be needed. This is why only
disposable `sleep` processes are used in the first experiment.

**What must never be claimed**: "Original cgroup was recreated." "Alternative
restore path was used." "PID was safely restored." "Rollback succeeded."

---

### Scenario F10: Target Cgroup Becomes Non-Empty Unexpectedly

**Trigger**: The target cgroup `cgroup.procs` contains PIDs that were not
written by this operation.

**Sub-scenarios**:
- **F10a**: Target is non-empty at verification time (step 6). An unexpected
  PID appeared in the target after our write.
- **F10b**: Target is non-empty at cleanup time (step 10). A PID appeared in
  the target between rollback and cleanup.

**Expected rollback behavior**: Rollback proceeds regardless of unexpected
PIDs in the target. The rollback writes our PID to the original cgroup; it
does not read or modify the target's `cgroup.procs`.

**Expected cleanup behavior**: Target MUST NOT be removed if non-empty. This
is a hard safety rule. The unexpected PID could belong to another process
or system component. Removing a non-empty cgroup is forbidden.

**Expected output honesty**:
- "Unexpected PID(s) in target cgroup: <pids>."
- "Target cleanup skipped: target cgroup is not empty."
- "Leftover target cgroup: <path>."
- PID location: depends on rollback outcome (separate from target contents).

**Target cgroup may remain**: Yes — explicitly and intentionally left because
it is non-empty.

**Manual recovery required**: Investigation of why unexpected PIDs appeared
in the target. This should not happen with a freshly created, empty target
cgroup. If it does, it indicates a kernel or system-level issue outside
Zelynic's control.

**What must never be claimed**: "Target was cleaned up." "All PIDs were
removed from target." "Target was force-removed." "Cleanup succeeded."

---

### Scenario F11: Permission Denied on cgroup.procs Write

**Trigger**: The `cgroup.procs` write returns EACCES.

**Sub-scenarios**:
- **F11a**: Move write (step 5) returns EACCES. The PID was not moved.
- **F11b**: Rollback write (step 8) returns EACCES. The PID was moved but
  cannot be restored.

**Expected rollback behavior**:
- F11a: No rollback needed. The PID was not moved. Report the write failure.
- F11b: Rollback write failed. Report loudly. Do NOT retry. The PID may be
  stranded in the target.

**Expected cleanup behavior**: Standard cleanup — remove target if empty and
operation-owned.

**Expected output honesty**:
- "cgroup.procs write to <path> failed: permission denied (EACCES)."
- F11b: "ROLLBACK FAILURE: cgroup.procs write to <original_cgroup> failed:
  permission denied (EACCES). PID <pid> may be stranded in target cgroup
  <target_path>."
- PID location: "not moved" (F11a), "unknown — rollback failed" (F11b).

**Target cgroup may remain**: Yes, especially in F11b if PID is still in the
target.

**Manual recovery required**: For F11b with a live PID, manual
`echo <pid> > /original/cgroup.procs` may be needed. For `sleep 3`, the PID
exits on its own.

**What must never be claimed**: "Write succeeded." "PID was moved." (F11a).
"PID was restored." "Rollback succeeded." (F11b).

---

### Scenario F12: Unexpected EBUSY / ENOENT / EACCES Behavior

**Trigger**: Kernel or cgroup subsystem returns an unexpected errno that is
not covered by the specific scenarios above.

**Sub-scenarios**:
- **F12a**: Move write (step 5) returns EBUSY. The cgroup subsystem may be
  locked by another process or internal kernel state.
- **F12b**: Move write (step 5) returns ENOENT. The target cgroup directory
  disappeared between creation and write.
- **F12c**: Rollback write (step 8) returns EBUSY.
- **F12d**: Rollback write (step 8) returns ENOENT.
- **F12e**: Target mkdir (step 4) returns ENOSPC. No space on the cgroup
  filesystem.
- **F12f**: Any read operation (steps 2, 3, 6, 9) returns an unexpected
  errno.

**Expected rollback behavior**:
- For move write failures (F12a, F12b): No rollback needed if PID was not
  moved. If the write partially succeeded (kernel wrote but returned error),
  attempt rollback to be safe — the PID may be in the target.
- For rollback write failures (F12c, F12d): Report loudly. Do NOT retry.
- For read failures (F12f): Treat as verification failure. Do NOT abort the
  transaction if the read failure occurs after a successful move (step 6 or
  9). Proceed to the next step or rollback as appropriate.

**Expected cleanup behavior**: Standard cleanup — remove target if empty and
operation-owned.

**Expected output honesty**:
- "Unexpected error at step <N>: <operation> returned <errno> (<message>)."
- "PID location: <not moved / unknown / rollback unverified>."
- For any rollback failure: "ROLLBACK FAILURE: <details>."

**Target cgroup may remain**: Yes.

**Manual recovery required**: Depends on the specific failure and whether the
PID is stranded.

**What must never be claimed**: "Operation succeeded." "All steps completed
normally." "No errors occurred."

## Universal Failure Rules

The following rules apply to ALL failure scenarios. No exceptions.

### Rule 1: No False Rollback Claims

If no PID move was performed (failure before step 5), the output MUST NOT
claim that rollback was performed. The specific phrases "Rollback was
performed.", "PID was restored.", or "PID was rolled back." are FORBIDDEN
when no `cgroup.procs` write occurred. The output must instead state "No
PID was moved." and "No cgroup.procs write was performed."

### Rule 2: Rollback Attempted Once If Move May Have Occurred

If step 5 (move write) was attempted, rollback (step 8) MUST be attempted
exactly once, regardless of whether the move write returned success or
failure. Even if the move write returned an error, the kernel may have
partially completed the write. A single rollback attempt is the maximum.

### Rule 3: Rollback Failure Reported Loudly

If rollback fails, the output MUST contain the exact phrase "ROLLBACK
FAILURE" in uppercase. The failure message must include the PID, the target
original cgroup path, and the errno or error description. The output must
explicitly state that no further rollback attempts will be made.

### Rule 4: No Limiter Attach After Partial Move

Under no circumstances may a limiter be attached after a partial or failed
PID move. The phrases "Limiter attached.", "Bandwidth limiting active.", or
"Enforcement applied." are FORBIDDEN in all failure outputs. The canonical
deny lines from phase 3d must still be present: "No limiter attach was
performed." and "Bandwidth limiting is not active from this command yet."

### Rule 5: No nftables/tc/Zelynic State Write in Failure Path

Failure handling MUST NOT write nftables rules, tc configuration, or Zelynic
persistent state. The only filesystem operations permitted in failure paths
are: reading `cgroup.procs`, writing `cgroup.procs` (move and rollback only),
creating/removing directories under `/sys/fs/cgroup/zelynic/`, and reading
`/proc/<pid>/cgroup`. No other filesystem writes are permitted.

### Rule 6: No Deletion of Non-Empty Cgroup

A cgroup directory whose `cgroup.procs` file contains any PID must NOT be
removed. A cgroup directory that contains child subdirectories must NOT be
removed. This rule applies in all failure scenarios, including cleanup
failure scenarios.

### Rule 7: No Deletion Outside /sys/fs/cgroup/zelynic/

Cleanup MUST NOT touch any directory outside `/sys/fs/cgroup/zelynic/`. The
original cgroup, the system root cgroup, and all other system cgroups are
never modified or removed by the operation, including in failure paths.

### Rule 8: No Retry Loops for PID Movement

A `cgroup.procs` write (move or rollback) must be attempted at most once.
No retry loops, no exponential backoff, no "try again with different flags."
A single write attempt is the maximum for each direction (move and rollback).

### Rule 9: PID Location Must Be Explicitly Stated

Every failure output MUST include an explicit statement about the PID's
location. The location must be one of exactly these labels:

| Label | Meaning |
|-------|---------|
| `not moved` | PID was never written to any cgroup.procs (failure before step 5). |
| `verified in target` | Move succeeded and verification confirmed PID in target (step 6 success). |
| `verified restored` | Rollback succeeded and verification confirmed PID in original (step 9 success). |
| `rollback unverified` | Rollback write completed but verification was skipped or failed. |
| `unknown` | Move may have occurred but PID location cannot be determined (e.g., PID exited, read failed). |

The output MUST NOT leave PID location ambiguous. If the location is unknown,
it must explicitly say "PID location: unknown" rather than omitting the
information.

## Test Plan

The following test categories are planned for the future implementation
phase (when the failure simulation code is written). Phase 4a does not
implement these tests; it specifies what must be tested.

### Fake Filesystem / Unit Tests for Rollback Decision Model

These tests use a fake filesystem abstraction (in-memory directory tree) to
simulate cgroup operations without touching real `/sys/fs/cgroup`. The
rollback decision model is a pure function that, given the transaction state
and a failure event, determines whether rollback should be attempted, what the
cleanup action should be, and what the output should say.

| Test ID | Scenario | Description |
|---------|----------|-------------|
| T-FS-01 | F1a | Gate failure before mkdir: decision model returns "abort, no rollback, no cleanup." |
| T-FS-02 | F1b | PID dead at step 2: decision model returns "abort, no rollback, no cleanup." |
| T-FS-03 | F1c | Original cgroup invalid: decision model returns "abort, no rollback, no cleanup." |
| T-FS-04 | F2a | PID dead after mkdir: decision model returns "abort, no rollback, cleanup target." |
| T-FS-05 | F3a | Move succeeded, PID dead before verify: decision model returns "attempt rollback, cleanup target." |
| T-FS-06 | F3b | Move succeeded, verify EACCES: decision model returns "attempt rollback, cleanup target." |
| T-FS-07 | F4a | Verified, PID dead before rollback: decision model returns "attempt rollback, cleanup target." |
| T-FS-08 | F4b | Verified, original cgroup gone: decision model returns "attempt rollback (will fail), cleanup target." |
| T-FS-09 | F5a | Rollback EACCES: decision model returns "rollback failed, leave target, report loudly." |
| T-FS-10 | F5b | Rollback ENOENT: decision model returns "rollback failed, leave target, report loudly." |
| T-FS-11 | F5c | Rollback EBUSY: decision model returns "rollback failed, leave target, report loudly." |
| T-FS-12 | F5d | Rollback ESRCH: decision model returns "rollback failed (PID dead), cleanup if empty." |

### What Is NOT Tested in Phase 4a

- No root smoke tests. Root smoke is deferred to the actual PID move
  implementation phase.
- No live PID move. All tests use fake filesystems and fake writers.
- No live `cgroup.procs` write. All writes are simulated.
- No real cgroup hierarchy interaction. All cgroup operations are mocked.
- No multi-PID scenarios. Phase 4a failure simulation is single-PID only.
- No limiter attach scenarios. Limiter attach is forbidden in v2.8.
