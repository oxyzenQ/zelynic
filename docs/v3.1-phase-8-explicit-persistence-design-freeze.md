## 1. Purpose

Phase 8 freezes the future persistence contract **before** any implementation
work begins. Persistence remains fully disabled today — no ledger files are read,
no ledger files are written, and no background process or daemon performs any
filesystem I/O on behalf of the user. This document defines the principles, gates,
and lifecycle that any future persistence implementation must satisfy before a
single byte is written to disk.

The goals are fivefold. First, **explicit user command only**: ledger file
operations must only occur when the user explicitly invokes a command such as
`zelynic ledger save`, never silently in the background. Second, **no hidden
state**: any ledger file location, format, or naming must be documented and
user-visible; there must be no undocumented state directory, no hidden cache,
and no implicit data retention. Third, **gate-tested design**: every persistence
operation must pass a defined set of safety gates before implementation begins,
and those gates must be codified as tests that run in every CI pass. Fourth,
**backward compatibility**: the existing v3.0 usage JSON schema (schema_version 1)
must remain frozen; any future ledger JSON must use a separate schema or report
mode that does not interfere with v3.0 output. Fifth, **progressive activation**:
persistence must be implemented incrementally (design freeze, seam model, read-only
inspect, explicit save) with each phase requiring the previous to be complete.

### Existing Persistence Infrastructure (All Hard-Blocked)

Three modules provide model-only persistence infrastructure:

1. **`src/accounting/ledger_path.rs`** — Pure path planning model. Validates
   ledger file paths structurally (string-based only, no filesystem access).
   Rejects empty paths, absolute filenames, parent traversal, suspicious
   characters, and paths outside the configured namespace. Produces a
   `LedgerPathPlan` with `model_only: true` and `persistence_enabled: false`.

2. **`src/accounting/ledger_persistence.rs`** — Hard-blocked persistence I/O
   contract seam. Every operation (ReadLedger, WriteLedger, AtomicReplace,
   Backup, ValidatePath) returns a `LedgerPersistencePlan` with
   `persistence_status: Blocked(...)` and `blocked_reason` set to a canonical
   hard-block message. No filesystem I/O is performed. All mutation flags
   (`filesystem_read_performed`, `filesystem_write_performed`,
   `directory_create_performed`, `file_create_performed`, `file_remove_performed`,
   `state_mutation_performed`, `persistence_enabled`) are hardcoded to `false`.

3. **`src/accounting/ledger.rs`** — Core ledger data model with full serde
   serialization/deserialization support. `Ledger` and `LedgerEntry` types
   are fully serializable. `serialize_ledger_to_json()` and
   `deserialize_ledger_from_json()` provide round-trip JSON support. The model
   is read-only: all entries have `read_only: true`, `attribution_scope:
   "interface-level only"`, and `enforcement_status: "inactive/not implemented"`.

## 3. Future Persistence Principles

The following principles are frozen as contract for all future persistence work.
No implementation phase may violate these principles.

### 3.1 Explicit User Command Only

Persistence operations must only occur when the user explicitly invokes a CLI
command that is documented, visible in `--help`, and clearly labeled as a
write operation. There must be no automatic persistence triggered by `--sample`,
`--json`, `--delta`, or any other existing or future flag.

### 3.2 No Silent Write

The user must be informed before any file is written. The target path must be
displayed. A `--dry-run` mode must be available that shows what would be
written without actually writing it. No file may be created, overwritten, or
modified without the user's explicit confirmation.

### 3.3 No Auto-Save

There must be no automatic saving of ledger data at program exit, on signal,
on timer, or on any implicit trigger. Saving is always an explicit user action.

### 3.4 No Background Process

There must be no daemon, no background watch loop, no auto-start service, and
no scheduled task. Zelynic is a single-shot CLI tool. Each invocation runs,
produces output, and exits. Persistence is a single explicit write per invocation.

### 3.5 No Network Enforcement

Persistence is purely observational. Ledger files record what was observed.
They do not enable enforcement, blocking, throttling, or quota management.
No persistence data is used to make network traffic decisions.

### 3.6 No Permission/Block/Allow

Ledger files do not store allow/block rules, permission grants, or access
control lists. Persistence is a usage history record, not a firewall configuration.

### 3.7 No Mutation of System State

Reading and writing ledger files must not mutate any system state: no process
killing, no cgroup writes, no nft/tc rule changes, no eBPF program loading,
no PID movement, no network namespace changes.

## 4. Future Ledger File Lifecycle

This section defines the design for the future ledger file lifecycle. All steps
remain not implemented in this phase.

### 4.1 Ledger File Format

The ledger file is a JSON document produced by `serialize_ledger_to_json()` and
consumed by `deserialize_ledger_from_json()`. The format is defined by the
existing `Ledger` and `LedgerEntry` types with `schema_version: 1`.

Key properties:
- **Human-readable**: JSON with pretty-printing.
- **Deterministic**: BTreeMap-based ordering for reproducible serialization.
- **Self-describing**: Contains `schema_version`, `created_at`, `updated_at`,
  `host_id`, and `session_id` for provenance tracking.
- **Read-only invariant**: Every entry has `read_only: true`. A validation
  function (`validate_safety()`) rejects entries without this flag.
- **Append-friendly**: New entries are appended to the `entries` array. The
  file is never overwritten from scratch unless explicitly requested.

### 4.2 Ledger File Location

The ledger file location is determined by the path planning model in
`ledger_path.rs`:
- **Base directory**: User-configurable, defaults to XDG data directory
  (`~/.local/share`).
- **Namespace label**: `zelynic`.
- **Default filename**: `network-ledger-v1.json`.
- **Full default path**: `~/.local/share/zelynic/network-ledger-v1.json`.

Path validation rules (already implemented in `build_ledger_path_plan()`):
- Base directory must not be empty.
- Filename must not be empty.
- Filename must be relative (not absolute).
- Filename must not contain parent traversal (`..`).
- Base path must not contain parent traversal.
- Filename must not contain suspicious characters (only `[a-zA-Z0-9._-]`).
- Full path must be within the configured namespace.

### 4.3 Atomic Write Design

When a ledger file is written, the following atomic write sequence must be
used to prevent corruption:
1. Write new content to a temporary file in the same directory
   (e.g., `network-ledger-v1.json.tmp`).
2. Sync the temporary file to disk (`fsync`).
3. Rename the temporary file to the target filename (`rename`/`rename2`).
4. Sync the parent directory to ensure the directory entry is persisted.

This ensures that a crash or power failure during write never leaves a
corrupted or partially-written ledger file. Either the old file or the new
file exists, never a partial state.

### 4.4 Backup Strategy Design

Before overwriting an existing ledger file, a backup must be created:
- Backup filename: `network-ledger-v1.json.bak` (single backup, overwritten
  on each save).
- Backup is created by copying the existing file before the atomic write.
- If no existing file exists, no backup is created.
- The backup preserves the previous state for manual recovery.

### 4.5 Corruption Handling Design

When reading a ledger file, the following corruption scenarios must be handled:
- **Parse error**: JSON syntax error → clear error message suggesting manual
  inspection or restore from backup.
- **Schema mismatch**: `schema_version` is not recognized → clear error
  message indicating the file was created by a different version.
- **Validation failure**: `validate_safety()` rejects an entry → error message
  identifying the invalid entry and reason.
- **Empty file**: Zero-byte file → treated as empty ledger (no entries).
- **Partial write**: File ends mid-JSON → treated as parse error with
  suggestion to restore from backup.

## 5. Persistence Safety Gates

The following gates must be satisfied before any future persistence
implementation phase begins. Each gate must be documented, tested, and verified
in CI.

### Gate 1: Path Validation

All ledger file paths must pass the structural validation defined in
`build_ledger_path_plan()`. This includes:
- Non-empty base directory and filename.
- Relative filename only (no absolute paths).
- No parent traversal components.
- No suspicious characters.
- Namespace containment.

**Existing**: Already implemented and tested in `ledger_path.rs`.

### Gate 2: Schema Validation

All ledger content must pass `validate_safety()` on both individual entries
and the full ledger. This includes:
- `schema_version` is recognized (currently only version 1).
- All entries have `read_only: true`.
- All entries have `attribution_scope: "interface-level only"`.
- All entries have `enforcement_status: "inactive/not implemented"`.
- `combined_bytes == rx_bytes + tx_bytes` (saturating).

**Existing**: Already implemented and tested in `ledger.rs`.

### Gate 3: Explicit Command

No persistence operation occurs unless the user explicitly invokes a CLI
command that is visible in `--help`. Hidden flags, default behaviors, and
implicit triggers are forbidden.

**Existing**: Already enforced by phase 6 gate tests.

### Gate 4: Explicit Output Path

When writing a ledger file, the output path must be explicitly specified by
the user (via `--output <path>` flag) or derived from a documented default
that is displayed before writing. No path is assumed or hidden.

**Future**: Must be enforced when save command is implemented.

### Gate 5: Dry-Run Mode First

Every write operation must support a `--dry-run` flag that shows what would be
written (path, size, entry count) without actually writing it. Dry-run output
must include the full target path and a clear statement that no file was written.

**Future**: Must be implemented before any save command is activated.

### Gate 6: No Overwrite Without Explicit Flag

Writing to an existing file requires either a `--force` flag or an explicit
confirmation prompt. Without `--force`, the command must refuse to overwrite
and suggest using `--force` or a different filename.

**Future**: Must be enforced when save command is implemented.

### Gate 7: Atomic Temp File Design

All writes must use the atomic write sequence (write to temp, fsync, rename,
fsync directory). No direct overwrite of the target file.

**Future**: Must be implemented in the persistence seam.

### Gate 8: Checksum or Integrity Marker Design

Each ledger file should contain a checksum or integrity marker that can be
verified on read to detect corruption. This could be:
- A SHA-256 hash of the entries array stored as a top-level field.
- A line-by-line hash for incremental verification.
- Or a simpler approach: file size + entry count stored in the ledger itself.

**Future**: Must be designed and tested before save implementation.

### Gate 9: Backup Design

Before any overwrite, a backup of the existing file must be created. The
backup filename must follow a documented convention. If the backup fails, the
write must be aborted.

**Future**: Must be implemented in the persistence seam.

### Gate 10: No Symlink Surprises

The persistence seam must not follow symlinks when writing. The target path
must be verified to be a regular file (or non-existent) before writing. If a
symlink is detected at the target path, the operation must be rejected with a
clear error.

**Future**: Must be enforced when save command is implemented.

### Gate 11: No Parent Traversal

Path validation must reject any path containing `..` components, as already
implemented in `build_ledger_path_plan()`. This prevents directory traversal
attacks where a malicious or mistaken path could write outside the intended
namespace.

**Existing**: Already implemented and tested.

### Gate 12: No Absolute Unsafe Path Unless Explicitly Allowed

Relative filenames are required for ledger files. Absolute paths are only
accepted if the user explicitly provides them via `--output <absolute-path>`,
and even then, the path must be validated for safety (no traversal, no symlink).

**Existing**: Partially implemented (relative-only in path model).
**Future**: Explicit absolute path handling in save command.

### Gate 13: No Hidden State Directory Without Documentation

Any directory created by Zelynic must be documented in `--help` output and in
the design documents. The default `~/.local/share/zelynic/` directory must
be visible to the user via `zelynic ledger save --dry-run` before any file
is actually created.

**Future**: Must be enforced when save command is implemented.

## 6. Future Command Candidates

The following commands are candidates for future persistence phases. All remain
**not implemented** in this phase. They are listed here for design completeness
and to establish the command shape that future phases will implement.

### 6.1 `zelynic ledger inspect`

Inspects the current in-memory ledger state (built from fixture data or,
in future phases, from a persisted ledger file). Produces a human-readable
report with all safety disclaimers.

**Status**: Shape defined in phase 6 (hidden clap variant). Preview content
defined in phase 7 (fixture-only). Live persistence read not implemented.

### 6.2 `zelynic ledger inspect --json`

Same as above but outputs the ledger state as JSON. Uses the existing
`Ledger` serialization format with `schema_version: 1`.

**Status**: Shape defined in phase 6. Preview JSON via report model defined
in phase 7. Live persistence read not implemented.

### 6.3 `zelynic ledger export --json`

Exports the full ledger data as JSON to stdout. Intended for piping to other
tools or saving manually via shell redirection. Does not write to disk itself.

**Status**: Shape defined in phase 6. Not implemented.

### 6.4 `zelynic ledger save --dry-run`

Preview mode for the save command. Shows what would be written (target path,
size, entry count) without actually writing. Includes all safety disclaimers.

**Status**: Not implemented. Requires phase 9 persistence seam.

### 6.5 `zelynic ledger save --output <path>`

Saves the current ledger state to the specified file path. Requires explicit
`--output` flag. Refuses to overwrite existing files without `--force`.
Uses atomic write sequence with backup.

**Status**: Not implemented. Requires phases 9-14.

### 6.6 `zelynic ledger validate <path>`

Validates an existing ledger file without loading it into the running ledger.
Checks JSON syntax, schema version, safety invariants, and corruption.
Reports validation result.

**Status**: Not implemented. Requires phase 9 persistence seam.

## 7. JSON/Schema Policy

### 7.1 Existing v3.0 Usage JSON (Frozen)

The v3.0 usage JSON schema (`schema_version: 1`) produced by
`zelynic usage --sample --json` and `zelynic usage --sample --delta --json`
remains frozen. No field is added, removed, or renamed. The schema is:

```json
{
  "schema_version": 1,
  "command": "usage --sample --json",
  "source_path": "/proc/net/dev",
  "source_label": "live_proc_net_dev",
  "interfaces": [...],
  "totals": {...},
  "honesty": {...},
  "warnings": [...]
}
```

This schema must continue working identically for all existing scripts and
automation. Any change to this schema is a breaking change.

### 7.2 Future Ledger JSON (Separate Schema)

Future ledger JSON output from `zelynic ledger inspect --json` or
`zelynic ledger export --json` uses the existing `Ledger` serialization format
with `schema_version: 1`. This is a separate schema from the usage JSON because
it has a different structure (entries array, host_id, session_id, etc.).

The key discriminator between usage JSON and ledger JSON is the `command` field
or the presence of `entries`/`host_id` keys. No field names collide between
the two schemas.

### 7.3 Old Ledger Schema Compatibility

When future persistence is implemented, the ability to read ledger files created
by older versions must be preserved. The `schema_version` field serves as the
discriminator:
- `schema_version: 1` → current format.
- Unknown schema versions → clear error message indicating the file was
  created by a different version.

Schema migration (if ever needed) must be explicit: a separate
`zelynic ledger migrate` command that converts old format to new format,
with the old file preserved as backup.

## 8. Non-Goals

The following are explicitly out of scope for v3.1 phase 8:

- No runtime code changes.
- No persistence implementation.
- No ledger file read.
- No ledger file write.
- No CLI runtime enablement for any persistence command.
- No live resolver (process scanning, identity resolution).
- No enforcement, blocking, or quota activation.
- No quota guard.
- No eBPF program loading or attachment.
- No nft/tc mutation or cgroup mutation.
- No PID movement between cgroups.
- No daemon/watch/background mode.
- No hidden state directory creation.
- No modification to existing v3.0 usage JSON schema.
- No version bump. Version remains `v3.0.1`.
- No git tag, GitHub release, or crate publish.
- No new dependencies added to `Cargo.toml`.
- No modification to existing accounting module behavior.

## 9. Future Implementation Sequence

The following phased sequence refines the v3.1 roadmap from the phase 1
contract and the phase 5 gate design, with persistence-specific detail:

### Phase 9: Persistence Seam Model Review

**Scope**: Review the existing `ledger_path.rs` and `ledger_persistence.rs`
modules for completeness. Ensure path validation covers all required gates.
Add any missing gate tests. No actual file I/O.

**Deliverable**: Updated gate test count. Any missing path validation rules
codified. Updated design document.

**Safety**: No code execution. Tests only. Design review only.

### Phase 10: Read-Only Inspect from In-Memory Fixture Behind Gated Command

**Scope**: Wire the phase 7 preview module to the existing hidden
`zelynic ledger inspect` clap variant. Replace the dispatch-level rejection
with fixture-driven output. The command remains hidden until readiness.

**Deliverable**: `zelynic ledger inspect` produces fixture-based preview
output (text and JSON). All 13 preview disclaimers present. Gate tests
expanded.

**Safety**: No live reads. No persistence. Fixture data only. Command
remains hidden initially.

### Phase 11: Path Validation Hardening

**Scope**: Implement additional path safety checks beyond the existing
structural validation. Add symlink detection, canonicalization tests,
and permission checks (read-only) in a test-only module.

**Deliverable**: Hardened `build_ledger_path_plan()` with additional safety
checks. Tests covering edge cases.

**Safety**: No filesystem writes. Read-only permission checks in tests
only (against temp directories).

### Phase 12: Explicit File Read Design/Implementation Gate

**Scope**: Design and implement the read-only file read seam for ledger
files. The seam reads a JSON file, validates schema and safety invariants,
and produces a `Ledger` in memory. No writes.

**Deliverable**: `read_ledger_file(path)` function behind a feature gate
or explicit CLI flag. Tests use temp directories with known content.

**Safety**: Read-only. No writes. No enforcement. Validates before loading.

### Phase 13: Export-Only JSON Command

**Scope**: Activate `zelynic ledger export --json` that writes the current
ledger state (built from fixture or loaded from file) to stdout as JSON.
No file write to disk — the user redirects stdout if desired.

**Deliverable**: Working `zelynic ledger export --json` command. JSON output
matches `serialize_ledger_to_json()` format.

**Safety**: stdout only. No file writes. No persistence.

### Phase 14+: Save/Write Only After Separate Freeze

**Scope**: Implement `zelynic ledger save --output <path>` and
`zelynic ledger save --dry-run`. Requires a separate design freeze phase
that specifies the exact atomic write sequence, backup strategy, and
confirmation prompt.

**Deliverable**: Working save command with atomic writes, backup creation,
`--dry-run` mode, `--force` for overwrite.

**Safety**: Explicit command. Dry-run first. Atomic write. Backup before
overwrite. No silent writes. No daemon.

### Future Beyond Phase 14

- **Identity resolver design and implementation**: Live process scanning for
  `zelynic usage --target`. Requires its own design freeze phase.
- **Extended usage flags**: `--session`, `--since-boot`, `--interface` activated
  only after their prerequisite modules are complete.
- **`zelynic usage --target <target>`**: Last to be implemented, as it requires
  persistence + identity resolution + report model.
