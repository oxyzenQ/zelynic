# v3.0 Phase 13: Delta JSON Output Contract Design

## Purpose

Phase 13 designs the machine-readable JSON schema for the future
`zelynic usage --sample --delta --json` command. This phase is **design-only**: it
produces this document and updates related documentation but does NOT implement
`--delta --json`, does NOT change CLI behavior, does NOT change runtime code,
does NOT add dependencies, and does NOT add live reads beyond existing behavior.

The design is explicit enough that the next implementation phase can add delta
JSON output safely without schema drift.

## Current State

At phase 12/12b, the command `zelynic usage --sample --delta --json` is
intentionally rejected before any live read. Phase 13 only designs the future
JSON contract so that the implementation phase has a frozen specification to
follow.

The current behavior (frozen at phase 12b):

| Command | Behavior |
|---------|----------|
| `zelynic usage --sample --delta` | Finite two-sample text output |
| `zelynic usage --sample --delta --json` | Rejected before live read |
| `zelynic usage --delta` | Rejected by clap (requires `--sample`) |

## Hard Safety Boundary

Phase 13 enforces the same hard safety boundary as all v3.0 phases. The
following remain hard-blocked:

| Prohibited Action | Status |
|-------------------|--------|
| `--delta --json` implementation | Hard-blocked in phase 13 (design only) |
| Configurable interval | Hard-blocked in v3.0 |
| Loop/watch mode | Hard-blocked in v3.0 |
| Interface filtering | Hard-blocked in v3.0 |
| Persistence / ledger read/write | Hard-blocked in v3.0 |
| eBPF program loading/attachment | Hard-blocked in v3.0 |
| Quota enforcement | Hard-blocked in v3.0 |
| Network traffic blocking | Hard-blocked in v3.0 |
| Limiter attach | Hard-blocked in v3.0 |
| nftables/tc rule mutation | Hard-blocked in v3.0 |
| Zelynic runtime state mutation | Hard-blocked in v3.0 |
| PID movement | Hard-blocked in v3.0 |
| `cgroup.procs` write | Hard-blocked in v3.0 |
| `/sys/fs/cgroup` access | Hard-blocked in v3.0 |
| Sysfs read | Hard-blocked in v3.0 |
| Arbitrary file path acceptance | Hard-blocked in v3.0 |
| Filesystem writes | Hard-blocked in v3.0 |
| Directory/file creation or removal | Hard-blocked in v3.0 |

The only allowed live filesystem read remains `/proc/net/dev`.

## Command

```
zelynic usage --sample --delta --json
```

This command will produce machine-readable JSON output containing the delta
between two samples of `/proc/net/dev`. It extends the existing snapshot JSON
contract (phase 6/7/8) with delta-specific fields.

## Proposed Top-Level JSON Shape

```json
{
  "schema_version": 1,
  "command": "usage --sample --delta --json",
  "source_path": "/proc/net/dev",
  "source_label": "live_proc_net_dev",
  "sample_mode": "delta",
  "sample_count": 2,
  "delta_wait_ms": 1000,
  "read_count": 2,
  "start_sample": { ... },
  "end_sample": { ... },
  "interfaces": [ ... ],
  "totals": { ... },
  "warnings": [ ... ],
  "honesty": { ... }
}
```

### Top-Level Field Descriptions

| Field | Type | Description |
|-------|------|-------------|
| `schema_version` | integer | Schema version for forward compatibility. Currently `1`. |
| `command` | string | The command that produced this output. Always `"usage --sample --delta --json"`. |
| `source_path` | string | Data source file path. Always `"/proc/net/dev"`. |
| `source_label` | string | Data source label. Always `"live_proc_net_dev"`. |
| `sample_mode` | string | Sampling mode. Always `"delta"` for delta output. Distinguishes from snapshot mode (`"single"`). |
| `sample_count` | integer | Number of samples taken. Always `2` for delta mode. |
| `delta_wait_ms` | integer | Wait duration between samples in milliseconds. Fixed at `1000` (1 second) in current implementation. |
| `read_count` | integer | Number of successful reads. `2` on success, `1` if second read failed, `0` if first read failed. |
| `start_sample` | object or null | Start snapshot data. Present only on success or partial success (first read succeeded). `null` if first read failed. |
| `end_sample` | object or null | End snapshot data. Present only on full success (both reads succeeded). `null` if second read failed. |
| `interfaces` | array | Per-interface delta data. Empty array on error. |
| `totals` | object | Aggregate delta totals. Zero totals on error. |
| `warnings` | array | Warning strings about counter resets, incomplete data, and general caveats. |
| `honesty` | object | Honesty flags. Present in every JSON output (success and error). |

## Timestamp Policy

**Timestamps are intentionally omitted from the `start_sample` and `end_sample`
objects.** The delta JSON does not include wall-clock timestamps for either the
start or end sample.

Rationale:

- The existing snapshot JSON contract (phase 6) establishes that `sampled_at`
  is only populated when the caller explicitly provides a timestamp and is never
  silently generated.
- The delta sampling implementation (phase 12) does not capture timestamps during
  the two-sample process.
- Adding wall-clock timestamps would require calling `SystemTime::now()` or
  `Instant::now()` in the production output path, which the phase 6 contract
  explicitly forbids ("no silent wall-clock timestamp generation").
- The `delta_wait_ms` field already communicates the time gap between samples
  in a deterministic, caller-controlled way.

If timestamps are needed in a future phase, they must be explicitly designed
with a caller-provided mechanism (e.g., `--sampled-at-start` and
`--sampled-at-end` flags). That design is out of scope for phase 13.

## Start Sample / End Sample Objects

### Start Sample Object

```json
{
  "start_sample": {
    "status": "success",
    "interface_count": 2,
    "total_rx_bytes": 1000000000,
    "total_tx_bytes": 500000000,
    "total_combined_bytes": 1500000000
  }
}
```

### End Sample Object

```json
{
  "end_sample": {
    "status": "success",
    "interface_count": 2,
    "total_rx_bytes": 1064497408,
    "total_tx_bytes": 523789824,
    "total_combined_bytes": 1588287232
  }
}
```

### Sample Object Fields

| Field | Type | Description |
|-------|------|-------------|
| `status` | string | `"success"` if the read and parse succeeded, `"error"` if the read or parse failed. |
| `interface_count` | integer | Number of interfaces in this sample. `0` on error. |
| `total_rx_bytes` | integer | Sum of all interface RX byte counters in this sample. `0` on error. |
| `total_tx_bytes` | integer | Sum of all interface TX byte counters in this sample. `0` on error. |
| `total_combined_bytes` | integer | Sum of all interface combined byte counters in this sample. `0` on error. |

**Important**: The sample objects contain summary-level data only (counts and
totals). They do NOT include per-interface counter values. The per-interface
counter values for both start and end snapshots are embedded in the delta
interface objects (see next section), which include `start_rx_bytes`,
`end_rx_bytes`, etc. This avoids duplicating the full per-interface data while
still providing the start/end summary context.

## Per-Interface Delta Object

```json
{
  "name": "wlan0",
  "loopback": false,
  "start_rx_bytes": 500000000,
  "start_tx_bytes": 250000000,
  "end_rx_bytes": 530000000,
  "end_tx_bytes": 265000000,
  "delta_rx_bytes": 30000000,
  "delta_tx_bytes": 15000000,
  "delta_combined_bytes": 45000000,
  "start_rx_packets": 400000,
  "start_tx_packets": 200000,
  "end_rx_packets": 421000,
  "end_tx_packets": 212000,
  "delta_rx_packets": 21000,
  "delta_tx_packets": 12000,
  "counter_reset_detected": false,
  "interface_added": false,
  "interface_removed": false
}
```

### Per-Interface Delta Field Descriptions

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Interface name (e.g., `wlan0`, `eth0`, `lo`). |
| `loopback` | boolean | `true` if this is the loopback interface (`lo`). |
| `start_rx_bytes` | integer | RX byte counter from the start sample for this interface. |
| `start_tx_bytes` | integer | TX byte counter from the start sample for this interface. |
| `end_rx_bytes` | integer | RX byte counter from the end sample for this interface. |
| `end_tx_bytes` | integer | TX byte counter from the end sample for this interface. |
| `delta_rx_bytes` | integer | RX byte delta (`end_rx_bytes - start_rx_bytes`, or `0` if counter reset detected). |
| `delta_tx_bytes` | integer | TX byte delta (`end_tx_bytes - start_tx_bytes`, or `0` if counter reset detected). |
| `delta_combined_bytes` | integer | Combined delta (`delta_rx_bytes + delta_tx_bytes`, saturating). |
| `start_rx_packets` | integer | RX packet counter from the start sample. |
| `start_tx_packets` | integer | TX packet counter from the start sample. |
| `end_rx_packets` | integer | RX packet counter from the end sample. |
| `end_tx_packets` | integer | TX packet counter from the end sample. |
| `delta_rx_packets` | integer | RX packet delta (`end_rx_packets - start_rx_packets`, or `0` if reset). |
| `delta_tx_packets` | integer | TX packet delta (`end_tx_packets - start_tx_packets`, or `0` if reset). |
| `counter_reset_detected` | boolean | `true` if any counter (bytes or packets, RX or TX) decreased between start and end, indicating a counter reset. When `true`, the corresponding delta fields are `0` and the exact usage for that counter is unknown. |
| `interface_added` | boolean | `true` if this interface was not present in the start sample but is present in the end sample. The start counter fields will be `0` and the delta equals the end counter value. |
| `interface_removed` | boolean | `true` if this interface was present in the start sample but is not present in the end sample. The end counter fields will be `0` and the delta is `0` (exact usage unknown). |

## Honesty Flags

The honesty object contains boolean flags that honestly describe what the
command did and did not do. These flags are always present in every JSON
output (success and error).

```json
"honesty": {
  "interface_level_only": true,
  "per_app_attribution": false,
  "quota_enforcement_active": false,
  "network_blocking_active": false,
  "limiter_attach_performed": false,
  "nft_tc_state_mutation_performed": false,
  "ledger_persistence_performed": false,
  "ebpf_used": false,
  "cgroup_mutation_performed": false,
  "pid_movement_performed": false,
  "filesystem_write_performed": false,
  "state_mutation_performed": false,
  "delta_json_output": true,
  "live_read_performed": true,
  "read_count": 2,
  "single_shot": true,
  "loop_watch_mode": false,
  "configurable_interval": false,
  "interface_filtering": false,
  "arbitrary_path_read": false
}
```

### Honesty Flag Descriptions

| Flag | Type | Value | Description |
|------|------|-------|-------------|
| `interface_level_only` | boolean | `true` | Delta data is interface-level aggregate counters, not per-app or per-process. |
| `per_app_attribution` | boolean | `false` | Per-process/per-app traffic attribution is not available from `/proc/net/dev`. |
| `quota_enforcement_active` | boolean | `false` | No quota enforcement is implemented or active. |
| `network_blocking_active` | boolean | `false` | No network traffic is blocked, dropped, or rejected. |
| `limiter_attach_performed` | boolean | `false` | No rate limiter was attached or configured. |
| `nft_tc_state_mutation_performed` | boolean | `false` | No nftables/tc/Zelynic state mutation was performed. |
| `ledger_persistence_performed` | boolean | `false` | No ledger file was read or written. |
| `ebpf_used` | boolean | `false` | No eBPF programs were loaded or attached. |
| `cgroup_mutation_performed` | boolean | `false` | No cgroup creation, modification, or `cgroup.procs` write was performed. |
| `pid_movement_performed` | boolean | `false` | No PIDs were moved between cgroups. |
| `filesystem_write_performed` | boolean | `false` | No filesystem write was performed. |
| `state_mutation_performed` | boolean | `false` | No system state mutation of any kind was performed. |
| `delta_json_output` | boolean | `true` | This is a delta JSON output (distinguishes from snapshot JSON). |
| `live_read_performed` | boolean | `true` or `false` | Whether at least one live `/proc/net/dev` read was performed. `false` if the very first read failed. |
| `read_count` | integer | `0`, `1`, or `2` | Number of reads that succeeded. `2` on full success, `1` if second read failed, `0` if first read failed. |
| `single_shot` | boolean | `true` | Command exits after producing output. No loop or repeated sampling. |
| `loop_watch_mode` | boolean | `false` | No continuous monitoring or watch mode. |
| `configurable_interval` | boolean | `false` | The interval between samples is not user-configurable in v3.0. |
| `interface_filtering` | boolean | `false` | No interface filtering is applied. All interfaces are included. |
| `arbitrary_path_read` | boolean | `false` | No arbitrary file path was read. Source is hardcoded to `/proc/net/dev`. |

### Honesty Flag Constants

In v3.0, all boolean honesty flags are constant. The following are always `false`
and must never be `true`:

- `per_app_attribution`
- `quota_enforcement_active`
- `network_blocking_active`
- `limiter_attach_performed`
- `nft_tc_state_mutation_performed`
- `ledger_persistence_performed`
- `ebpf_used`
- `cgroup_mutation_performed`
- `pid_movement_performed`
- `filesystem_write_performed`
- `state_mutation_performed`
- `loop_watch_mode`
- `configurable_interval`
- `interface_filtering`
- `arbitrary_path_read`

The following are always `true` in delta JSON success output:

- `interface_level_only`
- `delta_json_output`
- `single_shot`

The following are dynamic:

- `live_read_performed`: `true` if at least one read succeeded, `false` if the
  first read failed entirely.
- `read_count`: `0`, `1`, or `2` depending on how many reads succeeded.

## Error JSON Shape

### Error Top-Level Shape

Error JSON uses the same top-level structure as success JSON, with the `error`
object populated and data fields zeroed or empty:

```json
{
  "schema_version": 1,
  "command": "usage --sample --delta --json",
  "source_path": "/proc/net/dev",
  "source_label": "live_proc_net_dev",
  "sample_mode": "delta",
  "sample_count": 0,
  "delta_wait_ms": 1000,
  "read_count": 0,
  "start_sample": null,
  "end_sample": null,
  "interfaces": [],
  "totals": {
    "total_delta_rx_bytes": 0,
    "total_delta_tx_bytes": 0,
    "total_delta_combined_bytes": 0,
    "interface_count": 0,
    "loopback_interface_count": 0,
    "non_loopback_interface_count": 0,
    "counter_reset_count": 0,
    "interface_added_count": 0,
    "interface_removed_count": 0
  },
  "warnings": [ ... same 12 default warnings ... ],
  "honesty": {
    "interface_level_only": true,
    "per_app_attribution": false,
    ... all 19 flags preserved ...
    "live_read_performed": false,
    "read_count": 0
  },
  "error": {
    "type": "read_error",
    "message": "read error: failed to read /proc/net/dev: ..."
  }
}
```

### Error Object

| Field | Type | Description |
|-------|------|-------------|
| `error` | object or null | Present on error, `null` on success. |
| `error.type` | string | Canonical error type: `"read_error"`, `"parse_error"`, or `"unsupported_flag_error"`. |
| `error.message` | string | Human-readable error message. |

### Error Scenarios and Honesty Flag Behavior

| Error Scenario | Error Type | `read_count` | `live_read_performed` | `start_sample` | `end_sample` | `sample_count` |
|----------------|-------------|--------------|----------------------|----------------|--------------|----------------|
| First read fails (I/O error) | `read_error` | `0` | `false` | `null` | `null` | `0` |
| First sample parse fails | `parse_error` | `0` | `false` | `null` | `null` | `0` |
| Second read fails (I/O error) | `read_error` | `1` | `true` | present (summary) | `null` | `1` |
| Second sample parse fails | `parse_error` | `1` | `true` | present (summary) | `null` | `1` |

**Precision requirements for error payloads:**

- If the first read fails: `live_read_performed` MUST be `false`, `read_count`
  MUST be `0`, and `sample_count` MUST be `0`. The error payload must NOT
  claim that any read was successful.
- If the second read fails: `live_read_performed` MUST be `true`, `read_count`
  MUST be `1`, and `sample_count` MUST be `1`. The error payload must NOT
  claim that two reads were completed.
- Counter reset warnings are never fatal. When counter resets are detected, the
  output is a **success** payload (not error) with `counter_reset_detected:
  true` on affected interfaces and additional entries in the `warnings` array.
  The delta for the affected counter is `0` and the consumer is warned that the
  exact usage is unknown.
- Error payloads MUST preserve all 19 honesty flags. The honesty object must be
  identical in structure between success and error outputs; only the dynamic
  flags (`live_read_performed`, `read_count`) may differ.
- Error payloads MUST NOT claim partial success unless explicitly represented
  by the dynamic flags. For example, when the second read fails, the payload
  must not set `sample_count: 2` or `read_count: 2`.

## Relationship to Existing JSON Contract

The delta JSON schema extends the existing snapshot JSON contract (phases 6-8)
with delta-specific fields. Key differences:

| Aspect | Snapshot JSON (`--sample --json`) | Delta JSON (`--sample --delta --json`) |
|--------|----------------------------------|---------------------------------------|
| `command` | `"usage --sample --json"` | `"usage --sample --delta --json"` |
| `sample_mode` | Not present (or `"single"`) | `"delta"` |
| `sample_count` | `1` | `2` |
| `delta_wait_ms` | Not present | `1000` |
| `read_count` | `0` or `1` | `0`, `1`, or `2` |
| `start_sample` | Not present | Summary object or null |
| `end_sample` | Not present | Summary object or null |
| `interfaces[]` | Snapshot counters (rx_bytes, tx_bytes, combined_bytes) | Delta counters (start/end/delta bytes/packets, reset flags) |
| `totals` | Snapshot totals | Delta totals with reset/added/removed counts |
| `honesty.delta_json_output` | Not present or `false` | `true` |
| `honesty.read_count` | `0` or `1` | `0`, `1`, or `2` |

The error types are reused from the existing contract: `read_error`,
`parse_error`, `unsupported_flag_error`. No new error types are needed for
delta mode. The error handling granularity is extended to distinguish first-read
failure from second-read failure through the `read_count` and `sample_count`
fields.

## Phase 15 Implementation Note (Completed)

Phase 15 wires the frozen phase 14 pure delta JSON model to the CLI. The command
`zelynic usage --sample --delta --json` now produces valid JSON output using the exact
schema defined in this contract document. No schema changes were made. The implementation
uses `build_delta_json_success`, `build_delta_json_first_read_error`, and
`build_delta_json_second_read_error` from the frozen phase 14 model. Error handling
distinguishes read errors from parse errors, and second-read errors correctly report
`read_count = 1` with the first sample summary present.
