## 1. Purpose

Phase 7 creates a pure fixture-based preview path that proves the future report
pipeline shape. No CLI wiring, no live resolver, no filesystem I/O.

This phase builds on the phase 4 report model (`ledger_identity_report.rs`) by
adding a thin preview layer that:

1. **Accepts parallel slices** of existing `LedgerEntry` and `Option<ResolvedUsageTarget>`
   values, constructing `LedgerIdentityAttachment` values internally.
2. **Wraps the phase 4 base report** with 13 preview-specific disclaimers that make
   it unambiguous this is a fixture-only, read-only preview with no live behavior.
3. **Reuses existing JSON serialization** from phase 4 for round-trip and deterministic
   serialization tests.
4. **Proves the full pipeline shape** — from raw entry+identity slices through
   attachment construction, base report building, preview rendering, and JSON
   serialization — using deterministic in-memory fixtures only.

The goals are fourfold. First, **prove the pipeline shape end-to-end**: the preview
module exercises the full code path from raw input slices to rendered output and JSON
serialization without any CLI, live resolver, or filesystem involvement. Second,
**validate 13 preview-specific disclaimers**: the rendered preview output includes
disclaimers making it explicit that no live resolver, no filesystem read/write, no
ledger persistence, no enforcement, no network blocking, no quota, no eBPF, no
nft/tc mutation, no cgroup mutation, and no PID movement occurs. Third, **preserve
v3.0 behavior unchanged**: no existing command, JSON schema, or module is modified.
Fourth, **remain pure fixture-only**: all test data is constructed in-memory with no
external dependencies.

## 2. Implementation

### 2.1 Module: `src/accounting/ledger_identity_preview.rs` (148 LOC)

Four public functions:

| Function | Signature | Description |
|----------|-----------|-------------|
| `build_ledger_identity_preview_report` | `(entries: &[LedgerEntry], identities: &[Option<ResolvedUsageTarget>]) -> LedgerIdentityReport` | Constructs a preview report from parallel slices. Builds `LedgerIdentityAttachment` values internally, then calls `build_ledger_identity_report()`. Mismatched slice lengths use the shorter slice. |
| `render_ledger_identity_preview_report` | `(report: &LedgerIdentityReport) -> String` | Renders the base report via `render_ledger_identity_report()` and prepends/appends 13 preview-specific disclaimers. |
| `serialize_preview_report_json` | `(report: &LedgerIdentityReport) -> Result<String, serde_json::Error>` | Serializes a preview report to deterministic pretty JSON. Reuses phase 4 `serialize_report_json()`. |
| `deserialize_preview_report_json` | `(json: &str) -> Result<LedgerIdentityReport, serde_json::Error>` | Deserializes JSON into a report. Reuses phase 4 `deserialize_report_json()`. |

### 2.2 Design Decisions

1. **Pure helper functions accepting existing types.** The preview module does not
   define any new model types. It accepts `LedgerEntry` and `Option<ResolvedUsageTarget>`
   directly and produces `LedgerIdentityReport` from phase 4. This avoids schema
   duplication and keeps the type surface minimal.

2. **Mismatched slice lengths use the shorter slice.** If the `entries` and
   `identities` slices have different lengths, the shorter length is used.
   This is a safe default that prevents index-out-of-bounds panics and is
   tested explicitly.

3. **Render wraps base report render.** `render_ledger_identity_preview_report`
   calls `render_ledger_identity_report()` and adds 13 preview-specific disclaimers
   around the base output. The base report content (interface breakdown, target
   breakdown, totals, honesty flags) is preserved intact.

4. **JSON reuses existing report serialization.** `serialize_preview_report_json`
   and `deserialize_preview_report_json` delegate to the phase 4 serialization
   functions. No new JSON schema is introduced.
