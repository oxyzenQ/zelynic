## 1. Purpose

Phase 12 implements the explicit `--file <PATH>` file-read gate that was
designed and frozen in Phase 11. When a user passes `--file <PATH>` to
`ledger inspect`, the handler validates the path, reads the file
(read-only), deserializes and validates the ledger JSON using the existing
`deserialize_ledger_from_json()` pipeline, and renders the output as
human-readable text or JSON. This is the first time any Zelynic code
performs a filesystem read on a user-specified file.

Without `--file`, `ledger inspect` remains fixture-only (unchanged from
Phase 10). There is no implicit default file read, no XDG auto-discovery,
and no environment variable path.

## 2. Command Shape Implemented

```
zelynic ledger inspect --file <PATH>
zelynic ledger inspect --file <PATH> --json
```

These are the only new runtime behaviors. All prior behaviors are preserved:

| Command | Behavior | Changed? |
|---------|----------|----------|
| `zelynic ledger inspect` | Fixture-only text | No |
| `zelynic ledger inspect --json` | Fixture-only JSON | No |
| `zelynic ledger inspect --file <PATH>` | File-read text | **New** |
| `zelynic ledger inspect --file <PATH> --json` | File-read JSON | **New** |
| `zelynic ledger inspect --input <PATH>` | Rejected by clap | No |
| `zelynic ledger export --json` | Design-gated | No |

## 3. Implementation

### 3.1 CLI (`src/cli.rs`)

Added `file: Option<String>` field to `LedgerCommands::Inspect` with
`#[arg(long = "file", hide = true, value_name = "PATH")]`. The flag is
hidden and requires an explicit path value.

### 3.2 Path Validation (`validate_inspect_file_path`)

Conservative validation layer applied before any filesystem read:

1. **Empty path**: Rejected immediately.
2. **Parent traversal**: Any `..` path component is rejected.
3. **Suspicious filename characters**: Only `[a-zA-Z0-9._-]` allowed in
   the filename (basename). Characters like `;`, `|`, `\0`, spaces, etc.
   are rejected.
4. **Filesystem existence**: `std::fs::symlink_metadata()` is used (does
   not follow symlinks).
5. **Symlink rejection**: If the path is a symlink, it is rejected with a
   clear error message.
6. **Regular file check**: The path must be a regular file, not a
   directory, device, or socket.

### 3.3 File Read (`read_ledger_file`)

Uses `std::fs::read_to_string()` to read the file content, then delegates
to the existing `deserialize_ledger_from_json()` which performs:
- JSON syntax validation
- Schema version check (only version 1 supported)
- Per-entry `validate_safety()`:
  - `read_only` must be `true`
  - `attribution_scope` must be `"interface-level only"`
  - `enforcement_status` must be `"inactive/not implemented"`
  - `combined_bytes == rx_bytes + tx_bytes` (saturating)

### 3.4 Text Output (`render_file_read_inspect_text`)

File-read text output includes all the same aggregate statistics as the
fixture output but with file-read-specific disclaimers:

- Header says "source: explicit file read" (not "model only")
- Shows the file path
- Safety disclaimers affirm: read-only, no file write, no persistence
  save, no live /proc or sysfs read, no live app identity resolver,
  interface-level only attribution, inactive enforcement, inactive network
  blocking, no limiter attach, no nft/tc state mutation.

### 3.5 JSON Output

File-read JSON output uses the same `serialize_ledger_to_json()` as the
fixture mode. The output is deterministic, valid JSON, and does not alter
the v3.0 usage JSON schema.

## 4. Safety Model Limitations

The path validation is conservative but has documented limits:

- It does not check file permissions (the OS rejects unreadable files).
- It does not resolve mount points or filesystem boundaries.
- It does not check for hardlinks (a hardlinked file is treated as a
  regular file).
- Symlink detection uses `symlink_metadata` which does not follow
  symlinks. This means a path like `/tmp/ledger.json` that is a symlink
  to `/etc/shadow` would be rejected. But a path like `/tmp/real.json`
  that is a regular file (even if it was previously a symlink target)
  would pass validation.

These limitations are acceptable for a read-only inspect command. The
ledger data itself is validated by `deserialize_ledger_from_json()` which
catches corrupt, tampered, or version-mismatched content.

## 5. Non-Goals

This phase does NOT:

- Implement implicit/default ledger file read.
- Read from XDG/default path unless user explicitly passes that path.
- Implement `--input` flag.
- Implement `ledger export`.
- Implement `ledger save`, `ledger write`, backup, migration, truncate,
  rename, or delete.
- Create directories.
- Write any file.
- Enable persistence auto-load.
- Add live app identity resolver.
- Add permission/block/allow mode.
- Add quota guard.
- Add eBPF.
- Add daemon/watch mode.
- Mutate nft/tc/cgroup/PID state.
- Change v3.0 usage JSON schema.
- Bump version.
- Tag, release, or publish.
- Add dependencies.
- Unhide the `ledger` subcommand.

This is not full persistence — it is explicit read-only inspect.
