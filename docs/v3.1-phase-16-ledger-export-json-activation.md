## 1. Purpose

Phase 16 activates the `zelynic ledger export --json --file <PATH>` command
that was designed and gated in Phase 15. This is a small runtime activation:
the command now actually works instead of being rejected with a
"design-gated" message. The activation is intentionally narrow: it requires
both `--json` and `--file <PATH>`, reads the specified ledger file,
validates it, and emits the validated ledger JSON to stdout only. There is
no output file, no `--output`, no `--overwrite`, no persistence, and no
live system state reading.

This phase follows the Phase 15 activation checklist exactly. All 16 design
rules from Phase 15 are satisfied by this implementation.

## 2. Activated Command

```
zelynic ledger export --json --file <PATH>
```

### 2.1 Behavior

- `--json` is required. Without it, the export command fails honestly or
  remains gated (no human export output is provided in this phase).
- `--file <PATH>` is required. Without it, the export command fails
  honestly and does not read any default path, does not auto-discover any
  file, and does not access the filesystem.
- The path is validated using the same hardened rules as `ledger inspect
  --file`: no empty paths, no parent traversal (`..`), no suspicious
  filename characters (only `[a-zA-Z0-9._-]`), no symlinks, no
  directories, no nonexistent files.
- The file is read with `std::fs::read_to_string` (read-only).
- The JSON content is deserialized and validated via the existing
  `deserialize_ledger_from_json()` pipeline: schema_version must be 1,
  all entries must pass `validate_safety()` (read_only, attribution_scope,
  enforcement_status, combined_bytes consistency).
- If validation passes, the validated ledger is reserialized via
  `serialize_ledger_to_json()` and emitted to stdout as deterministic JSON.
- No human headers, no disclaimers, no extra text are emitted in JSON mode.
  The stdout output is valid JSON only.
- The operation is deterministic: reading the same valid file always
  produces the same JSON output.
- No file is written. No directory is created. No state is mutated.
  The command is pure: same input file always produces the same stdout
  output.

### 2.2 Export JSON Shape

The exported JSON is the ledger serialization schema. It preserves:
- `schema_version: 1`
- `host_id`
- `created_at`, `updated_at`
- `session_id` (if present)
- `entries` array with all entry fields

The export JSON is produced by `serialize_ledger_to_json()`, the same
function used by `ledger inspect --json`. Export JSON and inspect JSON
have the same serialization schema but are semantically different
operations: export reads a user-specified file; inspect may use fixture
data.

The export JSON does NOT wrap the ledger in a new envelope. The output
is the ledger JSON itself, as-is.

## 3. Current Rejected/Gated Behavior

The following command shapes remain rejected or gated:

| Command | Behavior |
|---------|----------|
| `zelynic ledger export --json` | Fails honestly: --file is required |
| `zelynic ledger export --file <PATH>` | Fails honestly or remains gated: --json is required |
| `zelynic ledger export --output <PATH>` | Rejected: flag does not exist |
| `zelynic ledger export --overwrite` | Rejected: flag does not exist |
| `zelynic ledger export` (no flags) | Fails honestly: --file and --json are required |

## 5. Safety Contract

### 5.1 Export Requires Explicit --file

The `--file <PATH>` flag is mandatory. There is no implicit default file
path, no XDG auto-discovery, no environment variable, and no auto-load.
The user must type the exact file path on the command line.

### 5.2 Export Requires --json

The `--json` flag is mandatory. Human export output is not provided in
this phase. Without `--json`, the command fails honestly.

### 5.3 Stdout-Only Output

Export writes JSON to stdout only. There is no `--output <PATH>`, no
`--write <PATH>`, no `--save <PATH>`. If a user wants to save the output,
they redirect stdout to a file using shell redirection (`>
output.json`). The export command itself never writes to the filesystem.

### 5.4 No Output File / No Overwrite

The `--output` and `--overwrite` flags do not exist. They are not
registered in the clap definition. Any attempt to use them produces a
clap parse error.

### 5.5 No Default Path Read

Export does not read from any default path unless the user explicitly
passes that path via `--file`. No XDG auto-discovery, no environment
variable, no implicit fallback.

### 5.6 No Persistence / Save

Export does not save ledger state. No persistence is enabled. The
persistence module returns `Blocked` for all operations. Export is a
read-only, one-shot operation.

### 5.7 No Live /proc/sysfs Read

Export does not read from `/proc/net/dev`, `/sys/class/net/`, or any
other live system interface. The data source is exclusively the
user-specified JSON file.

### 5.8 No nft/tc/cgroup/PID Mutation

Export does not modify nftables rules, tc classes/filters, cgroup
assignments, process state, or any Zelynic limiter state. No limiter
attach, no policy install, no cleanup.

## 6. Schema Boundary

### 6.1 Export JSON is the Ledger JSON Schema

The export JSON uses the ledger serialization schema
(`schema_version: 1`, with `entries` array, `host_id`, timestamps, etc.).
It is produced by the existing `serialize_ledger_to_json()` function.

### 6.2 v3.0 Usage JSON Remains Unchanged

The v3.0 usage JSON schema (schema_version 1) produced by `zelynic
usage --sample --delta --json` remains frozen. No field is added,
removed, or renamed. The export command does not affect the usage JSON
schema.

### 6.3 Ledger Inspect JSON Remains Unchanged

All existing `ledger inspect` behavior remains unchanged:
- `ledger inspect` (fixture text)
- `ledger inspect --json` (fixture JSON)
- `ledger inspect --file <PATH>` (explicit file text)
- `ledger inspect --file <PATH> --json` (explicit file JSON)

### 6.4 Export JSON is Not Inspect JSON

While export JSON and inspect JSON share the same serialization schema
(both use `serialize_ledger_to_json()`), they are semantically different
operations. Export reads a user-specified file; inspect may use fixture
data. The difference is in the command and the data source, not the JSON
schema.

### 6.5 Export JSON is Not v3.0 Usage JSON

The export JSON has `schema_version`, `host_id`, `entries`, etc. The
v3.0 usage JSON has `command`, `sample_mode`, `interfaces`, `totals`,
etc. These are completely different schemas.

## 7. Future Roadmap Boundary

- **Output file export remains not implemented.** There is no
  `--output <PATH>` flag. Users must use shell redirection to save
  export output to a file.
- **Persistence/save remains not implemented.** The persistence module
  returns `Blocked` for all operations. No `ledger save` command exists.
- **Default ledger path read remains not implemented.** Export requires
  explicit `--file <PATH>`.
- **Live resolver remains not implemented.** Attribution is
  "interface-level only."
- **Enforcement remains not implemented.** `enforcement_status` is
  always "inactive/not implemented."

## 9. Phase 15 Design Rules — Satisfaction

All 16 design rules from Phase 15 are satisfied:

1. **Rule 1 (Export was gated)**: Satisfied in Phase 15; activated here.
2. **Rule 2 (Explicit --file)**: `--file <PATH>` is required; export
   fails without it.
3. **Rule 3 (Read-only)**: Export reads a file and writes nothing.
4. **Rule 4 (Path validation before read)**: Reuses
   `validate_inspect_file_path()`.
5. **Rule 5 (Schema validation before output)**: Reuses
   `deserialize_ledger_from_json()`.
6. **Rule 6 (JSON to stdout only)**: Uses `println!()` of serialized
   JSON. No file write.
7. **Rule 7 (No file write)**: No `create_dir`, `remove_file`,
   `rename`, `File::create`, `OpenOptions`, or `fs::copy` in production
   export code.
8. **Rule 8 (No directory creation)**: No `create_dir` or `create_dir_all`.
9. **Rule 9 (No backup/migration/rename/truncate/delete/overwrite)**:
   None of these operations exist.
10. **Rule 10 (No silent default path read)**: `--file` is required;
    no fallback.
11. **Rule 11 (No live /proc/sysfs read)**: Data source is exclusively
    the user-specified file.
12. **Rule 12 (No nft/tc/cgroup/PID mutation)**: No mutation code.
13. **Rule 13 (Honest non-mutating errors)**: All errors are reported
    with stable prefix `ledger export:` or `ledger export --file:`.
14. **Rule 14 (Export JSON schema is separate)**: Documented as ledger
    serialization schema, separate from v3.0 usage and any future
    inspect-specific schema.
15. **Rule 15 (v3.0 usage JSON unchanged)**: No change to usage code.
16. **Rule 16 (Ledger inspect unchanged)**: No change to inspect handler.
