## 1. Purpose

Phase 13 freezes and hardens the error behavior of the explicit `--file`
read-only path implemented in Phase 12. This phase adds no new features;
it adds deterministic tests that prove every error path produces stable,
honest error messages before any future export, save, or persistence phase
can introduce regressions.

The goal is to establish a verified error matrix: for every class of bad
input (invalid path, malformed JSON, schema violations, safety constraint
violations), the error message contains a stable identifying keyword that
scripts and tests can rely on without overfitting to OS-specific errno text.

## 3. Error Wording Design

Error messages use a stable prefix `ledger inspect --file:` to identify the
command context. The error wording is honest about the specific problem
without overfitting to OS-specific errno text. For example:

- Path validation errors: `ledger inspect --file: parent traversal (..) is not allowed`
- File access errors: `ledger inspect --file: cannot access '<path>': <os error>`
- Parse errors: `ledger inspect --file: invalid ledger in '<path>': JSON parse error: ...`
- Schema errors: `ledger inspect --file: invalid ledger in '<path>': unsupported schema version: ...`
- Safety errors: `ledger inspect --file: invalid ledger in '<path>': safety violation: entry <id> has ...`

This layered approach means the outer prefix is stable, the error class
keyword is stable, and only the OS-specific detail varies.

## 4. Permission-Denied Testing

R-09 (permission denied) is not tested with a deterministic file-permission
setup because creating permission-denied files safely across environments
(root vs non-root, container vs host) is unreliable. The OS error propagates
honestly through the `failed to read` path. This is documented here as a
known testing gap.

## 5. Non-Goals

This phase does NOT:

- Implement default ledger path read.
- Implement ledger export.
- Implement ledger save/write.
- Create directories.
- Write files.
- Add persistence auto-load.
- Add migration, backup, rename, truncate, delete, or overwrite.
- Add live app identity resolver.
- Add permission/block/allow mode.
- Add quota guard.
- Add eBPF.
- Add daemon/watch mode.
- Mutate nft/tc/cgroup/PID state.
- Change v3.0 usage JSON schema.
- Bump version.
- Tag, release, or publish.
- Add dependencies.
- Change production code behavior (error wording refinements only if needed).

Error hardening only. No new features.
