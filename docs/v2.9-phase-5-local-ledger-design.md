# Zelynic v2.9 Phase 5: Local Ledger Design

This document captures the design for the future local usage ledger, a persistent
storage mechanism for network accounting snapshots, session deltas, and interface-level
usage history. Phase 5 is **design-only**: no Rust code for file I/O is implemented,
no ledger read/write occurs, and no filesystem mutations are performed. This document
establishes the schema, storage boundary, privacy constraints, honesty rules, and future
implementation roadmap for the ledger so that subsequent phases can build on a clearly
defined contract.

## Purpose

The local ledger exists to solve a fundamental gap in the current v2.9 accounting
model: all existing accounting types (`InterfaceCounterSnapshot`, `UsagePreview`,
`SessionDelta`) are ephemeral. They operate on in-memory snapshots parsed from
string content and produce rendered output, but they have no persistence mechanism.
Once the process exits, all computed deltas and usage summaries are lost. The ledger
design provides a forward-looking plan for persistent storage that bridges the gap
between the current pure model layer and the future runtime features that need
historical data.

The ledger's purpose spans several future capabilities, each of which requires some
form of persistent state:

1. **Session tracking**: Store reference snapshots so that `zelynic usage --session`
   can compute deltas spanning multiple invocations without requiring the user to
   manually capture and compare counter values.
2. **Historical usage reporting**: Allow `zelynic usage --since-boot` and
   `zelynic usage --interface wlan0` to show usage accumulated over time, not just
   a single point-in-time snapshot.
3. **Future quota guard decisions**: When v3.x introduces quota enforcement, the
   guard needs cumulative usage data to decide whether a budget is exhausted. The
   ledger provides the authoritative record of what has been consumed.
4. **Future background data guard**: When non-foreground applications are subject
   to separate quotas, the ledger stores per-interface usage that can be attributed
   (at the interface level, not per-app level) to background traffic periods.
5. **Boot-to-boot continuity**: Because `/proc/net/dev` counters reset on reboot,
   the ledger can preserve cumulative totals across reboots by storing snapshots
   with timestamps and host identifiers, enabling "all-time" reporting that survives
   counter resets.

Importantly, the ledger remains honest about its limitations: it stores interface-level
counters, which are aggregate traffic measurements for an entire network interface and
cannot attribute bytes to individual applications, PIDs, or cgroups without additional
attribution infrastructure (such as eBPF, which is deferred to v4.x).

## Ledger Data Model

The ledger schema is defined as a structured document format. The choice of concrete
serialization (JSON, TOML, bincode, SQLite, or custom binary) is deferred to the
implementation phase (phase 6). The logical schema is presented here as a structured
description that can be mapped to any serialization format.

### Root Ledger Structure

The root object represents a single ledger file. Each ledger is associated with a
specific host and contains an ordered list of entries representing accounting
snapshots and deltas over time.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `schema_version` | integer | yes | Schema format version. Initial value: 1. Used for future migration. |
| `created_at` | string (ISO 8601) | yes | Ledger creation timestamp in UTC. |
| `updated_at` | string (ISO 8601) | yes | Last modification timestamp in UTC. Updated on every write. |
| `host_id` | string | yes | Unique host identifier. Suggested: hostname + boot_id hash to distinguish reboots. |
| `session_id` | string or null | no | Optional user-defined session identifier for multi-session tracking. Null for default session. |
| `entries` | array of LedgerEntry | yes | Ordered list of accounting entries, newest last. |
| `metadata` | object | no | Optional key-value metadata for future extensions (e.g., Zelynic version that wrote the ledger). |

### LedgerEntry Structure

Each entry in the ledger represents either a raw counter snapshot (captured from
`/proc/net/dev` at a point in time) or a computed session delta. Entries are
ordered chronologically so that consumers can iterate through usage history without
needing external sorting.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `entry_id` | string (UUID v4) | yes | Unique identifier for this entry. Enables deduplication and idempotent writes. |
| `timestamp` | string (ISO 8601) | yes | When this entry was recorded, in UTC. |
| `entry_type` | enum | yes | One of: `"snapshot"`, `"delta"`, `"quota_config"`. Distinguishes raw data from computed deltas from quota definitions. |
| `source_label` | string | yes | Provenance marker matching the existing `SourceLabel` type. Example values: `"model-only"`, `"parsed_sample"`, `"future_live_read"`. |
| `interface` | string | yes | Network interface name. Example: `"wlan0"`, `"eth0"`, `"usb0"`. One entry per interface per timestamp. |
| `rx_bytes` | integer (u64) | yes | Received bytes on this interface. For snapshots: raw counter value. For deltas: computed difference. |
| `tx_bytes` | integer (u64) | yes | Transmitted bytes on this interface. For snapshots: raw counter value. For deltas: computed difference. |
| `rx_packets` | integer (u64) | no | Received packets. Optional; included for snapshots, may be omitted for deltas if not needed. |
| `tx_packets` | integer (u64) | no | Transmitted packets. Optional; same rationale as rx_packets. |
| `combined_bytes` | integer (u64) | yes | rx_bytes + tx_bytes (saturating). Stored explicitly to avoid recomputation and to handle overflow consistently. |
| `reset_detected` | boolean | yes | Whether a counter reset or decrease was detected when this entry was recorded. True if any counter decreased between consecutive snapshots. |
| `reset_details` | array of ResetDetail or null | no | If reset_detected is true, describes which counters reset and with what values. Null if no resets. |
| `read_only` | boolean | yes | Always true for v2.9 ledger entries. Marks that this entry is observational and not enforcement-related. |
| `provenance` | string | yes | Provenance description matching existing model patterns. Example: `"read-only session delta model"`, `"read-only parsed snapshot"`. |
| `target_app` | string or null | no | **Not implemented.** Reserved for future per-app attribution (v3.x/v4.x). Must be null in v2.9. |
| `target_cgroup` | string or null | no | **Not implemented.** Reserved for future per-cgroup attribution (v3.x/v4.x). Must be null in v2.9. |
| `target_pid` | integer or null | no | **Not implemented.** Reserved for future per-PID attribution. Must be null in v2.9. |

### ResetDetail Structure

When a counter reset is detected (end value < start value for a specific counter),
a ResetDetail record captures the specifics for honest reporting.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `counter_field` | string | yes | Which counter reset. Values: `"rx_bytes"`, `"tx_bytes"`, `"rx_packets"`, `"tx_packets"`. |
| `start_value` | integer (u64) | yes | The counter value in the previous snapshot. |
| `end_value` | integer (u64) | yes | The counter value in the current snapshot (lower than start_value). |
| `reason` | string | no | Human-readable reason hint. Example: `"counter decrease detected, possible interface reset or reboot"`. |

### QuotaConfigEntry Structure (Future, Not Implemented)

This entry type is reserved for future quota budget definitions. It is included in
the schema design so that the ledger format is forward-compatible with v3.x quota
guard features. No quota config entries are written in v2.9.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `entry_type` | enum | yes | Always `"quota_config"`. |
| `interface` | string | yes | Interface this quota applies to. |
| `budget_bytes` | integer (u64) | yes | Total byte budget for the quota period. |
| `budget_direction` | enum | yes | One of: `"combined"`, `"rx_only"`, `"tx_only"`. |
| `period` | string | no | Quota period description. Example: `"daily"`, `"monthly"`, `"session"`. |
| `enforcement_status` | string | yes | Always `"not_enforced"` in v2.9. |
| `created_at` | string (ISO 8601) | yes | When this quota config was defined. |

### Schema Versioning and Migration Strategy

The `schema_version` field enables forward-compatible schema evolution. The migration
strategy is designed to be simple and safe:

1. **Version 1**: Initial schema as defined above. All v2.9 ledgers use version 1.
2. **Future versions**: When schema changes are needed (e.g., adding per-app fields
   in v3.x), the version number is incremented. New fields are added as optional
   fields with sensible defaults (typically null or zero).
3. **Migration on read**: When a newer Zelynic version encounters a ledger with an
   older schema version, it reads the known fields and populates new fields with
   defaults. No destructive migration occurs; the ledger is only rewritten when the
   user explicitly requests it (e.g., `zelynic ledger migrate`).
4. **Unknown field preservation**: When reading a ledger with a newer schema version
   than the running Zelynic understands, unknown fields are preserved in-memory and
   written back unchanged. This prevents data loss when downgrading.
5. **No automatic migration**: Migration is never triggered automatically on
   startup. The user must explicitly request migration or the ledger is read in
   compatibility mode. This prevents unexpected filesystem writes.

The migration strategy prioritizes data safety over convenience. A corrupted or
partially migrated ledger is worse than an old-format ledger that is read in
compatibility mode. The design explicitly avoids background migration daemons,
auto-upgrade hooks, or silent schema rewrites.

## Storage Boundary

The storage boundary defines where the ledger lives on disk, how it is written, and
what guarantees are provided around data integrity and privacy.

### Recommended File Path

The ledger should be stored under the Zelynic state/cache namespace, following
XDG Base Directory conventions for user-local data:

```
$XDG_DATA_HOME/zelynic/accounting-ledger.json
```

Default (if `$XDG_DATA_HOME` is not set):

```
~/.local/share/zelynic/accounting-ledger.json
```

This location is chosen for the following reasons:

1. **XDG compliance**: `$XDG_DATA_HOME` is the standard location for persistent
   application data on Linux. Tools like `xdg-user-dir` and the freedesktop.org
   specification define this as `~/.local/share`.
2. **Zelynic namespace**: The `zelynic/` subdirectory isolates Zelynic's data from
   other applications. If Zelynic is ever uninstalled, the entire `zelynic/`
   directory can be removed without affecting other applications.
3. **Not in `/sys/fs/cgroup`**: The ledger is explicitly NOT stored under
   `/sys/fs/cgroup`. v2.9 does not touch the cgroup filesystem for accounting.
4. **Not in `/proc`**: The ledger is not a procfs file. It is a regular file on
   the user's filesystem.
5. **User-writable**: The ledger path should be writable by the user running
   Zelynic. No root privileges are required for ledger read/write (though root
   may be needed for the underlying counter reads in future phases).

### Atomic Write Strategy

To prevent corruption from partial writes (e.g., crash during write, disk full,
power loss), the ledger must use an atomic write strategy:

1. **Write to temporary file**: Write the complete ledger content to a temporary file
   in the same directory with a unique suffix (e.g.,
   `accounting-ledger.json.tmp.<PID>` or `accounting-ledger.json.new.<UUID>`).
2. **Sync to disk**: Call `fsync()` on the temporary file to ensure data is
   flushed to persistent storage before the rename.
3. **Atomic rename**: Use `rename()` to atomically replace the old ledger file with
   the new one. On Linux, `rename()` within the same filesystem is atomic: either
   the old file or the new file exists, never a partial combination.
4. **Sync directory**: Call `fsync()` on the parent directory to ensure the
   directory entry update is persisted.
5. **Verify**: After rename, read back the file and verify the schema version and
   host_id match expected values. If verification fails, log a warning and restore
   from the old file if a backup exists.

This strategy ensures that the ledger file is never in a partially-written state
from the perspective of any reader. Either the old complete version or the new
complete version is visible at all times.

### Corruption Handling

Despite atomic writes, corruption can still occur from disk errors, filesystem bugs,
or external tampering. The design defines the following corruption handling strategy:

1. **Schema validation on read**: When reading a ledger file, validate the schema
   version, required fields, and field types before processing entries. If validation
   fails, report a clear error with the specific field and reason.
2. **Fallback to backup**: If the ledger file fails validation and a backup file
   exists (e.g., `accounting-ledger.json.bak`), attempt to read the backup. If the
   backup is valid, use it and warn the user that the primary file was corrupted.
3. **No silent recovery**: If the ledger is corrupted and no valid backup exists,
   report the corruption to the user and offer to start a fresh ledger. Never silently
   discard or overwrite corrupted data, as the user may want to inspect or manually
   recover it.
4. **Entry-level validation**: Individual entries within a valid ledger are validated
   independently. A corrupted entry does not invalidate the entire ledger. Invalid
   entries are skipped with a warning, and valid entries are processed.
5. **Checksum**: Future schema versions may include a per-entry or per-ledger
   checksum (e.g., SHA-256 of the entry bytes excluding the checksum field). This
   is not required for version 1 but is reserved as an optional field for future
   integrity verification.

### Rotation and Cleanup

Over time, the ledger file will grow as new entries are added. Without cleanup,
it could consume significant disk space. The design defines the following rotation
strategy for future implementation:

1. **Time-based rotation**: Entries older than a configurable threshold (default:
   90 days) are eligible for pruning. A `zelynic ledger prune --older-than 90d`
   command (future, not implemented) would remove old entries.
2. **Size-based rotation**: If the ledger file exceeds a configurable size limit
   (default: 10 MiB), oldest entries are pruned until the file is below the limit.
3. **Aggregation**: Before pruning, old entries can be aggregated into summary
   entries (e.g., daily or weekly totals) to preserve historical trends while
   reducing entry count. Aggregated entries are marked with `entry_type:
   "aggregated_summary"` and include the time range they cover.
4. **No automatic rotation**: Rotation is never performed automatically. The user
   must explicitly invoke a prune or cleanup command. This prevents unexpected data
   loss from automatic cleanup.
5. **Backup before prune**: Before any prune operation, the current ledger is backed
   up to `accounting-ledger.json.pre-prune.<timestamp>`. This allows manual recovery
   if too much was pruned.

### Permission Expectations

The ledger file should be created with the following permissions:

1. **File mode**: `0600` (owner read/write only). The ledger contains network usage
   data that could reveal browsing patterns, application usage, and connectivity
   timing. Restricting access to the owner protects user privacy.
2. **Directory mode**: `0700` for the `zelynic/` directory. Same rationale: prevent
   other users from listing or accessing Zelynic's data directory.
3. **No setuid/setgid**: The ledger must never be created with setuid or setgid bits.
4. **No world-readable**: The ledger must never be world-readable (no `o+r`).
5. **Ownership**: The file should be owned by the user running Zelynic. If Zelynic
   runs as root (as `zelynic strict` does), the ledger should be written to the
   target user's home directory or a system-wide location with appropriate
   permissions, not to root's home directory.

### What the Ledger Must NOT Store

The ledger stores only interface-level aggregate counters. It must never store the
following data:

1. **No secrets**: The ledger must never contain passwords, API keys, tokens,
   certificates, encryption keys, or any form of secret material.
2. **No raw packet data**: The ledger stores aggregate byte and packet counts, not
   individual packets. Capturing raw packets would require pcap-level access and
   would raise serious privacy and legal concerns.
3. **No process command lines**: The ledger must not capture process command lines,
   arguments, environment variables, or working directories. If per-app attribution
   is added in the future (v3.x/v4.x), it would store only process names or
   binary identifiers, not full command lines, and only after an explicit privacy
   review.
4. **No DNS queries or URLs**: The ledger must not capture DNS queries, URL patterns,
   domain names, or any content-level information about network traffic. It stores
   only byte counts at the interface level.
5. **No IP addresses of remote hosts**: The ledger does not track which remote
   hosts were communicated with. It stores only aggregate counters per interface.
6. **No timestamps of individual connections**: The ledger stores entry timestamps
   for snapshots and deltas, not for individual network connections.

These restrictions protect user privacy and ensure that the ledger cannot be used
as a surveillance tool, even if accessed by unauthorized parties.

## Privacy and Honesty Constraints

The ledger design inherits all honesty constraints from the v2.9 Network Accounting
Lab design document and adds ledger-specific constraints.

### Interface Counters Are Not Per-App Attribution

Every entry in the ledger stores interface-level aggregate counters. The ledger
must never claim that these counters represent per-app, per-PID, or per-cgroup
traffic. The `target_app`, `target_cgroup`, and `target_pid` fields are reserved
for future use and must be null in v2.9.

**Required wording in ledger rendering**: "Interface-level aggregate counters, not
per-app attribution."
**Forbidden wording**: "App usage", "Process bandwidth", "Application data
consumption", or any implication that ledger entries represent per-application
traffic.

### Ledger Must Not Imply Enforcement

The ledger is a passive data store. It records what was observed, not what was
enforced. The `read_only` field must always be true in v2.9. The ledger must never
claim that it triggered any blocking, throttling, or rate-limiting action.

**Required wording**: "Read-only observational ledger. No enforcement actions."
**Forbidden wording**: "Quota enforced", "Traffic blocked", "Limit applied by ledger",
or any implication that the ledger caused a traffic action.

### Ledger Must Not Imply Quota Guard Is Active

Even when the schema includes `QuotaConfigEntry` structures, no quota enforcement
occurs in v2.9. The `enforcement_status` field in quota config entries must always
be `"not_enforced"`. If a future `zelynic quota status` command reads the ledger,
it must display "report only -- no enforcement active".

**Required wording**: "Quota configuration stored for future use. No enforcement is
active."
**Forbidden wording**: "Quota active", "Budget guard active", "Enforcement enabled",
or any implication that quota decisions are being made.

### Ledger Must Not Imply Network Blocking

The ledger must never claim that network traffic was blocked, dropped, or rejected
as a result of any ledger entry. All entries are observational.

**Required wording**: "No network blocking performed."
**Forbidden wording**: "Blocked", "Dropped", "Rejected by ledger", or any implication
that the ledger modified network traffic.

### Ledger Must Not Imply eBPF

The ledger does not use eBPF in v2.9. All data comes from interface counters
(`/proc/net/dev` or sysfs). If per-app attribution is added in v4.x using eBPF,
the `source_label` field would distinguish eBPF-sourced entries from interface
counter entries.

**Required wording (in v2.9)**: "No eBPF attribution. Interface counters only."
**Forbidden wording**: "eBPF observed", "Per-app via eBPF", or any implication
that eBPF was used.

### Counters May Reset After Reboot or Interface Reset

Because the ledger stores counter snapshots (not deltas), it inherits the
counter reset limitation. If the system reboots or an interface is brought down
and up, the kernel counters reset to zero. The ledger must detect and flag these
resets using the `reset_detected` and `reset_details` fields.

**Required behavior**: When a new snapshot has lower counter values than the previous
snapshot for the same interface, `reset_detected` must be set to true, the delta
must be computed as zero (not a negative number), and `reset_details` must describe
which counters decreased and by how much.
**Forbidden behavior**: Computing negative deltas, silently ignoring counter
decreases, or claiming continuity across a detected reset.

### Ledger May Be Incomplete

If Zelynic was not running continuously, the ledger may have gaps between entries.
The time between consecutive entries may span minutes, hours, or days. The ledger
must not claim that entries represent complete coverage of all network activity during
that period. Only activity that was actually captured (when Zelynic was running and
reading counters) is recorded.

**Required wording**: "Ledger entries are point-in-time observations. Gaps between
entries may exist if Zelynic was not running."
**Forbidden wording**: "Complete history", "All traffic recorded", "Uninterrupted
coverage", or any implication that the ledger captured everything.

## Future Commands (Design Surface)

The following commands are designed as part of the ledger feature but are **not
implemented** in v2.9 phase 5. They are documented here to define the future CLI
surface and ensure the design accounts for them.

### `zelynic usage --session`

Display usage for the current or most recent session, computed from ledger entries
since the session start snapshot. Requires the ledger to exist and contain at least
two entries (start and end) for the session.

```
$ zelynic usage --session
Session: default (since 2026-06-06T10:00:00Z)
Duration: 2h 34m
Interface     RX           TX           Combined
wlan0         345 MiB      89 MiB       434 MiB
eth0          0 bytes       0 bytes      0 bytes
Total:        345 MiB      89 MiB       434 MiB
Read-only session delta model. No enforcement. No network blocking.
```

**Not implemented.** Design only.

### `zelynic usage --since-boot`

Display usage since the most recent system boot, using ledger entries from the
current boot session identified by `host_id`. Falls back to a direct `/proc/net/dev`
read if no ledger entries exist for the current boot.

```
$ zelynic usage --since-boot
Since boot (host_id: abc123, booted 2026-06-06T08:00:00Z)
Interface     RX           TX           Since
wlan0         1.23 GiB     342 MiB      boot
eth0          0 bytes       0 bytes      boot
usb0          45.6 MiB     12.3 MiB     boot (tethering)
Read-only observational model. No enforcement.
```

**Not implemented.** Design only.

### `zelynic usage --interface wlan0`

Display usage history for a specific interface, filtering ledger entries to the named
interface. Shows the most recent snapshot and, if available, the session delta.

```
$ zelynic usage --interface wlan0
Interface: wlan0
Latest snapshot: RX 1.23 GiB, TX 342 MiB (2026-06-06T12:34:56Z)
Session delta:  RX 345 MiB,  TX 89 MiB  (since 2026-06-06T10:00:00Z)
Read-only observational model. No enforcement. No per-app attribution.
```

**Not implemented.** Design only.

### `zelynic usage --ledger`

Display a summary of the ledger itself: number of entries, time range, interfaces
tracked, schema version, and any detected counter resets. Does not display actual
usage data, only ledger metadata.

```
$ zelynic usage --ledger
Ledger: ~/.local/share/zelynic/accounting-ledger.json
Schema version: 1
Entries: 47
Time range: 2026-06-05T10:00:00Z to 2026-06-06T12:34:56Z
Interfaces: wlan0, eth0, usb0
Counter resets detected: 1 (wlan0, 2026-06-06T08:00:00Z, boot)
Read-only ledger metadata. No enforcement. No live reads.
```

**Not implemented.** Design only.

### `zelynic ledger inspect`

Display detailed ledger contents: all entries with timestamps, interface names, byte
counts, and reset flags. Intended for debugging and manual inspection. Output may
be verbose.

```
$ zelynic ledger inspect
Ledger: 47 entries (newest first)
[2026-06-06T12:34:56Z] wlan0 snapshot: RX=1323898624 TX=358637568 (read-only)
[2026-06-06T12:34:56Z] eth0 snapshot: RX=0 TX=0 (read-only)
[2026-06-06T12:30:00Z] wlan0 delta: RX=52428800 TX=10485760 (reset=false)
...
Read-only ledger data. No enforcement.
```

**Not implemented.** Design only.

### `zelynic ledger clear`

Remove all entries from the ledger, effectively resetting the accounting history.
Requires confirmation (e.g., `--confirm` flag or interactive prompt) to prevent
accidental data loss. Creates a backup before clearing.

```
$ zelynic ledger clear --confirm
Ledger cleared. 47 entries removed.
Backup saved: accounting-ledger.json.pre-clear.20260606T123456Z
Read-only. No enforcement. No network blocking.
```

**Not implemented.** Design only.

### `zelynic quota status`

Display quota budget and current usage for configured quotas. Reads quota config
entries from the ledger and computes remaining budget. Always states that no
enforcement is active.

```
$ zelynic quota status
Interface: wlan0 (tethering)
  Budget: 100 MiB
  Used:  45.6 MiB
  Remaining: 54.4 MiB
  Status: report only -- no enforcement active
No quota enforcement. No network blocking. No limiter attach.
```

**Not implemented.** Design only.

The ledger design defines a phased implementation roadmap. Each phase has explicit
entry criteria, deliverables, and safety boundaries.

### Phase 5: Design Only (Completed)

**Deliverables**:
- This design document (`docs/v2.9-phase-5-local-ledger-design.md`).
- Updates to `docs/v2.9-network-accounting-lab.md` marking phase 5 completed.
- Updates to `CHANGELOG.md` with Unreleased phase 5 entry.

**Safety boundary**: No Rust code. No file I/O. No CLI. No test changes. Docs only.

**Entry criteria met**: Phases 1 through 4b complete, all tests passing, CI green.

### Phase 6: Pure Ledger Model + Serialization Tests (Completed)

**Goal**: Implement the ledger data model as pure Rust types and test serialization
and deserialization without touching the filesystem.

**Deliverables**:
- `src/accounting/ledger.rs` (or `src/accounting/ledger/mod.rs` directory module)
  with pure Rust types: `Ledger`, `LedgerEntry`, `ResetDetail`, `QuotaConfigEntry`.
- Pure serialization functions: `serialize_ledger(ledger) -> String`,
  `deserialize_ledger(content: &str) -> Result<Ledger, LedgerError>`.
- Pure build functions: `build_ledger_from_snapshot(snapshot, ...)`,
  `build_ledger_entry_from_delta(delta, ...)`.
- Pure render function: `render_ledger_summary(ledger)` for future `zelynic
  usage --ledger` display contract.
- Unit tests for: serialization round-trip, deserialization validation, schema
  version handling, field defaults for null optional fields, entry ordering,
  reset detail rendering, quota config rendering (not enforced), honesty
  disclaimers in render output, corruption detection in deserialization.
- No filesystem I/O. All tests use in-memory strings.
- No CLI command added.

**Safety boundary**: Pure model and serialization tests only. No file read or write.
No live counter reads. No CLI paths.

**Entry criteria**: Phase 5 design complete, CI green, all existing tests passing.

## Relationship to Existing v2.9 Accounting Types

The ledger design builds on the existing pure types in the `src/accounting/`
module. The mapping between existing types and ledger entries is:

| Existing Type | Ledger Entry Type | Notes |
|--------------|-------------------|-------|
| `InterfaceCounterSnapshot` | `LedgerEntry` with `entry_type: "snapshot"` | Each interface in the snapshot becomes a separate ledger entry. |
| `SessionDelta` | `LedgerEntry` with `entry_type: "delta"` | Each `SessionDeltaRow` becomes a separate ledger entry. `CounterResetWarning` maps to `ResetDetail`. |
| `UsagePreview` | Not stored directly | Derived from ledger entries on read. The preview is a computed view, not a stored entry. |
| `SourceLabel` | `LedgerEntry.source_label` | Direct mapping. |
| `InterfaceCounter` | Embedded in `LedgerEntry` fields | `rx_bytes`, `tx_bytes`, `rx_packets`, `tx_packets` map directly to `InterfaceCounter` fields. |

The ledger is a write-ahead log of snapshots and deltas. Usage previews, quota
decisions, and session summaries are computed by reading ledger entries and applying
the same pure functions already implemented in the v2.9 model layer.

## Ledger Integrity Invariants

When the ledger is implemented in future phases, the following invariants must be
maintained:

1. **Monotonic timestamps**: Entries within a ledger must have non-decreasing
   timestamps. An entry with an older timestamp than the previous entry indicates
   a bug or tampering. The deserialization code should detect and warn about
   out-of-order entries.
2. **Schema version consistency**: All entries in a ledger should share the same
   schema version as the root. If mixed versions are detected, the reader should
   warn but continue processing known fields.
3. **Host ID consistency**: All entries should have the same host_id as the root
   ledger. Entries from a different host indicate that ledger files from multiple
   machines were incorrectly merged.
4. **No negative values**: `rx_bytes`, `tx_bytes`, `combined_bytes`, `rx_packets`,
   `tx_packets` must never be negative. For delta entries, a counter decrease is
   flagged as a reset with delta = 0, not a negative value.
5. **Combined bytes consistency**: `combined_bytes` must equal `rx_bytes + tx_bytes`
   (with saturating arithmetic). If they differ, the entry is flagged as
   inconsistent.
6. **Read-only flag always true**: In v2.9, the `read_only` field must always be
   true. If a v2.9 ledger contains an entry with `read_only: false`, it indicates
   tampering or a bug from a newer version that was incorrectly downgraded.

## Privacy Review Requirements

Before phase 8 (persistence) or any phase that stores per-app attribution
information, a formal privacy review must be conducted covering:

1. **Data minimization**: Does the ledger store only the minimum data needed for its
   stated purpose? Are there fields that could be removed without losing
   functionality?
2. **Access control**: Are file permissions set correctly (0600 for files, 0700
   for directories)? Is there any risk of the ledger being readable by other
   users or processes?
3. **Retention**: Is there a clear retention policy? Are old entries pruned
   automatically or only on explicit user request?
4. **Transparency**: Can the user easily inspect all data stored in the ledger?
   Is there a command to export or delete all ledger data?
5. **Third-party access**: Are there any conditions under which ledger data could
   be transmitted to a third party (e.g., crash reporting, telemetry)? The answer
   must be no for v2.9.
6. **Forensic value**: Does the ledger contain data that could be used to infer
   user behavior, browsing habits, or application usage patterns? If so, this
   must be documented and the user must be informed.

This review is not required for phase 5 (design only) but must be completed before
any filesystem writes occur in phase 8 or later.

## Summary

Phase 5 produces a comprehensive design for the future local usage ledger that
defines the data model, storage boundary, privacy constraints, honesty rules,
corruption handling, and implementation roadmap. No code is written, no files are
created, no tests are modified, and no CLI paths are added. The design is
intentionally forward-looking: it accounts for future quota guard (v3.x) and eBPF
observer (v4.x) integration while keeping v2.9 firmly within the read-only,
no-enforcement, no-persistence safety boundary.
