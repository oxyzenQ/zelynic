# v3.1 Phase 6: CLI Parser Gate Tests

## Status

**Phase 6 — Hidden clap variants with dispatch-level rejection. 29 deterministic gate tests. No new working commands. No behavior change for existing v3.0 commands. No version bump.**

## Source of Truth

- `docs/v3.1-phase-5-cli-gate-design.md` (phase 5 gate design)
- `src/cli.rs` (hidden clap variants, `DESIGN_GATED_DISCLAIMERS`, `render_design_gated_message()`)
- `src/cli/v31_gate_tests.rs` (29 deterministic gate tests)
- `src/commands/mod.rs` (dispatch-level rejection for gated commands)

## 1. Purpose

Phase 6 implements the CLI parser gate tests defined in the phase 5 gate design document.
It registers hidden clap variants for all 7 future v3.1 candidate commands, proves the
parser shape through 29 deterministic tests, and hard-blocks every gated command at the
dispatch layer with a design-gated rejection message containing 10 safety disclaimers.

The goals are fourfold. First, **test the parser shape**: by registering hidden flags
and subcommands in clap, we prove that the future CLI surface exists in the parser and
can be activated when prerequisites are met. Second, **prove dispatch-level rejection**:
every gated command that reaches the dispatch layer is explicitly rejected with a clear
error message — no gated command can execute any functionality. Third, **verify safety
disclaimer output**: the rejection message for every gated command includes all 10 safety
disclaimers, confirming that no live resolver, persistence, enforcement, mutation, eBPF,
or PID movement can occur through these commands. Fourth, **preserve v3.0 behavior**: all
existing v3.0 usage commands (`--sample`, `--sample --json`, `--sample --delta`,
`--sample --delta --json`) remain unchanged in parser behavior and dispatch behavior.

## 2. Implementation Approach

Phase 6 chose the **hidden clap variants with dispatch-level rejection** approach over
pure parser-level rejection tests (option 1 from the task specification). This is the
safest option because:

- **Parser shape is tested**: flags and subcommands exist in clap's model, proving
  the CLI surface is reserved and will compile when activated.
- **Dispatch gate is tested**: every gated command is explicitly matched in the dispatch
  function and rejected with `render_design_gated_message()`.
- **Safety disclaimers are tested**: 13 dedicated tests verify each of the 10 safety
  disclaimers appears in the rejection output.
- **No silent behavior drift**: hidden flags are invisible in `--help` output, so
  existing users never see them, but developers can test the full shape.

### Alternative Considered: Pure Parser-Level Rejection

The pure test-only approach (proving clap rejects unknown flags/subcommands) was
considered but rejected because it does not test the dispatch layer, does not produce
safety disclaimer output, and would need to be rewritten when the commands are
activated. The hidden variant approach tests both parser and dispatch in one pass.

## 3. Changes to CLI Parser (`src/cli.rs`)

### 3.1 Hidden Usage Flags

Four hidden flags were added to the existing `Commands::Usage` variant:

| Flag | clap name | Type | Requires | Purpose |
|------|-----------|------|----------|---------|
| `session` | `--session` | `bool` | `--sample` | Future session-scoped usage report |
| `since_boot` | `--since-boot` | `bool` | `--sample` | Future boot-scoped usage report |
| `usage_interface` | `--interface` | `Option<String>` | `--sample` | Future interface filter |
| `usage_target` | `--target` | `Option<String>` | `--sample` | Future per-target usage report |

All flags use `hide = true` to prevent appearing in `--help` output. All flags use
`requires = "sample"` to enforce the existing v3.0 contract that `--sample` is required.

### 3.2 Hidden Ledger Subcommand

A new `Commands::Ledger` variant was added with `#[command(hide = true)]`:

```
Ledger { command: LedgerCommands }
```

The `LedgerCommands` enum has two hidden subcommands:
- `Inspect { json: bool }` — for future ledger inspection (text and JSON).
- `Export { json: bool }` — for future full ledger data export.

Both use `#[command(hide = true)]` to prevent appearing in `--help`.

### 3.3 Design-Gated Disclaimer Constants

A `DESIGN_GATED_DISCLAIMERS` constant array contains exactly 10 safety disclaimers:

1. `design-gated: this command is not yet implemented`
2. `no live resolver: no process scanning or identity resolution`
3. `no ledger persistence: no file read/write for ledger data`
4. `no filesystem write: no files are written to disk`
5. `no enforcement: no blocking, throttling, or quota active`
6. `no network blocking: no traffic blocking or shaping`
7. `no nft/tc mutation: no nftables or traffic control changes`
8. `no cgroup mutation: no cgroup writes or PID movement`
9. `no eBPF: no eBPF program loading or attachment`
10. `no PID movement: no processes moved between cgroups`

A `render_design_gated_message(command_name)` function produces a multi-line error
message with the command name and all 10 disclaimers.

## 4. Changes to Dispatch (`src/commands/mod.rs`)

### 4.1 Ledger Dispatch Gate

The `Commands::Ledger` variant is matched and always rejected:

```rust
Some(Commands::Ledger { command }) => match command {
    LedgerCommands::Inspect { .. } => {
        Err(anyhow::anyhow!("{}", render_design_gated_message("ledger inspect")))
    }
    LedgerCommands::Export { .. } => {
        Err(anyhow::anyhow!("{}", render_design_gated_message("ledger export")))
    }
}
```

### 4.2 Usage Hidden Flag Gates

The existing `Commands::Usage` match was extended to check for hidden flags:

```rust
Some(Commands::Usage {
    sample: true, json, delta,
    session, since_boot, usage_interface, usage_target,
}) => {
    if session { return Err(render_design_gated_message("usage --session")); }
    if since_boot { return Err(render_design_gated_message("usage --since-boot")); }
    if usage_interface.is_some() { return Err(render_design_gated_message("usage --interface")); }
    if usage_target.is_some() { return Err(render_design_gated_message("usage --target")); }
    usage::handle_usage_sample(json, delta)
}
```

When no hidden flags are set, the existing `handle_usage_sample(json, delta)` is called
unchanged — preserving v3.0 behavior.

## 6. Compatibility Verification

### Protected v3.0 Contracts (Unchanged)

1. `zelynic usage --sample` — produces the same text output with 13 honesty disclaimers.
2. `zelynic usage --sample --json` — produces the same JSON with `schema_version: 1`.
3. `zelynic usage --sample --delta` — produces the same delta text output.
4. `zelynic usage --sample --delta --json` — produces the same delta JSON.

### Protected Rejection Contracts (Unchanged)

1. `zelynic usage` without `--sample` — still rejected by clap (`required = true`).
2. `zelynic usage --delta` without `--sample` — still rejected by clap (`requires = "sample"`).
3. `zelynic usage --json` without `--sample` — still rejected by clap (`requires = "sample"`).

### Hidden Surface (Invisible to Users)

- `zelynic ledger` — hidden, not in `--help`.
- `zelynic usage --session` — hidden, requires `--sample`, rejected at dispatch.
- `zelynic usage --since-boot` — hidden, requires `--sample`, rejected at dispatch.
- `zelynic usage --interface <name>` — hidden, requires `--sample`, rejected at dispatch.
- `zelynic usage --target <target>` — hidden, requires `--sample`, rejected at dispatch.

## 8. Non-Goals

The following are explicitly out of scope for v3.1 phase 6:

- No new working commands. All gated commands are rejected at dispatch.
- No behavior change for existing v3.0 commands.
- No version bump. Version remains `v3.0.1`.
- No persistence implementation. No ledger file writes.
- No live process scanning. No `/proc/<pid>` reads.
- No enforcement, blocking, or quota activation.
- No eBPF program loading or attachment.
- No nft/tc mutation or cgroup mutation.
- No PID movement between cgroups.
- No JSON schema change. v3.0 `schema_version: 1` remains frozen.
- No daemon, watch, or loop mode.
- No git tag, GitHub release, or crate publish.
- No new dependencies added to `Cargo.toml`.
- No modification to existing accounting modules.
