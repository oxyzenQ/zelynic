## 1. Purpose

Phase 17 freezes the Phase 16 export behavior so that future work cannot
accidentally weaken safety, schema validation, or stdout-only boundaries.
This phase adds deterministic tests that prove every error class, every
success field preservation guarantee, and every safety boundary. No new
runtime features are added; the sole purpose is regression protection.

The error matrix documented here covers every rejection path in the
export pipeline: missing flags, path validation failures, JSON parse
failures, schema validation failures, and invariant violations. Each
error class has at least one deterministic test proving it is rejected
with a stable error keyword.

## 2. Activated Command (Unchanged from Phase 16)

```
zelynic ledger export --json --file <PATH>
```

This is the only activated export shape. No other export command
variant is active, and no other variant will be activated without a
dedicated design phase.

### 2.1 Frozen Success Example

```
$ zelynic ledger export --json --file /tmp/zelynic-ledger-valid.json
{"schema_version":1,"created_at":"2026-01-01T00:00:00Z","updated_at":"2026-01-01T00:00:00Z","host_id":"phase17-export-host","entries":[...]}
```

Exit code 0. Output is valid ledger JSON on stdout only. No human
header, no disclaimers, no extra text.

## 3. Frozen Failure Examples

### 3.1 Missing --file

```
$ zelynic ledger export --json
Error: ledger export: --file <PATH> is required for export
```

The export command without --file fails honestly. It does not read any
default path and does not access the filesystem.

### 3.2 Missing --json

```
$ zelynic ledger export --file /tmp/zelynic-ledger-valid.json
Error: ledger export: --json is required for export
```

The export command without --json fails honestly. No human export
output is provided.

### 3.3 Parent Traversal Path

```
$ zelynic ledger export --json --file /tmp/legit/../etc/passwd.json
Error: ledger export --file: parent traversal (..) is not allowed
```

Any `..` component in the path is rejected before any filesystem read.

### 3.4 Suspicious Filename

```
$ zelynic ledger export --json --file /tmp/ledger;rm-rf.json
Error: ledger export --file: suspicious character ';' in filename 'ledger;rm-rf.json'
```

Only `[a-zA-Z0-9._-]` characters are allowed in the filename.

### 3.5 Directory Path

```
$ zelynic ledger export --json --file /tmp
Error: ledger export --file: '/tmp' is not a regular file
```

Directory paths are rejected. Only regular files are accepted.

### 3.6 Symlink Path

```
$ zelynic ledger export --json --file /tmp/symlink.json
Error: ledger export --file: '/tmp/symlink.json' is a symlink; symlinks are not followed
```

Symlinks are detected via `symlink_metadata` and rejected.

### 3.7 Nonexistent File

```
$ zelynic ledger export --json --file /tmp/nonexistent.json
Error: ledger export --file: cannot access '/tmp/nonexistent.json': ...
```

The `cannot access` keyword is stable. The OS-specific detail may vary
by platform.

### 3.8 Unreadable File

Note: unreadable-file testing is not deterministic under all CI
environments because the test runner may run as root or with elevated
permissions that bypass file permission checks. This error class is not
tested with a chmod-based test to avoid flaky failures. The behavior is
implicitly covered by the `std::fs::read_to_string` error path which
produces an honest OS error.

### 3.9 Malformed JSON

```
$ zelynic ledger export --json --file /tmp/bad.json
Error: ledger export --file: invalid ledger in '/tmp/bad.json': ...
```

The `invalid ledger` keyword identifies parse failures.

### 3.10 Unsupported Schema Version

```
$ zelynic ledger export --json --file /tmp/old.json
Error: ledger export --file: invalid ledger in '/tmp/old.json': unsupported schema version: 99 (supported: 1)
```

Only `schema_version: 1` is supported.

### 3.11 Missing Top-Level Required Field

```
$ zelynic ledger export --json --file /tmp/missing.json
Error: ledger export --file: invalid ledger in '/tmp/missing.json': ...
```

Missing `created_at`, `updated_at`, or `host_id` produces an
`invalid ledger` error.

### 3.12 Missing Entry Required Field

```
$ zelynic ledger export --json --file /tmp/noif.json
Error: ledger export --file: invalid ledger in '/tmp/noif.json': ...
```

Missing entry fields (e.g., `interface`) produce an `invalid ledger`
error during deserialization.

### 3.13 Invalid combined_bytes Invariant

```
$ zelynic ledger export --json --file /tmp/cb.json
Error: ledger export --file: invalid ledger in '/tmp/cb.json': safety violation: ...
```

`combined_bytes` must equal `rx_bytes + tx_bytes`.

### 3.14 read_only=false Rejected

```
$ zelynic ledger export --json --file /tmp/rw.json
Error: ledger export --file: invalid ledger in '/tmp/rw.json': ...
```

All entries must have `read_only: true`.

### 3.15 Active enforcement_status Rejected

```
$ zelynic ledger export --json --file /tmp/enf.json
Error: ledger export --file: invalid ledger in '/tmp/enf.json': ... enforcement_status ...
```

`enforcement_status` must be `"inactive/not implemented"`.

### 3.16 Invalid attribution_scope Rejected

```
$ zelynic ledger export --json --file /tmp/attr.json
Error: ledger export --file: invalid ledger in '/tmp/attr.json': ...
```

`attribution_scope` must be `"interface-level only"`.

### 3.17 --output Rejected

```
$ zelynic ledger export --output /tmp/out.json
error: unexpected argument '--output' found
```

The `--output` flag does not exist on the export subcommand.

### 3.18 --overwrite Rejected

```
$ zelynic ledger export --overwrite
error: unexpected argument '--overwrite' found
```

The `--overwrite` flag does not exist on the export subcommand.

## 4. Safety Boundary

The following safety invariants are frozen and must not be weakened by
any future phase without a dedicated design phase. In summary: no output
file, no overwrite, no write, no persistence, no default path read, no
live /proc/sysfs read, no live resolver, no enforcement, and no
nft/tc/cgroup/PID mutation.

### 4.1 No Output File

Export writes to stdout only. There is no `--output <PATH>`, no
`--write <PATH>`, no `--save <PATH>`. Users redirect stdout with shell
redirection (`> output.json`).

### 4.2 No Overwrite

The `--overwrite` flag does not exist. It is not registered in the clap
definition.

### 4.3 No File Write

The export handler contains no file write APIs: no `create_dir`,
`remove_file`, `remove_dir`, `rename`, `fs::copy`, `File::create`,
or `OpenOptions`.

### 4.4 No Persistence / Save

Export does not save ledger state. The persistence module returns
`Blocked` for all operations.

### 4.5 No Default Ledger Path Read

Export requires explicit `--file <PATH>`. No XDG auto-discovery, no
environment variable, no implicit fallback.

### 4.6 No Live /proc/sysfs Read

Export does not read from `/proc/net/dev`, `/sys/class/net/`, or any
other live system interface. The data source is exclusively the
user-specified JSON file.

### 4.7 No Live Resolver

Export does not resolve app identities from live system state.
Attribution is "interface-level only."

### 4.8 No Enforcement

Export does not apply any network quota, rate limit, blocking rule, or
traffic policy. `enforcement_status` is always
"inactive/not implemented."

### 4.9 No nft/tc/cgroup/PID Mutation

Export does not modify nftables rules, tc classes/filters, cgroup
assignments, process state, or any Zelynic limiter state.

## 5. Schema Boundary

### 5.1 Export JSON is Ledger JSON

The export JSON uses the ledger serialization schema
(`schema_version: 1`, with `entries` array, `host_id`, timestamps, etc.).
It is produced by the existing `serialize_ledger_to_json()` function.

### 5.2 Export JSON is Not Inspect JSON

While export JSON and inspect JSON share the same serialization schema
(both use `serialize_ledger_to_json()`), they are semantically different
operations: export reads a user-specified file; inspect may use fixture
data. The difference is in the command and the data source, not the JSON
schema.

### 5.3 Export JSON is Not v3.0 Usage JSON

The export JSON has `schema_version`, `host_id`, `entries`, etc. The
v3.0 usage JSON has `command`, `sample_mode`, `interfaces`, `totals`,
etc. These are completely different schemas. The export command does not
affect the usage JSON schema in any way.

### 5.4 v3.0 Usage JSON Unchanged

The v3.0 usage JSON schema (schema_version 1) produced by
`zelynic usage --sample --delta --json` remains frozen. No field is
added, removed, or renamed.

### 5.5 Ledger Inspect JSON Unchanged

All existing `ledger inspect` behavior remains unchanged:
- `ledger inspect` (fixture text)
- `ledger inspect --json` (fixture JSON)
- `ledger inspect --file <PATH>` (explicit file text)
- `ledger inspect --file <PATH> --json` (explicit file JSON)
