# Zelynic v3.0 Phase 7b: JSON Model Validation / Error-Type Contract Freeze

This document freezes the pure JSON output model and serialization tests from phase 7
before wiring `--json` to the CLI in phase 8. Phase 7b is a **validation-only** phase:
it verifies the JSON model against the phase 6 contract, confirms canonical error type
serialization strings, and documents the frozen state. No Rust behavior is changed.

## Purpose

Phase 7b resolves a documentation ambiguity about error type serialization. The phase 7
implementation log states that `UsageJsonErrorType` variants were fixed to serialize as
`read_error`, `parse_error`, and `unsupported_flag_error`, but also mentions "serde renamed
to snake_case in JSON" which could be misread as implying `read`, `parse`, and
`unsupported_flag` (without the `_error` suffix). This freeze document provides
definitive clarification and confirms the correct canonical strings through both code
inspection and existing test coverage.

## Verification: Error Type Serialization

### Code Inspection

The `UsageJsonErrorType` enum in `src/accounting/usage_json.rs` (lines 124-135) uses
explicit `#[serde(rename = "...")]` attributes on each variant:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub enum UsageJsonErrorType {
    #[serde(rename = "read_error")]
    Read,
    #[serde(rename = "parse_error")]
    Parse,
    #[serde(rename = "unsupported_flag_error")]
    UnsupportedFlag,
}
```

These rename attributes control both serialization and deserialization. The serde
attributes are explicit string literals, not derived from variant names. This means the
JSON output is guaranteed to use the exact strings specified in the rename attributes,
regardless of the Rust variant names (`Read`, `Parse`, `UnsupportedFlag`).

### Canonical Error Type Strings

The canonical error type strings serialized in JSON output are:

| Rust Variant | JSON Serialization | Renamed Via |
|-------------|-------------------|-------------|
| `UsageJsonErrorType::Read` | `"read_error"` | `#[serde(rename = "read_error")]` |
| `UsageJsonErrorType::Parse` | `"parse_error"` | `#[serde(rename = "parse_error")]` |
| `UsageJsonErrorType::UnsupportedFlag` | `"unsupported_flag_error"` | `#[serde(rename = "unsupported_flag_error")]` |

These are the **only** valid error type strings in v3.0 JSON output. No other
serialization form is possible given the explicit rename attributes.

### Existing Test Coverage

The following existing tests in `src/accounting/tests/usage_json.rs` already prove
correct error type serialization:

1. **`serializes_read_error_json`** (line 267): Asserts `json.contains("\"read_error\"")`.
   The error message is "read error: permission denied" (with space, not underscore), so
   the `"read_error"` string can only originate from the `type` field value.

2. **`serializes_parse_error_json`** (line 283): Asserts `json.contains("\"parse_error\"")`.
   The error message is "parse error: missing colon after interface name", so the
   `"parse_error"` string can only originate from the `type` field value.

3. **`serializes_unsupported_flag_error_json`** (line 298): Asserts
   `json.contains("\"unsupported_flag_error\"")`. The error message uses "unsupported flag"
   (with space), so the `"unsupported_flag_error"` string can only originate from the
   `type` field value.

4. **`round_trips_error_json`** (line 344): Full serialize-then-deserialize cycle with
   equality assertion. If the serialized form used any string other than what the
   deserializer expects (which is controlled by the same `#[serde(rename = "...")]`
   attribute), the round-trip would produce a different type or fail entirely.

5. **`error_type_display`** (line 567): Asserts `Display` trait output matches the
   canonical strings: `Read.to_string() == "read_error"`,
   `Parse.to_string() == "parse_error"`,
   `UnsupportedFlag.to_string() == "unsupported_flag_error"`.

### Verification Conclusion

The error type serialization is **correct and frozen**. No Rust behavior changes are
needed. The `#[serde(rename = "...")]` attributes provide explicit, literal control
over the JSON output strings. The existing tests provide multi-vector proof:

- Substring tests prove the exact string appears in serialized output.
- Round-trip tests prove structural consistency between serialization and deserialization.
- Display tests prove the canonical string representation.

Phase 7b confirms: the potential ambiguity about "serde renamed to snake_case" does not
apply to the error type enum. The error type enum uses explicit rename attributes with
the full `_error` suffix, not automatic snake_case conversion.

### sampled_at Policy

The `sampled_at` field follows the strict caller-provided-only policy:

- **Omitted when `None`**: The `#[serde(skip_serializing_if = "Option::is_none")]`
  attribute ensures the field is completely absent from JSON output when the caller
  does not provide a timestamp. It is not serialized as `null`.
- **Included when `Some(value)`**: The field appears as `"sampled_at": "value"`
  exactly when the caller provides a timestamp string.
- **No silent wall-clock generation**: The model functions never call
  `SystemTime::now()`, `Instant::now()`, or any other time API. The timestamp is
  passed exclusively through the `sampled_at` parameter by the caller.

This policy is tested by:
- `omits_sampled_at_when_none` and `sampled_at_absent_from_json_when_none`
- `includes_sampled_at_when_passed` and `sampled_at_present_in_json_when_some`
- `does_not_generate_timestamp_silently`
