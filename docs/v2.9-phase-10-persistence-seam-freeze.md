# Zelynic v2.9 Phase 10: Persistence Seam Freeze / Non-Exposure Audit

This document freezes and audits the v2.9 persistence boundary established in
phases 8 and 9. Phase 10 is **docs/report only**: no Rust code is modified, no
tests are added, and no runtime behavior changes. The purpose is to formally
document that the persistence seam remains hard-blocked, verify that no
filesystem I/O APIs are exposed, confirm non-exposure of persistence modules
from CLI or runtime paths, and establish explicit entry criteria that must be
satisfied before any future phase may implement actual filesystem persistence.

## Phase Summary

Phases 5 through 9 established the following persistence foundation:

- **Phase 5**: Local ledger design document — schema, storage boundary, privacy
  constraints, honesty rules, atomic write strategy, corruption handling,
  rotation/cleanup, permission expectations, and future implementation roadmap.
  Design-only phase with no Rust code changes.

- **Phase 6**: Pure ledger model and JSON serialization tests — `Ledger`,
  `LedgerEntry`, `ResetDetail`, and `LedgerError` types with
  `serialize_ledger_to_json()`, `deserialize_ledger_from_json()`, and
  `render_ledger_summary()`. All operations are pure in-memory; no filesystem I/O
  occurs. 33 tests added.

- **Phase 7**: Ledger inspect/render model — `LedgerInspect` type with
  `build_ledger_inspect()` and `render_ledger_inspect()`. Computes aggregate
  statistics from a `Ledger` with deterministic interface ordering and
  saturating arithmetic. No filesystem persistence occurs. 30 tests added.

- **Phase 8**: Persistence path design and safe-path model — `LedgerPathPlan`
  and `PathStatus` types with `build_ledger_path_plan()` and
  `render_ledger_path_plan()`. Validates future ledger persistence paths using
  pure string-based analysis. No filesystem access, no canonicalization, no
  `std::fs` APIs. 43 tests added.

- **Phase 9**: Persistence I/O contract and hard-blocked seam —
  `LedgerPersistencePlan`, `PersistenceOperation`, `PersistenceStatus`, and
  `PersistenceError` types with `build_ledger_read_plan()`,
  `build_ledger_write_plan()`, `build_ledger_persistence_plan()`, and
  `render_ledger_persistence_plan()`. All operations return hard-blocked status
  because persistence is not implemented in v2.9. No `std::fs` APIs. No filesystem
  I/O. No directory/file creation/removal. 34 tests added.

## Persistence Freeze Guarantees

The following guarantees are established by the current codebase and must remain
true until explicitly relaxed in a future implementation phase:

| Guarantee | Current Value | Enforced By |
|------------|---------------|-------------|
| `persistence_status` | Always `Blocked` or `Rejected` | `build_persistence_plan()` |
| `persistence_enabled` | `false` | Hardcoded in struct construction |
| `filesystem_read_performed` | `false` | Hardcoded in struct construction |
| `filesystem_write_performed` | `false` | Hardcoded in struct construction |
| `directory_create_performed` | `false` | Hardcoded in struct construction |
| `file_create_performed` | `false` | Hardcoded in struct construction |
| `file_remove_performed` | `false` | Hardcoded in struct construction |
| `state_mutation_performed` | `false` | Hardcoded in struct construction |
| `model_only` | `true` | Hardcoded in struct construction |
| Ledger file loaded from disk | Never | No `std::fs::read` calls |
| Ledger file saved to disk | Never | No `std::fs::write` calls |
| Atomic replace completed | Never | Hard-blocked seam |
| Backup completed | Never | Hard-blocked seam |

No `std::fs` read, write, create, or remove APIs are used in any accounting
persistence module (`ledger_path.rs` or `ledger_persistence.rs`). The modules
perform only string-based validation and construct in-memory model structs.

## Non-Exposure Audit

### Internal Model Only

- `ledger_path` module (`src/accounting/ledger_path.rs`) is registered as
  `pub(crate)` in `src/accounting/mod.rs`. It is not exposed outside the
  accounting crate module. No CLI command calls `build_ledger_path_plan()` or
  `render_ledger_path_plan()`.

- `ledger_persistence` module (`src/accounting/ledger_persistence.rs`) is
  registered as `pub(crate)` in `src/accounting/mod.rs`. It is not exposed
  outside the accounting crate module. No CLI command calls
  `build_ledger_persistence_plan()`, `build_ledger_read_plan()`,
  `build_ledger_write_plan()`, or `render_ledger_persistence_plan()`.

### No CLI Path

- No CLI subcommand, flag, or option references the persistence seam.
- `src/cli.rs` has no imports from `ledger_path` or `ledger_persistence`.
- The accounting module itself is internal (`mod accounting;` in main) with no
  public re-exports to CLI-facing code paths.

### No Runtime Path

- No runtime code path (systemd wrapper, scope runner, strict limiter backend)
  calls the persistence seam.
- The persistence modules are used only in tests (`src/accounting/tests/`).
- No background daemon, scheduler, or async task invokes persistence functions.

### No Filesystem I/O APIs

- `ledger_path.rs`: No `use std::fs` statement. No `std::fs::metadata()`,
  `std::fs::canonicalize()`, `std::fs::read()`, `std::fs::write()`, or any
  other filesystem API call. All validation is string-based.

- `ledger_persistence.rs`: No `use std::fs` statement. No filesystem API calls.
  All persistence plans are constructed in-memory with hardcoded false flags and
  blocked status.

### Strict Limiter Unchanged

- The existing `zelynic strict` path remains the only validated active limiter.
- v2.9 accounting code does not modify strict limiter behavior, nftables rules,
  tc classes, or cgroup membership.

### Accounting Remains Model/Read-Only

- All accounting modules (interface_counters, usage_preview, session_delta,
  ledger, ledger_inspect, ledger_path, ledger_persistence) are pure model
  and rendering code.
- No accounting module performs filesystem writes, network blocking, quota
  enforcement, or state mutation.

### No Active Guards

- No quota guard is active. The `Ledger` model has `enforcement_status:
  "inactive/not implemented"` on every entry.
- No background data guard is active. This feature is designed but not
  implemented; it requires per-app attribution (deferred to v3.x/v4.x).

## Future Phase Entry Criteria Before Actual Persistence

Any future phase that implements actual filesystem persistence must satisfy all
of the following entry criteria before any `std::fs` call is added to the
accounting persistence modules:

1. **Separate explicit implementation phase required**: Actual filesystem
   persistence must be implemented in a dedicated phase with its own design
   document, test plan, and safety review. It must not be added as a side effect
   of another phase.

2. **Safe path plan must be accepted**: The `LedgerPathPlan` must return
   `PathStatus::Accepted` for the target path. Path validation (string-based,
   no filesystem access) must pass: no empty base, no empty filename, no
   absolute filename, no parent traversal, no outside-namespace path, no
   suspicious filename characters.

3. **Persistence seam must be explicitly unblocked**: The `LedgerPersistencePlan`
   construction must be modified to allow non-blocked status when an explicit
   enable flag is set. The default must remain blocked.

4. **Atomic write strategy reviewed**: The atomic write strategy defined in the
   phase 5 design (temp file + fsync + rename + verify) must be reviewed and
   approved for implementation. The review must cover crash safety, disk full
   handling, and permission errors.

5. **Corruption handling reviewed**: The corruption handling strategy (schema
   validation, backup fallback, no silent recovery) must be reviewed and approved.
   Entry-level validation and fallback-to-backup behavior must be specified.

6. **Schema migration reviewed**: The schema versioning strategy (read in
   compatibility mode, explicit migration command, no auto-upgrade) must be
   reviewed. Forward and backward compatibility must be addressed.

7. **Privacy review completed**: A formal privacy review must be completed
   covering data minimization, access control (0600/0700 permissions),
   retention policy, user transparency, no third-party transmission, and
   forensic value of stored data. This review is defined in the phase 5 design
   document under "Privacy Review Requirements."

8. **No per-app attribution claim**: The ledger must not claim per-app
   attribution. Interface-level aggregate counters only. The `target_app`,
   `target_cgroup`, and `target_pid` fields must remain null in v2.9.

9. **No quota/enforcement claim**: The ledger must not claim that quota
   enforcement or traffic blocking is active. `enforcement_status` must remain
   `"inactive/not implemented"`.

10. **Exact file path reviewed before write**: The exact file path must be
    reviewed and approved before any filesystem write occurs. The path must be
    under `$XDG_DATA_HOME/zelynic/` (typically `~/.local/share/zelynic/`) and
    must not be under `/sys/fs/cgroup`, `/proc`, or any system directory.

11. **Explicit operator confirmation before first filesystem write**: The first
    filesystem write must require explicit operator confirmation (e.g., a
    `--confirm` flag or interactive prompt). No silent writes. No automatic
    writes on startup.
