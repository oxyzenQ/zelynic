## Summary

Phase 3 aligns the existing v2.9 ledger model (`Ledger`, `LedgerEntry`) with
the v3.1 identity model (`TargetIdentity`, `ResolvedUsageTarget`,
`UsageAttributionScope`) without enabling persistence, runtime resolution, CLI
wiring, or any changes to existing behavior. This is a pure model/alignment
phase — no new persistence write path is enabled, no runtime resolver is added,
no enforcement fields become active, and the existing v3.0 usage JSON schema
remains unchanged.

## Alignment Decision

### Chosen Approach: Separate Adapter Module

The smallest safe change is a **separate pure adapter module** that creates
identity attachments alongside ledger entries without modifying the existing
ledger schema. This approach was chosen because:

1. **Existing `LedgerEntry` serde schema must remain unchanged.** Adding optional
   identity fields to `LedgerEntry` would change its JSON serialization format,
   breaking backward compatibility with existing ledger JSON files.

2. **Existing safety invariants must remain intact.** The ledger validates
   `attribution_scope == "interface-level only"` and `enforcement_status ==
   "inactive/not implemented"` on deserialization. Adding identity fields that
   imply per-app attribution would violate these invariants.

3. **The attachment is a model-only construct.** The `LedgerIdentityAttachment`
   wraps a cloned `LedgerEntry` with optional `ResolvedUsageTarget` alongside it.
   The original entry is never mutated. The attachment exists only in memory.

4. **No duplicate ledger system.** The new module imports and reuses existing
   types from both `ledger` and `identity` modules. It does not re-define any
   ledger or identity types.

### Alternative Approaches Considered

| Approach | Rejected Because |
|----------|-----------------|
| Add optional identity fields to `LedgerEntry` | Would change serde JSON format, breaking backward compatibility. Would violate `attribution_scope` safety invariant. |
| Create a new `IdentityLedger` type parallel to `Ledger` | Would duplicate the entire ledger system. Unnecessary — a lightweight adapter suffices. |
| Modify `LedgerInspect` to include identity | Would change the inspect model's public API and render output format. Premature — identity is not yet available at runtime. |

## Model Types Added

| Type | Module | Description |
|------|--------|-------------|
| `LedgerIdentityAttachment` | `src/accounting/ledger_identity.rs` | Wraps a cloned `LedgerEntry` with optional `ResolvedUsageTarget` and `IdentityHonesty`. `identity: None` means interface-level only. `identity: Some(...)` means best-effort attribution is provided. |
| `build_interface_only_attachment(entry)` | `src/accounting/ledger_identity.rs` | Creates attachment with `InterfaceOnly` scope derived from the entry's interface name. Detects loopback (`"lo"`). |
| `build_identity_attachment(entry, identity)` | `src/accounting/ledger_identity.rs` | Creates attachment with a pre-built `ResolvedUsageTarget` (process/cgroup best-effort). |
| `build_no_identity_attachment(entry)` | `src/accounting/ledger_identity.rs` | Creates attachment with `identity: None` — the default case. |
| `render_ledger_identity_attachment(attachment)` | `src/accounting/ledger_identity.rs` | Human-readable render with best-effort disclaimers. When identity is absent: "interface-level only (no per-app identity)". When present: includes scope, identity fields, and "Identity attribution is best-effort." |
| `render_identity_summary(attachments)` | `src/accounting/ledger_identity.rs` | Summary for multiple attachments: count breakdown, attribution scopes. Includes "All identity attribution is best-effort." |
