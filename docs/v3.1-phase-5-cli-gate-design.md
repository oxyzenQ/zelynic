## 1. Purpose

Phase 5 defines future CLI command gates for v3.1 App Identity + Network Ledger
workflows **before** any implementation begins. This document serves as the
authoritative gate specification for every future command that will expose ledger
inspect, identity reports, or extended usage modes to the CLI surface.

The goals are threefold. First, **prevent accidental behavior drift**: by
documenting every candidate command and its activation prerequisites now, we
avoid ad-hoc CLI changes that could silently alter existing v3.0 usage behavior
or introduce ambiguity in command semantics. Second, **keep v3.0 usage behavior
stable**: the existing `zelynic usage --sample` command family must remain
exactly as-is, and any future `zelynic usage` flags or subcommands must be
additive without breaking backward compatibility. Third, **establish hard safety
gates** that every future implementation phase must satisfy before activating
any new CLI surface — no persistence writes, no enforcement, no mutation, no
background daemon, no hidden I/O.

This phase produces a design document only. No Rust code is changed, no CLI
flags are registered, no clap subcommands are added, and no runtime behavior
changes.

## 2. Current CLI Baseline

The v3.0.1 CLI exposes the following usage commands and nothing else in the
usage/ledger/identity domain:

### Active Commands (v3.0.1)

| Command | Source | Output | Scope |
|---------|--------|--------|-------|
| `zelynic usage --sample` | `/proc/net/dev` (single read) | Human text | Interface-level only |
| `zelynic usage --sample --json` | `/proc/net/dev` (single read) | JSON (`schema_version: 1`) | Interface-level only |
| `zelynic usage --sample --delta` | `/proc/net/dev` (two reads, 1s wait) | Human text | Interface-level delta |
| `zelynic usage --sample --delta --json` | `/proc/net/dev` (two reads, 1s wait) | JSON (`schema_version: 1`) | Interface-level delta |

### Rejected Commands (v3.0.1)

| Command | Rejection Reason |
|---------|-----------------|
| `zelynic usage` (no flags) | clap: `--sample` is required |
| `zelynic usage --json` (without `--sample`) | clap: `--json` requires `--sample` |
| `zelynic usage --delta` (without `--sample`) | clap: `--delta` requires `--sample` |

### CLI Surface That Does NOT Exist (v3.0.1)

- No `zelynic ledger` subcommand.
- No `zelynic ledger inspect` command.
- No `zelynic ledger export` command.
- No `zelynic usage --session` flag.
- No `zelynic usage --since-boot` flag.
- No `zelynic usage --interface <name>` flag.
- No `zelynic usage --target <target>` flag.
- No app identity CLI of any kind.
- No JSON output with `schema_version: 2`.

### Current JSON Schema (v3.0.1, Frozen)

The v3.0.1 JSON output uses `schema_version: 1` with the following top-level
fields: `schema_version`, `command`, `source_path`, `source_label`, `sampled_at`
(omitted when null), `interfaces`, `error`, `totals`, `honesty`, `warnings`.
This schema MUST NOT change in v3.1. Any future ledger/identity report JSON
must use a separate command and a separate schema version.

## 3. Future Command Candidates

The following commands are candidates for future implementation. Each is
classified by its dependency on existing infrastructure and future modules.

### 3.1 Extended Usage Commands

#### `zelynic usage --session`

**Description**: Report network usage accumulated since the start of the current
zelynic session (i.e., the current CLI invocation or a tracked session window).

**Dependencies**:
- **Existing live `/proc/net/dev` sample**: YES — requires at least one live
  read to gather interface counters.
- **Pure model only**: NO — needs live data.
- **Future ledger persistence**: YES — meaningful session reports require
  accumulated ledger entries across time, which in turn require persistence.
- **Future identity resolver**: NO — can report interface-level session usage
  without identity. Identity is additive.
- **Future report model**: YES — would use `LedgerIdentityReport` or a
  session-specific variant to produce the output.

**Classification**: `requires-persistence`, `read-only`, `future-gated-implementation`.

**Gate requirements**: Persistence must be implemented and explicitly enabled by
the user before this command produces meaningful output. Without persistence,
this command MUST either be rejected or return an empty/session-only response.

#### `zelynic usage --since-boot`

**Description**: Report network usage accumulated since the current system boot,
using `boot_id` from `/proc/sys/kernel/random/boot_id` as the partition key.

**Dependencies**:
- **Existing live `/proc/net/dev` sample**: YES — needs at least one live read.
- **Pure model only**: NO — needs live data and accumulated ledger.
- **Future ledger persistence**: YES — requires ledger entries spanning the boot
  session, which requires persistence across invocations.
- **Future identity resolver**: NO — interface-level boot usage is meaningful
  without identity.
- **Future report model**: YES — would use the report model with boot-scoped
  filtering.

**Classification**: `requires-persistence`, `read-only`, `future-gated-implementation`.

**Gate requirements**: Same as `--session`. Requires persistence to be
implemented. Additionally requires `boot_id` reading from
`/proc/sys/kernel/random/boot_id` — a new `/proc` read path that must be
explicitly documented and gate-tested.

#### `zelynic usage --interface <name>`

**Description**: Filter usage output to a specific network interface by name
(e.g., `--interface wlp1s0`).

**Dependencies**:
- **Existing live `/proc/net/dev` sample**: YES — reads the same source, but
  filters output to one interface.
- **Pure model only**: NO — needs live data.
- **Future ledger persistence**: NO — can filter a single live sample or delta
  without persistence.
- **Future identity resolver**: NO — interface filtering is orthogonal to
  identity.
- **Future report model**: NO — simple post-processing filter on existing output.

**Classification**: `read-only`, `future-gated-implementation`, `no-persistence-required`.

**Gate requirements**: Must not alter the JSON schema. Must not alter the text
output format for other interfaces. The `--interface` flag must be compatible
with `--sample` and optionally with `--json` and `--delta`. If the named
interface does not exist in the sample, output must indicate this honestly
(rather than silently omitting).

#### `zelynic usage --target <target>`

**Description**: Report usage attributed to a specific target (by process name,
PID, cgroup path, or systemd unit).

**Dependencies**:
- **Existing live `/proc/net/dev` sample**: YES — needs live data as the source
  for counters.
- **Pure model only**: NO — needs live data plus identity resolution.
- **Future ledger persistence**: YES — meaningful per-target usage requires
  accumulated attributed ledger entries.
- **Future identity resolver**: YES — requires a live process scanner that reads
  `/proc` to resolve PIDs to process names, executable paths, and cgroup
  memberships.
- **Future report model**: YES — would use `LedgerIdentityReport` filtered by
  target identity.

**Classification**: `requires-persistence`, `requires-resolver`, `read-only`,
`future-gated-implementation`.

**Gate requirements**: The most complex candidate. Requires both persistence
(phase 8+) and identity resolution (not yet phased). This command MUST be the
last to be implemented. It must NOT be activated until identity resolution is
fully designed, gate-tested, and proven safe (read-only, no PID movement, no
cgroup mutation).

### 3.2 Ledger Commands

#### `zelynic ledger inspect`

**Description**: Display a human-readable summary of the accumulated network
usage ledger. Shows entry counts, per-interface totals, attribution scope
breakdown, and all applicable honesty disclaimers.

**Dependencies**:
- **Existing live `/proc/net/dev` sample**: NO — operates on accumulated ledger
  data, not live samples. May optionally include a live sample for comparison,
  but the core command reads from the ledger model.
- **Pure model only**: YES — uses `LedgerInspect` from
  `src/accounting/ledger_inspect.rs` to compute aggregate statistics from a
  `Ledger` struct. No live system reads required for the model itself.
- **Future ledger persistence**: YES — requires a populated `Ledger` struct,
  which in turn requires persistence to have accumulated entries. Without
  persistence, this command shows an empty ledger.
- **Future identity resolver**: NO — `LedgerInspect` is interface-level only.
  Identity-enhanced inspect would be a separate mode or command.
- **Future report model**: YES — if identity-enhanced inspect is desired, would
  consume `LedgerIdentityReport`.

**Classification**: `future-gated-implementation`, `read-only`,
`requires-persistence`, `pure-model-available`.

**Gate requirements**: The pure `LedgerInspect` model already exists and is
tested. CLI wiring requires a `Ledger` struct with actual entries. Before
persistence is implemented, this command may be activated in a preview mode
using in-memory fixture data (phase 7 scope).

#### `zelynic ledger inspect --json`

**Description**: Display the ledger inspect summary as machine-readable JSON.

**Dependencies**: Same as `zelynic ledger inspect`, plus JSON serialization.

**Classification**: `future-gated-implementation`, `read-only`,
`requires-persistence`, `pure-model-available`.

**Gate requirements**: JSON output MUST use a separate schema version
(`schema_version: 2` or higher). It MUST NOT alter the existing v3.0 usage
JSON schema (`schema_version: 1`). The JSON structure must include all honesty
flags and safety disclaimers. This JSON schema is defined by the phase 1
contract (section 9) but is NOT implemented in this phase.

#### `zelynic ledger export --json`

**Description**: Export the full ledger data (all entries, not just the inspect
summary) as JSON for external tools, scripts, or long-term archival.

**Dependencies**:
- **Existing live `/proc/net/dev` sample**: NO — operates on accumulated data.
- **Pure model only**: YES — uses `serialize_ledger_to_json()` from
  `src/accounting/ledger.rs`.
- **Future ledger persistence**: YES — requires a populated ledger from
  persistence.
- **Future identity resolver**: NO — export is data-dumping, not identity
  analysis.
- **Future report model**: NO — exports raw ledger entries, not report summaries.

**Classification**: `future-gated-implementation`, `read-only`,
`requires-persistence`, `pure-model-available`, `writes-nothing-to-filesystem`.

**Gate requirements**: Export prints JSON to stdout. It does NOT write files to
disk (the user can redirect stdout). The JSON uses the existing `Ledger` serde
schema (`schema_version: 1` for ledger entries). No new schema version needed
for raw ledger export — this is a dump of the existing ledger format.

## 4. Command Status Matrix

The following matrix classifies every candidate command by its properties.

| Command | Status | Requires Persistence | Requires Resolver | Read-Only | Writes Files | Enforcement | Pure Model Available |
|---------|--------|----------------------|-------------------|-----------|-------------|-------------|---------------------|
| `zelynic usage --session` | future gated | YES | NO | YES | NO | NO | NO (needs live + persistence) |
| `zelynic usage --since-boot` | future gated | YES | NO | YES | NO | NO | NO (needs live + persistence) |
| `zelynic usage --interface <name>` | future gated | NO | NO | YES | NO | NO | NO (needs live sample filter) |
| `zelynic usage --target <target>` | future gated | YES | YES | YES | NO | NO | NO (needs live + persistence + resolver) |
| `zelynic ledger inspect` | future gated | YES | NO | YES | NO | NO | YES (`LedgerInspect` exists) |
| `zelynic ledger inspect --json` | future gated | YES | NO | YES | NO | NO | YES (needs JSON schema design) |
| `zelynic ledger export --json` | future gated | YES | NO | YES | NO | NO | YES (`serialize_ledger_to_json` exists) |

**Status definitions**:
- `design-only` — documented in this phase, no code.
- `future gated` — requires future implementation gated by prerequisites.
- `requires persistence` — needs ledger persistence to be implemented first.
- `requires resolver` — needs live app identity resolution to be implemented first.
- `read-only` — the command never writes to disk, never mutates system state.
- `writes files` — the command produces files on disk (none of the candidates do).
- `enforcement` — the command activates enforcement, blocking, or quota (none do).

## 5. Compatibility Policy

The v3.0 usage behavior is a frozen contract. The following scripts and
automations MUST continue to work exactly as they do today:

### Protected Command Contracts

1. **`zelynic usage --sample`** — produces human text output with all 13 honesty
   disclaimers. The text format, disclaimer wording, and interface-level-only
   scope MUST NOT change.

2. **`zelynic usage --sample --json`** — produces JSON with `schema_version: 1`,
   `command: "usage --sample --json"`, the full interface array, totals, honesty
   object with 12 boolean flags, and warnings array. The JSON schema MUST NOT
   change.

3. **`zelynic usage --sample --delta`** — produces human text delta output.
   The two-sample protocol, 1-second wait, delta computation, and output format
   MUST NOT change.

4. **`zelynic usage --sample --delta --json`** — produces delta JSON with
   `schema_version: 1`, `sample_mode: "delta"`, `sample_count: 2`,
   `read_count: 2`. The JSON schema MUST NOT change.

### Protected Rejection Contracts

1. **`zelynic usage`** without `--sample` MUST remain rejected by clap.
2. **`zelynic usage --json`** without `--sample` MUST remain rejected by clap.
3. **`zelynic usage --delta`** without `--sample` MUST remain rejected by clap.

### Future Flag Policy

New flags on the `usage` subcommand (`--session`, `--since-boot`, `--interface`,
`--target`) are subject to the following rules:

1. **No flag silently changes existing behavior.** Adding `--session` to
   `zelynic usage --session --sample` MUST produce session-scoped output that
   is clearly distinct from the single-sample output of
   `zelynic usage --sample` alone. The JSON schema for session output MUST use
   a separate `schema_version`.

2. **`--sample` remains required.** No future flag removes the `--sample`
   requirement. `zelynic usage --session` without `--sample` MUST be rejected
   by clap.

3. **Future flags are additive.** `zelynic usage --sample --json` continues to
   produce the same output as today regardless of what other flags are added
   to the parser in the future. New flags do not inject new fields into the
   v3.0 JSON schema.

4. **New subcommands do not alter old ones.** Adding `zelynic ledger inspect`
   does not change `zelynic usage --sample` in any way. The `ledger` subcommand
   is a separate top-level command, not a mode of `usage`.

### Decision: `zelynic usage` vs New Subcommand

The design must explicitly address whether future `zelynic usage --session`
becomes a ledger report command, remains sample-gated, or requires a new
subcommand to avoid ambiguity.

**Decision**: Future `zelynic usage` commands remain sample-gated. The `--sample`
flag is always required for the `usage` subcommand. This preserves the existing
clap contract and avoids ambiguity between live sampling and ledger reporting.

For ledger-specific operations, a new `zelynic ledger` top-level subcommand is
used. This cleanly separates the two domains:

- `zelynic usage --sample [--json] [--delta] [--interface <name>]` — live
  sampling domain. Always reads `/proc/net/dev`. Always requires `--sample`.
  JSON always uses `schema_version: 1`.

- `zelynic ledger inspect [--json]` — ledger reporting domain. Reads from
  accumulated ledger data (requires persistence). JSON uses `schema_version: 2`
  or higher.

This separation ensures that:
- Scripts relying on `zelynic usage --sample --json` are never surprised by a
  different JSON schema.
- `zelynic ledger inspect` is clearly a separate operation with its own output
  contract.
- No flag combination on `usage` silently switches from live sampling to ledger
  reporting.

## 6. JSON Output Policy

### v3.0 JSON Schema (Frozen)

The v3.0 usage JSON schema (`schema_version: 1`) is frozen. No field is added,
removed, renamed, or retyped. The schema covers:

```json
{
  "schema_version": 1,
  "command": "usage --sample --json",
  "source_path": "/proc/net/dev",
  "source_label": "live_proc_net_dev",
  "sampled_at": null,
  "interfaces": [...],
  "error": null,
  "totals": {...},
  "honesty": {...},
  "warnings": [...]
}
```

### Future v3.1 Ledger/Report JSON (Separate Schema)

Any future v3.1 JSON output that includes identity attribution, ledger
summaries, or report data MUST use a different `schema_version` (2 or higher)
and MUST be produced by a different command (e.g., `zelynic ledger inspect
--json` or `zelynic usage --session --sample --json` with a separate JSON
rendering path).

The v3.1 JSON schema direction is defined in the phase 1 contract (section 9)
and includes:

```json
{
  "schema_version": 2,
  "command": "ledger inspect --json",
  "report_mode": "since_boot",
  "source": {...},
  "session": {...},
  "interfaces": [...],
  "targets": [...],
  "totals": {...},
  "warnings": [],
  "honesty": {...},
  "error": null
}
```

### Policy Rules

1. **No field injection**: Adding `targets` or `report_mode` to the v3.0 schema
   is prohibited. These fields appear only in the v3.1 schema.

2. **Schema version is the discriminator**: Consumers can reliably detect the
   JSON format by reading `schema_version`. Version 1 is always v3.0 usage
   format. Version 2+ is always v3.1 ledger/report format.

3. **Honesty flags are extended, not modified**: The v3.1 honesty object includes
   additional identity-specific flags (attribution_is_best_effort,
   per_app_attribution_may_be_partial, etc.) that do not exist in the v3.0
   honesty object. The v3.0 honesty flags (12 boolean flags) remain unchanged.

4. **Error format is consistent**: Both v3.0 and v3.1 JSON use the same error
   pattern (error object with type and message, mutually exclusive with data).

## 7. Safety Gates

The following hard gates apply to ALL future implementation phases for v3.1 CLI
commands. No command may be activated without satisfying all applicable gates.

### 7.1 Persistence Gate

- **No persistence unless explicit.** Ledger file writes occur ONLY when the
  user invokes a command that explicitly requests persistence (e.g., a future
  `zelynic ledger record --sample` or similar). No command silently writes a
  ledger file as a side effect.
- **File path is user-visible.** Any ledger file path MUST be communicated in
  CLI output. No hidden files, no unexpected locations.
- **No silent writes.** Every persistence operation produces a visible log
  line or output message confirming the write.

### 7.2 Daemon and Background Gate

- **No background daemon.** Zelynic does not run as a background process,
  systemd service, or cron job. All commands are single-shot CLI invocations.
- **No auto-start.** Zelynic does not auto-start on boot, login, network
  change, or any system event. The user must invoke zelynic explicitly.
- **No watch/loop mode.** All commands execute once and exit. No polling loop,
  no interval timer, no long-running process.

### 7.3 Enforcement and Mutation Gate

- **No enforcement.** Zelynic does not block, throttle, limit, or shape
  network traffic. No quota enforcement. No permission mode (block/allow).
- **No nft/tc mutation.** Zelynic does not modify nftables rules, tc qdiscs,
  or tc filters. The `zelynic strict` command is the only validated limiter path
  and is outside the v3.1 scope.
- **No cgroup mutation.** Zelynic does not write to `cgroup.procs`, move PIDs
  between cgroups, or create/destroy cgroups.
- **No PID movement.** Zelynic does not move processes between cgroups or
  namespaces.
- **No eBPF.** Zelynic does not load, attach, or use eBPF programs.
- **No quota guard.** Zelynic does not enforce byte limits, rate limits, or
  time-based quotas.

### 7.4 I/O Gate

- **No hidden writes.** Every filesystem write is triggered by an explicit user
  command and produces visible output confirming the write.
- **No hidden reads.** Every filesystem read is documented. The only allowed
  live read paths in v3.0 are `/proc/net/dev`. Future v3.1 reads (e.g.,
  `/proc/sys/kernel/random/boot_id`, `/proc/<pid>/comm`) MUST be explicitly
  documented and gate-tested before activation.

### 7.5 Identity Resolver Gate

- **Read-only resolution.** Any future identity resolver reads `/proc` files
  only. It never writes to `/proc`, `/sys`, cgroups, or any filesystem.
- **Best-effort only.** Identity resolution is always best-effort. Partial
  resolution is acceptable and MUST be reported honestly.
- **No PID tracking.** Zelynic does not track PIDs across time, remember
  previous resolutions, or build a process database.

### 7.6 Command Activation Gate Checklist

Before any future command is activated, the following checklist MUST be
satisfied:

- [ ] Phase-specific design document exists and is reviewed.
- [ ] Pure model for the command exists and is tested (unit tests pass).
- [ ] CLI wiring code is reviewed and does not alter existing commands.
- [ ] Output format (text and/or JSON) is specified and tested.
- [ ] All honesty/safety disclaimers are present in output.
- [ ] No existing JSON schema is modified.
- [ ] No existing v3.0 command behavior is altered.
- [ ] No enforcement, blocking, or mutation is introduced.
- [ ] No persistence unless explicitly gated.
- [ ] No eBPF, no nft/tc, no cgroup mutation, no PID movement.
- [ ] `./build.sh check-all` passes.
- [ ] All new tests pass.
- [ ] LOC policy respected (every file under 1000 LOC).

## 8. Suggested Implementation Sequence

The following phased sequence maps to the v3.1 roadmap from the phase 1
contract, refined with CLI-specific gate details:

### Phase 6: CLI Parser Gate Tests Only

**Scope**: Add clap registration for future commands in a hidden or rejected
state. Commands are parsed but immediately rejected with a "not implemented"
message. No actual functionality.

**Deliverable**: clap variants for `Ledger { Inspect, Export }` subcommand
added to `Commands` enum. All variants rejected at dispatch with a clear error
message. Tests verify rejection.

**Safety**: No behavior change. No new I/O. No new reads.

### Phase 7: Read-Only Report Preview Using In-Memory Fixtures — **completed** (see `docs/v3.1-phase-7-read-only-report-preview-fixtures.md`)

**Scope**: Create a pure fixture-based preview module that proves the future
report pipeline shape. No CLI wiring, no live resolver, no filesystem I/O.

**Deliverable**: `src/accounting/ledger_identity_preview.rs` with 4 functions
that accept parallel slices of `LedgerEntry` + `Option<ResolvedUsageTarget>`,
wrap the phase 4 base report with 13 preview-specific disclaimers, and
reuse phase 4 JSON serialization. 33 deterministic tests.

**Safety**: No live reads. No persistence. No filesystem writes. Fixture data
only. No CLI command behavior enabled.

### Phase 8: Explicit Persistence Design Freeze — **completed** (see `docs/v3.1-phase-8-explicit-persistence-design-freeze.md`)

**Scope**: Design document defining the persistence file format, location
policy, write contract, safety gates, and lifecycle for future ledger file I/O.
No actual file writes.

**Deliverable**: `docs/v3.1-phase-8-explicit-persistence-design-freeze.md`.
Defines 13 persistence safety gates, atomic write design, backup strategy,
corruption handling, future command candidates, JSON/schema policy, and
implementation sequence (phases 9-14+). Docs/design only.

**Safety**: No code. No I/O. Design only.

### Phase 9: Persistence Seam Implementation (Still Disabled by Default)

**Scope**: Implement the actual file I/O seam for ledger read/write, but keep
it disabled by default behind an explicit flag or command. The seam is tested
with temp directories only.

**Deliverable**: File I/O functions behind a feature gate or explicit CLI flag.
Tests use temp directories. No default activation.

**Safety**: Disabled by default. No silent writes. Temp directories in tests.

### Phase 10+: Gated Ledger Inspect Implementation

**Scope**: Activate `zelynic ledger inspect` with real persisted ledger data.
Requires phases 8 and 9 to be complete. Activate `zelynic ledger export --json`
for full ledger data export.

**Deliverable**: Functional `ledger inspect` reading from persisted ledger file.
Functional `ledger export --json` printing full ledger to stdout.

**Safety**: Read-only commands. No enforcement. No mutation. Persistence writes
remain gated behind explicit commands.

### Future Beyond Phase 10

- **Identity resolver design and implementation**: Live process scanning for
  `zelynic usage --target`. Requires its own design freeze phase.
- **Extended usage flags**: `--session`, `--since-boot`, `--interface` activated
  only after their prerequisite modules are complete.
- **`zelynic usage --target <target>`**: Last to be implemented, as it requires
  persistence + identity resolution + report model.

## 9. Non-Goals

The following are explicitly out of scope for v3.1 phase 5 and ALL v3.1
sub-phases unless explicitly overridden by a future phase design document:

- **No runtime code.** This phase produces a design document only. No Rust
  source files are created or modified.
- **No CLI wiring.** No clap subcommands are registered. No dispatch handlers
  are added. No flags are added to existing commands.
- **No new CLI flags.** No `--session`, `--since-boot`, `--interface`,
  `--target`, or any other flag is added to the `usage` subcommand.
- **No behavior change.** The existing `zelynic usage --sample` command
  family behaves exactly as in v3.0.1.
- **No version bump.** The version remains `v3.0.1`.
- **No persistence implementation.** No ledger file write code is added or
  activated.
- **No ledger file write.** No file is written to disk.
- **No live resolver.** No process scanning, no `/proc/<pid>` reads, no
  identity resolution.
- **No enforcement.** No blocking, no quota, no permission mode.
- **No quota guard.** No byte limits, no rate limits, no time-based quotas.
- **No eBPF.** No eBPF program loading, attachment, or use.
- **No nft/tc mutation.** No nftables or tc modification.
- **No cgroup mutation.** No cgroup write, no PID movement.
- **No daemon.** No background process, no systemd service, no auto-start.
- **No tag.** No git tag is created.
- **No GitHub release.** No release is published.
- **No package publish.** No crate is published to crates.io.
- **No new dependencies.** No entries are added to `Cargo.toml`.
