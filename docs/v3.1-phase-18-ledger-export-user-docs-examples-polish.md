## 1. Purpose

Phase 18 polishes the user-facing documentation and examples for the
`ledger export` command. This phase adds no new runtime behavior, changes
no parsing logic, and touches no production code. Its sole purpose is to
make the `ledger export --json --file <PATH>` command understandable and
safe for users by documenting its purpose, showing expected output,
providing copy-paste examples, and clearly stating what is NOT
implemented.

The documentation also clarifies the distinction between the five ledger
command variants so users understand when to use each one.

## 2. Command Variants

The `ledger` subcommand is hidden from `--help` output. It is an
internal/development command that is not part of the stable CLI surface.
The five command variants are:

### 2.1 `zelynic ledger inspect`

Prints a human-readable text summary of the in-memory fixture ledger.
No `--file` flag is passed, so the handler builds a deterministic
fixture. No filesystem reads or writes occur. This is a **human
inspection** view with summary totals, interface lists, and safety
disclaimers.

### 2.2 `zelynic ledger inspect --json`

Prints the same fixture ledger as deterministic JSON to stdout. This is
an **inspect summary JSON** view. The JSON uses the ledger serialization
schema (schema_version 1, host_id, entries array).

### 2.3 `zelynic ledger inspect --file <PATH>`

Reads a ledger JSON file from the explicitly specified `<PATH>`,
validates the path and file, and prints a human-readable text summary.
This is the **explicit file human inspection** view. The text output says
"source: explicit file read" and includes the file path.

### 2.4 `zelynic ledger inspect --file <PATH> --json`

Same as above but outputs deterministic JSON instead of human-readable
text. This is the **explicit file inspect summary JSON** view.

### 2.5 `zelynic ledger export --json --file <PATH>`

Reads a ledger JSON file from the explicitly specified `<PATH>`,
validates the path and schema, and emits the **raw validated ledger
JSON** to stdout only. This is the **raw ledger export** — the full
ledger data, not a summary. The output is pure JSON with no human
header, no disclaimers, and no extra text. The JSON uses the same
serialization schema as inspect but the operation is semantically
different: export reads a user-specified file, inspect may use fixture
data.

### 2.6 Summary Table

| Variant | Data Source | Output Format | Purpose |
|---------|------------|---------------|---------|
| `ledger inspect` | In-memory fixture | Human text | Quick fixture preview |
| `ledger inspect --json` | In-memory fixture | Ledger JSON | Fixture as structured data |
| `ledger inspect --file <P>` | User-specified file | Human text | Inspect a specific ledger file |
| `ledger inspect --file <P> --json` | User-specified file | Ledger JSON | Inspect file as structured data |
| `ledger export --json --file <P>` | User-specified file | Ledger JSON | Raw validated ledger export |

## 3. Minimal Valid Ledger JSON File

This is a minimal valid ledger JSON file that can be used with both
`ledger inspect --file` and `ledger export --json --file`:

```json
{
  "schema_version": 1,
  "created_at": "2026-01-01T00:00:00Z",
  "updated_at": "2026-01-01T00:00:00Z",
  "host_id": "my-host",
  "entries": [
    {
      "entry_id": "snap-001",
      "timestamp": "2026-01-01T00:00:00Z",
      "entry_type": "snapshot",
      "source_label": "manual",
      "interface": "eth0",
      "rx_bytes": 1000000,
      "tx_bytes": 500000,
      "combined_bytes": 1500000,
      "read_only": true,
      "provenance": "manual-snapshot",
      "attribution_scope": "interface-level only",
      "enforcement_status": "inactive/not implemented",
      "reset_detected": false,
      "reset_details": []
    }
  ]
}
```

Required top-level fields: `schema_version` (must be 1), `created_at`,
`updated_at`, `host_id`, `entries`.

Required per-entry fields: `entry_id`, `timestamp`, `entry_type`,
`source_label`, `interface`, `rx_bytes`, `tx_bytes`, `combined_bytes`,
`read_only` (must be `true`), `provenance`, `attribution_scope` (must
be `"interface-level only"`), `enforcement_status` (must be
`"inactive/not implemented"`), `reset_detected`, `reset_details`.

The `combined_bytes` field must equal `rx_bytes + tx_bytes`. Any
violation of these constraints produces a clear error message.

## 4. Copy-Paste Examples

### 4.1 Create a Valid Ledger File

```bash
cat >/tmp/zelynic-ledger-valid.json <<'JSON'
{
  "schema_version": 1,
  "created_at": "2026-01-01T00:00:00Z",
  "updated_at": "2026-01-01T00:00:00Z",
  "host_id": "my-host",
  "entries": [
    {
      "entry_id": "snap-001",
      "timestamp": "2026-01-01T00:00:00Z",
      "entry_type": "snapshot",
      "source_label": "manual",
      "interface": "eth0",
      "rx_bytes": 1000000,
      "tx_bytes": 500000,
      "combined_bytes": 1500000,
      "read_only": true,
      "provenance": "manual-snapshot",
      "attribution_scope": "interface-level only",
      "enforcement_status": "inactive/not implemented",
      "reset_detected": false,
      "reset_details": []
    }
  ]
}
JSON
```

### 4.2 Inspect a File (Human Summary)

```
$ zelynic ledger inspect --file /tmp/zelynic-ledger-valid.json
Zelynic v2.9 ledger inspect (source: explicit file read)
File: /tmp/zelynic-ledger-valid.json
Schema version: 1
Created: 2026-01-01T00:00:00Z
Updated: 2026-01-01T00:00:00Z
Host: my-host
...
Entries: 1 (1 snapshots, 0 deltas)
...
```

### 4.3 Inspect a File (JSON)

```
$ zelynic ledger inspect --file /tmp/zelynic-ledger-valid.json --json | jq '.schema_version, .host_id, (.entries | length)'
1
"my-host"
1
```

### 4.4 Export Raw Validated Ledger JSON

```
$ zelynic ledger export --json --file /tmp/zelynic-ledger-valid.json | jq '.schema_version, .host_id, (.entries | length), .entries[0].entry_id'
1
"my-host"
1
"snap-001"
```

### 4.5 Pipe Export Through jq

```
$ zelynic ledger export --json --file /tmp/zelynic-ledger-valid.json | jq '.entries[0] | {interface, rx_bytes, tx_bytes, combined_bytes}'
{
  "interface": "eth0",
  "rx_bytes": 1000000,
  "tx_bytes": 500000,
  "combined_bytes": 1500000
}
```

### 4.6 Redirect Export to a File (User-Controlled Shell Behavior)

```
$ zelynic ledger export --json --file /tmp/zelynic-ledger-valid.json > /tmp/my-export.json
```

**Important**: Zelynic itself has no `--output` flag and does not write
files. The shell redirection (`>`) is performed by the user's shell,
not by Zelynic. Zelynic only writes to stdout. If the destination file
already exists, the shell may overwrite it — this is the user's shell
behavior and Zelynic has no control over it. Zelynic provides no
`--overwrite` flag because it never writes files in the first place.

## 6. Safety Wording

The following safety guarantees apply to `ledger export`:

- **Stdout-only**: Export writes JSON to stdout only. There is no
  internal file write, no output file flag, and no overwrite mechanism.
- **Explicit --file required**: The user must specify the source file
  via `--file <PATH>`. There is no implicit default file path, no XDG
  auto-discovery, and no environment variable.
- **--json required**: The `--json` flag is mandatory for export.
  Human export output is not provided.
- **No output file flag**: There is no `--output <PATH>`, `--write
  <PATH>`, or `--save <PATH>`. Users redirect stdout with shell
  redirection.
- **No overwrite**: The `--overwrite` flag does not exist.
- **No internal file write**: The export handler contains no file write
  APIs (no `create_dir`, `remove_file`, `rename`, `File::create`,
  `OpenOptions`).
- **No persistence/save**: Export does not save ledger state. The
  persistence module returns `Blocked` for all operations.
- **No default ledger path read**: Export requires explicit `--file`.
  No XDG auto-discovery, no environment variable, no implicit fallback.
- **No live /proc/sysfs read**: The data source is exclusively the
  user-specified JSON file. No live system interface is read.
- **No live resolver**: Attribution is "interface-level only." No app
  identity resolution from live system state.
- **No enforcement**: `enforcement_status` is always
  "inactive/not implemented." No quota, rate limit, or blocking.
- **No nft/tc/cgroup/PID mutation**: No limiter attach, no policy
  install, no cleanup.
- **No eBPF**: No eBPF code in the export path.
- **No quota**: No quota guard in the export path.
- **No daemon/watch**: No background process or interval-based mode.

## 7. Schema Boundary

The codebase has three distinct JSON output contexts:

| Context | Command | Schema | Key Fields |
|---------|---------|--------|------------|
| v3.0 usage | `usage --sample --delta --json` | Usage schema | command, sample_mode, interfaces, totals, error |
| Ledger inspect | `ledger inspect --json` | Ledger schema | schema_version, host_id, entries |
| Ledger export | `ledger export --json --file <P>` | Ledger schema | schema_version, host_id, entries |

These schemas are intentionally separate:

- **Export JSON is ledger JSON**: The export output uses the ledger
  serialization schema (schema_version 1, host_id, entries array). It
  is produced by `serialize_ledger_to_json()`.
- **Inspect JSON is inspect JSON**: While inspect JSON and export JSON
  share the same serialization function, the operations are semantically
  different. Inspect may use fixture data; export reads a user-specified
  file.
- **Usage JSON is v3.0 usage JSON**: The usage JSON has `command`,
  `sample_mode`, `interfaces`, `totals`, `error`. These fields do not
  appear in the export or inspect JSON. The schemas are completely
  different.

The v3.0 usage JSON schema (schema_version 1) produced by `zelynic
usage --sample --delta --json` remains frozen. No field is added,
removed, or renamed. The export command does not affect the usage JSON
schema.

## 8. Troubleshooting

### 8.1 Missing Field: `entry_id`

```
Error: ledger export --file: invalid ledger in '/tmp/bad.json': ...
```

Every entry must have an `entry_id`. If this field is missing,
deserialization fails and the export is rejected.

### 8.2 Missing Field: `created_at` / `updated_at`

```
Error: ledger export --file: invalid ledger in '/tmp/bad.json': ...
```

Both `created_at` and `updated_at` are required top-level fields. If
either is missing, the export is rejected.

### 8.3 Wrong `attribution_scope`

```
Error: ledger export --file: invalid ledger in '/tmp/bad.json': ...
```

The `attribution_scope` field must be exactly `"interface-level only"`.
Any other value (e.g., `"per-app"`, `"process-level"`) is rejected.

### 8.4 Wrong `enforcement_status`

```
Error: ledger export --file: invalid ledger in '/tmp/bad.json': ... enforcement_status ...
```

The `enforcement_status` field must be exactly `"inactive/not
implemented"`. Any other value (e.g., `"active"`, `"enforcing"`) is
rejected.

### 8.5 `combined_bytes` Mismatch

```
Error: ledger export --file: invalid ledger in '/tmp/bad.json': safety violation: entry <id> has combined_bytes (9999) != rx_bytes (100) + tx_bytes (50)
```

The `combined_bytes` field must equal `rx_bytes + tx_bytes`. If the
values are inconsistent, the export is rejected.

### 8.6 `read_only` = false

```
Error: ledger export --file: invalid ledger in '/tmp/bad.json': ...
```

All entries must have `read_only: true`. If any entry has
`read_only: false`, the export is rejected.

### 8.7 Parent Traversal Path

```
$ zelynic ledger export --json --file /tmp/legit/../etc/passwd.json
Error: ledger export --file: parent traversal (..) is not allowed
```

Any `..` component in the path is rejected before any filesystem read.

### 8.8 Symlink Path

```
$ zelynic ledger export --json --file /tmp/symlink.json
Error: ledger export --file: '/tmp/symlink.json' is a symlink; symlinks are not followed
```

Symlinks are detected and rejected. Only regular files are accepted.

### 8.9 Malformed JSON

```
$ zelynic ledger export --json --file /tmp/bad.json
Error: ledger export --file: invalid ledger in '/tmp/bad.json': JSON parse error: ...
```

The file must contain valid JSON. Syntax errors produce an `invalid
ledger` error.

## 9. Future Boundary

This section documents what is NOT implemented yet:

- **Output file export remains not implemented.** There is no
  `--output <PATH>` flag. Users must use shell redirection to save
  export output to a file.
- **Persistence/save remains not implemented.** No `ledger save`
  command exists. The persistence module returns `Blocked` for all
  operations.
- **Default ledger path read remains not implemented.** Export
  requires explicit `--file <PATH>`.
- **Live ledger capture remains not implemented.** Export does not read
  from `/proc/net/dev` or any live system interface. The data source
  is exclusively the user-specified JSON file.
- **This phase is documentation polish only.** No production code was
  changed, no new features were added, and no existing behavior was
  modified. The sole deliverable is this document and the guard tests
  that verify its content.
