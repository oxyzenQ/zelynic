The `ledger` subcommand remains `#[command(hide = true)]` in clap and is

## 2. Ledger Command Status

The `zelynic ledger` subcommand is a v3.1 experimental/hidden command. It
does not appear in normal `zelynic --help` output. It is intended for
development, testing, and controlled inspection of ledger JSON files. The
command has two active subcommands:

- `ledger inspect` — human-readable or JSON summary of a ledger.
- `ledger export` — raw validated ledger JSON output to stdout.

Both subcommands and all their variants remain hidden. The command surface
is documented here for discoverability but is not promoted to the stable
CLI surface.

## 3. Inspect vs Export Distinction

The `ledger inspect` and `ledger export` commands serve different purposes
and produce different output:

- **ledger inspect** produces a human-readable summary (text mode) or an
  inspect summary JSON. The inspect summary is derived from the ledger data
  and rendered for human consumption. When `--file` is not provided, the
  inspect command uses an in-memory fixture with no filesystem access.
  When `--file <PATH>` is provided, it reads and validates the specified
  file in read-only mode.
- **ledger export** produces the raw validated ledger JSON to stdout only.
  Export requires both `--json` and `--file <PATH>`. It reads the
  specified file, validates path and schema, and emits the full ledger JSON
  without any human header or disclaimer text.

In summary: inspect is for human-readable summary views, export is for raw
ledger JSON output.

## 4. Export Constraints

The `ledger export --json --file <PATH>` command operates under these
constraints:

- **stdout-only**: Export writes JSON to stdout only. There is no internal
  file write, no output file flag, and no overwrite mechanism.
- **Explicit --file required**: The user must specify the source file via
  `--file <PATH>`. There is no implicit default file path.
- **No --output, no --overwrite, no write**: There is no `--output
  <PATH>`, `--write <PATH>`, `--save <PATH>`, or `--overwrite` flag.
  Users redirect stdout with shell redirection if they wish to save the
  output.
- **No persistence/save/default path read**: Export does not save ledger
  state and does not read from any default ledger path.

## 5. Safety Guarantees

The following safety guarantees apply to all ledger commands (inspect and
export):

- **No output file flag, no overwrite, no write**: No file write APIs are
  used in the production code path. No `--output`, `--write`, `--save`,
  or `--overwrite` flags exist.
- **No persistence/save/default path read**: No automatic loading, saving,
  or caching of ledger data. No default ledger path is read.
- **No live resolver, no enforcement, no nft/tc/cgroup/PID mutation**: No
  limiter attach, no policy install, no cleanup. No live app identity
  resolution. `enforcement_status` is always "inactive/not implemented".
  No nft/tc/cgroup/PID state mutation is performed.
- **No live /proc/sysfs read**: The data source is either the in-memory
  fixture (no `--file`) or the user-specified JSON file (`--file`).
- **No eBPF**: No eBPF code in the ledger path.
- **No quota**: No quota guard in the ledger path.
- **No daemon/watch**: No background process or interval-based mode.

## 6. Schema Boundaries

The v3.0 usage JSON remains unchanged. The `zelynic usage --sample
--delta --json` command outputs `schema_version: 1` with its own schema
(command, sample_mode, interfaces, totals, error). The ledger JSON schema
(schema_version 1, host_id, entries) is a separate schema used by both
`ledger inspect --json` and `ledger export --json --file <PATH>`. These
schemas are intentionally separate and do not affect each other.
