## 1. Purpose

Phase 10b freezes and audits the Phase 10 hidden fixture-only inspect preview
behavior before any future file-backed inspect work. This phase adds 28
deterministic invariant tests that prove every required safety property holds
after the Phase 10 activation of `ledger inspect` with in-memory fixture data.

The 28 invariants cover five categories:

1. **Fixture data integrity** (invariants 1-2): Inspect uses only in-memory
   fixture data — no live `/proc` reads, no filesystem reads, no live system
   data of any kind. Both text and JSON output are derived exclusively from the
   deterministic 3-entry fixture ledger.

2. **No persistence coupling** (invariants 3-4): The inspect handler does not
   call ledger path planning or persistence read plans. These modules remain
   independently hard-blocked. The inspect output contains no path planning
   markers ("persistence path", "namespace", "full ledger path") and no
   persistence blocked markers ("HARD-BLOCKED").

3. **No filesystem I/O** (invariant 5): The inspect handler does not use
   `std::fs` read/write APIs. This is verified structurally (no `std::fs`
   import in the handler module) and by output analysis (no `/proc/`, `/sys/`,
   `/dev/`, `.json`, or "File I/O" patterns in rendered text).

4. **Output disclaimer completeness** (invariants 6-15): The inspect render
   output contains all required safety disclaimers: fixture preview ("ledger
   inspect model only"), no ledger file read ("no filesystem read performed"),
   no ledger file write ("no filesystem write performed"), no persistence
   ("ledger inspect model only" phrasing), no live resolver ("interface-level
   only (not per-app attribution)"), no enforcement ("quota enforcement:
   inactive/not implemented"), no network blocking ("network blocking:
   inactive/not implemented"), no nft/tc mutation ("no nft/tc/Zelynic state
   mutation performed"), no cgroup mutation ("no limiter attach performed"),
   no PID movement ("no limiter attach performed" + "Read-only: true").

5. **Schema and CLI boundary invariants** (invariants 16-28): Ledger inspect
   JSON is valid JSON (parsable by serde_json), uses the ledger schema (not
   v3.0 usage JSON schema), and does not contain v3.0 usage-specific fields
   ("command", "source_path", "totals", "honesty", "warnings"). The v3.0 usage
   JSON schema remains at schema_version 1. Ledger export remains
   design-gated/rejected. All hidden usage flags remain design-gated/rejected.
   The ledger subcommand remains hidden from public help. Usage help remains
   free of hidden v3.1 flags. No new visible public command is introduced.
   No file path argument, output path argument, overwrite/save flag, or
   daemon/watch/interval mode exists on ledger inspect. Existing v3.0 commands
   have no runtime behavior changes.
