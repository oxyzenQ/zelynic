# Zelynic v3.0 Live Read-Only Usage Lab — Design Document

This document captures the design for the v3.0 Live Read-Only Usage Lab, a
read-only milestone that builds a live data reader foundation and future CLI
contract for `zelynic usage`. Phase 1 is **design-only**: it produces this
document and updates related documentation but introduces no runtime changes, no
enforcement, no live system reads, and no CLI command registration.

## Purpose

v3.0 is a live read-only usage milestone that bridges the gap between the v2.9
Network Accounting Lab (pure model, parser, and renderer — no live I/O) and a
future release where `zelynic usage` can sample live interface counters from the
running system. Unlike v2.9 (which established the data model, persistence
design, and serialization tests entirely as pure Rust models without touching
any live system files), v3.0 plans the reader seam that will eventually connect
the existing parser to the live `/proc/net/dev` kernel interface.

The v3.0 milestone is intentionally incremental and conservative. It does not
jump directly to a working CLI command. Instead, it layers the work across
multiple phases: design, reader seam, tests with fake/injected sources, CLI
preview, JSON output contract, delta sampling design, and a final validation
freeze. Each phase has explicit entry criteria and explicit hard safety
boundaries that are inherited from v2.9 and extended where necessary.

v3.0 does NOT implement:

- eBPF
- quota enforcement
- network blocking
- limiter attach
- nftables/tc rule mutation
- Zelynic runtime state mutation
- PID movement
- `cgroup.procs` writes
- `/sys/fs/cgroup` access for usage
- filesystem persistence (no ledger file read or write)
- live CLI command registration in phase 1

`zelynic strict` remains the only validated active limiter path.

## Live Read-Only Usage Milestone Scope

v3.0 is scoped as a **live read-only usage milestone**. This means:

- **No enforcement**: Usage data is observed and reported, never used to
  trigger automatic blocking, throttling, or rate-limiting actions.
- **No blocking**: No network traffic is ever blocked, dropped, or rejected by
  v3.0 code. The `zelynic strict` path remains the only validated limiter.
- **No quota guard**: Quota budget tracking is deferred. No commands that
  enforce quota limits exist in v3.0.
- **No eBPF**: The design acknowledges eBPF as a future advanced backend
  for per-app attribution, but v3.0 uses only `/proc/net/dev` for live reads
  when the reader seam is implemented in phase 2+.
- **No persistence**: v3.0 does not read or write ledger files. The hard-blocked
  persistence seam from v2.9 remains in place.
- **No PID movement**: v3.0 does not move PIDs or interact with cgroups.
- **No `cgroup.procs` write**: v3.0 never writes to cgroup.procs files.
- **No nft/tc mutation**: v3.0 never adds, removes, or modifies nftables rules
  or tc qdisc/filters/classes.
- **No live CLI command in phase 1**: Phase 1 is design-only. The `zelynic
  usage` command is planned but not registered in the CLI parser.

## Future Command Contract (Planned, Not Implemented)

The following commands define the future CLI surface for the `zelynic usage`
command family. All commands are **planned** in this design document and are
**not implemented** in v3.0 phase 1. Implementation will occur incrementally
across phases 2–7.

### `zelynic usage`

Display a summary of live network usage across all interfaces. Default view
shows aggregate RX/TX since boot for all non-loopback interfaces. This is the
primary command of the v3.0 milestone.

```
$ zelynic usage
zelynic usage — live read-only interface counters
Source: live_proc_net_dev
Attribution: interface-level only (not per-app)
No quota enforcement active. No network blocking active.
No limiter attach performed. No nft/tc/Zelynic state mutation performed.
No ledger persistence performed. No eBPF used.
Counters may reset after reboot or interface reset.

Interface     RX           TX           Since
wlan0         1.23 GiB     342 MiB      boot
eth0          0 bytes       0 bytes      boot
usb0          45.6 MiB     12.3 MiB     boot (tethering)

Total (excl. loopback):  RX 1.28 GiB  TX 354 MiB
```

**Status**: Planned. Not implemented in phase 1.
**Constraints**: Read-only. Interface-level only. Not per-app attribution.
Not quota enforcement. Not network blocking.

### `zelynic usage --sample`

Perform a single sample of live interface counters and print the result. This
is the simplest invocation: one read of `/proc/net/dev`, one parse, one render.
No polling loop, no delta computation. Useful for scripting and automation where a
single snapshot is sufficient.

```
$ zelynic usage --sample
zelynic usage --sample: single live read-only snapshot
Source: live_proc_net_dev
[... same honesty labels as above ...]
```

**Status**: Planned. Not implemented in phase 1.
**Constraints**: Single read. No polling. No loop. Read-only.

### `zelynic usage --interface wlan0`

Show live usage for a specific interface only. Filters output to the named
interface. Validates that the interface exists in the parsed snapshot before
displaying. If the interface is not found in `/proc/net/dev`, reports an error
rather than showing empty data.

```
$ zelynic usage --interface wlan0
zelynic usage --interface wlan0: live read-only interface counters
Source: live_proc_net_dev
[... honesty labels ...]

Interface     RX           TX           Since
wlan0         1.23 GiB     342 MiB      boot
```

**Status**: Planned. Not implemented in phase 1.
**Constraints**: Filters to single interface. Read-only. No enforcement.

### `zelynic usage --json`

Output live usage data as structured JSON. The JSON schema is defined in the
JSON Output Contract section below. Machine-readable output for scripting,
monitoring integrations, and dashboard feeds.

```
$ zelynic usage --json
{
  "schema_version": 1,
  "source": "live_proc_net_dev",
  "interfaces": [
    {
      "name": "wlan0",
      "rx_bytes": 1320489368,
      "tx_bytes": 358637568,
      "combined_bytes": 1679126936,
      "loopback": false
    }
  ],
  "totals": {
    "rx_bytes": 1320489368,
    "tx_bytes": 358637568,
    "combined_bytes": 1679126936
  },
  "honesty": {
    "per_app_attribution": false,
    "quota_enforcement_active": false,
    "network_blocking_active": false,
    "persistence_performed": false,
    "ebpf_used": false
  }
}
```

**Status**: Planned. Not implemented in phase 1.
**Constraints**: Structured JSON output. Schema versioned. Includes honesty
flags. Read-only.

### `zelynic usage --delta --interval 5s`

Perform delta sampling: take two snapshots separated by the specified interval,
compute the per-interface usage delta, and display the result. This provides
"usage in the last N seconds" information rather than cumulative since-boot
counters. Uses the existing `SessionDelta` model from v2.9.

```
$ zelynic usage --delta --interval 5s
zelynic usage --delta (5s interval): live read-only delta
Source: live_proc_net_dev
[... honesty labels ...]

Interface     RX/s         TX/s         Since
wlan0         12.3 MiB/s   4.5 MiB/s   5s sample
eth0          0 bytes/s     0 bytes/s    5s sample
```

**Status**: Planned. Not implemented in phase 1.
**Constraints**: Two reads separated by interval. Read-only. No enforcement.
Delta computation uses existing SessionDelta model. Counter reset detection
applies.

### `zelynic usage --since-start`

Display usage since Zelynic was started (or since a reference snapshot was
captured). This requires storing a baseline snapshot at startup and computing
the delta against the current live counters. No persistence to disk — the
baseline is held in memory only for the lifetime of the process.

```
$ zelynic usage --since-start
zelynic usage --since-start: delta from process start
Source: live_proc_net_dev
Baseline: captured at process start
[... honesty labels ...]

Interface     RX           TX           Duration
wlan0         45.6 MiB     12.3 MiB     2h 14m
```

**Status**: Planned. Not implemented in phase 1.
**Constraints**: In-memory baseline only. No disk persistence. Read-only.

### `zelynic usage --no-loopback`

Exclude loopback (`lo`) interface from output. Loopback traffic is local
inter-process communication and is not representative of network usage. This
flag filters the loopback interface from both display and totals.

**Status**: Planned. Not implemented in phase 1.
**Constraints**: Filter flag. Read-only. No mutation.

### `zelynic usage --all-interfaces`

Show all interfaces including loopback. This is the default behavior but the
flag allows explicit opt-in when the user wants to see loopback traffic
included in the output.

**Status**: Planned. Not implemented in phase 1.
**Constraints**: Display flag. Read-only. No mutation.

## Live Read-Only Data Source Plan

The v3.0 data source strategy is centered on `/proc/net/dev` as the sole live
data source. This is a deliberate simplification: `/proc/net/dev` is always
available on Linux, requires no special permissions, and provides per-interface
RX/TX byte counters that are updated in real time by the kernel.

## `/proc/net/dev` Reader Boundary

The reader seam for `/proc/net/dev` operates under strict boundaries. These
boundaries ensure that the live reader cannot be abused or extended to read
arbitrary system files.

### What the Reader May Do (In Phase 2+ Implementation Only)

- Read the single file `/proc/net/dev` using standard file I/O.
- Pass the file content to the existing `parse_proc_net_dev` parser.
- Produce an `InterfaceCounterSnapshot` with `SourceLabel::LiveProcNetDev`.
- Report I/O errors (file not found, permission denied) without panicking.

### What the Reader Must NOT Do

| Prohibited Action | Reason |
|-------------------|--------|
| Read arbitrary files | Reader is scoped to `/proc/net/dev` only |
| Write any file | Reader is read-only by design |
| Mutate system state | Reader observes only, never modifies |
| Read directories | Reader reads a single file, never lists directories |
| Canonicalize paths | No path traversal — exact path only |
| Accept user-supplied paths | Path is hardcoded, not configurable |
| Follow symlinks | Direct read of `/proc/net/dev` only |

### Source Label Honesty

When the reader seam produces a snapshot from a live read, it must label the
source honestly:

- `SourceLabel::LiveProcNetDev` when reading from the real `/proc/net/dev`.
- `SourceLabel::InjectedTestContent` when using fake/injected content in tests.
- The label must never be falsified — a test snapshot must never claim a live
  source, and a live snapshot must never claim an injected source.

### Parser Reuse

The reader seam must reuse the existing parser from
`src/accounting/interface_counters.rs`. Specifically:

- `parse_proc_net_dev(content: &str)` for full snapshot parsing.
- `InterfaceCounterSnapshot` as the output type.
- `SourceLabel` as the observational marker.
- All existing 53 parser tests remain valid and unchanged.

No new parser is written for v3.0. The existing parser already handles the
`/proc/net/dev` format correctly, including header skipping, interface name
trimming, colon validation, field count validation, u64 parsing with overflow
detection, and all error cases.

## Output Honesty Requirements

All v3.0 output (human-readable and JSON) must include explicit honesty
disclaimers. These are inherited from v2.9 and extended for the live context.

### Required Honesty Statements in Human-Readable Output

Every render of live usage data must include:

1. **Interface-level only**: "Attribution: interface-level only (not per-app)"
   — the counters represent aggregate traffic on each interface, not traffic
   attributed to specific processes, PIDs, or cgroups.
2. **No per-app attribution**: "Not per-app attribution" — the data does not
   identify which process sent or received which bytes.
3. **No quota enforcement active**: "No quota enforcement active" — the
   usage display does not enforce any bandwidth budget or quota limit.
4. **No network blocking active**: "No network blocking active" — the usage
   display does not block, drop, or reject any network traffic.
5. **No limiter attach performed**: "No limiter attach performed" — the usage
   display does not attach or configure any rate limiter.
6. **No nft/tc/Zelynic state mutation performed**: "No nft/tc/Zelynic state
   mutation performed" — the usage display does not modify nftables rules, tc
   qdisc/filters/classes, or Zelynic runtime state.
7. **No ledger persistence performed**: "No ledger persistence performed" — the
   usage display does not read or write any ledger file.
8. **No eBPF used**: "No eBPF used" — the usage display does not load, attach,
   or interact with eBPF programs.
9. **Counters may reset**: "Counters may reset after reboot or interface reset"
   — cumulative counters are since boot/interface-up and are lost on reboot.

### Forbidden Wording in Output

| Forbidden | Reason |
|-----------|--------|
| "App usage" | Implies per-app attribution, which is not available |
| "Process bandwidth" | Implies per-process attribution, which is not available |
| "Quota active" | Implies enforcement, which is not implemented |
| "Enforced" | Implies traffic was blocked or throttled by usage |
| "Blocked by quota" | Implies enforcement, which is not implemented |
| "All-time usage" | Implies persistent tracking, which is not implemented |
| "Lifetime" | Implies persistent tracking, which is not implemented |

## Future JSON Output Contract

The JSON output for `zelynic usage --json` follows a structured schema. The
schema is versioned for forward compatibility.

### Schema Definition

```json
{
  "schema_version": 1,
  "source": "live_proc_net_dev",
  "sampled_at": "2026-06-07T12:00:00Z",
  "interfaces": [
    {
      "name": "wlan0",
      "rx_bytes": 1320489368,
      "tx_bytes": 358637568,
      "combined_bytes": 1679126936,
      "loopback": false
    },
    {
      "name": "lo",
      "rx_bytes": 524288,
      "tx_bytes": 524288,
      "combined_bytes": 1048576,
      "loopback": true
    }
  ],
  "totals": {
    "rx_bytes": 1321013656,
    "tx_bytes": 359162856,
    "combined_bytes": 1680176512
  },
  "honesty": {
    "per_app_attribution": false,
    "quota_enforcement_active": false,
    "network_blocking_active": false,
    "persistence_performed": false,
    "ebpf_used": false
  }
}
```

### Field Descriptions

| Field | Type | Description |
|-------|------|-------------|
| `schema_version` | integer | Schema version for forward compatibility. Currently `1`. |
| `source` | string | Data source identifier. `"live_proc_net_dev"` for live reads. |
| `sampled_at` | string (optional) | ISO 8601 timestamp if provided by caller. Not generated silently. Omitted if caller does not provide a timestamp. |
| `interfaces` | array | Per-interface counter data. |
| `interfaces[].name` | string | Interface name (e.g., `wlan0`, `eth0`, `lo`). |
| `interfaces[].rx_bytes` | integer | Cumulative received bytes since boot/interface-up. |
| `interfaces[].tx_bytes` | integer | Cumulative transmitted bytes since boot/interface-up. |
| `interfaces[].combined_bytes` | integer | `rx_bytes + tx_bytes`. Saturating addition for overflow safety. |
| `interfaces[].loopback` | boolean | `true` if this is the loopback interface (`lo`). |
| `totals` | object | Aggregate totals across all interfaces (including loopback unless `--no-loopback` is used). |
| `totals.rx_bytes` | integer | Sum of all interface RX bytes. |
| `totals.tx_bytes` | integer | Sum of all interface TX bytes. |
| `totals.combined_bytes` | integer | Sum of all interface combined bytes. |
| `honesty` | object | Honesty flags. Present in every JSON output. |
| `honesty.per_app_attribution` | boolean | Always `false`. Per-app attribution is not available from interface counters. |
| `honesty.quota_enforcement_active` | boolean | Always `false`. No quota enforcement is implemented. |
| `honesty.network_blocking_active` | boolean | Always `false`. No network blocking is performed. |
| `honesty.persistence_performed` | boolean | Always `false`. No ledger file read or write is performed. |
| `honesty.ebpf_used` | boolean | Always `false`. No eBPF programs are loaded or attached. |

### Honesty Flags Contract

The honesty flags are **always boolean constants** in the current implementation:

- `per_app_attribution`: Always `false`. Interface counters cannot attribute
  traffic to specific processes.
- `quota_enforcement_active`: Always `false`. No quota enforcement exists.
- `network_blocking_active`: Always `false`. No traffic is blocked.
- `persistence_performed`: Always `false`. No ledger I/O is performed.
- `ebpf_used`: Always `false`. No eBPF is used.

These flags must never be `true` in v3.0. If a future version implements
per-app attribution (e.g., via eBPF), the schema version must be incremented
and the flag documentation must be updated accordingly.

### sampled_at Semantics

The `sampled_at` field is **only populated if the caller explicitly provides
a timestamp**. The command does not silently generate a timestamp. This
prevents misleading precision in JSON output where the user did not request a
timestamped sample.

- `zelynic usage --json` (no timestamp argument): `sampled_at` is omitted.
- `zelynic usage --json --sampled-at "2026-06-07T12:00:00Z"`: `sampled_at`
  is set to the caller-provided value.

## What v3.0 Is Not

To prevent scope creep and honest communication about capabilities, v3.0
explicitly does NOT include:

- **Per-app attribution**: v3.0 reports interface-level counters only. Per-app
  or per-process attribution requires eBPF or PID-based socket mapping, which
  is explicitly deferred to a future milestone.
- **Quota enforcement**: v3.0 displays usage data but never blocks, throttles,
  or rate-limits traffic based on usage. Quota enforcement is deferred.
- **Network blocking**: v3.0 does not drop, reject, or shape any packets.
- **eBPF observer**: v3.0 does not load, attach, or interact with eBPF programs.
  The eBPF observer backend is deferred to a future milestone (v4.x or later).
- **Persistent ledger**: v3.0 does not read or write ledger files from disk. The
  v2.9 hard-blocked persistence seam remains in place.
- **Per-target usage via strict backend**: v3.0 does not read nftables or tc
  counters from `zelynic strict` rules. Reading those counters requires the
  strict infrastructure to be in place and is a separate feature.
- **Real-time monitoring dashboard**: v3.0 provides single-sample and delta
  output, not a continuous real-time TUI dashboard. The existing `zelynic list
  --live` TUI remains the real-time monitoring path.
- **Background data guard**: v3.0 does not implement the background data guard
  concept from the v2.9 design. This requires per-app attribution and quota
  enforcement, both deferred.
- **Allow/block mode**: v3.0 does not implement allow or block modes for
  network traffic. Traffic control remains exclusively through `zelynic strict`.
