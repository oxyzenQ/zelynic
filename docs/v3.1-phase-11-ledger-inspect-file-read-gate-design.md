## 1. Purpose

Phase 11 freezes the future contract for moving from fixture-only `ledger inspect`
toward explicit user-selected ledger file reads in a later phase. This phase
chooses and documents one future command shape, explicitly rejects alternative
shapes, and codifies the design rules that any future file-read implementation
must satisfy. No file reading is implemented in this phase — the command remains
fixture-only.

The primary design question answered in this phase is: **how does a user
explicitly select a ledger file to inspect?** The answer is frozen as contract
for all future phases.

## 2. Chosen Future Command Shape

```
zelynic ledger inspect --file <PATH>
zelynic ledger inspect --file <PATH> --json
```

This is the only accepted future shape for user-selected file reads. The `--file`
flag is explicit, positional, and required for any file-backed inspect. No
implicit default file path, no environment variable, no auto-discovery.

### 2.1 Design Rationale

The `--file <PATH>` shape was chosen for five reasons:

1. **Explicit consent**: The user must type `--file` and a path. There is no
   ambiguity about whether a file will be read. The flag name itself communicates
   the operation (reading a file).

2. **No implicit state**: Unlike a bare `zelynic ledger inspect` that might
   silently read a default file, `--file <PATH>` makes the data source visible
   in the command invocation. This aligns with the phase 8 principle of "no
   hidden state."

3. **Composable with `--json`**: The `--json` flag already exists on the
   current fixture-only `ledger inspect`. Adding `--file` as a peer flag means
   the user can request file-backed text output (`--file <PATH>`) or file-backed
   JSON output (`--file <PATH> --json`) without ambiguity.

4. **Compatible with existing clap shape**: The current `LedgerCommands::Inspect`
   variant has a single `json: bool` field. Adding a `file: Option<String>` field
   is a minimal, backward-compatible change. When `file` is `None`, the handler
   uses the in-memory fixture (current behavior). When `file` is `Some(path)`,
   the handler reads and validates the file (future behavior).

5. **Testable at the parser level**: Both `--file` presence and absence are
   easily tested with clap's `try_parse_from()` without any filesystem access.

## 3. Rejected Alternatives

### 3.1 Implicit Default File Read (Rejected)

**Shape**: `zelynic ledger inspect` reads `~/.local/share/zelynic/network-ledger-v1.json`
automatically.

**Rejection reason**: Violates the phase 8 principle of "no hidden state" and
"explicit user command only." A user running `zelynic ledger inspect` today
expects fixture-only output. Silently changing this to read a default file in a
future phase would be a behavioral breaking change for anyone who has scripts or
automation built around the fixture output. The user must explicitly opt into
file reading via `--file`.

### 3.2 Auto-Reading from XDG Directory Without Explicit Request (Rejected)

**Shape**: `zelynic ledger inspect` checks for `~/.local/share/zelynic/network-ledger-v1.json`
and reads it if present, falling back to fixture if absent.

**Rejection reason**: This combines the worst aspects of implicit default file read
with silent fallback behavior. The output would change unpredictably depending on
whether a ledger file happens to exist on the system. Scripts would produce
different results on different machines. This violates determinism and the
"explicit user command only" principle.

### 3.3 Environment Variable Path (Rejected)

**Shape**: `ZELYNIC_LEDGER_PATH=/path/to/file zelynic ledger inspect` reads from
the environment variable.

**Rejection reason**: Environment variables are invisible in `--help` output,
invisible in shell history, and invisible in scripts unless explicitly documented.
This is a form of hidden state that the phase 8 principles explicitly prohibit.
A user reading a script cannot tell that a different file is being read without
checking the environment. The path must be in the command invocation itself.

### 3.4 Live State Read (Rejected)

**Shape**: `zelynic ledger inspect` reads live system state (cgroup, nft counters,
/proc/net/dev) instead of a file.

**Rejection reason**: `ledger inspect` is a reporting/inspection command for
persisted ledger data, not a live monitoring command. Reading live state is the
domain of `zelynic list` or `zelynic usage --sample`. Mixing live state into
ledger inspect would blur the boundary between observation (usage/list) and
reporting (ledger), making it unclear what data source the output represents.

### 3.5 Combined Inspect/Export/Save in One Command (Rejected)

**Shape**: `zelynic ledger --file <PATH> --save --export --json` combines multiple
operations.

**Rejection reason**: Single-responsibility principle. Each command should do
one thing. `inspect` reads and displays; `export` writes to stdout; `save`
writes to disk. Combining these in one command creates ambiguity about side
effects (does `--file` mean read from or write to?), complicates error handling,
and makes it harder to reason about safety guarantees.

## 4. Design Rules

The following design rules are frozen as contract for all future file-read
implementation phases. Any implementation of `ledger inspect --file <PATH>`
must satisfy every rule before the file-read code path is activated.

### Rule 1: Fixture-Only by Default

Plain `zelynic ledger inspect` (without `--file`) remains fixture-only for all
future phases until a separate explicit decision is made to change this default.
No future phase may change the default behavior of bare `ledger inspect` without
a dedicated design freeze phase that documents the change.

### Rule 2: Explicit `--file <PATH>` Required

File reads must require the `--file <PATH>` flag. There is no short form, no
positional argument, no implicit default. The `PATH` argument must be a
non-empty string provided by the user on the command line.

### Rule 3: Path Validation Before Read

Before any file is read, the path must be validated through the existing path
model in `src/accounting/ledger_path.rs`. Specifically, the `build_ledger_path_plan()`
function (or its future hardened variant) must accept the path. This includes:

- Non-empty base directory and filename.
- Relative filename only (no absolute paths) for default namespace paths.
- No parent traversal (`..`) components.
- No suspicious characters (only `[a-zA-Z0-9._-]` in filename).
- Namespace containment (path must be within the configured base directory).

For user-provided absolute paths via `--file`, an additional validation layer
must verify: no parent traversal, no symlink target escaping, and the file
must be a regular file (not a directory, device, or symlink).

### Rule 4: No Symlink Following

The file-read path must not follow symlinks silently. If the user-provided
`--file <PATH>` resolves through a symlink, the implementation must either:

- Reject the path with a clear error message identifying the symlink, or
- Resolve the symlink and verify the canonical target is within the allowed
  namespace before reading.

Silent symlink following is prohibited. This aligns with phase 8 Gate 10
(No Symlink Surprises) and the phase 9 `symlink_resolution_performed` flag.

### Rule 5: No Directory Creation

The file-read path must not create any directories. If the directory containing
the ledger file does not exist, the implementation must return an error. It must
not create the directory, parent directories, or any other filesystem structure.
This aligns with the phase 9 `hidden_state_directory_created` flag.

### Rule 6: No Write Operations

The file-read path must not write, backup, rename, delete, truncate, or migrate
any file. The operation is strictly read-only. No temporary files are created.
No lock files are created. No modification timestamps are updated (beyond what
the OS does for `open()` with `O_RDONLY`). This aligns with the phase 8
principles of "No Silent Write" and "No Mutation of System State."

### Rule 7: Parse Then Validate Schema

After reading a file, the implementation must parse the JSON content, then
validate the parsed `Ledger` struct using the existing `Ledger::validate_safety()`
method before rendering any output. This catches corrupt, tampered, or
version-mismatched files before they produce misleading output. The validation
includes:

- `schema_version` is recognized (currently only version 1).
- All entries have `read_only: true`.
- All entries have `attribution_scope: "interface-level only"`.
- All entries have `enforcement_status: "inactive/not implemented"`.
- `combined_bytes == rx_bytes + tx_bytes` (saturating).

### Rule 8: Honest Non-Mutating Errors

All errors during file read (not found, permission denied, parse error,
schema validation failure, symlink detected) must be reported as honest
error messages. The error messages must identify the specific problem and
suggest a user action. No error path may silently mutate state, write files,
or fall back to fixture data.

### Rule 9: `ledger export --json` Remains Gated

The `ledger export --json` command remains design-gated and rejected at dispatch
until a separate export phase (phase 13 or later) implements it. This phase does
not ungate, design, or implement export.

### Rule 10: `ledger save/write` Remains Not Implemented

No save, write, or persist command is implemented, designed, or ungated in this
phase. All persistence operations remain hard-blocked via the existing
`LedgerPersistencePlan` with `persistence_status: Blocked(...)`.

### Rule 11: v3.0 Usage JSON Schema Unchanged

The v3.0 usage JSON schema (schema_version 1) produced by `zelynic usage
--sample --json` and `zelynic usage --sample --delta --json` remains frozen.
No field is added, removed, or renamed. The schema is the same as in v3.0.1.
