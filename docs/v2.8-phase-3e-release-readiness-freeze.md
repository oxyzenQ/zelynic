## Purpose

This document freezes the v2.8 Experimental PID Move Lab at its current
validated state. Phase 3e is a docs-only phase: it produces this report,
updates cross-reference markers, and establishes the release-readiness
baseline for v2.8. No runtime code is introduced, modified, or removed in
this phase.

The v2.8 release is **not** the release that implements live PID movement.
v2.8 is the design-and-scaffold milestone that establishes the gate
infrastructure, executor seam, transaction model, output honesty guarantees,
and negative-path coverage required before any future phase can safely
perform a real PID move.

### Phase 3a: Single PID Rollback Design

- **Scope**: Produced `docs/v2.8-phase-3a-single-pid-rollback-design.md`
  specifying the design for the first actual PID move experiment.
- **Key constraints (13 hard constraints)**:
  - Root-only, system-scope-only.
  - Single disposable PID only (`sleep 3` or `sleep 10`).
  - Immediate rollback required (PID must return to original cgroup).
  - No limiter attach, no nft/tc/state mutation, no persistent state write.
  - No multi-PID process trees, no user scope, no long-running apps.
  - No browser/terminal/desktop processes, no bandwidth limiting claim.
  - No cleanup of non-empty cgroups, no cleanup outside namespace.
- **Transaction model**: Exact 10-step model with full dependency graph:
  gate re-evaluation, PID liveness recheck, original cgroup validation,
  target cgroup preparation, move PID to target, verify in target, record
  move success, rollback PID to original, verify restored, cleanup target.
- **Failure policy**: Pre-move failures abort without rollback. Move
  failures attempt rollback. Rollback failures reported loudly, no retry.
  Cleanup failures leave target in place.
- **Status**: Completed. Docs/design only — no runtime changes.

## Freeze Safety Guarantees

The following invariants are **locked** at the phase 3e freeze boundary.
These guarantees are backed by automated tests (449 unit + 4 integration),
root smoke validation, and CI.

### Invariant 1: No PID Movement

No code path in the current codebase can move a PID between cgroups.
The `move_executor.rs` seam validates gate inputs and always returns
`status: blocked`. The `cgroup.procs` file is never opened for writing
in any code path. The canonical deny line "No PID was moved." is tested
across all 11 negative-path scenarios.

### Invariant 2: No cgroup.procs Write

No code path writes to any `cgroup.procs` file. The mkdir-live path
(phase 2b) creates empty cgroup directories and reads `cgroup.procs` for
PID discovery, but never writes to it. The canonical deny line "No
cgroup.procs write was performed." is tested across all negative-path
scenarios.

### Invariant 3: No Limiter Attach

No code path attaches a limiter from the experimental attach/run path.
No nftables rules, tc qdisc/class/filter changes, or bandwidth enforcement
are performed through any v2.8 code path. The canonical deny lines
"No limiter attach was performed." and "Bandwidth limiting is not active
from this command yet." are tested across all negative-path scenarios.

### Invariant 4: No nftables/tc/Zelynic State Mutation

No code path modifies nftables rules, tc configuration, or Zelynic
persistent state (`/run/zelynic/state.json`) through any v2.8 code path.
The canonical deny line "No nftables, tc, or Zelynic state changes were
made." is tested across all negative-path scenarios.

### Invariant 5: No Persistent State Write

No code path writes persistent operational state to disk. The operation
journal is model-only. The canonical deny line "No persistent state write
was performed." is tested across all negative-path scenarios.

### Invariant 6: No Bandwidth Limiting from Experimental Attach Path

No bandwidth limiting is active through the `zelynic run` / experimental
attach path. `zelynic strict` remains the only validated limiter path.
The canonical deny line "Bandwidth limiting is not active from this
command yet." is tested.

### Invariant 7: Mkdir-Only May Create/Cleanup Target Cgroup

The only real filesystem write in v2.8 is the mkdir-only path (phase 2b),
which may:
- Create `/sys/fs/cgroup/zelynic` if absent (namespace directory).
- Create `/sys/fs/cgroup/zelynic/target_<name>` (target cgroup).
- Remove `/sys/fs/cgroup/zelynic/target_<name>` if empty and
  operation-owned after scope exits.
- Leave `/sys/fs/cgroup/zelynic` in place (namespace directory persists).

This write is gated behind `--mkdir-live` plus all 7 consent flags, root,
and system scope.

## Phase 4 Entry Criteria

Phase 4 (rollback failure simulation / stale PID handling) is the next
implementation phase after the v2.8 freeze. Before phase 4 can begin, all
of the following conditions must be satisfied:

### Gate Conditions

| # | Criterion | Rationale |
|---|-----------|-----------|
| 1 | Root-only | All write operations require euid == 0. |
| 2 | System-scope-only | User-scope mutations are explicitly forbidden. |
| 3 | Single disposable sleep PID only | First PID move target must be `sleep 3` or `sleep 10`. |
| 4 | Immediate rollback required | PID must be restored to original cgroup within the same operation. |
| 5 | No limiter attach | Phase 4 tests rollback, not bandwidth enforcement. |
| 6 | No nft/tc/state | Phase 4 does not touch networking or state subsystems. |
| 7 | No persistent state | Phase 4 rollback data is in-memory only. |
| 8 | Failure simulation plan before broader use | Phase 4 must define explicit failure scenarios to inject and validate. |
| 9 | Exact manual smoke commands reviewed before execution | Every root smoke command must be reviewed and documented before execution. |

### Phase 4 Scope (Not Implemented — Entry Criteria Only)

Phase 4 will introduce tests and guarded paths for edge cases:
- PID exits between move and rollback.
- Original cgroup becomes unavailable after capture.
- Target cgroup is not empty on cleanup.
- Rollback `cgroup.procs` write fails.

Phase 4 will not introduce limiter attach, nftables/tc changes, or
persistent state writes.

## What v2.8 Delivers

v2.8 is a scaffold-and-validation release, not a live-PID-move release.
What v2.8 actually delivers:

1. **Gate infrastructure**: Multi-layered consent flags, runtime conditions,
   and model conditions that any future write path must satisfy.
2. **Executor seam**: A validated, hard-gated bridge between the gate
   checklist and the future live write path. Every gate failure blocks.
   Even with all valid inputs, the seam returns blocked.
3. **Transaction model**: A 10-step move-and-rollback transaction with
   explicit failure policy, documented in code and tested.
4. **Output honesty guarantees**: 7 canonical deny lines tested across
   11 negative-path scenarios. No output can falsely claim PID movement,
   cgroup.procs write, limiter attach, nft/tc/state mutation, persistent
   state write, or bandwidth enforcement.
5. **Mkdir-only experiment**: The first real write in Zelynic history,
   validated in root smoke, with honest output and safe cleanup.
6. **Comprehensive test coverage**: 449 unit tests + 4 integration tests
   covering all gates, output wording, seam behavior, transaction model,
   and negative paths.

## What v2.8 Does NOT Deliver

- No live PID movement.
- No `cgroup.procs` write.
- No limiter attach from the experimental path.
- No bandwidth limiting from `zelynic run`.
- No nftables/tc/Zelynic state mutation.
- No persistent state write.
- No rollback failure simulation (phase 4, future).
- No multi-PID support.
- No user-scope support for writes.
