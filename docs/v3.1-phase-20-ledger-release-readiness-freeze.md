## 1. Purpose

Phase 20 freezes the v3.1 ledger implementation as release-ready. This phase
adds no runtime behavior, changes no CLI parsing, and makes no hidden
command public. Its sole purpose is to document and test that the v3.1
ledger command surface is frozen and ready for final release review while
preserving all safety boundaries.

After this phase, the v3.1 ledger is in a documented, tested, and frozen
state. Any future release, version bump, tag, or publish decision is
intentionally deferred to a separate phase (Phase 21 or later) that
maintainers may choose to invoke.

## 2. No Runtime Behavior Changed

This phase changes no runtime behavior. The production code in
`src/commands/ledger.rs` is completely unchanged from Phase 16. The CLI
parsing in `src/cli.rs` is unchanged. The dispatch in `src/commands/mod.rs`
is unchanged. The only changes are documentation files and a new test
module.

## 3. No CLI Visibility Changed

The `zelynic ledger` subcommand remains `#[command(hide = true)]` in clap
and does not appear in normal `zelynic --help` output. The `Inspect` and
`Export` subcommands also remain hidden. No flag visibility is changed. No
argument parsing is changed. The hidden/experimental safety boundary is
fully preserved.

## 4. Ledger Command Surface (Frozen)

The v3.1 ledger has exactly five user-facing/hidden command shapes, all of
which remain frozen:

1. `zelynic ledger inspect` — In-memory fixture human-readable summary. No
   filesystem access.
2. `zelynic ledger inspect --json` — In-memory fixture inspect summary JSON.
   No filesystem access.
3. `zelynic ledger inspect --file <PATH>` — Explicit read-only file
   inspection with human-readable text output. Path validation before read.
   Schema validation before render.
4. `zelynic ledger inspect --file <PATH> --json` — Explicit read-only file
   inspection with JSON output. Path validation before read. Schema
   validation before render.
5. `zelynic ledger export --json --file <PATH>` — Raw validated ledger JSON
   to stdout only. Path validation before read. Schema validation before
   output.

No other command variant is active, and no other variant will be activated
without a dedicated design phase.

## 5. Frozen JSON Schemas

The following three JSON schemas are frozen and unchanged:

### 5.1 v3.0 Usage JSON Schema

Produced by `zelynic usage --sample --delta --json`. Key fields:
`schema_version` (1), `command`, `sample_mode`, `sample_count`,
`read_count`, `interfaces` (array with per-interface delta fields),
`totals` (aggregate delta), `error`. This schema is completely separate
from the ledger schemas and is not affected by any ledger command.

### 5.2 Ledger Inspect JSON Schema

Produced by `ledger inspect --json` (fixture or `--file`). Uses the
ledger serialization schema: `schema_version` (1), `created_at`,
`updated_at`, `host_id`, `entries` (array with `entry_id`, `timestamp`,
`entry_type`, `source_label`, `interface`, `rx_bytes`, `tx_bytes`,
`combined_bytes`, `read_only`, `provenance`, `attribution_scope`,
`enforcement_status`, `reset_detected`, `reset_details`). No field is
added, removed, or renamed.

### 5.3 Ledger Export JSON Schema

Produced by `ledger export --json --file <PATH>`. Uses the same ledger
serialization schema as inspect JSON. The output is the raw validated
ledger JSON — not a summary, not usage data. No field is added, removed,
or renamed. The export JSON is identical in schema to the inspect JSON;
the difference is in the command semantics (export reads a user-specified
file, inspect may use fixture data).

## 6. Inspect vs Export Boundary

### 6.1 Inspect is Summary

The `ledger inspect` command produces a human-readable summary (text mode)
or a structured summary JSON (JSON mode). The inspect output is derived
from ledger data and rendered for human consumption. The text output
includes disclaimers about safety guarantees, data provenance, and
attribution scope. When `--file` is not provided, the inspect command uses
an in-memory fixture with no filesystem access.

### 6.2 Export is Raw Ledger JSON

The `ledger export --json --file <PATH>` command produces the raw validated
ledger JSON to stdout only. No human header, no disclaimers, no extra text.
The output is pure JSON suitable for piping to `jq` or other tools. The
export reads a user-specified file, validates path and schema, and emits
the full ledger JSON. This is semantically distinct from inspect: inspect
is for human-readable summary views, export is for raw ledger data output.

## 7. Export Boundary

The `ledger export --json --file <PATH>` command operates under these
frozen constraints:

- **stdout-only**: Export writes JSON to stdout only. There is no internal
  file write, no output file flag, and no overwrite mechanism. Users
  redirect stdout with shell redirection if they wish to save the output.
- **Explicit --file required**: The user must specify the source file via
  `--file <PATH>`. There is no implicit default file path, no XDG
  auto-discovery, and no environment variable.
- **--json required**: The `--json` flag is mandatory for export. Human
  export output is not provided. Missing `--json` fails honestly with a
  clear error message.
- **Path validation before read**: Before any filesystem read, the handler
  validates the path for parent traversal, suspicious characters, symlink
  status, regular file status, and existence. Any violation produces a
  clear error before the file is read.
- **Schema validation before output**: After reading the file, the handler
  validates the ledger JSON schema: `schema_version` must be 1, all
  required fields must be present, `combined_bytes` must equal
  `rx_bytes + tx_bytes`, `read_only` must be true, `attribution_scope`
  must be "interface-level only", and `enforcement_status` must be
  "inactive/not implemented". Any violation rejects the export with a
  clear error.
- **No output file flag**: There is no `--output <PATH>`, `--write
  <PATH>`, or `--save <PATH>`. Users redirect stdout with shell
  redirection.
- **No overwrite**: The `--overwrite` flag does not exist and is not
  registered in the clap definition.

## 8. Safety Boundary

The following safety invariants are frozen and must not be weakened by any
future phase without a dedicated design phase:

- **No internal file write**: The production code path contains no file
  write APIs: no `create_dir`, `remove_file`, `remove_dir`, `rename`,
  `fs::copy`, `File::create`, or `OpenOptions`.
- **No persistence/save**: No automatic loading, saving, or caching of
  ledger data. The persistence module returns `Blocked` for all
  operations.
- **No default ledger path read**: All ledger commands require explicit
  user action. No XDG auto-discovery, no environment variable, no implicit
  fallback, no auto-read of any default directory.
- **No live /proc/sysfs read for ledger commands**: The data source for
  ledger commands is either the in-memory fixture (no `--file`) or the
  user-specified JSON file (`--file`). No `/proc/net/dev`, no
  `/sys/class/net/`, no live system interface reads.
- **No live app identity resolver**: Attribution is "interface-level only."
  No per-app identity resolution from live system state. No PID-to-app
  mapping, no cgroup-based attribution, no socket ownership resolution.
- **No enforcement**: `enforcement_status` is always "inactive/not
  implemented." No quota, rate limit, blocking rule, or traffic policy.
  No permission mode, no block mode, no allow mode.
- **No quota**: No quota guard in the ledger path. No bandwidth quota
  tracking, no threshold alerting, no usage-based blocking.
- **No eBPF**: No eBPF code in the ledger path. No BPF program loading,
  no map operations, no perf events.
- **No daemon/watch**: No background process or interval-based mode. No
  file watching, no periodic re-read, no live monitoring.
- **No nft/tc/cgroup/PID/Zelynic mutation**: No limiter attach, no policy
  install, no cleanup, no nftables rule modification, no tc class/filter
  modification, no cgroup assignment, no process state modification, no
  Zelynic state file modification.

## 10. Explicit Non-Goals

This phase does NOT and MUST NOT:

- **No version bump in Phase 20**: The version remains `3.0.1`. A version
  bump is a deliberate release action that requires its own phase.
- **No tag**: No git tag is created. Tagging is a deliberate release action.
- **No GitHub release**: No GitHub release is created. Release creation is
  a deliberate release action.
- **No package publish**: No crates.io, AUR, or other package publication.
- **No v3.2 permission mode**: Per-app permission/block/allow mode is not
  part of v3.1. It requires its own design phase.
- **No v3.3 quota guard**: Bandwidth quota tracking and threshold-based
  blocking is not part of v3.1. It requires its own design phase.
- **No v4 eBPF backend**: eBPF-based traffic shaping is not part of v3.1.
  It requires its own design phase and major version bump.

## 11. Next-Step Boundary

### 11.1 Phase 21 May Be v3.1 Final Release Prep

Phase 21 may be used for v3.1 final release preparation, which may include
a version bump (e.g., to `3.1.0`), a git tag, a GitHub release, and/or a
package publish. This is only if maintainers choose to release. There is
no obligation to release v3.1. The freeze in Phase 20 ensures that the
ledger is ready for release whenever that decision is made.

### 11.2 v3.2 Must Not Begin Until v3.1 Release Boundary Is Closed

Work on v3.2 features (permission mode, quota guard, per-app identity
resolution, enforcement) must not begin until the v3.1 release boundary is
intentionally closed. This means either v3.1 is released (tagged,
published) or explicitly deferred by maintainers. Starting v3.2 work
before closing v3.1 would create confusion about which features belong to
which version.
