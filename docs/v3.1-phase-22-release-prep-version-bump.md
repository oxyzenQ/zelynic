## 1. Phase 22 Performs the v3.1.0 Version Bump

Phase 22 is the release-preparation phase that bumps the project version from
`3.0.1` to `3.1.0` in `Cargo.toml` and `Cargo.lock`. This is the only
metadata change in this phase. No runtime behavior is changed. No CLI parsing
is changed except that the reported package version now reads `3.1.0` instead
of `3.0.1` when `zelynic --version` or `zelynic -V` is invoked.

All version assertions in existing test modules that previously checked for
`3.0.1` have been updated to check for `3.1.0`. Test function names that
previously said `no_version_bump` have been updated to say
`version_is_3_1_0` to reflect the new reality without adding misleading
language. The number of lines in each file remains under 1000 LOC.

## 2. Phase 22 Is Release Prep Only

Phase 22 changes version metadata and release-preparation documentation only.
No new runtime behavior is added. No CLI parsing is changed except the version
string. No hidden command is made public. No command behavior is changed. This
phase converts the project from a v3.1 release-ready candidate at version
`3.0.1` into a v3.1.0 release-prep commit with all safety boundaries intact.

## 3. No Tag / Release / Publish in Phase 22

No git tag is created in Phase 22. No GitHub release is created. No crates.io,
AUR, or other package publication occurs. Tagging, releasing, and publishing
are deliberate release actions that require their own dedicated phase (Phase
23 or later) with proper binary builds, checksum generation, and release
artifacts. Phase 22 only prepares the version state; it does not execute the
release.

## 4. v3.2 / v3.3 / v4 Are Not Started

Phase 22 does not start any work on v3.2 (per-app permission/block/allow mode,
per-app identity resolution, enforcement), v3.3 (quota guard, bandwidth quota
tracking, threshold-based blocking), or v4 (eBPF backend). These features
require their own dedicated design phases and must not begin until the v3.1
release boundary is intentionally closed.

## 5. Command Behavior Unchanged Except Package Version

All five frozen v3.1 ledger command shapes behave identically to Phase 21. The
only observable difference is that `zelynic --version` now reports
`Version: v3.1.0` instead of `Version: v3.0.1`. No command output format,
error message, parsing behavior, or dispatch logic has changed.

## 6. Frozen v3.1 Ledger Command Surface

The v3.1 ledger has exactly five user-facing/hidden command shapes, all of
which remain frozen:

1. `zelynic ledger inspect` — In-memory fixture human-readable summary. No
   filesystem access.
2. `zelynic ledger inspect --json` — In-memory fixture inspect summary JSON.
   No filesystem access.
3. `zelynic ledger inspect --file <PATH>` — Explicit read-only file
   inspection with human-readable text output. Path validation before read.
   Schema validation before render.
4. `zelynic ledger inspect --file <PATH> --json` — Explicit read-only file
   inspection with JSON output. Path validation before read. Schema
   validation before render.
5. `zelynic ledger export --json --file <PATH>` — Raw validated ledger JSON
   to stdout only. Path validation before read. Schema validation before
   output.

No other command variant is active, and no other variant will be activated
without a dedicated design phase.

## 7. Rejected / Absent Command Shapes

The following command shapes are explicitly rejected or absent from the
v3.1 ledger surface:

- `ledger export --json` without --file — rejected, fails honestly
  with "--file <PATH> is required for export"
- `ledger export --file <PATH>` without --json — rejected, fails honestly
  with "--json is required for export"
- `ledger export --output <PATH>` — rejected, the `--output` flag does not
  exist and is not registered in the clap definition
- `ledger export --overwrite` — rejected, the `--overwrite` flag does not
  exist and is not registered in the clap definition
- Save/write/persistence commands — not implemented, the persistence module
  returns `Blocked` for all operations
- Permission/block/allow commands — not part of v3.1, requires its own
  design phase (v3.2 scope)
- Quota commands — not part of v3.1, requires its own design phase
  (v3.3 scope)

These rejections are verified by deterministic tests and must not be
weakened without a dedicated design phase.

## 8. Schema Freeze

### 8.1 Package Version Changes to 3.1.0

The `Cargo.toml` package version field now reads `version = "3.1.0"`. The
`Cargo.lock` is updated to match. This is a metadata-only change.

### 8.2 JSON Schema Versions Do NOT Change

All three JSON schema versions remain at `schema_version: 1`:

- **v3.0 usage JSON** — `schema_version: 1`. Produced by `zelynic usage
  --sample --delta --json`. No field is added, removed, or renamed.
- **Ledger inspect JSON** — `schema_version: 1`. Produced by `ledger
  inspect --json` (fixture or `--file`). No field is added, removed, or
  renamed.
- **Ledger export JSON** — `schema_version: 1`. Produced by `ledger export
  --json --file <PATH>`. No field is added, removed, or renamed.

### 8.3 v3.0 Usage JSON Unchanged

The v3.0 usage JSON schema is completely separate from the ledger schemas
and is not affected by any version bump or ledger command. Key fields:
`schema_version` (1), `command`, `sample_mode`, `sample_count`,
`read_count`, `interfaces`, `totals`, `error`.

### 8.4 Ledger Inspect JSON Unchanged

The ledger inspect JSON uses the ledger serialization schema:
`schema_version` (1), `created_at`, `updated_at`, `host_id`, `entries`
(array with `entry_id`, `timestamp`, `entry_type`, `source_label`,
`interface`, `rx_bytes`, `tx_bytes`, `combined_bytes`, `read_only`,
`provenance`, `attribution_scope`, `enforcement_status`,
`reset_detected`, `reset_details`). No field is added, removed, or
renamed.

### 8.5 Ledger Export JSON Unchanged

The ledger export JSON uses the same ledger serialization schema as inspect
JSON. The output is the raw validated ledger JSON — not a summary, not usage
data. No field is added, removed, or renamed.

## 9. Safety Freeze

The following safety invariants are frozen and must not be weakened by any
future phase without a dedicated design phase:

- **No output file**: No `--output <PATH>`, no `--write <PATH>`, no
  `--save <PATH>`. Export writes to stdout only.
- **No overwrite**: The `--overwrite` flag does not exist and is not
  registered in the clap definition.
- **No internal file write**: The production code path contains no file
  write APIs: no `create_dir`, `remove_file`, `remove_dir`, `rename`,
  `fs::copy`, `File::create`, or `OpenOptions`.
- **No save**: No automatic saving of ledger data. The persistence module
  returns `Blocked` for all operations.
- **No persistence**: No automatic loading, saving, or caching of ledger
  data.
- **No default ledger path read**: All ledger commands require explicit
  user action. No XDG auto-discovery, no environment variable, no implicit
  fallback, no auto-read of any default directory.
- **No live /proc/sysfs read for ledger commands**: The data source for
  ledger commands is either the in-memory fixture (no `--file`) or the
  user-specified JSON file (`--file`). No `/proc/net/dev`, no
  `/sys/class/net/`, no live system interface reads.
- **No live resolver**: Attribution is "interface-level only." No per-app
  identity resolution from live system state. No PID-to-app mapping, no
  cgroup-based attribution, no socket ownership resolution.
- **No enforcement**: `enforcement_status` is always "inactive/not
  implemented." No quota, rate limit, blocking rule, or traffic policy.
  No permission mode, no block mode, no allow mode.
- **No permission/block/allow**: Per-app permission, block, and allow
  modes are not part of v3.1. They require their own design phase.
- **No quota**: No quota guard in the ledger path. No bandwidth quota
  tracking, no threshold alerting, no usage-based blocking.
- **No eBPF**: No eBPF code in the ledger path. No BPF program loading,
  no map operations, no perf events.
- **No daemon/watch**: No background process or interval-based mode. No
  file watching, no periodic re-read, no live monitoring.
- **No nft/tc/cgroup/PID/Zelynic mutation**: No limiter attach, no policy
  install, no cleanup, no nftables rule modification, no tc class/filter
  modification, no cgroup assignment, no process state modification, no
  Zelynic state file modification.

## 11. Next-Step Boundary

### 11.1 Phase 23 May Be Final Tag/Release Package Prep Only

Phase 23 may be used for final tag creation, GitHub release creation, and/or
package publication, but only after Phase 22 is locked and all its tests
pass. There is no obligation to release v3.1. The release-prep in Phase 22
ensures that the version state is correct and ready for release whenever
that decision is made.

### 11.2 v3.2 Permission Mode Must Not Begin Until v3.1 Release Boundary Is Closed

Work on v3.2 features (per-app permission/block/allow mode, quota guard,
per-app identity resolution, enforcement) must not begin until the v3.1
release boundary is intentionally closed. This means either v3.1 is released
(tagged, published) or explicitly deferred by maintainers. Starting v3.2
work before closing v3.1 would create confusion about which features belong
to which version. Phase 22 explicitly does not start any v3.2, v3.3, or v4
work.
