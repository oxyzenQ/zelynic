# Zelynic v2.8 Phase 5c — Guarded Real Move Implementation Design

## Overview

This document designs the future guarded real PID move implementation for the
Zelynic Experimental PID Move Lab. Phase 5c is **design-only**: it produces
this document and updates related documentation but introduces **no runtime
changes, no PID movement, no `cgroup.procs` writes, no limiter attach, and no
state mutation**.

The design specifies the module boundaries, transaction steps, gate
conditions, abort conditions, rollback path, cleanup path, output honesty
contract, manual recovery behavior, test strategy, and root smoke strategy
for the first real PID move. This design is the blueprint for a future
implementation phase; no code is written in phase 5c.

## Phase 5c Boundary

### What Phase 5c Produces

- This implementation design document
  (`docs/v2.8-phase-5c-guarded-real-move-implementation-design.md`).
- Updates to `docs/experimental-pid-move-lab.md` with phase 5b completed and
  phase 5c current status.
- Updates to `CHANGELOG.md` with a phase 5c entry.
- No Rust code changes. No runtime behavior changes. No new CLI flags.

### What Phase 5c Does NOT Do

- Does NOT implement live PID movement.
- Does NOT write `cgroup.procs`.
- Does NOT add runtime mutation of any kind.
- Does NOT add limiter attach, nftables/tc enforcement, or state writes.
- Does NOT change CLI behavior.
- Does NOT modify any Rust source file.
- Does NOT enable any code path for live PID move.
- Does NOT write test code (only a test strategy is specified).

## Allowed Future Live Target

The only acceptable target for the first real PID move is strictly defined.
No deviation from these constraints is permitted without a new design phase.

| Constraint | Requirement | Rationale |
|------------|-------------|-----------|
| Root-only | Must run as UID 0 with CAP_SYS_ADMIN | `cgroup.procs` writes require root for cross-cgroup moves |
| System-scope-only | PID must be in systemd system scope | User-scope cgroups have different delegation rules |
| Single disposable sleep PID | Target must be `sleep N` with no children | Short-lived, no state, no resources to lose |
| Operation-owned target | Target cgroup created by Zelynic under `/sys/fs/cgroup/zelynic/` | Ensures cleanup ownership and safety |
| Immediate rollback | PID must be written back to original cgroup right after move | Minimizes PID time in unfamiliar cgroup |
| No limiter attach | No BPF/tc/enforcement applied to target cgroup | Move is cgroup-only, no bandwidth control |
| No nft/tc/state mutation | No nftables, tc, or Zelynic persistent state changes | Move is cgroup-only, no network or state changes |
| No persistent state write | No journal entries, state files, or config changes | No side effects beyond the cgroup write itself |
| No browser/terminal/desktop | Target must not be any GUI process | Prevents user-visible disruption |
| No user app | Target must not be any application the user relies on | Prevents functional disruption |
| No multi-PID tree | Target must have no child processes | Prevents accidental multi-PID move |

### Writer Trait / Interface

The core abstraction is a `CgroupProcsWriter` trait that separates the
write operation from the transaction logic. Both a live writer (for real
cgroup writes) and a fake writer (for testing) implement this trait.

```rust
/// Abstraction for writing a PID to a cgroup.procs file.
/// The live writer performs real filesystem I/O.
/// The fake writer simulates outcomes for testing.
pub(crate) trait CgroupProcsWriter {
    /// Write a PID string to the cgroup.procs at the given path.
    /// Returns Ok(()) on success, Err(description) on failure.
    fn write_pid(&self, cgroup_path: &str, pid: u32) -> Result<(), String>;

    /// Read PIDs from a cgroup.procs file.
    /// Returns Ok(vec of PIDs) on success, Err(description) on failure.
    fn read_pids(&self, cgroup_path: &str) -> Result<Vec<u32>, String>;

    /// Create a directory (for target cgroup preparation).
    fn mkdir(&self, path: &str) -> Result<(), String>;

    /// Remove an empty directory (for cleanup).
    fn rmdir_if_empty(&self, path: &str) -> Result<(), String>;
}
```

### Live Writer: Narrow and Isolated

The live writer is intentionally narrow. It performs exactly two types of
filesystem operations: writing PID strings to `cgroup.procs` files, and
creating/removing directories under `/sys/fs/cgroup/zelynic/`. It must not:

- Know about limiters, BPF programs, tc classes, or enforcement.
- Know about nftables rulesets or tc qdisc configurations.
- Read or write Zelynic persistent state.
- Perform any network operation.
- Perform any operation outside `/sys/fs/cgroup/zelynic/` (for mkdir/rmdir)
  or the explicitly provided cgroup.procs paths (for read/write).

```rust
/// Live writer that performs real filesystem operations on cgroup.procs.

/// Isolated: no knowledge of limiters, nft/tc, or Zelynic state.

impl CgroupProcsWriter for LiveCgroupWriter {
    fn write_pid(&self, cgroup_path: &str, pid: u32) -> Result<(), String> {

        // Validates: path starts with /sys/fs/cgroup/
        // Does NOT: attach limiter, mutate nft/tc/state
        ...

        // Parses: one PID per line
        ...

    fn mkdir(&self, path: &str) -> Result<(), String> {
```

### Fake Writer Remains Test-Only

The existing fake writer (`failure_simulation/fake_writer.rs`) remains
unchanged and test-only. It implements the same `CgroupProcsWriter` trait
via a test adapter, but the fake writer's injectable failure modes and
pure-function semantics are preserved. The fake writer is never reachable
from production code paths.

### Transaction Controller

The transaction controller decides when rollback is required based on the
outcome of each step. It follows the decision logic validated by the phase
4b failure simulation model:

| Target Write Result | Rollback Required | Reason |
|---------------------|-------------------|--------|
| Not attempted (pre-target failure) | No | PID was never written |
| Succeeded | Yes | PID may be in target |
| Failed | Maybe | Kernel may have partially completed the write |

If rollback is required, the controller attempts exactly one rollback write.
No retry. No loop.

### Renderer

The renderer reports the verified PID location using the canonical labels
from the phase 4a taxonomy. It produces honest output for each transaction
step, including the 7 canonical deny lines with honest substitutions where
real operations occurred.

## Exact Transaction Steps

The guarded real move follows a 10-step transaction model (from phase 3a),
augmented with pre-flight gates and post-operation audit. Each step has
explicit entry conditions, actions, and exit conditions.

| Step | Name | Type | Writes? | Rollback if Failed? |
|------|------|------|---------|-------------------|
| 0 | Pre-flight gates | Gate | No | No (abort) |
| 1 | PID liveness re-check | Gate | No | No (abort) |
| 2 | Original cgroup validation | Gate | No | No (abort) |
| 3 | Target cgroup preparation | Write (mkdir) | Yes | Cleanup target if created |
| 4 | Move PID to target | Write (cgroup.procs) | Yes | See failure handling |
| 5 | Verify PID in target | Read | No | Rollback required |
| 6 | Record move success | In-memory | No | Rollback required |
| 7 | Rollback PID to original | Write (cgroup.procs) | Yes | Report loudly |
| 8 | Verify PID restored | Read | No | Report unverified |
| 9 | Cleanup target cgroup | Write (rmdir) | Yes | Leave and report |
| 10 | Post-smoke audit | Audit | No | N/A |

### Step 0: Pre-flight Gates

Before any write operation, all gate conditions must pass. If any gate
fails, the transaction aborts at step 0 with no rollback needed.

### Steps 3–6: Move Phase

Steps 3 through 6 constitute the move phase. If step 4 (the move write)
succeeds, rollback at step 7 MUST be attempted regardless of what happens
at steps 5 or 6. If step 4 fails, rollback is only required if the kernel
may have partially completed the write (which is always possible with
cgroup.procs).

### Steps 7–9: Rollback and Cleanup Phase

Steps 7 through 9 constitute the rollback and cleanup phase. Step 7 is
attempted exactly once. Step 8 verifies the rollback. Step 9 cleans up
the target cgroup only if it is empty and operation-owned.

### Step 10: Post-Smoke Audit

After all operations complete, a post-smoke audit verifies that no
unexpected state changes occurred: no nft/tc rules, no Zelynic state
changes, no persistent state writes.

## Exact Safety Gates

Each gate condition is checked before any write operation. If any gate
fails, the transaction aborts immediately with no PID move and no rollback.

| # | Gate | Check | Abort If |
|---|------|-------|----------|
| G1 | Root check | `getuid() == 0` | Not root |
| G2 | System scope check | PID's cgroup is NOT under `/user.slice/` | User scope detected |
| G3 | Single PID check | Target scope contains exactly one PID | Multiple PIDs |
| G4 | PID liveness re-check | `kill(pid, 0)` succeeds | PID exited |
| G5 | Original cgroup captured | `/proc/<pid>/cgroup` read succeeds | Read failure |
| G6 | Original cgroup verified safe | Path does NOT contain `zelynic/` | Already Zelynic-managed |
| G7 | Original cgroup exists | Directory at `/sys/fs/cgroup/<path>` exists | Missing |
| G8 | Original cgroup.procs writable | `test -w` on original `cgroup.procs` | Not writable |
| G9 | Target path under zelynic | Path starts with `/sys/fs/cgroup/zelynic/` | Outside namespace |
| G10 | Target empty or new | `cgroup.procs` at target is empty or dir does not exist | Non-empty |
| G11 | Target operation-owned | Target dir was created by this operation | Pre-existing and non-empty |
| G12 | Cgroup mount writable | `/sys/fs/cgroup` mount is `rw` | Read-only mount |
| G13 | Rollback consent present | `--rollback-required` flag is set | Not present |
| G14 | No limiter path reachable | No code path from transaction to limiter attach | Design violation |

Gate G14 is a design-time gate, not a runtime check. The implementation
architecture must ensure that no code path exists from the transaction
controller to the limiter module. This is enforced by the narrow live writer
interface that has no limiter-related methods.

## Abort Conditions

The following conditions cause an immediate abort of the transaction. No
`cgroup.procs` write is performed. No PID is moved. No rollback is claimed.

| # | Condition | Detection Point | Abort Action |
|---|-----------|-----------------|--------------|
| A1 | Not root | Gate G1 | Report "not root", stop |
| A2 | Not system scope | Gate G2 | Report "user scope forbidden", stop |
| A3 | Multiple PIDs discovered | Gate G3 | Report "multiple PIDs", stop |
| A4 | PID missing or stale | Gate G4, or step 1 | Report "PID dead", stop |
| A5 | Original cgroup missing | Gate G7, or step 2 | Report "original cgroup gone", stop |
| A6 | Original cgroup Zelynic-managed | Gate G6 | Report "already Zelynic-managed", stop |
| A7 | Target outside zelynic namespace | Gate G9 | Report "target path violation", stop |
| A8 | Target non-empty | Gate G10 | Report "target not empty", stop |
| A9 | Cgroup mount read-only | Gate G12 | Report "cgroup read-only", stop |
| A10 | Permissions unexpected | Any mkdir/write EACCES | Report "permission denied", stop |
| A11 | Rollback target unverifiable | Gate G8 | Report "rollback target uncertain", stop |
| A12 | Ambiguous output | Any step returns unexpected data | Report "ambiguous", stop |

## Failure Handling

Failure handling follows the 9 universal failure rules from phase 4a and
the 12 failure scenarios (F1–F12). The response for each failure point is:

### Pre-Target Failure (Steps 0–2)

If the transaction fails before step 4 (the move write), no PID was moved.
No rollback is needed. No rollback is claimed.

- Report: "Transaction aborted at step <N>: <reason>."
- Report: "No PID was moved."
- Report: "No cgroup.procs write was performed."
- PID location: "not moved"
- Cleanup: Remove target if it was created (step 3) and is empty.

### Target Write Failure (Step 4)

If the move write fails, the PID may or may not have been moved (the kernel
may have partially completed the write). Rollback MUST be attempted.

- Report: "cgroup.procs write to <target> failed: <errno>."
- Attempt rollback (step 7).
- If rollback succeeds: PID location "verified restored"
- If rollback fails: PID location "unknown", manual recovery required

### Verification Failure After Target Write (Step 5–6)

If target verification fails after the move write succeeded, the PID is in
an unknown state. Rollback MUST be attempted.

- Report: "Target verification failed: <reason>."
- Attempt rollback (step 7).
- PID location: "unknown" (if verification was skipped) or "verified in
  target" (if verification succeeded before the failure)

### Rollback Write Failure (Step 7)

If the rollback write fails, the PID may be stranded in the target cgroup.

- Report: "ROLLBACK FAILURE: cgroup.procs write to <original> failed:
  <errno>."
- Report: "No further rollback attempts. Single attempt was the maximum."
- PID location: "unknown"
- Manual recovery required: yes
- For a disposable `sleep` PID: the PID exits on its own. Safe to wait.

### Rollback Verification Failure (Step 8)

If the rollback write succeeded but verification failed, the rollback may or
may not have worked.

- Report: "Rollback write completed. Verification skipped: <reason>."
- PID location: "rollback unverified"
- Manual recovery required: no (write succeeded, verification is uncertain)

### Cleanup Failure (Step 9)

If target cleanup fails, the target directory remains.

- Report: "Target cleanup failed: <reason>."
- Report: "Leftover target cgroup: <path>."
- PID location: unchanged by cleanup failure
- Target may remain: yes

### Never Retry

No retry loop exists for PID movement. A `cgroup.procs` write (move or
rollback) is attempted at most once. No exponential backoff, no "try again
with different flags", no second attempt.

## Output Honesty Contract

Every transaction produces output that honestly describes what happened. The
output must satisfy the following contract.

### Required Fields

| Field | Values | When |
|-------|--------|------|
| PID moved status | `moved` / `not moved` | Based on step 4 outcome |
| PID location | `verified in target` / `verified restored` / `rollback unverified` / `unknown` / `not moved` | Based on verification outcomes |
| Rollback attempted | `yes` / `no` | `yes` if step 4 succeeded |
| Rollback outcome | `succeeded` / `failed` / `unverified` / `not attempted` | Based on step 7–8 outcome |
| Target cleanup status | `succeeded` / `failed` / `skipped` / `not needed` | Based on step 9 outcome |
| Manual recovery required | `yes` / `no` | Based on PID location and rollback outcome |

### Forbidden Claims (Never Present in Output)

- "Limiter was attached." — No limiter is ever attached.
- "Bandwidth limiting is active." — No bandwidth limiting is applied.
- "nft/tc/state mutation was performed." — No state mutation occurs.
- "Persistent state was written." — No persistent state is written.
- "PID was restored." (if rollback failed) — Must say "ROLLBACK FAILURE".
- "Target was cleaned up." (if rmdir failed) — Must say "cleanup failed".
- "No PID was moved." (if step 4 succeeded) — Must say "PID was moved".
- "Rollback was performed." (if step 4 failed and PID not moved) — Must say
  "No rollback was required."

### Canonical Deny Lines

Every output must include the 7 canonical deny lines, with honest
substitutions where real operations occurred:

1. No live PID move was performed → or "PID <pid> was moved to target."
2. No real cgroup.procs write was performed → or "cgroup.procs write details."
3. No limiter attach was performed. (always true)
4. No nftables, tc, or Zelynic state changes were made. (always true)
5. No persistent state write was performed. (always true)
6. No CLI path for live PID move was enabled. (always true for first smoke)
7. This is fake/model-only output → or "This is a real smoke experiment output."

## Manual Recovery Behavior

If the transaction encounters a failure after step 4 and the PID is
stranded, the manual recovery behavior follows the phase 5a/5b recovery
procedures.

### Automatic Recovery (In-Transaction)

The transaction controller automatically attempts rollback at step 7 if
step 4 succeeded. This is the only automatic recovery action. If it fails,
the transaction reports "ROLLBACK FAILURE" and sets manual recovery required.

### Manual Recovery (Operator-Initiated)

If automatic rollback fails and the PID is still alive:

1. Read `/proc/<pid>/cgroup` to determine current PID location.
2. Verify the original cgroup directory exists and is writable.
3. Write PID to original `cgroup.procs`: `echo <pid> > /sys/fs/cgroup/<original>/cgroup.procs`
4. Verify restoration: `cat /proc/<pid>/cgroup`
5. Check target emptiness: `cat /sys/fs/cgroup/zelynic/<target>/cgroup.procs`
6. If empty: `rmdir /sys/fs/cgroup/zelynic/<target>`
7. If non-empty: leave in place, do not force-remove.
8. Stop sleep scope: `systemctl stop zelynic-smoke`

Do NOT create new cgroups as restore targets. Do NOT write outside
`/sys/fs/cgroup`. Do NOT delete non-empty cgroups.

## Test Strategy

### Unit Tests Using Fake Writer

The fake writer from phase 4c (`failure_simulation/fake_writer.rs`) will be
adapted to implement the `CgroupProcsWriter` trait via a test adapter. All
transaction logic is tested through this adapter with no real I/O.

| Test Category | Count | Coverage |
|--------------|-------|----------|
| Gate checks | ~15 | Each gate condition (G1–G14) pass/fail |
| Abort conditions | ~12 | Each abort condition (A1–A12) |
| Happy path | ~5 | Full transaction succeeds, rollback succeeds |
| Target write failure | ~5 | EACCES, ENOENT, EBUSY, ESRCH, partial |
| Verification failure | ~3 | Read EACCES, PID exited, ambiguous output |
| Rollback failure | ~5 | EACCES, ENOENT, EBUSY, ESRCH, original gone |
| Cleanup failure | ~3 | EBUSY, ENOTEMPTY, EACCES |
| Output honesty | ~10 | Required fields, forbidden claims, deny lines |
| Determinism | ~3 | Same inputs produce same outputs |
| Render structure | ~5 | All sections present, correct format |

Estimated: ~66 new unit tests, all using fake writer, all in-memory.

### Real Writer Path: Compiled but Not Reachable

The live writer module is compiled as part of the normal build, but no code
path reaches it without all 14 gate conditions passing. The gate conditions
require:

- Root (UID 0) — never true in CI or non-root environments
- `--rollback-required` flag — explicit opt-in
- `--experimental-single-pid-attach` — explicit opt-in
- `--i-understand-this-moves-pids` — explicit opt-in
- System scope — not true for user sessions

These gates are ANDed together. The live writer path is dead code in all
normal execution contexts.

### CLI Remains Blocked

The CLI does not expose a direct command for the PID move. The move is only
reachable through the experimental attach gate path, which requires all
consent flags. No new CLI flags are added in the implementation phase.

### No Forbidden Claims

All tests verify that the output never contains forbidden claims (limiter
attach, bandwidth limiting active, nft/tc/state mutation performed).

### No Limiter/nft/tc/State Interaction

All tests verify that the live writer never calls any limiter, nftables,
tc, or Zelynic state module. The live writer type has no fields or methods
related to these subsystems.

## Root Smoke Strategy

The root smoke follows the operator checklist from phase 5b. The strategy
is:

1. **Prepare**: Run the 7 preflight environment checks from the phase 5b
   operator checklist. All must pass.

2. **Create target**: Start a disposable `sleep 30` in a systemd scope.
   Capture PID. Capture original cgroup. Create target cgroup.

3. **Execute**: Run the Zelynic binary (or a test binary) with all gate
   flags, targeting the disposable sleep PID. Observe output at each step.

4. **Verify**: After the transaction completes, verify PID location, target
   cleanup, and no state changes using the post-smoke audit checklist.

5. **Report**: Collect all output and compare against the expected
   observation table from phase 5b.

6. **Cleanup**: Stop the sleep scope. Verify no leftover state.

The root smoke is performed manually by the operator, not automated in CI.
The root smoke requires physical console or SSH access to a root shell.
