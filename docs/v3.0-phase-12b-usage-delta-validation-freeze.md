# v3.0 Phase 12b: Usage Delta Validation Freeze + LOC Split

## Purpose

Phase 12b freezes and audits the first two-sample delta CLI implementation
(phase 12), then reduces the LOC pressure on `src/commands/usage_delta.rs`
before future delta JSON work (phase 13). This phase is validation/documentation
plus a pure refactor/split — no new runtime behavior is introduced.

Phase 12b follows the same pattern as phase 3b (live_proc_net_dev test LOC
split) and phase 5b (usage --sample validation freeze): audit the current
implementation, produce a freeze document, and refactor for maintainability.

## Phase Summary: 9 through 12

### Phase 9: Delta Sampling Design

Defined the two-sample delta protocol: read `/proc/net/dev` twice, wait a
bounded duration, compute per-interface byte deltas using the existing
`SessionDelta` model from v2.9, render honest output with 16 safety
disclaimers. Designed counter reset handling (saturating delta = 0 with
warning), error handling at three points (first read, wait, second read),
and the JSON delta contract direction. Phase 9 was design-only — no Rust
code changes, no CLI flag registration, no test additions.

### Phase 10: Pure Delta Output Model and Render Tests

Implemented `src/accounting/usage_delta.rs` (284 LOC) with pure Rust model
for delta output rendering. Model types: `UsageDeltaOutput` (per-interface
delta rows, totals, warnings, interface count), `UsageDeltaRow` (interface
name, RX/TX/combined delta bytes, has_reset flag). Pure functions:
`build_usage_delta_from_session_delta()` transforms `SessionDelta` into
display-oriented output, `render_usage_delta()` renders human-readable text
with 14+ safety disclaimers. 33 tests in
`src/accounting/tests/usage_delta.rs` covering builds, totals, empty delta,
interface add/remove, counter reset warnings, all honesty lines,
determinism, source label, structural safety.

### Phase 11: CLI Gate Design / Blocked Flag Contract

Produced design/gate document defining 12 activation gates that must be
satisfied before `--delta` is enabled. Defined absolute constraints:
interface-level only, no per-app attribution, no quota enforcement, no
network blocking, no persistence, no eBPF, no nft/tc mutation, no
cgroup/PID mutation, no loop/watch, no arbitrary path, no filesystem
write, read-only. Defined delta-specific constraints: exactly two reads,
bounded wait, no background loop, no daemon behavior, counter reset
handling, single-shot. Phase 11 was design-only.

### Phase 12: Single-Shot Two-Sample Delta Implementation

Implemented actual `zelynic usage --sample --delta` command. Added `--delta`
flag to CLI with `requires = "sample"`. Created `src/commands/usage_delta.rs`
(~920 LOC) with `handle_usage_delta()` that reads `/proc/net/dev` exactly
twice via existing reader seam, waits 1 second, computes deltas, renders
live text with 16 safety disclaimers, and exits. `--delta --json` rejected
before any live read. 32 new tests with injected `DualSampleReader` and
`SampleSleeper` traits for test injection without touching the filesystem.

## Delta CLI Behavior Proof

### `zelynic usage --sample --delta`

- Reads `/proc/net/dev` exactly twice via existing `read_live_proc_net_dev()`
- Waits bounded default 1 second between samples (`DEFAULT_DELTA_WAIT_DURATION`)
- Computes per-interface deltas using existing `build_session_delta()` model
- Renders text output with 16 safety disclaimers
- Exits after second sample — no loop, no watch, no daemon

### `zelynic usage --sample --delta --json`

- Rejected before any live read with error message
- Delta JSON is deferred to phase 13
- No `/proc/net/dev` read occurs when this combination is used

### `zelynic usage --delta`

- Rejected by clap because `--delta` has `requires = "sample"`
- No `/proc/net/dev` read occurs — the flag is never accepted without `--sample`

### Negative Behavior Proofs

- No loop/watch mode: `--watch` flag not accepted by clap
- No interval flag: `--interval` flag not accepted by clap
- No interface filter: `--interface` flag not accepted by `usage` subcommand
- No arbitrary path input: `DualSampleReader` and `ContentReader` traits have
  no path parameter; source path hardcoded to `/proc/net/dev`
- No sysfs read: no sysfs paths referenced anywhere in the module
- No ledger persistence: no ledger file read/write functions called
- No state mutation: no Zelynic runtime state is written
- No enforcement/blocking: all 16 honesty disclaimers explicitly deny these

## LOC Split: `src/commands/usage_delta.rs`

### Before Split

| File | LOC |
|------|-----|
| `src/commands/usage_delta.rs` | ~920 (single file) |

### After Split

| File | LOC | Content |
|------|-----|---------|
| `src/commands/usage_delta/mod.rs` | ~35 | Module doc, submodule declarations, re-exports `handle_usage_delta` and `render_usage_delta_live` |

All files remain under 1000 LOC. Total LOC slightly increased due to repeated
module docs and imports per file, but each file is now focused and under the
policy limit with comfortable margin for future phase 13 work.

### What Changed

- File structure: single file → directory module with 4 focused files
- Module-level doc comment appears in `mod.rs` instead of on the single file
- Sub-module doc comments added to `handler.rs` and `render.rs`
- Import paths adjusted for new module structure (e.g., `super::render::render_usage_delta_live`)
- `handle_usage_delta` re-exported from `handler.rs` via `mod.rs` so external
  importers (`usage.rs`) continue to work without changes

### What Did NOT Change

- All 65 usage_delta tests preserved with identical behavior
- All test assertions unchanged
- All output wording in `render_usage_delta_live()` unchanged character-for-character
- Public API: `handle_usage_delta()` remains `pub(crate)` at the same path
- Runtime behavior: identical — same two reads, same wait, same delta computation,
  same render output
- No new tests added (preserving exact test count)
- No new dependencies
- No new CLI flags
- No changes to any other source files except the split itself
