## 1. Purpose

Phase 21 is the final audit gate before any version bump, tag, GitHub
release, or package publish for v3.1. This phase performs one last
comprehensive audit proving that v3.1 is release-ready as a candidate. The
audit verifies documentation completeness, schema freeze integrity, safety
boundary preservation, command surface stability, and test coverage.

This phase is explicitly final audit only. No version bump occurs in Phase
21. No tag is created. No GitHub release is published. No package is
published. These actions are intentionally deferred to a separate phase
(Phase 22 or later) that maintainers may choose to invoke. The purpose of
this phase is solely to prove readiness, not to execute the release itself.

## 2. No Version Bump in Phase 21

The version remains `3.0.1` throughout Phase 21. A version bump is a
deliberate release action that requires its own dedicated phase. Phase 21
does not modify `Cargo.toml`, does not change the version string, and does
not create any version-related artifacts. The version bump decision and
execution belong to a future release-preparation phase.

## 3. No Tag / Release / Publish in Phase 21

No git tag is created in Phase 21. No GitHub release is created. No
crates.io, AUR, or other package publication occurs. Tagging, releasing,
and publishing are deliberate release actions that require their own
dedicated phase with proper changelog finalization, binary builds, and
checksum generation. Phase 21 only audits readiness; it does not execute
the release.

## 4. v3.1 Is Release-Ready Candidate After Phase 20

After Phase 20, the v3.1 ledger implementation is in a documented, tested,
and frozen state. Phase 20 established the release-readiness freeze by
documenting all five frozen command shapes, all three frozen JSON schemas,
the full safety boundary, and a release readiness checklist. Phase 21
builds on top of that freeze by performing a final audit that re-confirms
every boundary and checklist item before any release action.

The v3.1 release-ready candidate status means: all ledger commands work as
documented, all schemas are frozen, all safety boundaries are preserved,
all tests pass, the production code has no forbidden APIs, and the
documentation is complete and cross-referenced.

## 5. Phase 21 Is the Audit Gate Before Version Bump / Release Prep

Phase 21 serves as the final quality gate between the frozen v3.1
candidate and any future release-preparation phase. No release-preparation
work (version bump, tag, release, publish) should begin without Phase 21
being locked and its tests passing. This ensures that the release-prep
phase starts from a known-good, fully audited state.

After Phase 21 is locked, a future phase (Phase 22) may proceed with
release preparation activities, but only if maintainers explicitly choose
to do so. There is no obligation to release v3.1.

## 6. Frozen v3.1 Ledger Command Surface

The v3.1 ledger has exactly five user-facing/hidden command shapes, all of
which remain frozen:

1. `zelynic ledger inspect` — In-memory fixture human-readable summary. No
   filesystem access.
2. `zelynic ledger inspect --json` — In-memory fixture inspect summary JSON.
   No filesystem access.
3. `zelynic ledger inspect --file <PATH>` — Explicit read-only file
   inspection with human-readable text output. Path validation before read.
   Schema validation before render.
4. `zelynic ledger inspect --file <PATH> --json` — Explicit read-only file
   inspection with JSON output. Path validation before read. Schema
   validation before render.
5. `zelynic ledger export --json --file <PATH>` — Raw validated ledger JSON
   to stdout only. Path validation before read. Schema validation before
   output.

No other command variant is active, and no other variant will be activated
without a dedicated design phase.

## 7. Rejected / Absent Command Shapes

The following command shapes are explicitly rejected or absent from the
v3.1 ledger surface:

- `ledger export --json` without --file — rejected, fails honestly
  with "--file <PATH> is required for export"
- `ledger export --file <PATH>` without --json — rejected, fails honestly
  with "--json is required for export"
- `ledger export --output <PATH>` — rejected, the `--output` flag does not
  exist and is not registered in the clap definition
- `ledger export --overwrite` — rejected, the `--overwrite` flag does not
  exist and is not registered in the clap definition
- Persistence save/write commands — not implemented, the persistence module
  returns `Blocked` for all operations
- Permission/block/allow commands — not part of v3.1, requires its own
  design phase (v3.2 scope)
- Quota commands — not part of v3.1, requires its own design phase
  (v3.3 scope)

These rejections are verified by deterministic tests and must not be
weakened without a dedicated design phase.

## 8. Schema Freeze

The following three JSON schemas are frozen and unchanged:

### 8.1 v3.0 Usage JSON

Produced by `zelynic usage --sample --delta --json`. Key fields:
`schema_version` (1), `command`, `sample_mode`, `sample_count`,
`read_count`, `interfaces` (array with per-interface delta fields),
`totals` (aggregate delta), `error`. This schema is completely separate
from the ledger schemas and is not affected by any ledger command. No
field is added, removed, or renamed.

### 8.2 Ledger Inspect JSON

Produced by `ledger inspect --json` (fixture or `--file`). Uses the
ledger serialization schema: `schema_version` (1), `created_at`,
`updated_at`, `host_id`, `entries` (array with `entry_id`, `timestamp`,
`entry_type`, `source_label`, `interface`, `rx_bytes`, `tx_bytes`,
`combined_bytes`, `read_only`, `provenance`, `attribution_scope`,
`enforcement_status`, `reset_detected`, `reset_details`). No field is
added, removed, or renamed.

### 8.3 Ledger Export JSON

Produced by `ledger export --json --file <PATH>`. Uses the same ledger
serialization schema as inspect JSON. The output is the raw validated
ledger JSON — not a summary, not usage data. No field is added, removed,
or renamed. The export JSON is identical in schema to the inspect JSON;
the difference is in the command semantics (export reads a user-specified
file, inspect may use fixture data).

## 9. Safety Freeze

The following safety invariants are frozen and must not be weakened by any
future phase without a dedicated design phase:

- **No output file**: No `--output <PATH>`, no `--write <PATH>`, no
  `--save <PATH>`. Export writes to stdout only.
- **No overwrite**: The `--overwrite` flag does not exist and is not
  registered in the clap definition.
- **No internal file write**: The production code path contains no file
  write APIs: no `create_dir`, `remove_file`, `remove_dir`, `rename`,
  `fs::copy`, `File::create`, or `OpenOptions`.
- **No save**: No automatic saving of ledger data. The persistence module
  returns `Blocked` for all operations.
- **No persistence**: No automatic loading, saving, or caching of ledger
  data.
- **No default ledger path read**: All ledger commands require explicit
  user action. No XDG auto-discovery, no environment variable, no implicit
  fallback, no auto-read of any default directory.
- **No live /proc/sysfs read for ledger commands**: The data source for
  ledger commands is either the in-memory fixture (no `--file`) or the
  user-specified JSON file (`--file`). No `/proc/net/dev`, no
  `/sys/class/net/`, no live system interface reads.
- **No live resolver**: Attribution is "interface-level only." No per-app
  identity resolution from live system state. No PID-to-app mapping, no
  cgroup-based attribution, no socket ownership resolution.
- **No enforcement**: `enforcement_status` is always "inactive/not
  implemented." No quota, rate limit, blocking rule, or traffic policy.
  No permission mode, no block mode, no allow mode.
- **No permission/block/allow**: Per-app permission, block, and allow
  modes are not part of v3.1. They require their own design phase.
- **No quota**: No quota guard in the ledger path. No bandwidth quota
  tracking, no threshold alerting, no usage-based blocking.
- **No eBPF**: No eBPF code in the ledger path. No BPF program loading,
  no map operations, no perf events.
- **No daemon/watch**: No background process or interval-based mode. No
  file watching, no periodic re-read, no live monitoring.
- **No nft/tc/cgroup/PID/Zelynic mutation**: No limiter attach, no policy
  install, no cleanup, no nftables rule modification, no tc class/filter
  modification, no cgroup assignment, no process state modification, no
  Zelynic state file modification.

## 11. Next-Step Boundary

### 11.1 Phase 22 May Be v3.1 Release Prep / Version Bump Only

Phase 22 may be used for v3.1 release preparation, which may include a
version bump (e.g., to `3.1.0`), a git tag, a GitHub release, and/or a
package publish. This is only permissible after Phase 21 is locked and all
its tests pass. There is no obligation to release v3.1. The audit in
Phase 21 ensures that the ledger is ready for release whenever that
decision is made.

### 11.2 v3.2 Permission Mode Must Not Begin Until v3.1 Release Boundary Is Closed

Work on v3.2 features (per-app permission/block/allow mode, quota guard,
per-app identity resolution, enforcement) must not begin until the v3.1
release boundary is intentionally closed. This means either v3.1 is
released (tagged, published) or explicitly deferred by maintainers.
Starting v3.2 work before closing v3.1 would create confusion about which
features belong to which version. Phase 21 explicitly does not start any
v3.2, v3.3, or v4 work.
