# Zelynic v2.9 Network Accounting Lab — Design Document

This document captures the design for the v2.9 Network Accounting Lab, a
read-only accounting milestone that establishes the data model, data sources,
and architecture for future quota enforcement, per-app usage history,
allow/block modes, and an eventual eBPF observer backend. The v2.9 phase 1 is
**design-only**: it produces this document and updates related documentation but
introduces no runtime changes, no enforcement, and no system mutations.

## Purpose

v2.9 is a read-only accounting foundation. Unlike v2.8 (which explored PID
movement and cgroup writes as a safety/research milestone), v2.9 focuses on
observing and recording network usage without blocking, shaping, or enforcing
any traffic policy. The design is intentionally observational:

- v2.9 does NOT implement quota enforcement.
- v2.9 does NOT block any network traffic.
- v2.9 does NOT attach limiters or mutate nftables/tc rules.
- v2.9 does NOT move PIDs or write `cgroup.procs`.
- v2.9 does NOT implement eBPF (reserved for a future milestone).
- v2.9 does NOT write Zelynic runtime state.
- v2.9 does NOT touch `/sys/fs/cgroup` for accounting purposes.

The goal is to build a clean, honest data model for network usage that can be
consumed by future features such as per-app quota budgets, background data
guards, tethering-aware limits, and session-based usage tracking. All accounting
in v2.9 is read-only: it reads system counters, parses them, and renders
structured output. No writes, no enforcement, no mutation.

## Hard Safety Boundary

v2.9 enforces the following hard safety boundary throughout all phases:

| Prohibited Action | Status |
|-------------------|--------|
| eBPF program loading/attachment | Hard-blocked in v2.9 |
| Quota enforcement (block/throttle on budget) | Hard-blocked in v2.9 |
| Network traffic blocking | Hard-blocked in v2.9 |
| Limiter attach (strict/backend attach) | Hard-blocked in v2.9 |
| nftables/tc rule mutation | Hard-blocked in v2.9 |
| Zelynic runtime state mutation | Hard-blocked in v2.9 |
| PID movement | Hard-blocked in v2.9 |
| `cgroup.procs` write | Hard-blocked in v2.9 |
| `/sys/fs/cgroup` access | Hard-blocked in v2.9 |
| Persistent state write | Hard-blocked in v2.9 (phase 5 may design a ledger, not implement it) |

These boundaries are absolute for v2.9. Any feature that requires mutation is
explicitly deferred to a future version milestone.

## Read-Only Accounting Milestone Scope

v2.9 is scoped as a **read-only accounting milestone**. This means:

- **No enforcement**: Accounting data is observed and reported, never used to
  trigger automatic blocking, throttling, or rate-limiting actions.
- **No blocking**: No network traffic is ever blocked, dropped, or rejected by
  v2.9 code. The `zelynic strict` path remains the only validated limiter and
  is not modified by v2.9.
- **No quota guard yet**: Future quota budget tracking is designed but not
  implemented. No commands that enforce quota limits exist in v2.9.
- **No eBPF yet**: The design acknowledges eBPF as a future advanced backend
  for per-app attribution, but v2.9 uses only existing kernel counter
  interfaces (`/proc/net/dev`, sysfs, nftables counters where available).
- **No PID movement**: v2.9 does not move PIDs or interact with cgroups for
  accounting. Per-app attribution is a future challenge.
- **No `cgroup.procs` write**: v2.9 never writes to cgroup.procs files.
- **No nft/tc mutation**: v2.9 never adds, removes, or modifies nftables rules
  or tc qdisc/filters/classes. If nftables counters are read, it is a read-only
  operation on rules that already exist from `zelynic strict`.
- **No persistent state mutation in phase 1**: Phase 1 is design-only. A local
  ledger design in phase 5 would be design-only. No file writes occur in v2.9
  unless explicitly approved in a later phase.

## Future Accounting Targets

The v2.9 accounting model is designed to eventually support the following
targets. Each target is listed with its current feasibility and limitations.

### Global Network Usage

Track aggregate RX/TX bytes and packets across all interfaces. This is the
simplest accounting target: read `/proc/net/dev` or sysfs interface counters
and sum across all non-loopback interfaces. Useful for overall data budget
monitoring (e.g., "I used 2.3 GB today") but provides no per-app breakdown.

Data source: `/proc/net/dev` (cumulative since boot), sysfs
`/sys/class/net/<iface>/statistics/`.

### Per-Interface Usage (Especially Wi-Fi/Tethering)

Track RX/TX bytes per network interface. This is critical for users who switch
between Wi-Fi, Ethernet, and mobile tethering and want per-interface budgets
(e.g., "limit background downloads on tethering to 100 MB"). Interface counters
are readily available and reset only on interface down/up or reboot.

Data source: `/proc/net/dev` per-interface lines, sysfs
`/sys/class/net/<iface>/statistics/rx_bytes` and `tx_bytes`.

### Per-Session Usage

Track usage within a defined session (e.g., "since I opened this terminal" or
"since the last billing cycle reset"). This requires computing deltas between
counter snapshots. Session accounting needs a reference point (snapshot) and a
clock. No persistence is needed for a simple since-boot session; persistent
session tracking would require a local ledger.

Data source: delta between two `/proc/net/dev` reads, with timestamps.

### Since-Boot Usage

Track usage since system boot. Since `/proc/net/dev` counters are cumulative
since interface creation (typically since boot), this is the natural default.
No snapshot or ledger is needed; the counter itself is the since-boot total.

Data source: `/proc/net/dev` directly (cumulative counters).

### Per-Target/App/Cgroup Usage

Track usage attributable to a specific application, binary name, PID, or cgroup.
This is the hardest accounting target and requires per-flow attribution:

- **cgroup-based**: If a process is already in a Zelynic-managed cgroup (via
  `zelynic strict`), tc class byte/packet counters on the associated HTB class
  provide per-cgroup usage. However, this only works for processes that have
  been explicitly limited, and it requires the tc infrastructure to be in place.
- **nftables counter-based**: If the `zelynic strict` backend has already
  created nftables rules with packet/byte counters for a target, those counters
  provide per-target usage. This is read-only access to existing counters.
- **PID-based**: Mapping sockets to PIDs via `ss` and `/proc/<pid>/fd/` is the
  approach already used by `zelynic list`. It can provide a snapshot of
  current per-process usage but has significant limitations (see honesty
  constraints below).

Data source: tc `tc -s class show dev <iface>` for HTB class counters,
`nft list counters` for nftables counters (read-only), `ss` + `/proc` for
per-PID socket attribution.

### Future Quota Budget Tracking

Design the data model for quota budgets: a defined byte limit per target
(interface, app, session) with a current-usage counter and a remaining-budget
counter. The model is designed in v2.9 but enforcement (blocking or throttling
when the budget is exhausted) is explicitly deferred. A `zelynic quota status`
  command would show budget and usage but never enforce.

### Future Background Data Guard

Design the concept of a "background data guard" — a mode where non-foreground
applications are subject to a separate (lower) quota budget. This requires
per-app attribution (hard) and quota enforcement (deferred). The v2.9 design
captures the concept as a future feature with clear prerequisites: per-app
attribution must be reliable, quota enforcement must be implemented, and the
user must explicitly enable the guard.

## Data Sources and Tradeoffs

v2.9 evaluates multiple data sources for network accounting. Each has strengths
and limitations that must be honestly documented.

### `/proc/net/dev` — Interface Counters

**Path**: `/proc/net/dev`

**What it provides**: Cumulative per-interface RX/TX bytes, packets, errors,
drops, multicast, and frame counters since interface creation. Updated by the
kernel in real time.

**Strengths**:
- Always available on any Linux system with network interfaces.
- No special permissions required to read (world-readable).
- Provides per-interface breakdown (Wi-Fi, Ethernet, tethering, etc.).
- No dependency on nftables, tc, cgroups, or eBPF.
- Granularity: one line per interface, updated continuously.
- Can be polled at any interval for session-based delta computation.

**Limitations**:
- **Not per-app**: Counts all traffic on an interface regardless of which
  process sent or received it. There is no way to attribute bytes to a specific
  application from `/proc/net/dev` alone.
- **Cumulative since boot**: Counters reset only on interface down/up or system
  reboot. Computing "last hour" requires delta between snapshots.
- **No protocol breakdown**: Counts all protocols (TCP, UDP, ICMP, etc.)
  together on each interface.
- **Loopback traffic**: `lo` interface counters include local loopback traffic,
  which may need to be excluded from user-visible usage reports.
- **VLAN/sub-interface**: May double-count if both physical and VLAN interfaces
  are reported.

### sysfs Network Interface Statistics

**Path**: `/sys/class/net/<iface>/statistics/rx_bytes`, `tx_bytes`, etc.

**What it provides**: Same data as `/proc/net/dev` but in individual sysfs
files per counter per interface. Each file contains a single numeric value.

**Strengths**:
- Same data as `/proc/net/dev`, but easier to parse (one number per file).
- Can be watched with `inotify` for changes (though polling is simpler).
- Explicit per-counter files: `rx_bytes`, `tx_bytes`, `rx_packets`,
  `tx_packets`, `rx_errors`, `tx_errors`, `rx_dropped`, `tx_dropped`.
- Well-documented sysfs interface, stable across kernel versions.

**Limitations**:
- Same limitations as `/proc/net/dev` regarding per-app attribution.
- Same cumulative-since-boot behavior.
- No additional data beyond what `/proc/net/dev` provides; it is a different
  view of the same kernel counters.

### nftables Counters (If Already Used by Strict Backend)

**Path**: `nft list ruleset` (read-only), or `nft list table inet zelynic`
(if the Zelynic nftables table exists).

**What it provides**: Per-rule packet and byte counters for nftables rules
that have been created by `zelynic strict`. If a target process has been
limited, the corresponding nftables rules (output chain for upload, input chain
for download) accumulate byte counters for that target's traffic.

**Strengths**:
- Provides per-target usage if `zelynic strict` is active.
- Read-only operation: reading counters does not modify rules or traffic.
- Already available if the strict backend has set up rules for a target.
- Can distinguish upload (output/egress) and download (input/ingress) per
  target.

**Limitations**:
- **Only works for limited targets**: Counters exist only for processes that
  have been explicitly limited with `zelynic strict`. Unrestricted processes
  have no per-target counters.
- **Requires nftables rules to be present**: If `zelynic clean` or
  `zelynic unstrict` removes the rules, the counters are lost.
- **Not cumulative across rule recreation**: If a target is unstricted and
  re-stricted, the counters reset to zero.
- **nftables table may not exist**: On systems that have never run
  `zelynic strict`, the Zelynic nftables table does not exist and there are
  no counters to read.
- **Requires root for `nft list ruleset`**: On some configurations, reading
  the full ruleset requires CAP_NET_ADMIN.

### tc Statistics (If Applicable)

**Path**: `tc -s class show dev <iface>` (read-only), `tc -s qdisc show dev
<iface>`.

**What it provides**: Per-HTB-class byte and packet counters for traffic that
  has been shaped by the tc subsystem. If `zelynic strict` has created HTB
  classes for a target, the class statistics show how many bytes and packets
  have passed through that class.

**Strengths**:
- Provides per-class usage if `zelynic strict` is active.
- Read-only operation: `tc -s show` does not modify qdisc, filters, or classes.
- Complements nftables counters: tc counters reflect traffic that was actually
  shaped, while nftables counters reflect traffic that matched the mark/policer.
- Can show rate (bytes per second) in recent interval.

**Limitations**:
- **Only works for limited targets**: Same as nftables counters; only
  available for processes with active `zelynic strict` limits.
- **Requires tc infrastructure**: If no HTB qdisc and classes exist, there
  are no counters to read.
- **Root required**: Reading tc statistics typically requires root or
  CAP_NET_ADMIN.
- **Counter reset on qdisc change**: If the HTB hierarchy is recreated, class
  counters reset.

### cgroup-Based Accounting Limitations

**Path**: `/sys/fs/cgroup/<cgroup>/io.stat`, memory controller, or
`net_cls`/`bpf` controllers (if available).

**What it provides**: Some cgroup controllers can report network-related
statistics. The `net_cls` controller can tag packets with a class ID, and an
eBPF program could count per-cgroup traffic. However, these are limited:

- **cgroup v2 `io.stat`**: Reports block I/O statistics, not network I/O.
- **`net_cls` controller**: Only tags packets; does not count them. Requires
  tc filter matching on the class ID to actually shape or count traffic.
- **`pids` controller**: Counts PIDs in a cgroup, not network usage.
- No standard cgroup v2 controller provides per-cgroup network byte counters
  without eBPF.

**Strengths**:
- Could be useful in the future with eBPF integration.
- Cgroup membership is already tracked by Zelynic for the strict backend.

**Limitations**:
- **No built-in network byte counters in cgroup v2**: This is a fundamental
  limitation. Without eBPF, cgroups do not track per-cgroup network usage.
- **Not useful for v2.9**: Without eBPF, cgroup-based network accounting is
  not feasible. This is why v2.9 does NOT touch `/sys/fs/cgroup` for
  accounting.

### Process/PID Attribution Limitations

**Path**: `ss -tnp` (TCP), `ss -unp` (UDP), `/proc/<pid>/net/tcp`,
  `/proc/<pid>/fd/`.

**What it provides**: Maps network sockets to owning PIDs and process names.
This is the approach already used by `zelynic list` to show per-process
bandwidth.

**Strengths**:
- Provides per-process (per-PID) socket attribution.
- Works without any special kernel features beyond `/proc` and `ss`.
- Can show per-socket byte counters (on kernels 4.6+) via
  `ss -tnpi` which reads `tcp_info` byte counters.

**Limitations**:
- **Snapshots only**: `ss` and `/proc` provide point-in-time snapshots. There
  is no cumulative per-PID byte counter in the kernel. Computing per-PID usage
  over time requires polling and differencing, which is expensive and noisy.
- **Browser/app multiprocess behavior**: Modern browsers (Chromium, Firefox)
  spawn dozens of helper processes (renderer, GPU, sandbox, utility). A
  single "Firefox" or "Brave" session may involve 20-50 PIDs, each with
  their own sockets. Attributing total usage to "Firefox" requires summing
  across all PIDs in the browser's process group, which is fragile.
- **Short-lived processes**: A download manager or `curl` that finishes
  quickly may not be caught in any poll interval. Its socket counters disappear
  when the process exits.
- **Stale PIDs**: A PID may exit between polls. The accounting code must handle
  the case where a PID seen in poll N is gone in poll N+1.
- **Not cumulative**: Kernel socket byte counters (`tcp_info`) are per-socket
  and cumulative since socket creation. They disappear when the socket closes.
  There is no kernel-maintained total-per-PID byte counter.
- **UDP attribution**: UDP socket attribution is harder than TCP because UDP
  sockets may be shared or unconnected.
- **Cannot attribute to cgroup without eBPF**: Without eBPF, mapping network
  bytes to a cgroup requires correlating PIDs with cgroup membership, which is
  an approximation.

### Future eBPF Observer as Advanced Backend Only

**Status**: Not implemented in v2.9. Explicitly deferred to a future milestone
  (v4.x or later).

**What it could provide**: eBPF programs attached to network hooks (XDP,
  `tc` ingress/egress, cgroup skb hooks) could provide precise per-cgroup,
  per-PID, per-app, per-protocol network byte counters without modifying
  traffic. eBPF maps could accumulate counters efficiently in kernel space and
  expose them to userspace via BPF_MAP_LOOKUP_ELEM.

**Strengths**:
- The most precise per-app attribution possible on Linux.
- Low overhead: counters are maintained in kernel space, no polling needed.
- Can count per-cgroup, per-PID, per-UID, per-comm (process name), per-IP,
  per-port, per-protocol.
- Does not modify traffic (observability only, if only counting).
- Well-supported on modern kernels (5.4+).

**Limitations**:
- **Requires kernel support**: eBPF requires a relatively modern kernel (5.4+
  for most features, 5.8+ for some). Some minimal kernels or container
  environments restrict eBPF.
- **Requires CAP_BPF or root**: Loading eBPF programs requires privileges.
- **Complexity**: eBPF programs are harder to develop, test, and debug than
  simple `/proc/net/dev` parsing.
- **Must be read-only**: An eBPF observer must never modify packets (no
  XDP_DROP, no TC_ACT_SHOT). This is a hard safety boundary.
- **Out of scope for v2.9**: v2.9 does not load, attach, or interact with
  eBPF programs in any way.

## Honesty Constraints

The v2.9 design imposes strict honesty constraints on all accounting output.
These constraints prevent the code from claiming capabilities it does not have
  or accuracy it cannot guarantee.

### Interface Counters Are Not Per-App Attribution

When reporting interface-level counters (from `/proc/net/dev` or sysfs), the
output must clearly state that this is aggregate interface traffic, not
per-app traffic. The rendering must not imply or suggest that the bytes are
attributed to any specific application, PID, or cgroup.

**Required wording**: "Interface total" or "Aggregate interface traffic."
**Forbidden wording**: "App usage," "Process bandwidth," or any implication
  that interface counters are per-application.

### PID Attribution Can Be Stale or Incomplete

When reporting per-PID usage (via `ss` or `/proc` polling), the output must
acknowledge that:

- PID attribution is a snapshot, not a cumulative total.
- PIDs may exit between polls, losing their socket counters.
- Browser multiprocess behavior makes per-app totals unreliable.
- Short-lived connections may be missed entirely.

**Required wording**: Must include "snapshot" or "point-in-time" when reporting
  per-PID data. Must note limitations if a process has multiple PIDs.
**Forbidden wording**: "Total bytes transferred by this app" (unless backed
  by eBPF or persistent tracking).

### Browser/App Multiprocess Behavior Makes App Accounting Hard

Modern applications (browsers, Electron apps, messaging apps) use multiple
processes. Attributing network usage to "the app" requires summing across all
child PIDs, which is fragile because:

- Child processes spawn and exit dynamically.
- Some child processes may use different cgroups.
- Sandboxed processes may have restricted `/proc` access.
- The main process may not be the one generating the most traffic.

The design must document this limitation in any per-app accounting output and
must not claim precise per-app totals without eBPF.

### Counters May Reset After Reboot/Interface Reset

All `/proc/net/dev` and sysfs counters are cumulative since interface
creation (typically since boot). They reset when:

- The system reboots.
- The interface is brought down and up (`ip link set <iface> down` then
  `up`).
- The driver is reloaded.
- A network namespace is created or destroyed.

Since-boot accounting must acknowledge this limitation. Session-based accounting
that spans a reboot must note that the reference point is lost on reboot.

**Required wording**: Must state "since boot" or "since interface up" for
  cumulative counters.
**Forbidden wording**: "All-time usage" or "lifetime" (unless backed by a
  persistent ledger that survives reboots).

### Read-Only Accounting Must Not Claim Enforcement

All v2.9 accounting output must clearly state that it is observational only. No
output may imply that traffic is being blocked, throttled, or shaped by the
accounting system. If a limit exists (from `zelynic strict`), it must be
attributed to the strict backend, not to the accounting system.

**Required wording**: Must include "read-only" or "observational" in
  accounting output headers.
**Forbidden wording**: "Enforced," "blocked," "limited by quota," or any
  suggestion that accounting output triggered a traffic action.

### Quota Guard Must Not Be Claimed Until Enforcement Exists

Future quota budget tracking may display a budget and current usage, but it
must never claim that the quota is being enforced. The output must explicitly
state "quota enforcement is not implemented" or "this is a usage report only;
  no automatic enforcement."

**Required wording**: Must state "not enforced" or "report only" for all
  quota-related output.
**Forbidden wording**: "Quota active," "budget enforced," "blocked by quota
  guard," or any implication that a quota limit has caused traffic action.

## Proposed Future Commands (Not Implemented)

The following commands are designed as part of the v2.9 accounting model but are
**not implemented** in v2.9 phase 1. They are documented here to define the
future CLI surface and ensure the design accounts for them. No CLI path exists
for these commands in v2.9.

### `zelynic usage`

Display a summary of network usage across all interfaces. Default view shows
aggregate RX/TX since boot for all non-loopback interfaces.

```
$ zelynic usage
Interface     RX           TX           Since
wlan0         1.23 GiB     342 MiB      boot
eth0          0 bytes       0 bytes      boot
usb0          45.6 MiB     12.3 MiB     boot (tethering)
```

**Not implemented.** Design only.

### `zelynic usage --since-boot`

Explicit since-boot view. Equivalent to the default but makes the time range
explicit in the output. Useful for scripting and automation.

**Not implemented.** Design only.

### `zelynic usage --interface wlan0`

Show usage for a specific interface only. Filters output to the named
interface. Validates that the interface exists before reading counters.

```
$ zelynic usage --interface wlan0
Interface     RX           TX           Since
wlan0         1.23 GiB     342 MiB      boot
```

**Not implemented.** Design only.

### `zelynic usage --target brave`

Show usage attributed to a specific target (process name or PID). This requires
per-PID socket attribution via `ss` or `/proc`, which has the honesty
constraints documented above (snapshot only, multiprocess limitations).

```
$ zelynic usage --target brave
Target: brave (PIDs: 2345, 2346, 2350, 2401)
Warning: per-PID attribution is snapshot-based and may be incomplete.
RX (snapshot): 12.3 MiB    TX (snapshot): 4.5 MiB
Active sockets: 23
```

**Not implemented.** Design only.

### `zelynic quota status`

Display quota budget and current usage for configured quotas. In v2.9, no
quotas are enforced. If a quota configuration exists (future feature), this
command shows the budget, current usage, and remaining balance but explicitly
states that no enforcement is active.

```
$ zelynic quota status
Interface: wlan0 (tethering)
  Budget: 100 MiB
  Used:  45.6 MiB
  Remaining: 54.4 MiB
  Status: report only — no enforcement active
```

**Not implemented.** Design only.

### `zelynic quota reset`

Reset quota usage counters for a specific interface or target. This would set
the current usage back to zero (or record a new baseline) so the quota period
restarts. No enforcement is implied by the reset.

**Not implemented.** Design only.

### `zelynic limit-background --allow-all 100mb`

Design surface for the future background data guard. This would set a combined
(upload + download) quota for background (non-foreground) applications. The
quota is not enforced in v2.9; the command is a design placeholder.

```
$ zelynic limit-background --allow-all 100mb
Background data guard: not implemented
No enforcement active. This is a design placeholder only.
```

**Not implemented.** Design only.

### `zelynic limit-background --allow-down 100mb`

Design surface for download-only background quota. Same constraints as above:
not implemented, no enforcement.

**Not implemented.** Design only.

### `zelynic limit-background --allow-up 25mb`

Design surface for upload-only background quota. Same constraints as above:
not implemented, no enforcement.

**Not implemented.** Design only.

## Phase Plan

v2.9 is decomposed into six sequential phases. Each phase must pass CI, pass
local validation, and be documented before the next phase begins. All phases
are design/docs-only unless explicitly noted.

### Phase 1: Accounting Design / Read-Only Architecture (Completed)

- Produce this design document (`docs/v2.9-network-accounting-lab.md`).
- Update `CHANGELOG.md` with Unreleased entry for v2.9.
- Update `docs/experimental-pid-move-lab.md` next-milestone section.
- Define data model, data sources, honesty constraints, future command surface.
- No runtime changes. No Rust code modifications. No Rust tests added.
- This phase establishes the accounting targets, data source tradeoffs, safety
  boundaries, and future command designs that all subsequent phases must follow.

### Phase 2: Interface Counter Model + Pure Parser Tests (Completed)

- Added `src/accounting/` module with pure Rust types and parsers for
  `/proc/net/dev`-style content. No live system reads, no filesystem access.
- Model types: `InterfaceCounter` (interface name, RX/TX bytes, RX/TX packets),
  `InterfaceCounterSnapshot` (ordered list of counters with source label),
  `ParseError` (structured error enum), `SourceLabel` (observational marker).
- Pure parser: `parse_proc_net_dev(content: &str)` parses standard
  `/proc/net/dev` format with header skipping, interface name trimming,
  colon validation, field count validation, u64 parsing with overflow detection.
- Pure single-line parser: `parse_proc_net_dev_line(line: &str)` for
  individual line parsing.
- Render helper: `render_interface_counter_snapshot()` produces
  human-readable output with honesty labels: "read-only parsed sample",
  "aggregate interface traffic, not per-app", "No enforcement",
  "No quota guard", "No per-app attribution".
- 53 pure unit tests: single/multiple interface parsing, header skipping,
  lo/wlan0/eth0/wlp/enp/usb0 name support, whitespace trimming,
  missing colon rejection, too-few-fields rejection, non-numeric RX/TX
  rejection, overflow rejection, negative number rejection, empty input
  handling, determinism, input order preservation, snapshot aggregation
  helpers, render output honesty (read-only, no enforcement, no quota
  guard, no per-app attribution, source label, loopback tagging), error
  display formatting, design contract verification (no live /proc read).
- No CLI command added. No runtime I/O. No enforcement.
- Module registered in `src/main.rs` as `mod accounting;` (internal only).
- All tests use const sample strings — no live `/proc/net/dev` or sysfs reads.

### Phase 3: Read-Only Usage Preview Renderer (Completed)

- Added `src/accounting/usage_preview.rs` with pure model and renderer for
  the future `zelynic usage` command display contract.
- Model types: `UsagePreview` (total RX/TX/combined bytes, interface count,
  per-interface rows, source label, attribution scope, enforcement status),
  `UsagePreviewRow` (interface name, raw + human-readable RX/TX/total bytes,
  loopback flag).
- Pure functions: `build_usage_preview(snapshot)` builds structured preview
  with saturating arithmetic; `render_usage_preview(preview)` renders
  human-readable output with IEC binary prefixes (B/KiB/MiB/GiB/TiB).
- `format_bytes_human(bytes)` helper for IEC binary prefix formatting.
- Rendered output includes comprehensive safety disclaimers: "read-only parsed
  snapshot/model", "interface-level only", "not per-app attribution",
  "no quota enforcement active", "no network blocking active",
  "no limiter attach performed", "no nft/tc/Zelynic state mutation performed",
  "no live /proc or sysfs read performed".
- 30 pure unit tests: format_bytes_human (zero, bytes, KiB, MiB, GiB, TiB,
  u64::MAX no-panic), build_usage_preview (one interface, multiple interfaces,
  empty snapshot, interface order preservation, source label, attribution scope,
  enforcement status, loopback flag, human-readable fields), render_usage_preview
  (read-only statement, interface-level only, per-app denial, quota enforcement
  denial, network blocking denial, limiter attach denial, nft/tc/state mutation
  denial, live /proc/sysfs read denial, source label, human-readable bytes,
  empty snapshot, determinism), overflow-safe saturating totals (u64::MAX).
- No CLI command added. No runtime I/O. No enforcement.

## Future Version Roadmap

The following milestones are explicitly deferred beyond v2.9:

### v3.x: Quota Guard / Allow-Block App Mode

- Implement quota enforcement: when a budget is exhausted, trigger a traffic
  policy (block, throttle, or notify).
- Implement allow-block app mode: whitelist/blacklist specific applications
  for network access.
- Requires: reliable per-app attribution (likely eBPF), quota budget model
  (from v2.9 ledger design), and an enforcement backend (nftables/tc or eBPF).
- Safety requirements: explicit user consent, clear audit trail, rollback
  capability, no silent blocking.

### v4.x: eBPF Observer / Classifier Backend

- Implement an eBPF observer that provides precise per-cgroup, per-PID,
  per-UID, per-comm network byte counters.
- The observer is read-only: it counts packets/bytes but never modifies or
  drops them.
- Requires: kernel 5.4+, CAP_BPF or root, eBPF toolchain (libbpf or Aya).
- Safety requirements: eBPF programs must be verified as non-mutating,
  no XDP_DROP, no TC_ACT_SHOT, no packet modification.
- Integration with v2.9 accounting model: eBPF counters replace or supplement
  `/proc/net/dev` and `ss`-based attribution.

## Relationship to Existing Zelynic Features

v2.9 does not modify any existing Zelynic features:

- **`zelynic strict`**: Remains the only validated active limiter path. v2.9
  may read counters created by `zelynic strict` (nftables counters, tc class
  statistics) but never modifies them.
- **`zelynic list`**: Remains the per-process bandwidth monitor. v2.9's
  future `usage` command is a separate interface-level accounting view, not a
  replacement for `zelynic list`.
- **`zelynic status`**: Remains the active limits viewer. v2.9 does not add
  quota status to `zelynic status`; a separate `zelynic quota status` command
  is designed for the future.
- **`zelynic run` / Scope Runner**: Remains unchanged. v2.9 does not interact
  with the Scope Runner, systemd scopes, or the experimental PID move path.
- **`zelynic log`**: Remains the bandwidth history log. v2.9's future
  accounting features are a separate system from the existing log.
- **Guarded real writer seam**: Remains hard-blocked. v2.9 does not call or
  modify the guarded real writer in any way.
