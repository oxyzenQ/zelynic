# Zelynic v2.8 Phase 3d: Output Audit + Negative-Path Smoke Coverage

## Purpose

Phase 3d audits and locks output honesty for every experimental attach and move
executor seam path before any future real PID move phase. This phase does not
introduce runtime changes or code behavior changes — it strengthens the test
suite and documents the negative-path output guarantees.

## Scope

### What This Phase Does

1. **Canonical deny-line strengthening**: Adds two new canonical deny lines to
   the move executor seam disclaimers:
   - `Experimental PID move is not implemented yet.`
   - `Bandwidth limiting is not active from this command yet.`

   These join the existing five deny lines (no PID moved, no cgroup.procs write,
   no limiter attach, no nft/tc/state changes, no persistent state write) to
   form a complete set of seven canonical safety denials.

2. **Negative-path deny-line tests**: Adds tests verifying that all seven
   canonical deny lines appear in rendered output for every negative path:
   - Non-root gate blocked
   - User scope gate blocked
   - Multi-PID discovery blocked
   - Missing original cgroup capture blocked
   - Missing consent flags blocked (all three individually)
   - Stale PID liveness blocked
   - Self-protection blocked
   - Missing --probe-live blocked
   - Missing --attach-live blocked

3. **Output wording audit tests**: Adds tests verifying that no negative-path
   output ever claims:
   - PID was moved
   - cgroup.procs was written
   - Limiter was attached
   - Bandwidth limiting is active
   - nftables/tc/Zelynic state was mutated
   - Persistent state was written
   - Rollback was performed

4. **Error constant honesty tests**: Adds tests verifying that error message
   constants (`EXPERIMENTAL_PID_MOVE_NOT_IMPLEMENTED`,
   `MKDIR_LIVE_PID_MOVE_NOT_IMPLEMENTED`) never claim mutation.

### What This Phase Does NOT Do

- No live PID move.
- No cgroup.procs write.
- No limiter attach.
- No nftables/tc/Zelynic state mutation.
- No persistent state write.
- No CLI enablement for live PID move.
- No runtime behavior changes.

## Canonical Deny Lines

The following seven statements are guaranteed to appear in move executor seam
output for all code paths (both valid and blocked):

| # | Deny Line | Source |
|---|-----------|--------|
| 1 | `no PID was moved` | Seam disclaimer |
| 2 | `no cgroup.procs write was performed` | Seam disclaimer |
| 3 | `no limiter attach was performed` | Seam disclaimer |
| 4 | `no nftables/tc/Zelynic state changes were made` | Seam disclaimer |
| 5 | `no persistent state write was performed` | Seam disclaimer |
| 6 | `Experimental PID move is not implemented yet.` | Seam disclaimer (new in 3d) |
| 7 | `Bandwidth limiting is not active from this command yet.` | Seam disclaimer (new in 3d) |

## Output Wording Prohibitions

Every negative path (gate check, error message, rendered seam section) must
avoid claiming any of the following:

| Prohibited Phrase | Rationale |
|-------------------|-----------|
| "PID was moved." | No PID movement occurs in any phase through 3d |
| "cgroup.procs written" | No cgroup.procs write occurs in any phase through 3d |
| "limiter attached" / "limiter attach was performed" | No limiter attach occurs |
| "bandwidth limiting active" | No bandwidth limiting is applied |
| "nftables rule" / "nftables" (positive context) | No nftables changes occur |
| "state file written" / "persistent state written" | No persistent state writes |
| "rollback performed" / "PID was restored" | No rollback is needed when no move happened |

## Why This Phase Is Still Non-Mutating

Phase 3d is a pure test-and-audit phase. It adds no runtime code paths, no
execution branches, and no write operations. The only production code change is
adding two new disclaimer strings to the seam result model — these are rendered
as text in the gate output section and have no side effects.

All new tests verify that output text never claims mutation that did not happen.
This strengthens the invariant that every experimental attach/move path produces
honest output regardless of which gate blocks or how many gates pass.

## Relationship to Future Phases

Phase 3d is the last phase before a future phase may implement live PID
movement. The canonical deny lines established here serve as the output honesty
baseline: any future phase that performs a real PID move must update the
disclaimers to accurately reflect what happened, and the tests must be updated to
verify the new output wording is honest about both what happened and what did not
happen.

Phase 3d does not remove any gate or change any blocking condition. The seam
still returns `status: blocked` and `execution: not implemented` for all inputs.
All safety gates remain in place.
