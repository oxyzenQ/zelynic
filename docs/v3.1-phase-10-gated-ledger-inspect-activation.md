**Phase 10b note**: Phase 10b added 28 deterministic validation freeze invariant tests
to `src/cli/v31_gate_tests.rs` (Section G) that audit and freeze the Phase 10 hidden

1481 (was 1453 after Phase 10, +28).

## 1. Purpose

Wire the existing phase 7 `build_ledger_inspect()` and `render_ledger_inspect()` to
the hidden `zelynic ledger inspect` CLI dispatch gate. The command produces output
from an in-memory fixture `Ledger` — no filesystem reads, no filesystem writes,
no live system reads. This is the smallest safe activation of a previously
hard-rejected hidden command.

## 2. Phase Decision

**Chosen**: Pure in-memory fixture inspect preview (not docs-only gate design).

**Rationale**: Phase 8 design freeze defined Phase 10 as "read-only inspect from
fixture behind gated command." The phase 7 `LedgerInspect` model and phase 6 clap
shape are complete. The only remaining gap is the dispatch wiring. This phase
bridges that gap with no new model types, no new persistence operations, and no
filesystem access.

**Scope boundary**: `ledger inspect` is activated; `ledger export` remains
hard-rejected. All hidden usage flags (`--session`, `--since-boot`, `--interface`,
`--target`) remain hard-rejected. Command remains `#[command(hide = true)]`.

## 3. Changes

### 3.1 New file: `src/commands/ledger.rs` (272 LOC)

Handler module for `zelynic ledger inspect`.

**Functions**:
- `build_fixture_ledger() -> Ledger`: Constructs a deterministic 3-entry
  in-memory ledger (2 snapshots: wlan0 at 00:00, eth0 at 00:05; 1 delta:
  wlan0 at 00:10). No filesystem access.
- `handle_ledger_inspect(json: bool) -> Result<()>`: Dispatch handler. Builds
  fixture ledger, computes inspect statistics via `build_ledger_inspect()`,
  renders text (default) or JSON (`--json`). Returns `Ok(())`.

**Tests**: 16 deterministic tests covering fixture structure (entry counts,
snapshot/delta counts, sorted interfaces, read-only flag, totals, provenance,
enforcement status), output correctness (text content, JSON validity, round-trip,
determinism), safety (no filesystem access, no persistence, no live /proc read,
no enforcement), and handler return values (Ok for text and JSON).
