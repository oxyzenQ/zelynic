# v3.0 Phase 15b: Usage Delta JSON CLI Validation Freeze + Release Readiness Audit

## Purpose

Phase 15b is a validation/documentation/audit phase that freezes the newly public
`usage --sample --delta --json` CLI behavior before any v3.0 release preparation.
This phase confirms that the CLI wiring performed in phase 15 produces correct,
honest, and safe output consistent with the phase 13 output contract, and that
no existing CLI behavior was altered by the wiring.

This phase does NOT change runtime behavior unless a test reveals a real bug. It
does NOT change the JSON schema. It does NOT change existing CLI outputs. It does
NOT add new public CLI flags. It does NOT add dependencies.

### Tests

Two cross-command isolation tests were added to
`src/commands/usage_delta/tests_json.rs`:

1. `delta_json_output_distinct_from_snapshot_json`: Verifies that delta JSON output
   contains fields that distinguish it from snapshot JSON (sample_mode="delta",
   delta_wait_ms present, command="usage --sample --delta --json", sample_count=2,
   read_count=2). This proves that `usage --sample --delta --json` does not produce
   snapshot JSON.
2. `delta_text_output_is_not_json`: Verifies that delta text output (via the text
   path injection point) contains human-readable headers and does not start with `{`.
   This proves that `usage --sample --delta` text output remains human text, not JSON.

## Release Readiness Assessment

The `usage --sample --delta --json` command is release-ready:

1. **Schema conformance**: Output matches the phase 13 contract exactly. No schema
   drift detected between pure model tests (66), CLI wiring tests (32), and manual
   live invocation.
2. **Error handling**: First-read and second-read errors produce valid JSON with
   appropriate error objects, correct read_count values, and preserved honesty flags.
3. **Cross-command isolation**: Delta JSON output is structurally distinct from
   snapshot JSON. Delta text output remains human-readable. No existing command
   behavior was altered by the phase 15 wiring.
4. **Safety boundary**: All 15 hard negative constraints are maintained. No
   configurable interval, no loop/watch, no daemon, no eBPF, no quota, no
   persistence, no network blocking, no limiter, no nft/tc mutation, no state
   mutation, no PID move, no cgroup write, no sysfs read, no filesystem write,
   no arbitrary path read.
5. **Test coverage**: 32 tests in `tests_json.rs` (30 phase 15 + 2 phase 15b)
   cover CLI parsing, JSON structure, field correctness, honesty flags, error
   handling, structural safety, and cross-command isolation. 66 pure model tests
   in `tests/usage_delta_json/` cover serialization, round-trip, error payloads,
   counter reset, interface add/remove, and all honesty flags.
6. **Documentation**: Phase 13 contract document, phase 14b freeze document, and
   this phase 15b freeze document provide a complete audit trail from design through
   implementation to frozen release readiness.
7. **CI**: All CI pipelines are green.
