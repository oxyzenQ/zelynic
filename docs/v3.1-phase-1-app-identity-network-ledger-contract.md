# v3.1 Phase 1: App Identity + Network Ledger Contract Design

## Status

**Phase 1 — Design only. No runtime code. No CLI changes. No persistence.**

## 1. Purpose

v3.1 connects usage data to app/target identity and prepares ledger/history reporting.
This phase defines the conceptual contracts and type names for two future subsystems:

1. **App Identity** — resolve a network-using target into a structured identity.
2. **Network Ledger** — accumulate, attribute, and report usage per identity.

v3.1 does NOT block network. It does NOT enforce quota. It does NOT mutate
nftables, tc, or cgroup state. It follows the Zelynic safety progression:

```
observe (v3.0) → identify (v3.1) → record (v3.1) → report (v3.1) → enforce (future)
```

v3.1 covers `identify → record → report` only. Enforcement is explicitly
out of scope for v3.1 and all its sub-phases.

## 2. Baseline

v3.0.1 provides the following read-only, single-shot usage commands:

| Command | Source | Output | Scope |
|---------|--------|--------|-------|
| `zelynic usage --sample` | `/proc/net/dev` | Human text | Interface-level only |
| `zelynic usage --sample --json` | `/proc/net/dev` | JSON | Interface-level only |
| `zelynic usage --sample --delta` | `/proc/net/dev` | Human text | Interface-level delta |
| `zelynic usage --sample --delta --json` | `/proc/net/dev` | JSON | Interface-level delta |

All v3.0.1 usage commands:

- Read only `/proc/net/dev` (hardcoded, no arbitrary path).
- Perform no per-app attribution.
- Perform no persistence (no ledger file write).
- Perform no enforcement (no nft/tc/cgroup mutation).
- Are single-shot (no loop, no watch, no daemon).
- Report honest disclaimers (12+ boolean honesty flags in JSON, 13+ text disclaimers).

v3.1 builds on this foundation by introducing identity resolution and usage
attribution concepts, but does not change any existing v3.0.1 behavior.

## 3. App/Target Identity Model Concept

The following type names represent the future identity model. They are defined
here as contracts; they are NOT implemented in v3.1 phase 1.

### `TargetIdentity`

Top-level identity wrapper for a network-observed target. Composed from one or
more sub-identities. A `TargetIdentity` is the result of resolving "what is this
network-using thing?" into a structured representation.

Fields (conceptual):

- `process: Option<ProcessIdentity>` — resolved from PID, if available.
- `cgroup: Option<CgroupIdentity>` — resolved from cgroup path, if available.
- `interface: Option<InterfaceIdentity>` — the network interface the traffic was observed on.
- `resolved: bool` — whether identity resolution was attempted (not whether it succeeded).
- `confidence: IdentityConfidence` — how reliable the attribution is.

### `ProcessIdentity`

Per-process identity derived from `/proc` filesystem reads.

Fields (conceptual):

- `pid: Option<u32>` — process ID (unstable: PIDs are recycled).
- `comm: Option<String>` — `comm` name from `/proc/<pid>/comm` (truncated to 15 chars by kernel, relatively stable within a session).
- `argv0: Option<String>` — first element of `/proc/<pid>/cmdline` (may differ from comm).
- `exe_path: Option<String>` — symlink target of `/proc/<pid>/exe` (stable: canonical path, but may be deleted/overwritten).
- `cmdline_full: Option<String>` — full command line (may contain sensitive arguments, best-effort only).
- `uid: Option<u32>` — owning user ID.

### `CgroupIdentity`

Cgroup-based identity derived from `/proc/<pid>/cgroup` or systemd introspection.

Fields (conceptual):

- `cgroup_path: Option<String>` — full cgroup v2 path (e.g., `/sys/fs/cgroup/user.slice/user-1000.slice/...`).
- `systemd_unit: Option<String>` — systemd unit or scope name, if determinable (best-effort).
- `is_system_scope: bool` — whether the cgroup is a system-level scope.
- `is_user_scope: bool` — whether the cgroup is a user-level scope.

### `InterfaceIdentity`

Network interface identity.

Fields (conceptual):

- `name: String` — interface name (e.g., `wlp1s0`, `eth0`).
- `is_loopback: bool` — whether the interface is a loopback device.

### `ResolvedUsageTarget`

A fully resolved usage target combining identity with observed usage data.

Fields (conceptual):

- `identity: TargetIdentity` — the resolved identity.
- `usage: UsageAttribution` — attributed usage counters.
- `attribution_scope: UsageAttributionScope` — the scope of attribution.
- `first_seen: Option<Instant>` — when this target was first observed in the session.
- `last_seen: Option<Instant>` — when this target was last observed.

### `UsageAttributionScope`

Enum describing the granularity of usage attribution.

Variants (conceptual):

- `InterfaceOnly` — current v3.0.1 behavior: no per-app attribution, counters are interface-level only.
- `PerProcess` — usage attributed to a specific process identity.
- `PerCgroup` — usage attributed to a cgroup.
- `PerSystemdUnit` — usage attributed to a systemd unit/scope.
- `Mixed` — combination of multiple scopes (e.g., process + cgroup).

## 4. Identity Sources

The following are possible future read-only identity sources. No identity
resolution is implemented in v3.1 phase 1.

### Stable Sources

| Source | Path/Method | Stability |
|--------|-------------|-----------|
| `comm` | `/proc/<pid>/comm` | Moderate — truncated to 15 chars, unique within a session. |
| `exe_path` | `/proc/<pid>/exe` (readlink) | High — canonical filesystem path, stable across process lifetime unless binary is deleted. |
| `cgroup_path` | `/proc/<pid>/cgroup` | High — assigned by systemd/kernel, stable within session. |
| `interface_name` | `/proc/net/dev` header | High — interface names are stable across reboots on most systems. |

### Unstable Sources

| Source | Path/Method | Instability Reason |
|--------|-------------|-------------------|
| `pid` | `/proc/<pid>` | PIDs are recycled by the kernel. A PID alone is not a stable identity across boot sessions or long timeframes. |
| `argv0` | `/proc/<pid>/cmdline` first element | May be set arbitrarily by the process (argv[0] spoofing). |
| `cmdline_full` | `/proc/<pid>/cmdline` | May contain sensitive data (passwords, tokens). Also mutable by the process. |

### Best-Effort Sources

| Source | Path/Method | Caveat |
|--------|-------------|--------|
| `systemd_unit` | `dbus` introspection or `/proc/<pid>/cgroup` heuristic parsing | Requires systemd. May not be available in containers, non-systemd hosts, or user namespaces. |
| `boot_id` | `/proc/sys/kernel/random/boot_id` | Identifies a boot session. Useful for partitioning ledger data by boot. Unstable across reboots by design. |
| `session_id` | Not defined yet | Concept for partitioning ledger data within a boot session. Requires future design decision. |

### Identity Resolution Policy

All future identity resolution must obey these constraints:

- Identity resolution is **read-only**: never write to `/proc`, `/sys`, cgroups, or any filesystem.
- Identity resolution is **best-effort**: partial resolution is acceptable and must be reported honestly.
- PID recycling must be acknowledged: any PID-based identity must include a disclaimer that PIDs are not stable across reboots.
- Sensitive data must not be logged: `cmdline_full` must not be persisted to ledger files in plaintext. If persisted, it must be truncated or omitted.
- No identity source may block or slow down the usage sampling path. Identity resolution must be async or deferred, never synchronous on the hot path.

## 5. Network Ledger Concept

The following type names represent the future ledger model. They are defined
here as contracts; they are NOT implemented in v3.1 phase 1.

### `UsageLedger`

The top-level ledger container. A ledger accumulates usage entries over time
and provides query/report interfaces.

Fields (conceptual):

- `id: LedgerId` — unique ledger identifier.
- `created_at: Instant` — when the ledger was created.
- `sessions: Vec<UsageSession>` — accumulated usage sessions.
- `provenance: LedgerProvenance` — metadata about how the ledger was built.
- `honesty: LedgerHonesty` — honesty flags for the ledger.

### `UsageSession`

A bounded time window of usage observation. A session may span from process
start to process end, from boot to shutdown, or an arbitrary user-defined window.

Fields (conceptual):

- `session_id: SessionId` — unique session identifier.
- `started_at: Instant` — session start time.
- `ended_at: Option<Instant>` — session end time (None if still active).
- `entries: Vec<UsageSampleEntry>` — individual sample entries.
- `boot_id: Option<String>` — boot session identifier, if available.

### `UsageSampleEntry`

A single point-in-time usage sample, attributed to a target.

Fields (conceptual):

- `sampled_at: Instant` — when the sample was taken.
- `source: SourceLabel` — where the data came from (e.g., `live_proc_net_dev`).
- `targets: Vec<TargetUsageEntry>` — per-target usage for this sample.
- `interfaces: Vec<InterfaceUsageEntry>` — per-interface usage for this sample.

### `UsageDeltaEntry`

A computed delta between two samples, attributed to targets.

Fields (conceptual):

- `start: UsageSampleEntry` — the earlier sample.
- `end: UsageSampleEntry` — the later sample.
- `interval: Duration` — time between samples.
- `targets: Vec<TargetDeltaEntry>` — per-target delta usage.
- `interfaces: Vec<InterfaceDeltaEntry>` — per-interface delta usage.
- `warnings: Vec<String>` — counter reset/decrease warnings.

### `TargetUsageEntry`

Usage attributed to a specific target for a single sample.

Fields (conceptual):

- `target: ResolvedUsageTarget` — the resolved target identity.
- `rx_bytes: u64` — received bytes.
- `tx_bytes: u64` — transmitted bytes.
- `combined_bytes: u64` — total bytes.
- `rx_packets: u64` — received packets.
- `tx_packets: u64` — transmitted packets.

### `InterfaceUsageEntry`

Usage for a single interface in a sample.

Fields (conceptual):

- `interface: InterfaceIdentity` — the interface.
- `rx_bytes: u64` — received bytes.
- `tx_bytes: u64` — transmitted bytes.
- `combined_bytes: u64` — total bytes.
- `rx_packets: u64` — received packets.
- `tx_packets: u64` — transmitted packets.
- `loopback: bool` — whether this is a loopback interface.

### `LedgerProvenance`

Metadata about the origin and integrity of ledger data.

Fields (conceptual):

- `source_label: String` — how data was collected (e.g., `live_proc_net_dev`).
- `hostname: Option<String>` — host machine name, if available.
- `boot_id: Option<String>` — boot session identifier.
- `kernel_version: Option<String>` — kernel version at time of collection.
- `zelynic_version: String` — zelynic version that created the ledger.
- `collection_method: String` — e.g., `single_shot`, `delta`, `poll`.
- `identity_resolution_used: bool` — whether app identity was attempted.
- `identity_resolution_confidence: Option<IdentityConfidence>` — confidence level.

### `LedgerHonesty`

Honesty flags for the ledger, following the same principle as v3.0.1's
12 boolean honesty flags but extended for ledger-level reporting.

Fields (conceptual):

- `attribution_is_best_effort: bool` — per-app attribution may be partial.
- `interface_level_data_source: bool` — data source is interface-level counters.
- `per_app_attribution_may_be_partial: bool` — process identity may be incomplete.
- `persistence_performed: bool` — whether ledger data was written to disk.
- `filesystem_write_performed: bool` — whether any filesystem write occurred.
- `network_blocking_active: bool` — whether any network blocking is active.
- `quota_enforcement_active: bool` — whether any quota enforcement is active.
- `nft_tc_mutation_performed: bool` — whether nftables/tc were mutated.
- `cgroup_mutation_performed: bool` — whether cgroups were mutated.
- `ebpf_used: bool` — whether eBPF was used for data collection.
- `daemon_mode: bool` — whether a daemon/long-running process collected the data.
- `single_shot: bool` — whether the data was collected in a single invocation.

## 6. Ledger Storage Boundary

### Design Principles

v3.1 may later introduce ledger files. This phase (phase 1) does NOT write files.
Future persistence must obey the following constraints:

1. **Explicit**: Any file write must be triggered by an explicit user command. No silent writes. No background writes.
2. **User-visible**: The file path must be communicated to the user in CLI output. No hidden files in unexpected locations.
3. **Test-gated**: All persistence code paths must have test coverage. File writes in tests must use temp directories only.
4. **No hidden background write**: zelynic must NOT write files in the background without an explicit foreground command.
5. **No daemon**: Ledger persistence must NOT require a daemon or long-running process. Single-shot writes only.
6. **No auto-start**: Ledger collection must NOT auto-start on boot, login, or any system event. The user must invoke zelynic explicitly.
7. **No silent state mutation**: If a ledger file exists, reading it must not modify it. Appending to a ledger must be the only mutation, and it must be explicit.
8. **No lock contention**: Ledger files must not require exclusive locks that prevent concurrent reads. Append-only writes are preferred.

### Future File Location (Conceptual)

A future ledger file location is NOT defined in this phase. Candidates for future
consideration:

- `~/.local/share/zelynic/ledger/` — XDG data directory.
- `/var/lib/zelynic/ledger/` — system-wide data directory (root only).
- User-specified path via CLI flag.

No file location is committed in v3.1 phase 1.

## 7. Future Report Views

The following are candidate report views for future implementation. They are NOT
implemented in v3.1 phase 1.

### Session Reports

| Report | Description | Scope |
|--------|-------------|-------|
| `since-session` | Usage since the start of the current zelynic session. | Single invocation. |
| `since-boot` | Usage since the current boot (using `boot_id` as partition key). | Boot session. |
| `since-time` | Usage since a user-specified timestamp or duration. | Arbitrary window. |

### Per-Entity Reports

| Report | Description | Scope |
|--------|-------------|-------|
| `per-interface` | Usage breakdown by network interface. | Interface-level. |
| `per-target` | Usage breakdown by resolved target identity. | Per-app/process/cgroup. |
| `per-cgroup` | Usage breakdown by cgroup path. | Cgroup-level. |
| `top-consumers` | Ranked list of targets by usage (highest first). | Aggregated. |

### Output Formats

| Format | Description |
|--------|-------------|
| `human-readable` | Formatted text output with tables, summaries, and disclaimers. |
| `JSON report` | Machine-readable JSON with full attribution data, honesty flags, and provenance. |

### Report Honesty Requirements

All future reports must include:

- Attribution scope (interface-level, per-process, per-cgroup, or mixed).
- Attribution confidence (how reliable the identity resolution is).
- Data source label (where the counters came from).
- Time bounds (start, end, duration of the reporting window).
- All applicable honesty flags from `LedgerHonesty`.
- Warnings about counter resets, incomplete attribution, or missing identity data.

## 8. Future Command Candidates

The following are candidate CLI commands for future implementation. They are NOT
implemented in v3.1 phase 1.

### Extended Usage Commands

| Command | Description |
|---------|-------------|
| `zelynic usage --session` | Report usage since the start of the current session. |
| `zelynic usage --since-boot` | Report usage since the current boot. |
| `zelynic usage --interface wlp1s0` | Report usage for a specific interface. |
| `zelynic usage --target brave` | Report usage attributed to a specific target (by name or PID). |

### Ledger Commands

| Command | Description |
|---------|-------------|
| `zelynic ledger inspect` | Display a human-readable ledger summary. |
| `zelynic ledger inspect --json` | Display ledger summary as JSON. |
| `zelynic ledger export --json` | Export full ledger data as JSON for external tools. |

### Important: Backward Compatibility

The existing `zelynic usage` command requires `--sample` in v3.0.x. This behavior
MUST NOT change casually. Future flags (`--session`, `--since-boot`, `--interface`,
`--target`) must be additive and must NOT silently alter the meaning of existing
flag combinations. Any new flag that changes the required flag set must go through
a dedicated CLI gate design phase (as done for v3.0 `--delta`).

Specifically:

- `zelynic usage` without `--sample` remains rejected by clap in v3.1.
- `zelynic usage --sample` retains its current v3.0.1 behavior unchanged.
- New flags are added via new phases with dedicated gate documents.

  },

### Schema Version Policy

- v3.0.1 JSON schema uses `schema_version: 1`.
- Future v3.1 ledger/identity JSON will use `schema_version: 2`.
- Schema version is bumped only when the top-level JSON structure changes in a
  backward-incompatible way.
- Adding new optional fields does NOT bump the schema version.
- Removing or renaming fields, or changing field types, DOES bump the schema version.

## 10. Honesty/Safety Flags

The following honesty principles extend the v3.0.1 honesty model for v3.1.
All flags default to their most conservative value (false for action flags,
true for limitation flags).

### Identity Honesty Flags

| Flag | Type | Default | Meaning |
|------|------|---------|---------|
| `attribution_is_best_effort` | bool | `true` | Per-app attribution is best-effort and may be incomplete or incorrect. |
| `interface_level_data_source` | bool | `true` | The underlying data source is interface-level counters, not per-app. |
| `per_app_attribution_may_be_partial` | bool | `true` | Process identity may be incomplete (e.g., PID recycling, missing exe_path). |

### Mutation Honesty Flags (Carried Forward)

| Flag | Type | Default | Meaning |
|------|------|---------|---------|
| `persistence_performed` | bool | `false` | Whether ledger data was written to disk. |
| `filesystem_write_performed` | bool | `false` | Whether any filesystem write occurred. |
| `network_blocking_active` | bool | `false` | Whether any network blocking is active. |
| `quota_enforcement_active` | bool | `false` | Whether any quota enforcement is active. |
| `nft_tc_mutation_performed` | bool | `false` | Whether nftables/tc were mutated. |
| `cgroup_mutation_performed` | bool | `false` | Whether cgroups were mutated. |
| `ebpf_used` | bool | `false` | Whether eBPF was used for data collection. |

### Mode Honesty Flags

| Flag | Type | Default | Meaning |
|------|------|---------|---------|
| `daemon_mode` | bool | `false` | Whether a daemon/long-running process collected the data. |
| `single_shot` | bool | `true` | Whether the data was collected in a single invocation. |

## 11. Non-Goals

The following are explicitly out of scope for v3.1 phase 1 and all v3.1
sub-phases through phase 7 (see section 12):

- No runtime code implementation.
- No CLI behavior change.
- No version bump.
- No persistence implementation (no ledger file write).
- No ledger file read/write.
- No app identity resolver implementation.
- No permission/block/allow mode.
- No quota guard.
- No eBPF.
- No nft/tc mutation.
- No cgroup mutation.
- No PID movement.
- No daemon/watch/loop mode.
- No hidden background process.
- No JSON schema change (v3.0.1 schema remains stable).
- No change to existing `usage --sample` behavior.
- No change to existing `usage --sample --delta` behavior.
- No change to existing `usage --sample --json` behavior.
- No change to existing `usage --sample --delta --json` behavior.

## 12. v3.1 Implementation Roadmap

v3.1 is divided into small, sequential phases. Each phase produces a design
document or a minimal code increment. No phase may introduce enforcement,
persistence, or mutation.

### Phase 1 (Current): Contract Design

**Status**: This document.

**Deliverable**: Design document defining identity types, ledger types, identity
sources, storage boundary, report views, command candidates, JSON contract
direction, and honesty flags.

**Scope**: Docs only. No code. No tests. No CLI changes.

### Phase 2: Pure Identity Model

**Deliverable**: Rust type definitions for `TargetIdentity`, `ProcessIdentity`,
`CgroupIdentity`, `InterfaceIdentity`, `ResolvedUsageTarget`,
`UsageAttributionScope`, `IdentityConfidence`. Pure data types with `serde`
derive. No filesystem reads. No `/proc` access. Constructor functions only.

**Scope**: New module `src/identity/mod.rs` with type definitions. Unit tests for
serialization/deserialization and construction. No live reads. No CLI wiring.

### Phase 3: Pure Ledger Schema

**Deliverable**: Rust type definitions for `UsageLedger`, `UsageSession`,
`UsageSampleEntry`, `UsageDeltaEntry`, `TargetUsageEntry`,
`InterfaceUsageEntry`, `LedgerProvenance`, `LedgerHonesty`. Pure data types
with `serde` derive. No filesystem reads or writes.

**Scope**: New module `src/ledger/mod.rs` with type definitions. Unit tests for
serialization/deserialization. No live reads. No CLI wiring. No file I/O.

### Phase 4: Ledger Report Model

**Deliverable**: Renderer functions that convert ledger data into human-readable
and JSON report output. No actual ledger data — uses injected test data only.

**Scope**: New functions in `src/ledger/` or `src/commands/`. Tests verify
output format, honesty flags, and disclaimer presence. No live reads. No CLI
wiring. No file I/O.

### Phase 5: CLI Gate Design

**Deliverable**: Design document defining activation gates for future `ledger`
subcommand and extended `usage` flags (`--session`, `--since-boot`, `--interface`,
`--target`). Follows the same gate pattern as v3.0 phase 4 and phase 11.

**Scope**: Docs only. No code. No CLI registration.

### Phase 6: Read-Only Preview

**Deliverable**: Optional `zelynic ledger inspect` (text) and
`zelynic ledger inspect --json` commands that display empty/mock ledger data
to prove the CLI wiring works. No real identity resolution. No real ledger
accumulation. No file I/O.

**Scope**: CLI registration of `ledger` subcommand. Mock data only. Tests prove
output format and honesty flags. No live reads. No persistence.

### Phase 7: Explicit Persistence Design

**Deliverable**: Design document for future ledger persistence. Defines file
format, location policy, append-only write contract, and test gates for file
I/O. No actual file writes.

**Scope**: Docs only. No code. No file I/O. No persistence.

### Phase 8+: Gated Persistence Implementation

**Deliverable**: Actual ledger file write, gated behind explicit user command.
Append-only writes to a user-visible file path. Full test coverage with temp
directories. Only after design freeze from phase 7.

**Scope**: File I/O code with tests. No enforcement. No mutation. No daemon.
No auto-start.
