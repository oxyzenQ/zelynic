## 1. Purpose

Phase 14 polishes the user-facing documentation and examples for the
`ledger inspect` command family. This phase adds no new runtime behavior,
changes no parsing logic, and touches no production code. Its sole
purpose is to make the existing `ledger inspect` commands understandable
and safe for users by documenting the four command variants, showing
expected output, providing valid and invalid file examples, and clearly
stating what is NOT implemented yet.

The documentation is structured as a reference that a user can read to
understand exactly what happens when they run each variant of
`ledger inspect`, what errors they may encounter, and what safety
guarantees are in place.

## 2. Commands

The `ledger` subcommand is hidden from `--help` output. It is an
internal/development command that is not part of the stable CLI surface.
The four command variants are:

### 2.1 `zelynic ledger inspect`

Prints a human-readable text summary of the in-memory fixture ledger.
No `--file` flag is passed, so the handler builds a deterministic
fixture with 3 entries (2 snapshots on wlan0 and eth0, 1 delta on
wlan0). No filesystem reads or writes occur. The output includes:

- Header: "Zelynic v2.9 ledger inspect (source: ledger inspect model only)"
- Schema version, timestamps, host ID
- Entry counts (3 entries: 2 snapshots, 1 deltas)
- Interface list: eth0, wlan0
- Aggregate totals (RX, TX, combined bytes, saturating arithmetic)
- Attribution scope: "interface-level only (not per-app attribution)"
- Enforcement status: "inactive/not implemented"
- Read-only: true
- 9 safety disclaimers including: no filesystem read performed, no
  filesystem write performed, no persistence, no live /proc or sysfs
  read, no live app identity resolver, no enforcement, no network
  blocking, no limiter attach, no nft/tc state mutation.

### 2.2 `zelynic ledger inspect --json`

Prints the same fixture ledger as deterministic JSON to stdout. The
output is valid JSON with `schema_version: 1`, `host_id: "fixture-host"`,
and an `entries` array containing the 3 fixture entries. No persistence
metadata, no path information, and no v3.0 usage schema fields are
included. The JSON uses the ledger serialization schema (separate from
the v3.0 usage JSON schema).

### 2.3 `zelynic ledger inspect --file <PATH>`

Reads a ledger JSON file from the explicitly specified `<PATH>`,
validates the path and file, deserializes the ledger, and prints a
human-readable text summary. This is the **explicit read-only file
inspect** mode. Key behaviors:

- The `--file <PATH>` flag is required for any file read. There is no
  implicit default file path and no XDG auto-discovery.
- Path validation rejects: empty paths, parent traversal (`..`),
  suspicious filename characters (only `[a-zA-Z0-9._-]` allowed),
  symlinks, directories, and nonexistent files.
- After path validation, the file is read with `std::fs::read_to_string`.
- The JSON content is deserialized and validated: schema_version must
  be 1, all entries must have `read_only: true`, `attribution_scope:
  "interface-level only"`, `enforcement_status: "inactive/not implemented"`,
  and `combined_bytes == rx_bytes + tx_bytes`.
- Text output says "source: explicit file read" and includes the file
  path in the header.
- Safety disclaimers affirm: read-only, no file write, no persistence
  save, no live /proc or sysfs read, no live app identity resolver,
  interface-level only attribution, inactive enforcement, inactive
  network blocking, no limiter attach, no nft/tc state mutation.

### 2.4 `zelynic ledger inspect --file <PATH> --json`

Same as `--file <PATH>` but outputs deterministic JSON instead of
human-readable text. The JSON uses the same `serialize_ledger_to_json()`
as the fixture mode. It does not include the file path or any
persistence metadata. The output is deterministic: reading the same
file twice produces identical JSON.

## 3. Fixture Preview vs Explicit File Read

The critical distinction is whether `--file <PATH>` is passed:

| Aspect | No `--file` (fixture preview) | With `--file <PATH>` (explicit file read) |
|--------|-------------------------------|------------------------------------------|
| Data source | In-memory fixture (deterministic) | User-specified file on disk |
| Filesystem read | None | Read-only (`read_to_string`) |
| Filesystem write | None | None |
| Path validation | Not applicable | Full validation (see section 4) |
| Schema validation | Not applicable (fixture is correct) | Full validation via `deserialize_ledger_from_json` |
| Text header | "source: ledger inspect model only" | "source: explicit file read" |
| File path in output | No | Yes |
| Use case | Development, testing, CI verification | Inspecting a real ledger file |

**Plain `zelynic ledger inspect` (without `--file`) is fixture-only.** It
does not read any file from disk, does not auto-discover any default
ledger path, and does not interact with the filesystem in any way.
This behavior is stable and will not change without a dedicated design
phase.

**`zelynic ledger inspect --file <PATH>` is explicit read-only file
inspect.** The user must provide the exact file path. There is no
shortcut, no environment variable, and no auto-discovery. The operation
is strictly read-only: no writes, no directory creation, no persistence
save, no backup, no migration.

## 4. Minimal Valid Ledger JSON Example

This is a minimal valid ledger JSON file that can be inspected with
`zelynic ledger inspect --file <PATH>`:

```json
{
  "schema_version": 1,
  "created_at": "2026-06-11T00:00:00Z",
  "updated_at": "2026-06-11T00:05:00Z",
  "host_id": "my-host",
  "session_id": "sess-001",
  "entries": [
    {
      "entry_id": "snap-001",
      "timestamp": "2026-06-11T00:00:00Z",
      "entry_type": "snapshot",
      "source_label": "manual",
      "interface": "eth0",
      "rx_bytes": 1000000,
      "tx_bytes": 500000,
      "combined_bytes": 1500000,
      "read_only": true,
      "provenance": "manual-snapshot",
      "attribution_scope": "interface-level only",
      "enforcement_status": "inactive/not implemented",
      "reset_detected": false,
      "reset_details": []
    }
  ]
}
```

Required top-level fields: `schema_version` (must be 1), `created_at`,
`updated_at`, `host_id`, `entries`. The `session_id` field is optional.

Required per-entry fields: `entry_id`, `timestamp`, `entry_type`,
`source_label`, `interface`, `rx_bytes`, `tx_bytes`, `combined_bytes`,
`read_only` (must be `true`), `provenance`, `attribution_scope` (must
be `"interface-level only"`), `enforcement_status` (must be
`"inactive/not implemented"`), `reset_detected`, `reset_details`.

The `combined_bytes` field must equal `rx_bytes + tx_bytes` (saturating
addition). Any violation of these constraints produces a clear error
message identifying the specific problem.

## 6. Invalid Input Examples

The following examples show the error messages produced for common
invalid inputs. All errors use the stable prefix `ledger inspect --file:`
followed by a keyword identifying the error class. These keywords are
frozen as contract and will not change without a dedicated design phase.

### 6.1 Parent Traversal Path

```
$ zelynic ledger inspect --file /tmp/legit/../etc/passwd.json
Error: ledger inspect --file: parent traversal (..) is not allowed
```

Any `..` component in the path is rejected before any filesystem read.
This prevents directory traversal attacks.

### 6.2 Nonexistent File

```
$ zelynic ledger inspect --file /tmp/nonexistent-ledger.json
Error: ledger inspect --file: cannot access '/tmp/nonexistent-ledger.json': No such file or directory (os error 2)
```

The `cannot access` keyword is stable. The OS-specific detail after the
colon may vary by platform.

### 6.3 Malformed JSON

```
$ zelynic ledger inspect --file /tmp/bad.json
Error: ledger inspect --file: invalid ledger in '/tmp/bad.json': JSON parse error: ...
```

The `invalid ledger` keyword identifies parse failures. The specific
JSON parse error detail may vary.

### 6.4 Unsupported Schema Version

```
$ zelynic ledger inspect --file /tmp/old-schema.json
Error: ledger inspect --file: invalid ledger in '/tmp/old-schema.json': unsupported schema version: 99 (supported: 1)
```

Only `schema_version: 1` is supported. Any other version is rejected
with the `unsupported schema version` keyword.

### 6.5 Missing Required Field

```
$ zelynic ledger inspect --file /tmp/missing-field.json
Error: ledger inspect --file: invalid ledger in '/tmp/missing-field.json': ...
```

Missing top-level fields (e.g., `created_at`, `host_id`) or missing
entry fields (e.g., `entry_id`, `interface`) produce an `invalid ledger`
error.

### 6.6 Invalid Invariant (e.g., combined_bytes mismatch)

```
$ zelynic ledger inspect --file /tmp/invalid-invariant.json
Error: ledger inspect --file: invalid ledger in '/tmp/invalid-invariant.json': safety violation: entry <id> has combined_bytes (9999) != rx_bytes (100) + tx_bytes (50)
```

The specific invariant violation is named in the error message, making
it clear what needs to be fixed.

## 7. Safety Guarantees

The following safety guarantees apply to all `ledger inspect` commands:

- **No default ledger path read**: `ledger inspect` without `--file`
  uses an in-memory fixture only. It does not read any file from disk,
  does not auto-discover any default ledger path, and does not check
  XDG directories.
- **No implicit persistence**: There is no automatic loading, saving,
  or caching of ledger data. Every operation is explicit.
- **No file write**: `ledger inspect` never writes to any file. There
  is no `--output`, `--save`, or `--overwrite` flag. No temporary
  files, lock files, or backup files are created.
- **No export**: `zelynic ledger export --json` is design-gated and
  rejected at dispatch with a clear message. Export is reserved for a
  future dedicated phase.
- **No save/write persistence**: `ledger save`, `ledger write`,
  `ledger backup`, `ledger migrate`, and similar commands are not
  implemented. The persistence module returns `Blocked` for all
  operations.
- **No migration**: No schema migration, format conversion, or data
  transformation is performed or attempted.
- **No backup**: No backup files are created, renamed, or managed.
- **No rename**: No file rename operations are performed.
- **No truncate**: No file truncation is performed.
- **No delete**: No file deletion is performed.
- **No overwrite**: No file overwrite is performed, even for the file
  being read (the read is strictly `O_RDONLY`).
- **No live resolver**: `ledger inspect` does not resolve app
  identities from live system state. Attribution is "interface-level
  only" — it does not identify individual applications or processes.
- **No enforcement**: `ledger inspect` does not apply any network
  quota, rate limit, blocking rule, or traffic policy. The
  `enforcement_status` is always "inactive/not implemented".
- **No nft/tc/cgroup/PID mutation**: `ledger inspect` does not
  modify nftables rules, tc classes/filters, cgroup assignments, or
  process state. No limiter attach is performed.
- **No /proc or sysfs read**: `ledger inspect` does not read from
  `/proc/net/dev`, `/sys/class/net/`, or any other live system
  interface for traffic data. The data source is either the in-memory
  fixture (no `--file`) or the user-specified JSON file (`--file`).
- **v3.0 usage JSON unchanged**: The `zelynic usage --sample --delta
  --json` command outputs `schema_version: 1` and its schema is frozen.
  `ledger inspect` uses a separate JSON schema that does not affect the
  v3.0 usage schema.

## 8. Future Roadmap Boundary

This section documents what is NOT implemented yet and what the future
phases may address:

- **`ledger export` remains gated for a later phase.** The `zelynic
  ledger export --json` command is registered in the CLI parser as a
  hidden subcommand but is hard-rejected at dispatch with a
  "design-gated" message. A dedicated phase will be required to design,
  implement, and test the export functionality.
- **`ledger save` / `ledger write` remains not implemented.** No
  command exists to write ledger data to disk. The persistence module
  returns `Blocked` for all operations. A dedicated phase will be
  required to design the write path, including atomic write (temp file
  + fsync + rename), backup strategy, and corruption handling.
- **Persistence remains not enabled.** There is no auto-save,
  background persistence, or daemon-mode writing. All persistence
  operations are hard-blocked.
- **Per-app attribution is not implemented.** The `attribution_scope`
  is always "interface-level only". Per-app identity resolution using
  live process scanning, cgroup membership, or socket matching is a
  future feature that requires its own design phase.
- **This phase is documentation polish only.** No production code was
  changed, no new features were added, and no existing behavior was
  modified. The sole deliverable is this document and the guard tests
  that verify its content.
