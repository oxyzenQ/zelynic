# Zelynic v2.9 Phase 11: Release-Candidate Freeze / Full Validation Index

This document freezes v2.9 as a read-only Network Accounting Lab release
candidate. Phase 11 is **docs/report only**: no Rust code is modified, no tests
are added, and no runtime behavior changes. The purpose is to produce a
comprehensive validation index of all v2.9 phases, confirm the release-candidate
safety state, and formally document what v2.9 delivers and what remains deferred
to future version milestones.

## Release-Candidate Decision

v2.9 is frozen as a read-only accounting foundation milestone. The release
candidate can proceed with the following understanding:

### What v2.9 Delivers

- A complete pure model layer for network usage observation: interface counters,
  usage previews, session deltas, ledger entries, ledger inspection, persistence
  path planning, and a hard-blocked persistence I/O contract seam.
- 256 accounting tests (901 unit tests total) covering all model types, parsers,
  renderers, validation logic, and safety disclaimer presence.
- Comprehensive honesty constraints ensuring no false claims of per-app
  attribution, enforcement, quota guard activity, or network blocking.
- A clean architectural boundary between model code (accounting module) and
  enforcement code (strict limiter backend), with no cross-contamination.
- Forward-looking design for future persistence (phase 5 design document),
  future quota guard (v3.x), and future eBPF observer (v4.x).

### What v2.9 Does Not Implement

- v2.9 does not implement a live `zelynic usage` command. The usage preview
  model exists but no CLI path renders it.
- v2.9 does not implement actual filesystem persistence. The persistence seam is
  hard-blocked; no ledger file is read from or written to disk.
- v2.9 does not implement quota guard or budget tracking. The ledger model
  supports quota config entries as a future type but none are created.
- v2.9 does not implement allow/block app mode or background data guard.
- v2.9 does not implement eBPF, per-app attribution, or per-cgroup accounting.
- v2.9 does not modify the `zelynic strict` limiter, nftables rules, tc
  classes, or cgroup membership in any way.

### Future Work Requires Separate Explicit Phases

- Future actual persistence (filesystem read/write) must be a separate explicit
  post-RC phase with its own design document, test plan, and safety review. It
  must satisfy all 11 entry criteria established in the phase 10 freeze report.
- Future quota guard, allow/block mode, and background data guard are deferred to
  v3.x and require per-app attribution infrastructure.
- Future eBPF observer/classifier is deferred to v4.x and requires kernel
  5.4+, CAP_BPF or root, and a verified non-mutating eBPF program.
